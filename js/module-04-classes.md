# Модуль 4: Классы и ООП в JavaScript

## Для кого этот модуль

Этот модуль для тебя, если ты уже много раз писал классы в Java, Spring или Kotlin, смотришь на `class` в JavaScript и думаешь: «Ну наконец-то, здесь всё по-человечески». И вот тут как раз начинается главный педагогический подвох. Снаружи действительно похоже. Но под капотом работает не та машина, к которой ты привык на JVM.

Если в модуле 2 мы много говорили про `this`, а в модуле 3 — про объекты, прототипы и цепочку делегирования, то здесь мы наконец собираем всё вместе. Классы в JS нельзя понять изолированно. Они как красивый интерфейс в приложении: если не знать, какие сервисы и базы стоят сзади, ты очень быстро упрёшься в странное поведение.

Поэтому в этом модуле мы будем не просто перечислять синтаксис `class`, `extends`, `super`, `static` и `#private`. Мы будем постоянно задавать вопрос: **а почему язык устроен именно так?** Что он хотел облегчить? Какие старые боли закрывал? И где Java-разработчик может ошибиться именно потому, что синтаксис слишком уж знакомый?

**Ожидаемый уровень на входе**: ты уже понимаешь объекты, функции, методы, наследование как идею и не пугаешься слова «прототип». Очень желательно пройти модуль 2 про `this` и модуль 3 про прототипы. Без них классы в JS будут казаться либо магией, либо «какой-то недо-Java». А это неправда.

## Главная установка модуля

**Класс в JavaScript — это не новый фундамент, а удобная приборная панель над старым мотором прототипов.**

Запомни эту мысль почти телом. Она будет возвращаться снова и снова. Когда ты видишь `class`, мозг после Java хочет представить «настоящий» класс, как будто язык внезапно сменил философию. Но JavaScript не сделал революцию. Он просто добавил более аккуратный способ описывать то, что и раньше делалось через функции-конструкторы и `prototype`.

Есть хорошая бытовая метафора. Представь современную стиральную машину с сенсорным экраном. На экране красивые режимы: «быстрая», «деликатная», «хлопок». Но внутри всё равно крутятся тот же барабан, насос, вода и мотор. `class` — это сенсорный экран. Прототипы — мотор и барабан. Если ты забудешь про мотор и будешь думать только экраном, при первой поломке не поймёшь, что произошло.

Вторая метафора — автомобиль. На панели есть удобные кнопки, значки и ассистенты. Но машина едет не благодаря красивой панели, а благодаря двигателю, коробке и колёсам. Так и здесь: `class` делает код понятнее человеку, но механизм наследования поведения по-прежнему держится на прототипной цепочке.

## 4.1. Классы ES6 — синтаксический сахар над прототипами

### Сначала боль: почему вообще понадобился `class`

До ES6 разработчики на JavaScript тоже создавали «типы объектов». Но выглядело это шумно: function constructor, ручное навешивание методов на `prototype`, отдельная настройка цепочки наследования, ловушки с `new`, странности с `this`. Для движка это было нормально. Для человека — не всегда.

Представь кухню после готовки большой семьи. Все продукты есть, ножи острые, плита работает, еда получится. Но стол завален, шаги неочевидны, и новичок теряется. Синтаксис классов сделал на этой кухне порядок: разложил ложки по ящикам, подписал банки и дал понятный маршрут. Он не изобрёл новую кухню. Он убрал когнитивный шум.

Для Java-разработчика проблема другая. Ты видишь аккуратный `class` и расслабляешься слишком рано. Кажется, что дальше всё как дома: классы, конструкторы, наследники, методы. Но JavaScript использует знакомую вывеску для другой объектной модели. Это как открыть в другой стране кафе с надписью «борщ» и ожидать вкус точь-в-точь как у бабушки. Может быть очень похоже. А может отличаться в самых важных деталях.

### Что такое класс в JS простыми словами

**Класс** в JavaScript — это удобный шаблон, по которому создаются объекты одного семейства. Не «шаблон для компилятора», как в Java, а скорее компактный способ договориться: какие данные будут у экземпляра, какие действия он умеет выполнять и от какого прототипа он будет наследовать поведение.

Метафора из жизни: класс — это форма заказа пиццы в приложении. Там уже предусмотрены поля «размер», «тесто», «добавки». Ты не рисуешь форму с нуля каждый раз. Но сама пицца не живёт в форме — форма только помогает собрать понятный заказ. В JavaScript `class` помогает описать объект, но не меняет фундаментальную механику языка.

Метафора из Java: если в Java класс похож на строгий контракт с жёстким участием компилятора и JVM, то в JavaScript класс больше напоминает **удобный DSL поверх уже существующей прототипной модели**. Он человеко-дружелюбный, но не превращает язык в статически типизированный OOP-мир.

### Первый пример класса

```javascript
class Task {
  constructor(title) {
    this.title = title;
  }

  complete() {
    this.done = true;
  }
}

const task = new Task('Изучить классы');
```

### Разбор по шагам

1. `class Task { ... }` создаёт специальную функцию-конструктор с именем `Task`.
2. `constructor(title)` — это метод, который будет вызван при `new Task(...)`.
3. Внутри конструктора строка `this.title = title` кладёт данные в сам экземпляр.
4. Метод `complete()` **не копируется в каждый объект**. Он попадает в `Task.prototype`.
5. `const task = new Task(...)` создаёт новый объект, связывает его с `Task.prototype`, вызывает конструктор и возвращает экземпляр.

Стоп. Вот здесь и живёт ключевая мысль. Когда ты смотришь на `complete()`, легко подумать: «Ну, метод класса, значит он внутри объекта как в Java». Но нет. Экземпляр хранит данные вроде `title`, а общие методы обычно живут в прототипе. Это экономит память и делает поведение разделяемым.

### Тот же код на старом стиле

Теперь давай снимем красивую упаковку.

```javascript
function Task(title) {
  this.title = title;
}

Task.prototype.complete = function() {
  this.done = true;
};

const task = new Task('Изучить классы');
```

### Разбор по шагам

1. `function Task(title) { ... }` — это функция, которую можно вызывать с `new`.
2. При `new Task(...)` создаётся новый объект и связывается с `Task.prototype`.
3. `this.title = title` записывает свойство в экземпляр.
4. `Task.prototype.complete = function() { ... }` добавляет общий метод в прототип.
5. При вызове `task.complete()` движок сначала ищет `complete` в самом объекте, а потом — в его прототипе.

Теперь сравни оба фрагмента не глазами, а смыслом. Они описывают **один и тот же механизм**. В одном случае ты говоришь языку: «Пожалуйста, настрой всё красиво сам». В другом — настраиваешь вручную. Вот что значит фраза «класс — это синтаксический сахар над прототипами».

### Side-by-side: один смысл, два синтаксиса

Ниже — прямое сравнение. Очень полезно читать по строкам.

```javascript
// ES6 class
class TaskA {
  constructor(title) {
    this.title = title;
  }

  complete() {
    this.done = true;
  }
}

// Старый стиль
function TaskB(title) {
  this.title = title;
}

TaskB.prototype.complete = function() {
  this.done = true;
};
```

### Разбор по шагам

1. `TaskA` и `TaskB` оба умеют создавать объекты через `new`.
2. В обоих случаях поле `title` окажется в экземпляре.
3. В обоих случаях метод `complete` будет общим, а не индивидуальным для каждого объекта.
4. В обоих случаях поиск метода идёт через прототипную цепочку.
5. Главное различие здесь — не в модели объектов, а в удобстве записи и ряде языковых ограничений.

Чтобы это почувствовать физически, полезно проверить несколько фактов.

```javascript
class Task {
  constructor(title) {
    this.title = title;
  }

  complete() {
    this.done = true;
  }
}

const task = new Task('Сравнить механизмы');

console.log(typeof Task); // function
console.log(Object.getPrototypeOf(task) === Task.prototype); // true
console.log(Task.prototype.complete === task.complete); // true
console.log(Object.hasOwn(task, 'complete')); // false
console.log(Object.hasOwn(task, 'title')); // true
```

### Разбор по шагам

1. `typeof Task` даёт `function`, потому что класс в JS создаёт специальную функцию-конструктор.
2. `Object.getPrototypeOf(task) === Task.prototype` показывает связь экземпляра с прототипом класса. В учебных целях иногда пишут `__proto__` для наглядности, но в production-коде используй `Object.getPrototypeOf(obj)` — это стандартный и рекомендуемый метод.
3. `Task.prototype.complete === task.complete` даёт `true`, потому что метод найден в прототипе.
4. `Object.hasOwn(task, 'complete')` возвращает `false`: метод не лежит прямо в объекте.
5. `Object.hasOwn(task, 'title')` возвращает `true`: поле `title` лежит в самом экземпляре.

Если модуль 3 про прототипы уже улёгся, сейчас должен случиться приятный момент узнавания. Мы не открыли новый мир. Мы просто надели на уже известный механизм более дружелюбный костюм.

🔗 **Классы в Angular**

Angular **построен** на классах. Каждый компонент, сервис, pipe — это класс:

```typescript
@Injectable({ providedIn: 'root' })
export class TaskService {
  private http = inject(HttpClient);

  getTasks(): Observable<Task[]> {
    return this.http.get<Task[]>('/api/tasks');
  }
}
```

**Декораторы** (`@Component`, `@Injectable`) — это метаданные класса для DI-контейнера Angular. Аналог в Spring: `@Service` / `@Controller` помечают класс для контейнера; `@Autowired` же ближе к Angular `inject()` или constructor injection — это **точка внедрения**, а не декоратор класса.

**Lifecycle hooks** (`ngOnInit`, `ngOnDestroy`) — Angular вызывает эти методы **по имени**, а не через интерфейс. Интерфейсы `OnInit`, `OnDestroy` опциональны и нужны только для type-checking в TypeScript (в отличие от Java, где интерфейс обязателен).

### Что `class` улучшает кроме красоты

Было бы нечестно говорить, что `class` — это только сахар. Он ещё делает синтаксис строже: методы неперечисляемы, код внутри класса работает в `strict mode`, а вызов без `new` сразу падает. То есть механизм тот же, но защита от случайных ошибок заметно лучше.

### Temporal Dead Zone: почему класс нельзя использовать раньше объявления

Ещё один важный момент для собеседований: объявление класса ведёт себя как `let`/`const`, а не как старое function declaration. То есть имя класса попадает в **Temporal Dead Zone** до строки объявления.

Если в модуле 1 ты уже разбирал hoisting и TDZ, здесь всё должно показаться знакомым. Имя как будто уже существует, но пользоваться им до фактической инициализации нельзя. Это ещё один пример того, как `class` старается сделать код более строгим и менее магическим.

```javascript
const bug = new Task('Упаду раньше времени'); // ReferenceError

class Task {}
```

### Разбор по шагам

1. Парсер знает, что имя `Task` будет объявлено позже.
2. Но до строки `class Task {}` это имя находится в TDZ.
3. Попытка выполнить `new Task(...)` до объявления приводит к `ReferenceError`.
4. Это отличие от function declaration, которую можно вызвать раньше её текста.
5. Так язык защищает тебя от неочевидного порядка инициализации.

Сравни со старым поведением функций:

```javascript
const ok = new Task('Сработало');

function Task(title) {
  this.title = title;
}
```

### Разбор по шагам

1. `function Task(...)` всплывает целиком как function declaration.
2. Поэтому `new Task(...)` может сработать даже до текстового объявления.
3. Для старого JS это было привычно, но часто усложняло чтение файла сверху вниз.
4. Классы выбирают более строгую модель: сначала объяви, потом используй.
5. Это ближе к интуиции «прочитал — потом вызвал».

### Ещё одно отличие: без `new` класс не прощает

```javascript
class Task {
  constructor(title) {
    this.title = title;
  }
}

Task('Без new'); // TypeError
```

### Разбор по шагам

1. `Task` выглядит как функция, но классы нельзя вызывать как обычные функции.
2. Попытка `Task('Без new')` вызывает `TypeError`.
3. Это встроенная защита: экземпляр должен создаваться через `new`.
4. В старом стиле function constructor можно было случайно вызвать без `new` и получить сломанный `this`.
5. `class` убирает этот целый класс ошибок.

### Разница `class` и function constructor коротко

| Что сравниваем | `class` | Function constructor |
|---|---|---|
| Синтаксис | Современный и декларативный | Старый и более шумный |
| Методы | Пишутся внутри тела класса | Вешаются на `prototype` вручную |
| Hoisting | TDZ, как у `let` | Function declaration всплывает |
| Вызов без `new` | Ошибка | Может сработать криво или сломать `this` |
| `strict mode` | Всегда внутри класса | Зависит от контекста |
| Ментальная модель | Удобнее читать человеку | Лучше показывает механику напрямую |

### Самая опасная иллюзия для Java-разработчика

Опасность не в том, что классы в JS плохие. Опасность в том, что они слишком похожи на знакомую тебе модель. Поэтому на собеседовании безопаснее честно говорить про прототипы и ограничения языка, чем повторять фразу «здесь всё точно как в Java».

## 4.2. Конструктор, поля класса, методы

### Зачем вообще разделять «данные» и «поведение»

Когда начинающий разработчик смотрит на класс, он видит просто коробку с кодом. Но сильный разработчик видит более важный вопрос: **что у объекта является состоянием, а что — действием?** Если смешать это бездумно, модель расползётся.

Представь чемодан в поездке. Внутри лежат вещи — это состояние. Сам чемодан умеет открываться, закрываться, катиться на колёсиках — это поведение. Будет странно, если ты начнёшь хранить «открыться» как футболку внутри чемодана. Так же странно, когда мы не различаем поля и методы.

В Java ты привык к чёткому виду класса: поля сверху, конструкторы, методы, maybe getters/setters. В JavaScript всё похоже, но есть нюанс: объектная модель здесь гибче, а значит легче случайно положить не туда то, что должно быть общим или индивидуальным. Поэтому сейчас мы будем медленно и тщательно раскладывать: что создаётся на экземпляре, что хранится в прототипе и где на практике бывают ошибки.

### Конструктор: момент первого заселения объекта

**Конструктор** — это специальный метод, который выполняется при создании объекта через `new`. Это момент первой настройки экземпляра: разложить поля, проверить входные данные, проставить дефолты, возможно — подготовить внутренние структуры.

Метафора из быта: заселение в квартиру. Когда новый жилец заезжает, ему выдают ключ, фиксируют показания счётчиков, записывают телефон и номер договора. Конструктор — это вот такая регистрация. Объект уже «родился», но ещё не настроен. Конструктор доводит его до рабочего состояния.

Метафора из Java-мира: `constructor` в JS выполняет знакомую роль, но перегрузки конструкторов как в Java нет. У тебя только один `constructor` на класс. Если нужно несколько вариантов создания, часто помогают `static` фабрики или объект `options`.

```javascript
class Task {
  constructor(title, assignee) {
    this.title = title;
    this.assignee = assignee;
  }
}

const task = new Task('Подготовить демо', 'Denis');
```

### Разбор по шагам

1. `class Task` объявляет шаблон семейства объектов.
2. `constructor(title, assignee)` принимает входные данные при создании.
3. `this.title = title` и `this.assignee = assignee` записывают свойства прямо в экземпляр.
4. `new Task(...)` создаёт новый объект и вызывает конструктор с переданными аргументами.
5. В результате у тебя появляется конкретный объект с собственными полями `title` и `assignee`.

### Поля класса: где живёт состояние

**Поле** — это данные, принадлежащие экземпляру или самому классу. Пока что говорим про поля экземпляра: заголовок задачи, статус, исполнитель, дата создания. Это то, что делает один объект отличным от другого.

Представь бейджи сотрудников. У каждого бейджа есть имя, отдел, должность. У одного сотрудника «Анна», у другого «Денис». Было бы странно хранить имя сотрудника в общем корпоративном регламенте. Имя — это вещь конкретного экземпляра. Так же и поля объекта.

С современным синтаксисом JS ты можешь объявлять поля прямо в теле класса. Это удобно, потому что дефолтные значения видны сразу, а не прячутся внутри конструктора.

```javascript
class Task {
  status = 'open';
  tags = [];

  constructor(title, assignee) {
    this.title = title;
    this.assignee = assignee;
  }
}

const task = new Task('Подготовить демо', 'Denis');
```

### Разбор по шагам

1. `status = 'open'` задаёт значение по умолчанию для каждого нового экземпляра.
2. `tags = []` создаёт отдельный массив для каждого экземпляра, а не один общий на всех.
3. `constructor(...)` добавляет остальные поля, зависящие от входных данных.
4. После `new Task(...)` у объекта будут свои `status`, `tags`, `title`, `assignee`.
5. Важно: поля класса здесь — свойства экземпляра, а не значения в `prototype`.

Стоп! Здесь частая ошибка мышления. Кто-то видит `status = 'open'` в теле класса и думает: «О, это как будто лежит где-то рядом с методами, значит общее». Нет. У каждого экземпляра будет своё поле `status`. То же касается `tags`. Если `taskA.tags.push('urgent')`, это не добавит тег в `taskB.tags`.

### Методы: общее поведение, которое делится между объектами

**Метод** — это функция, ассоциированная с объектом. Она описывает, что объект умеет делать: завершать задачу, переименовывать её, рассчитывать следующий запуск.

Метафора: если поле — это карман куртки, то метод — это действие руки. В кармане лежит ключ. Рука достаёт ключ, закрывает дверь, открывает машину. Состояние и действие работают вместе, но это не одно и то же.

Обычные методы класса обычно попадают в `prototype`, а значит разделяются всеми экземплярами.

```javascript
class Task {
  status = 'open';

  constructor(title) {
    this.title = title;
  }

  complete() {
    this.status = 'done';
  }

  rename(newTitle) {
    this.title = newTitle;
  }
}
```

### Разбор по шагам

1. `status` создаётся у каждого экземпляра отдельно.
2. `complete()` объявлен как обычный метод класса.
3. `rename(newTitle)` тоже обычный метод класса.
4. Оба метода окажутся в `Task.prototype`, а не в каждом объекте по отдельности.
5. Поэтому все экземпляры будут использовать одну общую реализацию методов.

### Проверяем, что методы не копируются в объект

```javascript
class Task {
  constructor(title) {
    this.title = title;
  }

  complete() {
    this.done = true;
  }
}

const task1 = new Task('A');
const task2 = new Task('B');

console.log(task1.complete === task2.complete); // true
console.log(Object.hasOwn(task1, 'complete')); // false
```

### Разбор по шагам

1. Создаются два разных экземпляра: `task1` и `task2`.
2. Оба имеют доступ к методу `complete`.
3. Сравнение `task1.complete === task2.complete` даёт `true`, потому что это одна и та же функция из прототипа.
4. `Object.hasOwn(task1, 'complete')` возвращает `false`, потому что собственного свойства `complete` у экземпляра нет.
5. Это и есть экономия памяти и разделение поведения.

### `this`: старый знакомый, который опять может укусить

Если модуль 2 про `this` ещё свеж в памяти — отлично. Если нет, сейчас напомню главное. Внутри метода класса `this` указывает не на «класс вообще», а на **конкретный объект вызова**.

Это очень похоже на Java только до тех пор, пока ты вызываешь метод в нормальной форме `task.complete()`. Как только метод передан отдельно как callback, обёртка может исчезнуть, и `this` потеряется. В Java этот сценарий обычно не так коварен. В JS — это один из самых популярных источников ошибок.

```javascript
class Task {
  constructor(title) {
    this.title = title;
  }

  printTitle() {
    console.log(this.title);
  }
}

const task = new Task('Релиз');
const printer = task.printTitle;
printer(); // TypeError в strict mode, потому что this потерян
```

### Разбор по шагам

1. У экземпляра `task` есть метод `printTitle` через прототип.
2. `const printer = task.printTitle` берёт ссылку на функцию отдельно от объекта.
3. Когда вызывается просто `printer()`, нет объекта слева от точки, который стал бы `this`.
4. В `strict mode` внутри метода `this` становится `undefined`.
5. Попытка прочитать `this.title` приводит к ошибке.

Стоп. Это не «странность классов». Это продолжение темы `this` из модуля 2. Просто в классовом синтаксисе ошибка выглядит особенно обидно, потому что код кажется очень «Java-подобным» и поэтому кажется безопасным.

### Что делают в реальном коде

Есть несколько путей. Первый — явно биндинговать метод там, где он уходит в callback. Второй — использовать стрелочную функцию в поле класса. Третий — не передавать голый метод, а обернуть его вызов в лямбду там, где нужно.

```javascript
class Task {
  constructor(title) {
    this.title = title;
    this.printTitle = this.printTitle.bind(this);
  }

  printTitle() {
    console.log(this.title);
  }
}
```

### Разбор по шагам

1. В конструкторе берётся текущая функция `printTitle`.
2. `bind(this)` создаёт новую функцию с навсегда привязанным контекстом.
3. Эта новая функция записывается прямо в экземпляр как собственное свойство.
4. Теперь даже если вынести `const printer = task.printTitle`, `this` сохранится.
5. Цена: у каждого экземпляра будет своя отдельная версия забинженного метода.

### Стрелочные методы в полях: удобно, но не бесплатно

Ещё одна популярная форма:

```javascript
class Task {
  constructor(title) {
    this.title = title;
  }

  printTitle = () => {
    console.log(this.title);
  };
}
```

### Разбор по шагам

1. `printTitle` здесь не обычный метод прототипа, а поле экземпляра.
2. Значением поля становится стрелочная функция.
3. Стрелочные функции не имеют собственного `this`, а берут его лексически.
4. Для каждого нового объекта будет создана новая функция `printTitle`.
5. Удобно для callback-сценариев, но дороже по памяти и менее «прототипно».

### История типичной ошибки

Классическая история: метод класса передали в `addEventListener`, а потом удивились потерянному `this`. Проблема не в DOM, а в ожидании, что метод сам по себе привязан к объекту. В JS нужно каждый раз спрашивать: кто именно вызовет эту функцию?

### Ещё одно отличие от Java: нет перегрузки методов

В Java можно перегружать методы по сигнатуре. В JavaScript — нет: второе определение с тем же именем просто перезапишет первое. Поэтому вместо перегрузки здесь обычно используют объект `options`, проверки аргументов или отдельные методы с разными именами.

## 4.3. 🎯 Наследование (`extends`, `super`) — Interview Focus

### Почему тема наследования в JS так коварна

Когда Java-разработчик впервые видит `extends` и `super`, у него часто возникает облегчение: «О, вот это уже мой родной район». И именно в этот момент интервьюер может начать довольно улыбаться. Потому что тема правда знакомая снаружи — и достаточно другая внутри.

Наследование в Java — часть class-based модели языка. Есть жёсткая иерархия типов, проверки компилятора, контрактные ожидания JVM, привычное поведение `super(...)`. В JavaScript `extends` — это **настройка прототипной цепочки плюс немного синтаксической магии вокруг функций-конструкторов**. То есть синтаксис очень похож, а фундамент — всё ещё тот же, прототипный.

Представь двухэтажный дом. В Java второй этаж строится по архитектурному проекту на капитальном каркасе. В JavaScript второй этаж может выглядеть так же красиво, но держится на более гибкой системе модульных соединений. Дом нормальный. Просто логика сборки другая. Если путать их, начнёшь не там искать причины странного поведения.

### Что делает `extends`

**Наследование** — это механизм, при котором один класс переиспользует и расширяет поведение другого. В JavaScript запись `class PriorityTask extends Task` означает: новый класс должен использовать базовую логику `Task` и строить поверх неё свою.

Метафора из жизни: есть базовый рецепт пиццы — тесто, соус, сыр. А потом появляется вариант «пепперони», который не переписывает рецепт с нуля, а добавляет свою начинку. `extends` говорит: «Возьми базовую основу и расширь её».

Метафора из Java: `extends` в JS похож на `extends` в Java на уровне идеи «ребёнок является более конкретной формой родителя». Но под капотом реализация не через ту же систему типов, а через связи `prototype` и `[[Prototype]]`.

```javascript
class Task {
  constructor(title) {
    this.title = title;
    this.status = 'open';
  }

  complete() {
    this.status = 'done';
  }
}

class PriorityTask extends Task {
  constructor(title, priority) {
    super(title);
    this.priority = priority;
  }

  isCritical() {
    return this.priority === 'high';
  }
}
```

### Разбор по шагам

1. `Task` описывает базовую сущность: у неё есть `title`, `status`, метод `complete()`.
2. `PriorityTask extends Task` создаёт дочерний класс, связанный с базовым.
3. В конструкторе наследника сначала вызывается `super(title)`.
4. После этого можно добавить собственное поле `priority`.
5. Метод `isCritical()` живёт уже в `PriorityTask.prototype`, а метод `complete()` будет найден через родительский прототип.

### Почему `super()` обязателен до `this`

Вот место, где многих ловят. В дочернем классе ты **не можешь** использовать `this` до вызова `super()`. Не «нежелательно». Не «лучше сначала». Именно нельзя.

Почему? Потому что в производном классе объектная основа ещё не готова. JavaScript хочет, чтобы сначала сработал конструктор родителя, который инициализирует базовую часть экземпляра. Только после этого ты получаешь доступ к `this`.

Представь аренду автомобиля. Пока не подписан базовый договор и тебе не выдали саму машину, бессмысленно обсуждать, какую наклейку ты приклеишь на дверь. Сначала должна появиться машина как объект аренды. Потом уже можно добавлять свои детали. `super()` — это шаг, который «делает машину доступной».

```javascript
class Base {
  constructor() {
    this.id = 1;
  }
}

class Child extends Base {
  constructor() {
    super();
    this.name = 'task';
  }
}
```

### Разбор по шагам

1. `new Child()` запускает конструктор дочернего класса.
2. Поскольку `Child` наследуется от `Base`, сначала нужно вызвать `super()`.
3. `super()` выполняет код конструктора `Base` и подготавливает базовую часть объекта.
4. Только после этого становится доступен `this` текущего экземпляра.
5. Затем можно безопасно выполнить `this.name = 'task'`.

А теперь плохой вариант.

```javascript
class Base {
  constructor() {
    this.id = 1;
  }
}

class Child extends Base {
  constructor() {
    this.name = 'task';
    super();
  }
}
```

### Разбор по шагам

1. Движок видит, что в производном конструкторе используется `this`.
2. Но `super()` ещё не был вызван.
3. Значит базовая объектная инициализация не завершена.
4. Это нарушает правила производного конструктора.
5. Код падает с ошибкой из-за обращения к `this` до `super()`.

Стоп. Очень важно не заучивать это как ритуал «ну просто так принято». Смысл не в церемонии. Смысл в порядке подготовки объекта.

### Что делает `super.method()`

`super` нужен не только в конструкторе. Через него можно обратиться к методам родителя, когда ты хочешь не полностью заменить поведение, а **расширить** его.

Это напоминает ситуацию в компании: есть общий процесс закрытия задачи. Но у отдела безопасности после закрытия надо ещё записать аудит, у отдела поддержки — отправить письмо клиенту. Ты не переписываешь весь процесс с нуля. Ты берёшь базовый шаг и добавляешь свой хвост.

```javascript
class Task {
  constructor(title) {
    this.title = title;
    this.status = 'open';
  }

  complete() {
    this.status = 'done';
  }
}

class RecurringTask extends Task {
  constructor(title, intervalDays) {
    super(title);
    this.intervalDays = intervalDays;
    this.nextRunAt = this.calculateNextRun();
  }

  calculateNextRun() {
    const next = new Date();
    next.setDate(next.getDate() + this.intervalDays);
    return next;
  }

  complete() {
    super.complete();
    this.nextRunAt = this.calculateNextRun();
    this.status = 'scheduled';
  }
}
```

### Разбор по шагам

1. `RecurringTask` наследует базовые поля и методы `Task`.
2. В конструкторе вызывается `super(title)`, чтобы инициализировать общую часть.
3. Метод `calculateNextRun()` вычисляет следующую дату запуска.
4. Переопределённый `complete()` сначала вызывает `super.complete()`.
5. Потом дочерний класс добавляет своё поведение: пересчитывает дату и меняет статус на `'scheduled'`.

### Side-by-side: наследование через `class` и через прототипы

Сейчас будет один из самых важных фрагментов всего модуля. Посмотри на два варианта одного и того же поведения.

```javascript
// Современный синтаксис
class TaskA {
  constructor(title) {
    this.title = title;
    this.status = 'open';
  }

  complete() {
    this.status = 'done';
  }
}

class PriorityTaskA extends TaskA {
  constructor(title, priority) {
    super(title);
    this.priority = priority;
  }
}

// Старый стиль
function TaskB(title) {
  this.title = title;
  this.status = 'open';
}

TaskB.prototype.complete = function() {
  this.status = 'done';
};

function PriorityTaskB(title, priority) {
  TaskB.call(this, title);
  this.priority = priority;
}

PriorityTaskB.prototype = Object.create(TaskB.prototype);
PriorityTaskB.prototype.constructor = PriorityTaskB;
```

### Разбор по шагам

1. В версии с `class` настройка наследования скрыта за `extends` и `super`.
2. В старой версии `TaskB.call(this, title)` вручную выполняет логику родительского конструктора.
3. `PriorityTaskB.prototype = Object.create(TaskB.prototype)` вручную строит прототипную цепочку для методов.
4. `PriorityTaskB.prototype.constructor = PriorityTaskB` возвращает корректную ссылку на конструктор после замены прототипа.
5. Обе версии выражают один и тот же смысл: базовые данные плюс связь прототипов между родителем и наследником.

Вот это и нужно уметь проговаривать на интервью. Не просто «extends наследует». А конкретно: **`extends` под капотом настраивает прототипные связи между классами и их прототипами, а `super` даёт удобный доступ к родительской логике**. Есть ещё одно отличие, которое часто упускают: `extends` настраивает не только `Child.prototype → Parent.prototype`, но и `Child → Parent` (сам конструктор наследует от родительского). Это нужно для наследования **static-методов**. В старом стиле для полного эквивалента нужно добавить `Object.setPrototypeOf(Child, Parent)`.

### Чем похоже на Java, а чем отличается

Сходство видно в синтаксисе: `class`, `extends`, `super`, переопределение методов, статические члены. Но отличие остаётся фундаментальным: под капотом у JS прототипная цепочка, `this` легче потерять, перегрузки нет, а многие гарантии проверяются уже во время выполнения.

### Маленький, но важный промежуточный пример

Вот ситуация между «простым» и «сложным» кодом. В Java её часто решают наследованием автоматически. В JS стоит подумать, насколько это действительно удобно.

```javascript
class Task {
  constructor(title) {
    this.title = title;
    this.status = 'open';
  }

  reopen() {
    this.status = 'open';
  }
}

class BugTask extends Task {
  constructor(title, severity) {
    super(title);
    this.severity = severity;
  }

  escalate() {
    if (this.severity !== 'critical') {
      this.severity = 'critical';
    }
  }
}
```

### Разбор по шагам

1. `BugTask` получает всё базовое поведение `Task`.
2. Ему не нужно заново писать `title` и `status` с нуля — это делает родитель.
3. Наследник добавляет своё поле `severity`.
4. Также он добавляет новый метод `escalate()`, которого не было в базе.
5. Это хороший пример, где наследование читается естественно: bug-задача действительно является частным случаем общей задачи.

### История частой ошибки №1: «extends копирует методы»

Разработчик приходит на собеседование и уверенно говорит: «Когда мы делаем `extends`, методы родителя копируются в ребёнка». Фраза звучит гладко, но она неверна. Методы не копируются как файлы из папки в папку. Они находятся через цепочку прототипов.

Почему это важно? Потому что из идеи «копирования» легко родятся дальнейшие ошибки: про память, про переопределение, про то, почему изменение в одном месте может отражаться на всех экземплярах. Кандидат вроде бы использует знакомые слова, но механика у него в голове другая.

Лучший ответ звучит так: `extends` настраивает прототипную связь, благодаря которой экземпляры дочернего класса могут находить методы в прототипе родителя.

### История частой ошибки №2: забытый `super()`

Реальная разработческая ситуация. Человек быстро пишет наследника, добавляет свои поля, запускает код — и получает ошибку. Он думает: «Но я же потом вызываю `super()`, почему не работает?» Потому что нельзя «потом». Порядок важен.

Это похоже на вход в офис по пропуску. Нельзя сначала открыть дверь в серверную, а потом поднести бейдж к турникету. Система хочет сначала базовую авторизацию, потом доступ к внутренним зонам. `super()` — это и есть эта базовая авторизация.

### Почему спрашивают

Наследование в JavaScript проверяет, понимаешь ли ты разницу между **знакомым синтаксисом** и **реальным механизмом**. Интервьюер почти всегда хочет увидеть, слышал ли ты только слова `extends/super`, или действительно понимаешь прототипную модель и порядок инициализации `this`.

Ещё это тема на архитектурную зрелость. Сильный разработчик не просто умеет наследоваться, а понимает, когда наследование помогает, а когда лучше взять композицию, фабрику или миксин. И это уже не синтаксис, а мышление.

💡 **В Angular: композиция > наследование**

Хотя TypeScript поддерживает наследование классов, в Angular-сообществе предпочитают **композицию**:
- Вместо базового класса → **сервисы** (DI)
- Вместо наследования шаблонов → **content projection** (`ng-content`)
- Вместо mixin → **directives** и **standalone pipes**

Наследование компонентов работает, но усложняет тестирование и понимание потока данных.

## 4.4. Статические методы и свойства

### Какую боль решают `static`

В какой-то момент в модели появляются вещи, которые относятся не к одной конкретной задаче, а ко всему классу задач. Например, список допустимых статусов. Или логика создания объекта из ответа API. Или общий счётчик созданных экземпляров. Если пытаться засунуть это в обычные методы экземпляра, код начинает путать уровни ответственности.

Представь университет. Конкретный студент может сдать экзамен, сменить группу, подать заявление. Но правила приёма, список факультетов и процедура импорта абитуриентов относятся не к одному студенту, а к университету как системе. Статические методы — это функции уровня университета, а не уровня студента.

В Java ты это уже видел как `static`. В JavaScript идея очень похожа, и это один из приятных островков сходства. Но всё равно стоит держать в голове вопрос: **эта логика работает с конкретным экземпляром или с классом в целом?**

### Базовый пример со статикой

```javascript
class Task {
  static statuses = ['open', 'done', 'scheduled'];

  constructor(title, status = 'open') {
    this.title = title;
    this.status = status;
  }

  static fromApi(payload) {
    return new Task(payload.title, payload.status);
  }

  static isValidStatus(status) {
    return Task.statuses.includes(status);
  }
}

const task = Task.fromApi({ title: 'Синк с командой', status: 'open' });
```

### Разбор по шагам

1. `static statuses` создаёт свойство у самого класса `Task`, а не у его экземпляров.
2. `constructor(...)` по-прежнему отвечает за создание конкретных объектов.
3. `static fromApi(payload)` вызывается как `Task.fromApi(...)` и создаёт экземпляр из внешних данных.
4. `static isValidStatus(status)` тоже живёт на классе и работает с общей информацией.
5. `task`, созданный через `fromApi`, получит обычные поля экземпляра, но сам не будет владельцем `fromApi`.

### Проверяем на уровне свойств

```javascript
class Task {
  static statuses = ['open', 'done'];

  constructor(title) {
    this.title = title;
  }
}

const task = new Task('Demo');

console.log(Task.statuses); // ['open', 'done']
console.log(task.statuses); // undefined
```

### Разбор по шагам

1. `statuses` объявлено как статическое свойство.
2. Значит оно хранится на `Task`, а не на `task`.
3. `Task.statuses` работает, потому что свойство принадлежит самому классу.
4. `task.statuses` вернёт `undefined`, если у экземпляра нет одноимённого свойства.
5. Это ещё раз показывает: экземпляр и класс — разные уровни.

### Когда `static` особенно полезен

Есть несколько очень типичных случаев.

**1. Фабрики и альтернативные способы создания.**
Когда объект приходит из JSON, localStorage, формы, query params. Вместо чудовищного конструктора с кучей веток можно сделать `Task.fromApi(...)`, `Task.fromStorage(...)`, `Task.empty()`.

**2. Общие константы и справочники.**
Например, `Task.statuses`, `Task.maxTitleLength`, `Task.priorityOrder`.

**3. Утилиты, которые логически относятся к классу.**
Например, `Task.isValidStatus(value)`.

### Промежуточный пример с валидацией

```javascript
class Task {
  static allowedStatuses = new Set(['open', 'in-progress', 'done']);

  constructor(title, status = 'open') {
    if (!Task.allowedStatuses.has(status)) {
      throw new Error(`Недопустимый статус: ${status}`);
    }

    this.title = title;
    this.status = status;
  }

  static isValidStatus(status) {
    return Task.allowedStatuses.has(status);
  }
}
```

### Разбор по шагам

1. Класс хранит общий набор допустимых статусов.
2. Конструктор использует это общее знание при создании экземпляра.
3. Если статус плохой, выбрасывается ошибка ещё на входе.
4. `isValidStatus` позволяет проверять данные и вне конструктора.
5. Логика уровня класса остаётся централизованной и не размазывается по коду.

### Важный нюанс: `this` внутри `static`

В статическом методе `this` указывает не на конкретный экземпляр, а обычно на сам класс, через который вызван метод. Это может быть полезно при наследовании статических фабрик.

```javascript
class Task {
  constructor(title) {
    this.title = title;
  }

  static create(title) {
    return new this(title);
  }
}

class PriorityTask extends Task {
  constructor(title) {
    super(title);
    this.priority = 'normal';
  }
}

const task = PriorityTask.create('Наследуемая фабрика');
```

### Разбор по шагам

1. `Task.create(title)` — статический метод.
2. Внутри него используется `new this(title)`.
3. Если вызвать `Task.create(...)`, `this` будет равен `Task`.
4. Если вызвать `PriorityTask.create(...)`, `this` будет равен `PriorityTask`.
5. Это позволяет писать более гибкие фабрики, которые наследуются вместе с классом.

### История типичной ошибки

Разработчик переносит мышление «всё через экземпляр» и пишет `new Task(...).fromApi(payload)`. Почему так хочется сделать? Потому что в голове нет чёткого разделения: что относится к объекту, а что — к самому семейству объектов.

Но экземплярная задача не должна знать, как вообще создаются задачи из API. Это уровень класса, а часто даже уровень фабрики или отдельного маппера. Если держать это различие, модель становится чище.

### Ещё одна история: статическое ≠ глобальное

`static` — это не склад для любого общего состояния. Хорошо хранить там константы, фабрики и знания уровня класса. Плохо — хаотичные флаги UI и псевдо-глобальные данные, которые не относятся к модели.

## 4.5. Приватные поля (`#private`) и методы

### Сначала боль: как жили до `#private`

Сейчас префикс `#` выглядит как нормальная встроенная возможность. Но важно почувствовать историческую боль. Долгое время в JavaScript **не было настоящей встроенной приватности полей**. Разработчики всё равно хотели скрывать внутреннее состояние, но язык не давал прямого механизма как `private` в Java.

Что делали раньше? Использовали `_secret` как договорённость. Делали замыкания. Использовали `WeakMap`. Писали комментарии «не трогать это поле руками». То есть не было единого простого пути, а была коллекция обходных манёвров.

Представь квартиру без внутреннего замка на кабинете. Можно повесить табличку «Не входить». Можно договориться с семьёй, что этот ящик не открывают. Можно даже спрятать ключ в другой комнате. Но если нужен настоящий замок — ты всё равно чувствуешь нехватку нормального механизма. Вот так JavaScript долго жил без `#private`.

### `_field` — это не приватность, а вежливая просьба

Очень важно проговорить это вслух. Поле `_history` или `_secret` — это **соглашение**, а не защита языка. Оно как наклейка «служебный вход». Хорошие люди, может, и не полезут. Но дверь остаётся обычной.

```javascript
class LegacyTask {
  constructor(title) {
    this._history = [];
    this.title = title;
  }
}

const task = new LegacyTask('Старый стиль');
console.log(task._history); // доступ есть
```

### Разбор по шагам

1. Поле `_history` создаётся как обычное публичное свойство экземпляра.
2. Подчёркивание в имени — только сигнал для человека.
3. Снаружи можно спокойно прочитать `task._history`.
4. Снаружи можно и изменить это поле.
5. Язык не защищает тебя от такого доступа.

### Настоящая приватность через `#`

С появлением class fields язык получил встроенный механизм приватных полей и методов. Если у поля имя `#history`, к нему можно обращаться **только внутри тела того же класса**.

Метафора из быта: не табличка «не входить», а электронный замок, который открывается только картой нужного уровня доступа. Если ты стоишь снаружи, ручка просто не сработает.

```javascript
class Task {
  #history = [];

  #normalizeTitle(title) {
    return title.trim();
  }

  constructor(title) {
    this.title = this.#normalizeTitle(title);
  }

  rename(newTitle) {
    this.#history.push({ oldTitle: this.title, changedAt: new Date() });
    this.title = this.#normalizeTitle(newTitle);
  }

  getHistoryCount() {
    return this.#history.length;
  }
}
```

### Разбор по шагам

1. `#history` — приватное поле, доступное только внутри класса `Task`.
2. `#normalizeTitle` — приватный метод, который нельзя вызвать извне.
3. Конструктор использует приватный метод для нормализации заголовка.
4. `rename()` записывает историю переименований в приватный массив.
5. Снаружи можно узнать только число записей через публичный метод `getHistoryCount()`.

### Что произойдёт снаружи

```javascript
class Task {
  #history = [];
}

const task = new Task();
// console.log(task.#history); // SyntaxError
```

### Разбор по шагам

1. У класса есть приватное поле `#history`.
2. Экземпляр `task` действительно его содержит.
3. Но синтаксис `task.#history` вне тела класса запрещён.
4. Ошибка возникает не как «вернулось undefined», а как синтаксическая проблема.
5. Это и отличает настоящую приватность языка от обычной договорённости.

Стоп. Это важный психологический момент. Многие ожидают, что «ну если нельзя, наверное вернётся `undefined`». Нет. Язык даже не хочет запускать такой код. Это не мягкий отказ. Это запрет на уровне синтаксиса.

### Как жили до `#private`: замыкание и WeakMap

Чтобы ты почувствовал, какую боль закрыла новая возможность, посмотри на более старый подход через `WeakMap`.

```javascript
const _history = new WeakMap();

class OldTask {
  constructor(title) {
    this.title = title;
    _history.set(this, []);
  }

  rename(newTitle) {
    _history.get(this).push(this.title);
    this.title = newTitle;
  }

  getHistoryCount() {
    return _history.get(this).length;
  }
}
```

### Разбор по шагам

1. Создаётся внешний `WeakMap`, в котором ключом будет экземпляр.
2. При создании `OldTask` для текущего объекта записывается пустая история.
3. В `rename()` история достаётся через `_history.get(this)`.
4. Снаружи у объекта нет публичного поля `history`, и это действительно скрывает данные.
5. Но код тяжелее, менее очевиден и требует внешней структуры, тогда как `#history` делает то же самое проще.

### Почему это важно не только эстетически

Приватные поля — не просто синтаксическая красота. Они защищают инварианты объекта. Если история переименований, внутренний кеш, токен, счётчик или флаг жизненного цикла должны обновляться только по определённым правилам, прямой доступ снаружи опасен.

Представь банковский счёт. Если любой внешний код может написать `account.balance = 999999`, модель уже не модель, а бумажный домик. Конечно, не всё в классе надо прятать. Но то, что не должно меняться произвольно, лучше защищать нормальным механизмом, а не надеждой на дисциплину команды.

### Геттеры и сеттеры в классах

Если ты привык к Java-геттерам (`getName()`, `setName()`), то в JS есть более элегантный способ — accessor properties через `get` и `set`. Снаружи они выглядят как обычные свойства, но внутри вызывается метод.

```javascript
class Temperature {
  #celsius;

  constructor(celsius) {
    this.#celsius = celsius;
  }

  get fahrenheit() {
    return this.#celsius * 9/5 + 32;
  }

  set fahrenheit(f) {
    this.#celsius = (f - 32) * 5/9;
  }

  get celsius() {
    return this.#celsius;
  }

  set celsius(c) {
    if (c < -273.15) throw new RangeError('Ниже абсолютного нуля!');
    this.#celsius = c;
  }
}

const temp = new Temperature(100);
console.log(temp.fahrenheit); // 212 — вызвался getter
temp.fahrenheit = 32;         // вызвался setter
console.log(temp.celsius);    // 0
```

Для Java-разработчика это как Lombok `@Getter/@Setter`, только встроенное в язык. Геттеры/сеттеры идеально сочетаются с `#private` полями: снаружи красивый интерфейс, внутри — валидация и приватное хранилище.

⚠️ **Осторожно с геттерами в Angular templates!**

Если геттер используется в шаблоне Angular, он вызывается на каждом цикле change detection. Тяжёлая логика внутри такого геттера быстро начинает тормозить UI:

```typescript
// ❌ Angular будет дёргать это снова и снова
get filteredTasks() {
  return this.tasks.filter(t => t.status === 'active');
}

// ✅ Лучше computed() signal или явное кеширование
filteredTasks = computed(() => this.tasks().filter(t => t.status === 'active'));
```

### Важные ограничения `#private`

Теперь — честно о границах.

- Приватные поля не участвуют в обычном доступе по строковому имени.
- Их нельзя получить через `obj['#field']` — это просто другая строка, а не настоящий доступ.
- Приватность жёстко привязана к телу класса.
- Они не являются «protected» как в Java: у наследников нет автоматического доступа к приватным полям родителя.

Вот это место часто удивляет Java-разработчиков сильнее всего. В Java `private` закрыт для наследников, а `protected` открыт. В чистом JS есть жёсткий `#private`, но отдельного встроенного `protected` нет. Приходится проектировать API аккуратнее.

### Промежуточный пример с инвариантом

```javascript
class Counter {
  #value = 0;

  increment() {
    this.#value += 1;
  }

  decrement() {
    if (this.#value === 0) {
      return;
    }

    this.#value -= 1;
  }

  getValue() {
    return this.#value;
  }
}
```

### Разбор по шагам

1. Внутреннее значение счётчика хранится в `#value`.
2. Менять его можно только методами класса.
3. `decrement()` защищает инвариант: значение не уходит ниже нуля.
4. Снаружи можно только читать результат через `getValue()`.
5. Такой дизайн делает объект предсказуемее и надёжнее.

### История типичной ошибки: underscore приняли за защиту

Команда пишет класс `UserSession`, внутри есть `_token`. Один разработчик думает: «Подчёркивание — это ведь private-ish, значит нормально». Другой в соседнем файле решает быстро «пофиксить баг» и напрямую пишет `session._token = 'fake'`. Через неделю авторизация ломается странным образом.

Проблема не в злонамеренности. Проблема в том, что соглашение не равно механизму. Подчёркивание — это дорожный знак. `#private` — это шлагбаум.

### История типичной ошибки: ожидание `protected`

Java-разработчик создаёт `BaseTask` с `#history`, потом в наследнике хочет дополнить историю и пишет `this.#history.push(...)`. Код не работает. Он удивляется: «Но я же наследник!» Да, но приватное поле привязано к телу класса `BaseTask`, а не «открыто семейству».

Если тебе нужен расширяемый контракт для наследников, чаще лучше дать защищённый по смыслу публичный/внутренний метод вроде `_addHistoryEntry()` или отдельный API, а не рассчитывать на поведение Java `protected`, которого в языке нет.

## 4.6. 🎯 Миксины и множественное наследование — Interview Focus

### История-проблема: один класс, а поведения хочется несколько

Представь, что ты делаешь `Task` для pet project. Потом приходит новая потребность: задачи надо тегировать. Потом ещё одна: задачи надо помечать временем последнего изменения. Потом ещё: задачи надо логировать. И тут возникает вопрос: как собрать несколько независимых способностей в один объект?

Если ты долго жил в классическом ООП, рука тянется строить иерархию. Но очень быстро получается монстр: `TimestampedTaggableAuditablePriorityRecurringTask`. Уже смешно, да? А теперь представь это в проде.

Стоп. Вот где JavaScript мягко подталкивает тебя к более композиционному мышлению. У класса только один `extends`. То есть **множественного наследования классов в JS нет**. И это даже неплохо: язык избавляет тебя от кучи конфликтов и ромбов, но оставляет пространство для сборки поведения другими способами.

### Что такое миксин

**Миксин** — это переиспользуемый кусок поведения, который можно подмешать в класс или объект без построения жёсткой иерархии. Не второй настоящий родитель, а добавочный слой возможностей.

Метафора из кухни: у тебя есть базовая паста. Отдельно есть соус песто, отдельно — пармезан, отдельно — острый перец. Ты не создаёшь новый вид макарон ради каждой комбинации. Ты подмешиваешь нужные свойства к уже существующей базе.

Метафора из Java-мира: миксин чем-то напоминает комбинацию default-методов, декораторов и композиции. Но это не интерфейс и не настоящий второй `extends`. Это способ дать объекту ещё одну способность, не превращая архитектуру в лес наследников.

### Современный вариант: mixin как функция от базового класса

```javascript
const Timestamped = Base => class extends Base {
  touch() {
    this.updatedAt = new Date();
  }
};

const Taggable = Base => class extends Base {
  addTag(tag) {
    this.tags ??= []; // Оператор `??=` (nullish assignment) — это сокращение для `if (this.tags == null) this.tags = []`. Присваивает значение только если текущее — `null` или `undefined`.
    this.tags.push(tag);
  }
};

class Task {
  constructor(title) {
    this.title = title;
  }
}

class RichTask extends Taggable(Timestamped(Task)) {}
```

### Разбор по шагам

1. `Timestamped` — функция, которая принимает базовый класс и возвращает новый класс с методом `touch()`.
2. `Taggable` делает то же самое, но добавляет `addTag(tag)`.
3. `Task` остаётся простой базовой сущностью.
4. `RichTask extends Taggable(Timestamped(Task))` строит цепочку: сначала добавляется timestamp-поведение, потом tag-поведение.
5. В итоге `RichTask` получает и `touch()`, и `addTag()`, не имея двух родителей в `extends`.

### Что это даёт на практике

Такой подход хорош, когда способности ортогональны друг другу. То есть «уметь тегироваться» и «уметь обновлять timestamp» — это не разные виды задач, а независимые характеристики, которые можно комбинировать.

Это делает миксин ближе к композиции, чем к классическому наследованию «родитель → ребёнок».

### Старый стиль: `Object.assign` в прототип

Есть и более старый способ — просто копировать методы в прототип.

```javascript
const auditMixin = {
  audit(message) {
    this.auditLog ??= []; // Оператор `??=` (nullish assignment) — это сокращение для `if (this.auditLog == null) this.auditLog = []`. Присваивает значение только если текущее — `null` или `undefined`.
    this.auditLog.push(message);
  }
};

class Task {
  constructor(title) {
    this.title = title;
  }
}

Object.assign(Task.prototype, auditMixin);
```

### Разбор по шагам

1. Создаётся объект `auditMixin` с методом `audit`.
2. `Task` объявляется как обычный класс.
3. `Object.assign(Task.prototype, auditMixin)` копирует методы из миксина в прототип `Task`.
4. После этого все экземпляры `Task` смогут вызывать `audit()`.
5. Метод работает, но происхождение поведения становится менее прозрачным, а конфликты имён возможнее.

### Почему не сделать множественное наследование как в некоторых других языках

Потому что вместе с множественным наследованием приходит старая проблема ромба. Если два родителя дают один и тот же метод `save()`, кто выигрывает? В каком порядке? Что делать с состоянием? Это быстро становится не учебником, а сериалом с запутанными родственными связями.

JavaScript решает это жёстко и просто: один `extends`. Всё остальное — осознанно через композицию, миксины, делегирование, декораторный стиль. Это снижает магию и заставляет тебя проектировать поведение более явно.

### Миксин ≠ второй настоящий родитель

Это принципиальный момент для собеседования. Не говори: «Миксин — это способ множественного наследования». Лучше сказать точнее: **миксин позволяет добавлять поведение без полноценного множественного наследования классов**.

Почему важно так формулировать? Потому что миксин обычно либо копирует методы, либо строит цепочку классов-обёрток. Но у тебя не появляется два независимых базовых класса в формальном смысле языка.

### Промежуточный пример из реальной предметной области

```javascript
const Assignable = Base => class extends Base {
  assignTo(user) {
    this.assignee = user;
  }
};

const Archivable = Base => class extends Base {
  archive() {
    this.archived = true;
  }
};

class Task {
  constructor(title) {
    this.title = title;
  }
}

class BoardTask extends Archivable(Assignable(Task)) {}

const task = new BoardTask('Подготовить релиз');
task.assignTo('Denis');
task.archive();
```

### Разбор по шагам

1. `Assignable` добавляет поведение назначения пользователя.
2. `Archivable` добавляет поведение архивирования.
3. `Task` даёт базовое поле `title`.
4. `BoardTask` собирает обе способности поверх базы.
5. Экземпляр получает комбинированное поведение без глубокой иерархии наследования.

### История типичной ошибки: бесконечная иерархия

Команда начинает с `Task`, потом появляется `PriorityTask`, потом `RecurringPriorityTask`, потом `RecurringPriorityTaskWithTags`, потом `RecurringPriorityTaskWithTagsAndAudit`. В какой-то момент становится страшно открывать файл. Любое новое требование означает ещё один подкласс.

Это момент, когда миксины и композиция начинают спасать не синтаксис, а психическое здоровье команды. Если способность живёт независимо, не надо превращать каждую комбинацию в новый биологический вид класса.

### История типичной ошибки: `Object.assign` затёр существующий метод

Разработчик добавил миксин `save()` через `Object.assign`, не заметив, что в `Task.prototype` уже есть свой `save()`. Код не упал сразу. Просто поведение тихо поменялось. Потом полдня расследования.

Вот почему современная форма `Base => class extends Base` часто безопаснее и яснее. Она лучше показывает происхождение поведения и меньше напоминает «подкрался ночью и подменил метод в коробке передач».

### Почему спрашивают

Миксины показывают, понимаешь ли ты ограничения классического наследования в JavaScript и умеешь ли мыслить композицией. Интервьюер здесь проверяет зрелость: не только знаешь ли ты синтаксис, но и можешь ли выбрать адекватный инструмент для модели, где одного `extends` мало.

## 4.7. Абстрактные паттерны: `abstract` нет, но можно имитировать

### Какая здесь боль для Java-разработчика

В Java ты можешь довольно рано выразить архитектурное намерение языком: `abstract class`, `abstract method`, интерфейс. Сам язык помогает тебе не забыть обязательные части контракта. В JavaScript такого встроенного compile-time каркаса нет.

И вот здесь у новичков бывает два неверных вывода. Первый: «Ну раз нет `abstract`, значит в JS нельзя строить нормальные архитектурные контракты». Второй: «Ладно, буду притворяться, что они есть сами собой». Оба плохие.

Правильная мысль такая: **в чистом JavaScript нет встроенного `abstract`, но идею абстрактной базы можно реализовать через рантайм-проверки, договорённости и тесты**. Это менее жёстко, чем в Java, но всё ещё полезно.

### Запрет прямого создания базового класса через `new.target`

```javascript
class Task {
  constructor(title) {
    if (new.target === Task) {
      throw new Error('Task — абстрактная база, создавай наследника');
    }

    this.title = title;
  }

  calculateScore() {
    throw new Error('Метод calculateScore() должен быть переопределён');
  }
}

class PriorityTask extends Task {
  constructor(title, priority) {
    super(title);
    this.priority = priority;
  }

  calculateScore() {
    return this.priority === 'high' ? 100 : 50;
  }
}
```

### Разбор по шагам

1. В конструкторе `Task` проверяется `new.target`.
2. Если попытались вызвать именно `new Task(...)`, выбрасывается ошибка.
3. Это запрещает прямое создание базового класса.
4. Метод `calculateScore()` в базе тоже бросает ошибку, если наследник не переопределил поведение.
5. `PriorityTask` вызывает `super(title)` и уже даёт рабочую реализацию `calculateScore()`.

### Что такое `new.target`

`new.target` — это специальная ссылка на конструктор, через который реально вызвали `new`. Не на «текущий текстовый класс», а на фактическую цель создания объекта.

Метафора: это как штамп в пропускной системе. Если на пропуске написано `Task`, значит ты вошёл именно через базовую дверь. Если там `PriorityTask`, значит вошёл через наследника, и база лишь участвует в цепочке настройки.

### Почему это только имитация, а не полный аналог Java `abstract`

Потому что защита срабатывает во время выполнения, а не на этапе компиляции. Ты можешь написать код, который выглядит нормально, и узнать об ошибке только когда он реально вызовется. В Java многие такие вещи компилятор отловил бы раньше.

Это не «плохо» само по себе. Просто меняется баланс ответственности: меньше гарантий от языка, больше ответственности у тестов, код-ревью и дисциплины дизайна.

### Ещё один подход: базовый метод как явный контракт

Иногда даже без запрета `new.target` полезно сделать базовый метод, который по умолчанию бросает ошибку. Это честный сигнал: «Наследник обязан меня определить».

```javascript
class Exporter {
  export() {
    throw new Error('Метод export() должен быть реализован наследником');
  }
}

class JsonExporter extends Exporter {
  export(data) {
    return JSON.stringify(data);
  }
}
```

### Разбор по шагам

1. `Exporter` описывает общий контракт: некий объект умеет экспортировать данные.
2. Но базовая реализация `export()` намеренно нерабочая.
3. Если наследник забыл её переопределить, ошибка всплывёт при вызове.
4. `JsonExporter` даёт конкретную реализацию контракта.
5. Это простой способ показать обязательность поведения даже без ключевого слова `abstract`.

### Когда лучше не строить «псевдо-абстрактный» класс

Если база почти пустая и только требует «override me», спроси себя: может, здесь лучше функция, стратегия или фабрика? Не стоит строить абстрактный класс только по привычке из Java. Абстракция полезна, когда реально снимает дублирование и проясняет контракт.

**2. Что проверяет `new.target === Task` внутри конструктора `Task`?**

- A) Что объект уже удалён сборщиком мусора
- B) Что метод вызван статически
- C) Что пытаются создать именно `Task`, а не его наследника
- D) Что класс приватный

## 4.8. `instanceof` и `Symbol.hasInstance`

### Почему просто "посмотреть на имя класса" недостаточно

Когда ты работаешь с объектами в динамическом языке, постоянно хочется ответить на вопрос: «Что это вообще такое?» В Java часть ответа приходит из системы типов и компилятора. В JavaScript вопрос часто решается уже во время выполнения.

Новичок может подумать: «Ну, наверное, `instanceof` просто сравнивает имя класса». Но нет. Если бы было так просто, JavaScript не был бы JavaScript. Здесь всё опять упирается в прототипную цепочку — привет модулю 3 и разделу 4.1.

Представь паспортный контроль. Пограничник не верит тому, что ты сам сказал про гражданство. Он смотрит на документы, цепочку подтверждений и штампы. `instanceof` делает что-то похожее: не верит словам, а проверяет происхождение объекта через его прототипы.

### Базовый пример `instanceof`

```javascript
class Task {}
class PriorityTask extends Task {}

const task = new PriorityTask();

console.log(task instanceof PriorityTask); // true
console.log(task instanceof Task); // true
```

### Разбор по шагам

1. `task` создан через `new PriorityTask()`.
2. Его непосредственный прототип — `PriorityTask.prototype`.
3. Дальше в цепочке прототипов находится `Task.prototype`.
4. Поэтому `task instanceof PriorityTask` даёт `true`.
5. И `task instanceof Task` тоже даёт `true`, потому что прототип родителя тоже встречается в цепочке.

### Что именно проверяет `instanceof`

Короткая формула звучит так: оператор `instanceof` проверяет, встречается ли `Constructor.prototype` в прототипной цепочке объекта.

Это важно проговаривать именно так, а не размыто вроде «проверяет принадлежность классу». Потому что в JS принадлежность — не абстрактная сущность, а конкретная цепочка ссылок между объектами и прототипами.

### `Symbol.hasInstance`: когда хочется свои правила

Иногда стандартного правила мало, и класс хочет сам решить, что считать своим «экземпляром». Для этого существует хук `Symbol.hasInstance`.

Это уже продвинутый, не повседневный инструмент. Но для собеседований и глубинного понимания он полезен. Представь охранника на входе, который вместо обычной турникетной проверки использует отдельный список критериев: «если есть бейдж, форма и код доступа — пропускаем».

```javascript
class JsonTaskLike {
  static [Symbol.hasInstance](obj) {
    return obj && typeof obj.title === 'string' && typeof obj.complete === 'function';
  }
}

const candidate = {
  title: 'Псевдо-задача',
  complete() {}
};

console.log(candidate instanceof JsonTaskLike); // true
```

### Разбор по шагам

1. Класс `JsonTaskLike` определяет статический метод по ключу `Symbol.hasInstance`.
2. Когда выполняется `candidate instanceof JsonTaskLike`, движок использует этот хук.
3. Хук проверяет не прототипную цепочку, а форму объекта: есть ли строковый `title` и функция `complete`.
4. Объект `candidate` проходит эту проверку.
5. Поэтому `instanceof` возвращает `true`, даже без реального наследования от `JsonTaskLike`.

### Это уже похоже на duck typing

Здесь мы приближаемся к идее: «если объект крякает как утка, возможно, это достаточно хорошая утка». В JavaScript такая форма мышления естественнее, чем в Java. Иногда тебе важнее не происхождение объекта, а наличие нужных методов и данных.

Но не увлекайся. В обычном бизнес-коде часто проще и честнее написать явную функцию `isTaskLike(obj)`, чем перегружать `instanceof` экзотикой. `Symbol.hasInstance` — это скорее тонкий инструмент, а не повседневный молоток.

### Промежуточный пример с явной проверкой

```javascript
function isTaskLike(obj) {
  return obj && typeof obj.title === 'string' && typeof obj.complete === 'function';
}
```

### Разбор по шагам

1. Функция принимает любой кандидат-объект.
2. Сначала проверяется, что объект вообще существует.
3. Потом проверяется наличие `title` нужного типа.
4. И наличие метода `complete`.
5. Это простой, прозрачный и часто достаточный способ проверки формы объекта.

### История типичной ошибки

Разработчик думает, что `instanceof` — это универсальная проверка «объект похож на нужный». Потом получает данные из другого окна браузера, другого realm или просто plain-object после JSON-десериализации, и проверка ведёт себя не так, как он ожидал.

С этого момента полезно задавать себе вопрос: **что я на самом деле хочу проверить — происхождение по прототипной цепочке или наличие нужного контракта поведения?** Это два разных вопроса.

## 4.9. Сравнение: классы JavaScript vs классы Java

### Зачем вообще делать это сравнение отдельно

Если ты переходишь с Java, тебе мало просто выучить синтаксис. Нужно ещё научиться **не доверять слишком быстро знакомым словам**. Самый надёжный способ — поставить языки рядом и честно проговорить сходства и различия.

Это похоже на пересадку с механики на автомат. У машины по-прежнему руль, сиденье, колёса и педали. Но некоторых привычных движений делать уже не надо, а некоторые старые рефлексы даже мешают. С классами в JS та же история: похожий интерфейс, другие акценты и иные опасные зоны.

### Таблица прямого сравнения

| Аспект | JavaScript | Java |
|---|---|---|
| Базовая модель | Прототипы, `class` как синтаксический сахар | Классическая class-based модель |
| Типизация | Динамическая | Статическая |
| Проверка ошибок | В основном во время выполнения | Многие ошибки ловятся компилятором |
| Конструктор | Один `constructor` на класс | Можно перегружать несколько конструкторов |
| Перегрузка методов | Нет настоящей перегрузки по сигнатуре | Есть |
| Модификаторы доступа | Публичность по умолчанию, `#private` для жёсткой приватности | `public`, `protected`, `private`, package-private |
| Наследование классов | Только один `extends` | Только один `extends` |
| Множественное поведение | Миксины, композиция | Интерфейсы, `default` методы, композиция |
| Статические члены | Есть через `static` | Есть через `static` |
| Абстрактные классы | Нативно нет, имитация через рантайм | Есть `abstract` |
| Интерфейсы | Нативно нет в чистом JS | Есть |
| `instanceof` | Основан на прототипной цепочке и может быть кастомизирован | Основан на системе типов JVM |
| Hoisting | Классы в TDZ | Нет аналога |
| `this` | Зависит от способа вызова, легко теряется | Жёстко привязан к экземпляру метода |
| Методы | Живут в `prototype` | Связаны с классом на уровне JVM |
| Поля экземпляра | Часто задаются в конструкторе или class fields | Описываются в теле класса |

### Разбор таблицы человеческим языком

Самая главная строка здесь — первая. Всё остальное во многом вырастает из неё. В JavaScript класс — это удобный синтаксис поверх прототипов. В Java класс — фундаментальная единица модели языка.

Из этого следуют и более слабые compile-time гарантии, и особая роль `this`, и отсутствие обычной перегрузки, и иная природа `instanceof`. Если держать в голове только похожие слова, различия будут раздражать. Если держать в голове разные основания модели, различия становятся логичными.

### Что у Java стоит позаимствовать как мышление, а что — не тащить буквально

Из Java полезно взять уважение к инвариантам, разделение состояния и поведения и привычку думать уровнями ответственности. Опасно переносить буквально веру в compile-time гарантии, ожидание перегрузки и стремление решать всё глубокой иерархией классов.

## 4.10. Паттерны: Singleton, Factory, Builder на JavaScript

### Сначала честный разговор о паттернах

Паттерны — это не коллекционные фигурки, которые надо обязательно собрать все. Это повторяемые ответы на повторяемые проблемы. Если нет проблемы — паттерн превращается в лишний костюм на жаре.

Java-разработчики особенно уязвимы в этой теме. После мира enterprise-приложений рука иногда автоматически тянется строить Factory, Builder, Singleton и ещё пол-GOF-каталога там, где хватило бы функции и объекта `options`. Поэтому мы пойдём от обратного: **сначала проблема, потом решение**.

Представь ящик с инструментами. Хороший мастер не забивает шуруп кувалдой только потому, что кувалда дорогая. Так и с паттернами: сила не в количестве умных названий, а в точности применения.

### Singleton

#### Проблема сначала

У тебя есть реестр задач, общий конфиг, единая точка логирования, один менеджер WebSocket-подключения. Если таких объектов окажется несколько несинхронизированных, начнётся хаос: разные экземпляры будут хранить разное состояние, и ты будешь долго ловить баги «почему в одном месте список пустой, а в другом заполнен».

То есть проблема Singleton не в любви к ограничениям. Проблема в требовании: **для этой роли в системе нужен один координирующий экземпляр**.

#### Классический вариант через класс

```javascript
class TaskRegistry {
  static #instance;

  constructor() {
    if (TaskRegistry.#instance) {
      return TaskRegistry.#instance;
    }

    this.tasks = [];
    TaskRegistry.#instance = this;
  }

  add(task) {
    this.tasks.push(task);
  }
}

const registryA = new TaskRegistry();
const registryB = new TaskRegistry();
console.log(registryA === registryB); // true
```

### Разбор по шагам

1. Класс хранит приватное статическое поле `#instance`.
2. При первом вызове `new TaskRegistry()` экземпляр создаётся нормально.
3. Он записывается в `TaskRegistry.#instance`.
4. При следующем вызове конструктор возвращает уже существующий экземпляр.
5. Поэтому `registryA === registryB` даёт `true`.

Как это работает? В JavaScript, если конструктор (функция, вызванная с `new`) **явно возвращает объект**, то `new` вернёт именно этот объект, а не свежесозданный `this`. Это фундаментальное отличие от Java, где конструктор НИКОГДА не возвращает значение. Именно поэтому `return TaskRegistry.#instance` работает.

#### Почему в современном JS часто есть способ проще

Теперь важная адаптация к JavaScript. Во многих случаях тебе вообще не нужен классический Singleton-паттерн, потому что **сам модуль уже ведёт себя как singleton**: модуль загружается один раз и кэшируется.

То есть вместо сложной конструкции иногда достаточно сделать:

```javascript
export const taskRegistry = {
  tasks: [],
  add(task) {
    this.tasks.push(task);
  }
};
```

### Разбор по шагам

1. Модуль экспортирует один объект `taskRegistry`.
2. При импорте в разных местах приложения будет использоваться один и тот же экземпляр экспорта.
3. Свойство `tasks` остаётся общим состоянием этого объекта.
4. Метод `add(task)` работает с этим общим состоянием.
5. Так модульная система часто закрывает задачу singleton проще и естественнее, чем отдельный класс.

#### История типичной ошибки

Разработчик по привычке из Java строит тяжёлый singleton-класс для конфигурации фронтенда, хотя мог просто экспортировать один объект из модуля. В итоге лишний код, лишняя церемония и никакой реальной выгоды.

Мораль простая: паттерн должен уважать экосистему языка. В Java модульная система и жизненный цикл другие. В JS многие старые задачи решаются проще.

### Factory

#### Проблема сначала

Теперь представь другую ситуацию. Данные о задачах приходят из формы, localStorage, API, импортированного JSON. У части задач есть `priority`, у части — `intervalDays`, у части — вообще только `title`. Если UI начнёт сам решать, какой конструктор вызвать, логика выбора расползётся по приложению.

Вот здесь и появляется реальная проблема: **нужно одно место, которое принимает сырой ввод и решает, какой объект создавать**.

#### Классическая фабрика

```javascript
class TaskFactory {
  static create(data) {
    switch (data.type) {
      case 'priority':
        return new PriorityTask(data.title, data.priority);
      case 'recurring':
        return new RecurringTask(data.title, data.intervalDays);
      default:
        return new Task(data.title);
    }
  }
}
```

### Разбор по шагам

1. `TaskFactory.create(data)` получает сырые входные данные.
2. По полю `type` она решает, какой конкретный класс нужен.
3. Для `'priority'` создаётся `PriorityTask`.
4. Для `'recurring'` создаётся `RecurringTask`.
5. Во всех остальных случаях создаётся обычный `Task`.

#### Почему это полезно

Фабрика собирает в одном месте правила преобразования данных в объекты. UI не знает про конкретные конструкторы. localStorage-слой не знает, чем `RecurringTask` отличается от `PriorityTask`. Всё это скрыто внутри точки создания.

Метафора: не каждый покупатель заходит в цех и сам выбирает станки. Он приходит на стойку заказа и говорит, что ему нужно. Фабрика сама решает, какой именно процесс собрать.

#### Промежуточный пример: разные источники данных

```javascript
class TaskFactory {
  static fromStorage(raw) {
    return TaskFactory.create(raw);
  }

  static fromForm(raw) {
    return TaskFactory.create({
      ...raw,
      type: raw.type || 'basic'
    });
  }

  static create(data) {
    switch (data.type) {
      case 'priority':
        return new PriorityTask(data.title, data.priority ?? 'normal');
      case 'recurring':
        return new RecurringTask(data.title, Number(data.intervalDays) || 7);
      default:
        return new Task(data.title);
    }
  }
}
```

### Разбор по шагам

1. `fromStorage(raw)` и `fromForm(raw)` дают более выразительные входные точки.
2. Обе сводятся к общей функции `create(data)`.
3. Фабрика не только выбирает класс, но и нормализует данные.
4. Для `priority` подставляется дефолт `'normal'`, если значение не пришло.
5. Для `recurring` строковое значение интервала превращается в число с запасным значением `7`.

#### История типичной ошибки

Разработчик не делает фабрику, и через месяц логика создания задач размазывается по шести файлам: форма создаёт один вид задачи, API-модуль другой, localStorage — третий. Потом меняется правило и приходится чинить всё в нескольких местах.

Фабрика — это не про «красивый паттерн ради паттерна». Это про централизацию решения «что именно создавать». Если такой проблемы нет — фабрика может быть лишней. Если проблема есть — она очень спасает.

### Builder

#### Проблема сначала

Иногда объект слишком сложно создавать одним конструктором. Много необязательных параметров, часть зависит друг от друга, нужна пошаговая валидация, хочется fluent API. Да, в JS часто помогает объект `options`, и это уже очень сильная альтернатива. Но не всегда её хватает.

Представь сборку компьютера. Можно теоретически в одном крике перечислить: корпус, материнка, память, SSD, видеокарта, блок питания, кулеры. Но человеку удобнее собирать по шагам и видеть промежуточный результат. Builder — это такой управляемый пошаговый сборщик.

#### Пример билдера

```javascript
class TaskBuilder {
  #title = 'Без названия';
  #priority = 'normal';
  #intervalDays = null;

  setTitle(title) {
    this.#title = title;
    return this;
  }

  setPriority(priority) {
    this.#priority = priority;
    return this;
  }

  setIntervalDays(intervalDays) {
    this.#intervalDays = intervalDays;
    return this;
  }

  build() {
    return this.#intervalDays == null
      ? new PriorityTask(this.#title, this.#priority)
      : new RecurringTask(this.#title, this.#intervalDays);
  }
}
```

### Разбор по шагам

1. `TaskBuilder` хранит внутреннее промежуточное состояние в приватных полях.
2. Методы `setTitle`, `setPriority`, `setIntervalDays` меняют это состояние и возвращают `this`.
3. Благодаря `return this` можно собирать fluent-цепочку вызовов.
4. `build()` смотрит на накопленные параметры и решает, какой итоговый объект создать.
5. Это полезно, когда создание идёт по шагам и требует осмысленной сборки.

#### Когда builder всё-таки уместен

- когда нужна строгая пошаговая сборка;
- когда каждый шаг должен валидироваться отдельно;
- когда хочется fluent API для читаемости;
- когда есть несколько сценариев сборки одного семейства объектов.

#### История типичной ошибки

Разработчик пишет builder для объекта из трёх полей, которые прекрасно помещались в обычный `options`-объект. В итоге кода в два раза больше, пользы в нём меньше, а читаемость не выросла.

## Финальная мысль модуля

Если в Java класс — это фундамент здания, то в JavaScript класс — это удобный фасад над уже существующей конструкцией из объектов и прототипов. Это не делает подход хуже. Это просто значит, что тебе нужно видеть сразу два слоя: красивый синтаксис сверху и реальный механизм снизу.

Когда ты это видишь, странности перестают быть странностями. `super()` уже не кажется ритуалом, `instanceof` — не кажется мистикой, а `#private` — не кажется просто забавным символом. Всё встаёт на свои места.

Попробуй унести из модуля не список ключевых слов, а одну взрослую мысль: **в JavaScript синтаксис часто дружелюбнее, чем фундаментальная модель, и именно поэтому нельзя останавливаться на внешнем сходстве**.

Если хочешь, можешь даже проверить себя прямо сейчас. Закрой файл на минуту и попробуй вслух объяснить другу или пустой комнате: почему `class` — это сахар над прототипами? Если объяснение получилось спокойно и без магии — модуль сработал.
