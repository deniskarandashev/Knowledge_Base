## Главная установка модуля

**В JavaScript структура данных — это не только контейнер, но и встроенный набор операций над потоком значений. Если в Java ты часто думаешь через `Collection + Stream`, то в JS эта идея часто живёт прямо на массиве.**

Стоп. Это важная мысль. Не пролистывай. В Java ты часто мысленно разделяешь: вот коллекция, а вот отдельно поверх неё запускается `stream()`. В JavaScript массив уже приехал к тебе с половиной нужного инструментария. Поэтому хороший JS-код часто выглядит не как серия циклов, а как цепочка небольших преобразований.

Вторая важная установка: **умение выбрать структуру данных — это не «синтаксис», а инженерное мышление**. Если ты хранишь уникальные теги в массиве и каждый раз проверяешь `includes`, ты решаешь задачу не тем инструментом. Если ты используешь `reduce` там, где нужен обычный `map`, ты тоже перегибаешь. Этот модуль не только про «как написать», но и про «почему именно так».

## Блок 1: Массивы как мини-Stream API

### 5.1. Массивы: полный обзор методов

**Массив (`Array`)** — это упорядоченный контейнер, как поезд, где у каждого вагона есть место. В Java ближайший родственник — `ArrayList`. Но в JavaScript массив важен не только как хранилище, а как готовая лента обработки, потому что на нём уже висят `map`, `filter`, `reduce` и другие операции, очень похожие на `Stream API`.

Представь кухню доставки. Есть коробки с блюдами, и тебе надо: переименовать позиции для чека, отфильтровать только вегетарианские заказы, найти первый просроченный, посчитать итоговую выручку, собрать статистику по ресторанам. Вот здесь массив — не просто полка. Это как конвейер на складе, где у каждой операции уже есть собственная станция.

Есть и вторая метафора. Массив — это не только `ArrayList`, но ещё и кусочек `Stream` прямо внутри. Как будто тебе в Java дали `ArrayList` с кнопками `map`, `filter`, `find`, `some`, `reduce`.

#### Почему это вообще важно до кода

Новички часто пытаются выучить массивные методы как список заклинаний. Это почти всегда плохо работает. Правильнее видеть намерение:

- `map` — меняю форму каждого элемента;
- `filter` — оставляю только нужные;
- `find` — ищу первый подходящий и сразу останавливаюсь;
- `some` / `every` — задаю коллекции вопрос да/нет;
- `reduce` — сворачиваю много значений в один итог;
- `flatMap` — из одного элемента делаю несколько и сразу расплющиваю результат.

Почему это полезно на проекте? Потому что код начинает читаться как фраза: взяли задачи, оставили активные, отсортировали, превратили в строки для UI. Это не только красиво, но и уменьшает шанс перепутать индекс, забыть `push` или случайно мутировать исходный массив.

#### Немутирующие и мутирующие методы

Вот первая большая идея, которую надо держать в голове: массивные методы делятся на **немутирующие** и **мутирующие**.

- **Немутирующие** работают как ксерокопия документа: берут исходник и возвращают новую версию.
- **Мутирующие** работают как редактор прямо на оригинале: пришёл и подчистил ручкой на месте.

Почему это критично? Представь, что один и тот же массив задач используется и в UI, и в аналитике, и в сохранении в localStorage. Если ты по дороге внезапно вызвал `sort()` или `splice()` на оригинале, ты изменил данные не только для текущей функции, но и для всех, кто держит ссылку на тот же массив. Это как переставить книги на общей полке и удивиться, почему коллега теперь не может ничего найти.

Для Java-разработчика здесь полезно включить знакомую мысль: иногда ты работаешь как с `Collections.unmodifiableList()` по духу — аккуратно создаёшь новый результат и не трогаешь исходник. А иногда осознанно делаешь операцию «на месте». Важно не то, чтобы всегда избегать мутаций, а то, чтобы они были **явными**, а не случайными.

### `map`: меняем форму, не меняя длину

#### Зачем нужен `map`

`map` очень легко почувствовать на реальной задаче. Представь интернет-магазин. У тебя есть массив товаров заказа, а фронтенду для списка нужен не весь объект, а только строка вида `"Ноутбук — 2 шт."`. Ты не удаляешь товары, не ищешь один товар, не суммируешь всё в итог. Ты просто превращаешь **каждый** элемент из одной формы в другую.

Вот главный критерий: если на входе один элемент и на выходе тоже один элемент, но в другой форме, это почти наверняка `map`. Как покрасочная линия на заводе: каждая коробка проезжает станцию и уезжает дальше, коробок столько же, просто они стали другими.

В Java это очень похоже на `stream().map(...).toList()`. Для Java-глаза это полезный якорь: `map` не фильтрует, не группирует, не уменьшает количество элементов. Он **преобразует**.

#### Пример: готовим строки для чека

**JavaScript**

```javascript
const orderItems = [
  { id: 1, title: 'Ноутбук', quantity: 1 },
  { id: 2, title: 'Мышь', quantity: 2 },
  { id: 3, title: 'Кабель USB-C', quantity: 3 }
];

const receiptLines = orderItems.map(item => `${item.title} — ${item.quantity} шт.`);

console.log(receiptLines);
// ['Ноутбук — 1 шт.', 'Мышь — 2 шт.', 'Кабель USB-C — 3 шт.']
```

Разберём по шагам:

1. `orderItems` — это массив объектов с деталями заказа.
2. `map` берёт **каждый** объект `item` по очереди.
3. Для каждого `item` мы строим новую строку через шаблонный литерал.
4. Длина массива остаётся той же: было 3 элемента, стало 3 строки.
5. Исходный массив `orderItems` не изменился. Мы получили новый массив `receiptLines`.

Стоп. Спрогнозируй: если в массиве было бы 50 товаров, сколько строк окажется на выходе? Тоже 50. Это главный инстинкт `map`.

**Java (Stream API)**

```java
List<String> receiptLines = orderItems.stream()
    .map(item -> item.getTitle() + " — " + item.getQuantity() + " шт.")
    .toList();
```

Разберём по шагам:

1. `stream()` открывает поток поверх коллекции.
2. `map(...)` превращает каждый объект в строку.
3. `toList()` собирает результат обратно в список.
4. Семантика ровно та же, что и у JS-версии: форма поменялась, количество элементов сохранилось.

#### История типичной ошибки

Новичок Лена решила сделать `map`, но внутри начала `push`-ить во внешний массив и ничего не возвращать:

```javascript
const result = [];
orderItems.map(item => {
  result.push(item.title);
});
```

Разберём по шагам:

1. Этот код **работает побочным эффектом**, а не через смысл `map`.
2. `map` здесь не нужен. Он возвращает новый массив, но Лена его игнорирует.
3. Если тебе нужен побочный эффект, это ближе к `forEach`.
4. Если нужен новый массив значений, надо **возвращать** результат из `map`.

Правильнее так:

```javascript
const titles = orderItems.map(item => item.title);
```

Разберём по шагам:

1. Теперь намерение видно сразу: берём каждый товар и вытаскиваем `title`.
2. Нет внешнего накопителя, нет лишнего шума.
3. Это читается в одну секунду.

### `filter`: оставляем только нужное

#### Зачем нужен `filter`

Представь, что у тебя список доставок за день, но менеджеру нужен только список опоздавших. Или есть массив задач, а на экран надо вывести только незавершённые. Ты не меняешь форму каждого элемента, ты не ищешь один элемент, ты просто **просеиваешь** массив.

`filter` — это сито, дуршлаг, фильтр кофемашины. Всё, что не прошло условие, уходит. Всё, что прошло, остаётся. Здесь очень полезно мысленно слышать вопрос: «Этот элемент остаётся или нет?»

Для Java-разработчика это прямой родственник `stream().filter(...).toList()`. Логика абсолютно знакомая: проверка-предикат, на выходе — новый массив меньшего или равного размера.

#### Пример: оставляем только заказы с доставкой сегодня

**JavaScript**

```javascript
const deliveries = [
  { id: 1, customer: 'Ирина', isToday: true },
  { id: 2, customer: 'Максим', isToday: false },
  { id: 3, customer: 'Олег', isToday: true }
];

const todayDeliveries = deliveries.filter(delivery => delivery.isToday);

console.log(todayDeliveries);
// [
//   { id: 1, customer: 'Ирина', isToday: true },
//   { id: 3, customer: 'Олег', isToday: true }
// ]
```

Разберём по шагам:

1. `filter` проходит по каждому объекту `delivery`.
2. Если выражение `delivery.isToday` даёт `true`, элемент остаётся.
3. Если `false`, элемент выбрасывается из результата.
4. Исходный массив `deliveries` не меняется.
5. На выходе — новый массив только из доставок на сегодня.

**Java (Stream API)**

```java
List<Delivery> todayDeliveries = deliveries.stream()
    .filter(Delivery::isToday)
    .toList();
```

Разберём по шагам:

1. В поток попадают все доставки.
2. Предикат `Delivery::isToday` пропускает только нужные.
3. `toList()` собирает отфильтрованный список.

#### История типичной ошибки

Петя хотел получить одного пользователя по email и написал:

```javascript
const user = users.filter(user => user.email === 'a@example.com')[0];
```

Разберём по шагам:

1. `filter` всегда возвращает массив.
2. Потом Петя берёт первый элемент этого массива.
3. Это значит, что он обходит **всю** коллекцию, хотя ему нужен только первый подходящий.
4. Если задача — найти один элемент, лучше `find`.

Запомни маленькое правило: если в голове звучит «дай мне **первый** подходящий», это уже не `filter`, а `find`.

### `find` и `findIndex`: найти и остановиться

#### Зачем нужен `find`

Есть ситуации, где тебе не нужен новый массив. Например, на складе ты ищешь первую коробку с кодом `BX-104`. Или в списке задач ищешь задачу с `id = 42`, чтобы открыть карточку редактирования. Здесь мысль другая: не «оставь все подходящие», а «найди первый и остановись».

`find` похож на охранника у входа, который смотрит на людей по одному и прекращает проверку, как только увидел нужного человека. Это не сито, а целевой поиск.

`findIndex` делает почти то же самое, но возвращает не сам элемент, а его позицию. Это полезно, если ты потом хочешь заменить элемент, удалить его или подсветить нужную строку в таблице.

#### Пример: ищем заказ для открытия карточки

```javascript
const orders = [
  { id: 101, customer: 'Мила', status: 'paid' },
  { id: 102, customer: 'Тимур', status: 'new' },
  { id: 103, customer: 'Нора', status: 'shipped' }
];

const orderToEdit = orders.find(order => order.id === 102);
const orderIndex = orders.findIndex(order => order.id === 102);

console.log(orderToEdit); // { id: 102, customer: 'Тимур', status: 'new' }
console.log(orderIndex);  // 1
```

Разберём по шагам:

1. `find` идёт по массиву слева направо.
2. Первый заказ `101` не подходит.
3. Второй заказ `102` подходит.
4. `find` немедленно останавливается и возвращает объект заказа.
5. `findIndex` делает тот же поиск, но возвращает индекс `1`.

Стоп. А что будет, если ничего не найдётся? `find` вернёт `undefined`, а `findIndex` — `-1`. Это надо помнить руками.

### `some`, `every`, `includes`: быстрые вопросы к коллекции

#### Зачем они нужны

Иногда тебе не нужен ни новый массив, ни конкретный элемент, ни итоговый объект. Тебе нужен быстрый ответ на вопрос.

- Есть ли хотя бы один просроченный счёт?
- Все ли обязательные поля заполнены?
- Есть ли у пользователя роль `ADMIN`?

Это уже не похоже на конвейер преобразования. Это похоже на короткий опрос коллекции. `some` — «есть ли хотя бы один?». `every` — «все ли подходят?». `includes` — «содержится ли конкретное значение?». В Java аналогии очень приятные: `anyMatch`, `allMatch`, `contains`.

#### Пример: проверяем доступ и качество данных

```javascript
const tasks = [
  { title: 'Сверить отчёт', done: true },
  { title: 'Подготовить демо', done: false },
  { title: 'Написать письмо', done: true }
];

const roles = ['USER', 'ADMIN'];

console.log(tasks.some(task => !task.done));   // true
console.log(tasks.every(task => task.title.length > 0)); // true
console.log(roles.includes('ADMIN')); // true
```

Разберём по шагам:

1. `some(task => !task.done)` спрашивает: есть ли хотя бы одна незавершённая задача?
2. Как только такая задача находится, проверка может остановиться.
3. `every(task => task.title.length > 0)` спрашивает: у всех ли задач непустой заголовок?
4. `includes('ADMIN')` проверяет, есть ли в массиве роль `ADMIN`.
5. Каждый метод отвечает своей формой: `true` или `false`.

### `reduce`: сворачиваем много значений в один итог

#### Почему `reduce` так пугает

Стоп. Здесь обычно начинаются вздохи. `reduce` — это тот самый метод, от которого многие новички физически отодвигаются. И не потому, что он какой-то мистический. А потому что его часто объясняют слишком быстро: «Ну это когда массив сворачивается в одно значение, короче сам поймёшь». Нет, не поймёшь. И это нормально.

Самая полезная мысль: `reduce` — это **копилка промежуточного результата**. Бухгалтер идёт по стопке чеков и после каждого обновляет общий итог. Для Java-разработчика это что-то между `stream().reduce(...)` и `collect(...)`: результатом может быть число, строка, объект, массив или `Map`.

Почему новички спотыкаются? Потому что пытаются смотреть сразу на весь колбэк целиком. А надо смотреть покадрово:

1. какое стартовое значение у аккумулятора;
2. какой текущий элемент пришёл;
3. что мы вернули как новый аккумулятор;
4. что будет на следующей итерации.

Если ты поймёшь эту покадровость, `reduce` перестанет быть страшным. Поэтому мы пойдём от очень простого к более сложному.

#### Первый уровень: обычная сумма

Начнём с бытового примера. У тебя есть покупки в кофейне за неделю: 190, 250, 310 рублей. Надо получить общую сумму. Здесь `reduce` выглядит максимально честно: была кучка чисел, стал один итог.

**JavaScript**

```javascript
const expenses = [190, 250, 310];

const total = expenses.reduce((acc, amount) => {
  return acc + amount;
}, 0);

console.log(total); // 750
```

Разберём по шагам:

1. Стартовое значение аккумулятора — `0`. Это значит: до начала обхода у нас пока ничего не накоплено.
2. Первая итерация: `acc = 0`, `amount = 190`. Возвращаем `190`.
3. Вторая итерация: `acc = 190`, `amount = 250`. Возвращаем `440`.
4. Третья итерация: `acc = 440`, `amount = 310`. Возвращаем `750`.
5. Массив закончился, итоговое значение `750` возвращается в `total`.

Если хочется увидеть это в виде таблички мыслью, то картина такая:

- старт: `acc = 0`
- после `190`: `acc = 190`
- после `250`: `acc = 440`
- после `310`: `acc = 750`

Стоп. Попробуй предсказать без запуска: что будет, если стартовое значение поставить `100`? Правильно: итог будет `850`, потому что ты начал не с пустой копилки, а уже со ста рублей внутри.

**Java (Stream API)**

```java
int total = expenses.stream()
    .reduce(0, Integer::sum);
```

Разберём по шагам:

1. `0` — это стартовое значение аккумулятора.
2. `Integer::sum` — способ объединять текущий итог и очередное число.
3. Логика идентична JS-примеру.

#### Второй уровень: собираем объект-итог

Теперь чуть интереснее. Представь, что у тебя список продаж за день, и ты хочешь получить объект со статистикой: общую выручку и количество продаж. Вот здесь `reduce` особенно хорош, потому что аккумулятор может быть не числом, а объектом.

```javascript
const sales = [
  { title: 'Кофе', price: 190 },
  { title: 'Сэндвич', price: 320 },
  { title: 'Чай', price: 140 }
];

const summary = sales.reduce((acc, sale) => {
  acc.totalRevenue += sale.price;
  acc.itemsCount += 1;
  return acc;
}, { totalRevenue: 0, itemsCount: 0 });

console.log(summary); // { totalRevenue: 650, itemsCount: 3 }
```

Разберём по шагам:

1. Стартовый аккумулятор — объект `{ totalRevenue: 0, itemsCount: 0 }`.
2. Первая продажа `Кофе` увеличивает `totalRevenue` до `190`, а `itemsCount` до `1`.
3. Вторая продажа увеличивает сумму до `510`, количество до `2`.
4. Третья продажа доводит сумму до `650`, количество до `3`.
5. В итоге `summary` — уже не число, а объект со сводкой.

Обрати внимание: мы мутируем **аккумулятор**, но это контролируемая мутация внутреннего временного объекта, а не случайная порча исходного массива. Это допустимый и частый паттерн внутри `reduce`.

#### Третий уровень: объект-словарь по категориям

Теперь представь супермаркет. Есть продажи с категориями, и менеджеру нужен объект вида:

- `food` → сумма по еде,
- `electronics` → сумма по электронике,
- `books` → сумма по книгам.

Вот здесь `reduce` начинает чувствоваться как ручной сборщик отчёта.

```javascript
const purchases = [
  { category: 'food', amount: 450 },
  { category: 'books', amount: 700 },
  { category: 'food', amount: 150 },
  { category: 'electronics', amount: 4000 }
];

const totalsByCategory = purchases.reduce((acc, purchase) => {
  const currentTotal = acc[purchase.category] ?? 0;
  acc[purchase.category] = currentTotal + purchase.amount;
  return acc;
}, {});

console.log(totalsByCategory);
// { food: 600, books: 700, electronics: 4000 }
```

Разберём по шагам:

1. Стартуем с пустого объекта `{}`.
2. Первая покупка: категория `food`. В объекте пока ничего нет, поэтому берём `0` и записываем `450`.
3. Вторая покупка: категория `books`. Там тоже пока пусто, записываем `700`.
4. Третья покупка: снова `food`. Уже есть `450`, прибавляем `150`, получаем `600`.
5. Четвёртая покупка: `electronics`. Создаём новое поле со значением `4000`.
6. В финале получаем словарь сумм по категориям.

Видишь, что произошло? Мы прошли по списку один раз и на лету собрали структуру данных, которая нам нужна. Это очень типичный `reduce`-сценарий.

#### Четвёртый уровень: группировка — любимая interview-тема

А теперь классика. У тебя есть задачи с разными статусами: `todo`, `in-progress`, `done`. Ты хочешь сгруппировать их по статусу. Это уже не сумма и не плоский объект статистики, а объект, в котором лежат массивы.

Перед кодом важный вопрос: что будет аккумулятором? Не число. Не строка. И даже не просто словарь чисел. Это будет объект, где по каждому статусу хранится массив задач.

Стоп. Попробуй предсказать: что надо сделать, когда мы встречаем статус впервые? Правильно — сначала создать пустой массив. И только потом положить в него задачу.

**JavaScript**

```javascript
const tasksForBoard = [
  { title: 'Fix auth', status: 'todo' },
  { title: 'Write tests', status: 'in-progress' },
  { title: 'Deploy', status: 'todo' }
];

const grouped = tasksForBoard.reduce((acc, task) => {
  if (!acc[task.status]) {
    acc[task.status] = [];
  }

  acc[task.status].push(task);
  return acc;
}, {});

console.log(grouped);
// {
//   todo: [
//     { title: 'Fix auth', status: 'todo' },
//     { title: 'Deploy', status: 'todo' }
//   ],
//   'in-progress': [
//     { title: 'Write tests', status: 'in-progress' }
//   ]
// }
```

Разберём по шагам:

1. Стартовый аккумулятор — пустой объект `{}`.
2. Первая задача имеет статус `todo`. Поля `acc.todo` ещё нет, значит создаём пустой массив `[]`.
3. Кладём задачу `Fix auth` в `acc.todo`.
4. Вторая задача имеет статус `in-progress`. Такого поля ещё нет — создаём `acc['in-progress'] = []`.
5. Кладём туда `Write tests`.
6. Третья задача снова со статусом `todo`. Поле уже есть, новый массив создавать не надо.
7. Просто `push`-им туда `Deploy`.
8. Возвращаем аккумулятор после каждой итерации.
9. В конце получаем объект, где ключи — статусы, а значения — массивы задач.

Для Java-глаза это очень близко к ручной реализации `Collectors.groupingBy(Task::getStatus)`.

**Java (Stream API)**

```java
Map<String, List<Task>> grouped = tasksForBoard.stream()
    .collect(Collectors.groupingBy(Task::getStatus));
```

Разберём по шагам:

1. Поток проходит по задачам.
2. `groupingBy` смотрит на статус каждой задачи.
3. Под каждый статус создаётся список.
4. Задачи раскладываются по этим спискам.
5. По смыслу это то же самое, что и наш JS `reduce`, только в Java большую часть механики прячет `Collector`.

#### Частые ошибки с `reduce`

История первая. Новичок Оля забыла вернуть аккумулятор:

```javascript
const total = expenses.reduce((acc, amount) => {
  acc + amount;
}, 0);
```

Разберём по шагам:

1. Выражение `acc + amount` вычисляется.
2. Но результат никуда не возвращается.
3. После первой итерации аккумулятор становится `undefined`.
4. Дальше всё ломается.

Правильно так:

```javascript
const total = expenses.reduce((acc, amount) => {
  return acc + amount;
}, 0);
```

Разберём по шагам:

1. Теперь новый аккумулятор явно возвращается.
2. Каждая итерация получает корректный накопленный результат.

История вторая. Игорь выбрал `reduce` для простого `map`, потому что хотел выглядеть «продвинутым»:

```javascript
const titles = orderItems.reduce((acc, item) => {
  acc.push(item.title);
  return acc;
}, []);
```

Разберём по шагам:

1. Код рабочий.
2. Но по смыслу это обычный `map`.
3. Читатель тратит больше усилий, чем нужно.
4. Здесь лучше простота: `orderItems.map(item => item.title)`.

Это важный взрослый вывод: **`reduce` мощный, но не обязан побеждать в каждой задаче**. Инженерная зрелость — не в том, чтобы всё сворачивать через `reduce`, а в том, чтобы выбрать самый ясный инструмент.

#### `reduce` как ментальная модель

Если тебе всё ещё некомфортно, держи простую формулу:

> `reduce` = «возьми аккумулятор, обнови его текущим элементом, верни новый аккумулятор».

И ещё один проверочный вопрос перед использованием:

- Нужен один итог? → возможно, `reduce`.
- Нужен новый массив той же длины? → скорее `map`.
- Нужен новый массив меньшей длины? → скорее `filter`.
- Нужен один найденный элемент? → скорее `find`.

Помнишь §5.1 про «вопрос к коллекции» и «сито»? `reduce` — это не вопрос и не сито. Это именно **сборка итога**.

⚠️ **Ловушка**: `[].reduce(fn)` без начального значения выбросит `TypeError`! Всегда передавай initial value, если массив может быть пустым. В Java это аналог `Stream.reduce(identity, accumulator)` — identity обязателен.

### `flat` и `flatMap`: расплющить вложенность

#### Зачем это нужно

В реальных системах данные редко лежат красиво в один слой. У тебя может быть список заказов, а внутри каждого заказа — список товаров. Или расписание конференции, где у каждого дня есть массив докладов. Иногда тебе нужен один общий плоский список.

`flat` — это каток, который выравнивает многоэтажный массив. `flatMap` — это комбо: сначала преобразовать элементы, а потом расплющить один уровень вложенности. В Java аналогия ближе всего к `flatMap` в Stream API.

#### Пример: из дней конференции делаем единый список докладов

```javascript
const conferenceDays = [
  ['Открытие', 'JavaScript и память'],
  ['TypeScript на проекте'],
  ['Node.js в production', 'Q&A']
];

const allTalks = conferenceDays.flat();
console.log(allTalks);
// ['Открытие', 'JavaScript и память', 'TypeScript на проекте', 'Node.js в production', 'Q&A']
```

Разберём по шагам:

1. Изначально у нас массив массивов.
2. `flat()` берёт каждый внутренний массив и вытаскивает его элементы на один уровень выше.
3. На выходе получается один общий список докладов.

#### Пример: товары из заказов в один список

```javascript
const ordersWithItems = [
  { id: 1, items: ['Ноутбук', 'Мышь'] },
  { id: 2, items: ['Клавиатура'] },
  { id: 3, items: ['Монитор', 'Кабель'] }
];

const allItems = ordersWithItems.flatMap(order => order.items);
console.log(allItems);
// ['Ноутбук', 'Мышь', 'Клавиатура', 'Монитор', 'Кабель']
```

Разберём по шагам:

1. `map`-часть берёт каждый заказ и возвращает его `items`.
2. Промежуточно получилось бы что-то вроде `[['Ноутбук', 'Мышь'], ['Клавиатура'], ...]`.
3. `flatMap` сразу же убирает один уровень вложенности.
4. В итоге получаем один плоский массив товаров.

### `slice`, `splice`, `sort`, `toSorted`

#### Почему эти методы постоянно путают

Потому что они стоят рядом в голове, но ведут себя по-разному. Особенно `slice` и `splice`. Разница всего в одной букве, а эффект — как между «аккуратно отрезать кусок торта» и «вырезать кусок из торта прямо на праздничном столе».

Ещё одна ловушка — сортировка. Многие новички не знают, что `sort()` мутирует массив и без компаратора сравнивает элементы как строки. Это очень частый реальный баг.

#### `slice`: берём кусок без порчи оригинала

```javascript
const queue = ['Анна', 'Борис', 'Вика', 'Глеб'];
const firstTwo = queue.slice(0, 2);

console.log(firstTwo); // ['Анна', 'Борис']
console.log(queue);    // ['Анна', 'Борис', 'Вика', 'Глеб']
```

Разберём по шагам:

1. `slice(0, 2)` говорит: возьми элементы с индекса `0` до `2`, не включая `2`.
2. Возвращается новый массив `['Анна', 'Борис']`.
3. Исходная очередь не меняется.

#### `splice`: хирургия на месте

```javascript
const queueForAdmin = ['Анна', 'Борис', 'Вика', 'Глеб'];
const removed = queueForAdmin.splice(1, 2, 'Дина');

console.log(removed);       // ['Борис', 'Вика']
console.log(queueForAdmin); // ['Анна', 'Дина', 'Глеб']
```

Разберём по шагам:

1. Начинаем с индекса `1`.
2. Удаляем `2` элемента: `Борис` и `Вика`.
3. На их место вставляем `Дина`.
4. `splice` возвращает массив удалённых элементов.
5. Исходный массив мутирует.

Стоп. Если ты работаешь с данными состояния в UI, такой метод надо использовать очень осознанно.

#### `sort` и `toSorted`

```javascript
const numbers = [10, 9, 2, 30];

console.log(numbers.sort()); // [10, 2, 30, 9] — лексикографически! Не [2, 9, 10, 30]
console.log(numbers);
```

Разберём по шагам:

1. Без компаратора `sort()` сравнивает элементы как строки.
2. Поэтому `'30'` оказывается между `'2'` и `'9'`: сравнение идёт лексикографически, а не как у чисел.
3. Метод ещё и меняет исходный массив `numbers`.

Правильнее для чисел:

```javascript
const scores = [1, 10, 2];
const sortedScores = scores.toSorted((a, b) => a - b);

console.log(sortedScores); // [1, 2, 10]
console.log(scores);       // [1, 10, 2]
```

Разберём по шагам:

1. Компаратор `(a, b) => a - b` задаёт числовую сортировку по возрастанию.
2. `toSorted` возвращает новый массив.
3. Исходный массив `scores` остаётся нетронутым.

`toSorted` (ES2023, Node 20+, Chrome 110+). Если нужна совместимость со старыми средами, используй `[...arr].sort(compareFn)` или `arr.slice().sort(compareFn)`.

Для Java-разработчика это тот же дух, что и `Comparator`, только синтаксис короче.

### 🆕 Иммутабельные методы нового поколения

Если тебе нравится идея «не трогай оригинал, верни новую версию», обрати внимание на свежие методы массива. Это как набор безопасных хирургических инструментов: работаешь с копией, а исходник остаётся цел.

```javascript
const arr = [3, 1, 2];

const sorted = arr.toSorted((a, b) => a - b);
const reversed = arr.toReversed();
const spliced = arr.toSpliced(1, 1, 'new');
const changed = arr.with(0, 'replaced');

console.log(arr);      // [3, 1, 2]
console.log(sorted);   // [1, 2, 3]
console.log(reversed); // [2, 1, 3]
console.log(spliced);  // [3, 'new', 2]
console.log(changed);  // ['replaced', 1, 2]
```

Разберём по шагам:

1. `toSorted()` — иммутабельная версия `sort()`.
2. `toReversed()` — иммутабельная версия `reverse()`.
3. `toSpliced()` — иммутабельная версия `splice()`.
4. `with()` — точечно заменяет элемент по индексу и возвращает новый массив.

Ещё один свежий помощник:

```javascript
const tasks = [
  { title: 'Fix bug', status: 'active' },
  { title: 'Write docs', status: 'done' },
  { title: 'Review PR', status: 'active' }
];

const grouped = Object.groupBy(tasks, task => task.status);
console.log(grouped);
// { active: [...], done: [...] }
```

`Object.groupBy` (ES2024) группирует элементы по правилу. Очень удобно, когда раньше ты бы писал `reduce` руками.

💡 **Поддержка окружения:** это свежие методы (`toSorted` / `toReversed` / `toSpliced` / `with` — ES2023, `Object.groupBy` — ES2024). В современных браузерах и Node они уже есть, но для старых окружений всегда проверяй поддержку или используй полифилы/запасной вариант через `slice`, `spread`, `reduce`.

🔗 **Связь с Angular (Change Detection)**

Выбор структуры данных влияет на производительность Angular:

- **Массивы**: при `OnPush` нужен **новый массив** для обнаружения изменений → `[...old, newItem]`
- **Map/Set**: Angular не отслеживает мутации внутри `Map`/`Set`! Нужно пересоздавать: `new Map(oldMap)`
- **Immutable patterns**: `toSorted()`, `toReversed()`, `spread` — идеальны для `OnPush`
- **trackBy / track**: помогает Angular не пересоздавать DOM при перерисовке списков

### `forEach`, `join`, `at`

#### `forEach`: когда нужен побочный эффект

Если `map` — это преобразование, то `forEach` — обход ради действия. Например, отправить логи, вызвать внешний API, показать уведомление. То есть не построить новый результат, а **что-то сделать** для каждого элемента.

```javascript
const notifications = ['Новый заказ', 'Новый отзыв', 'Новая задача'];

notifications.forEach(message => {
  console.log(`Отправляем уведомление: ${message}`);
});
```

Разберём по шагам:

1. `forEach` проходит по всем сообщениям.
2. Для каждого сообщения выполняется побочный эффект — `console.log`.
3. Новый массив не создаётся.
4. Возвращаемое значение `forEach` — `undefined`.

#### `join`: склеиваем в строку

```javascript
const breadcrumb = ['Каталог', 'Ноутбуки', 'Игровые'];
const path = breadcrumb.join(' > ');

console.log(path); // Каталог > Ноутбуки > Игровые
```

Разберём по шагам:

1. `join` берёт элементы массива.
2. Вставляет между ними разделитель `' > '`.
3. Возвращает одну итоговую строку.

#### `at`: аккуратный доступ по индексу

```javascript
const statuses = ['new', 'paid', 'shipped', 'done'];

console.log(statuses.at(0));  // 'new'
console.log(statuses.at(-1)); // 'done'
```

Разберём по шагам:

1. `at(0)` работает как обычный доступ к первому элементу.
2. `at(-1)` удобно берёт последний элемент.
3. Это особенно приятно, когда не хочется писать `arr[arr.length - 1]`.

### Как быстро выбирать метод

Если меняется форма каждого элемента — `map`. Если оставляешь только часть — `filter`. Если нужен первый найденный — `find`. Если нужен ответ да/нет — `some` или `every`. Если нужен один итог из многих — `reduce`. Если один элемент превращается в несколько и их надо объединить — `flatMap`.

Запомни короткую шпаргалку:

- **покрасить каждую коробку** → `map`;
- **просеять коробки** → `filter`;
- **найти первую коробку с красной наклейкой** → `find`;
- **спросить «есть ли хоть одна?»** → `some`;
- **спросить «все ли?»** → `every`;
- **сложить всё в один итог** → `reduce`.

Если вручную пишешь `forEach` и `push` в новый массив, почти всегда можно выразить мысль чище. Не всегда. Но очень часто.

</details>

### 🎯 5.2. Деструктуризация массивов и объектов (Interview Focus)

#### Почему спрашивают

Тема короткая по синтаксису, но очень показательная по мышлению. Когда человек умеет деструктурировать, он обычно лучше видит форму данных. Он понимает: здесь важен порядок, а здесь имя; здесь можно дать значение по умолчанию, а здесь стоит переименовать поле.

Если смотреть по-бытовому, деструктуризация — это распаковка чемодана: ты сразу раскладываешь нужные вещи по полкам. Для Java-разработчика это похоже на распаковку полей DTO прямо в месте использования. Это не просто сокращалка, а способ честно показать, какие части данных тебе важны.

#### Главная идея одним предложением

- У массива важен **порядок**.
- У объекта важны **имена свойств**.

Это фундамент. Всё остальное — надстройки.

### Прогрессия 1: от старого длинного способа к простой объектной деструктуризации

Начнём с самого житейского. У тебя есть пользователь, и тебе нужны имя и роль.

Старый подробный способ:

```javascript
const user = { id: 7, name: 'Denis', role: 'ADMIN' };

const userName = user.name;
const userRole = user.role;

console.log(userName, userRole);
```

Разберём по шагам:

1. Мы создали объект `user`.
2. Потом по отдельности обратились к `user.name` и `user.role`.
3. С точки зрения результата всё нормально.
4. Но если таких полей 5–6, код начинает разрастаться в однообразный шум.

Теперь тот же смысл через деструктуризацию:

```javascript
const user = { id: 7, name: 'Denis', role: 'ADMIN' };

const { name, role } = user;

console.log(name, role);
```

Разберём по шагам:

1. Фигура слева `{ name, role }` говорит: «возьми из объекта свойства `name` и `role`».
2. Значения этих свойств окажутся в локальных переменных с теми же именами.
3. Порядок свойств в объекте не важен. Важно имя.
4. Код стал короче и честнее показывает, какие поля нам реально нужны.

Стоп. Предсказание: если в объекте сначала идёт `role`, а потом `name`, код всё равно сработает? Да. Потому что у объекта ключевое — имя, а не позиция.

### Прогрессия 2: массивная деструктуризация — когда важен порядок

Теперь возьмём массив цветов. Здесь уже не имена, а позиции.

Старый способ:

```javascript
const colors = ['red', 'green', 'blue'];

const primary = colors[0];
const secondary = colors[1];

console.log(primary, secondary);
```

Разберём по шагам:

1. Мы вручную обращаемся к нулевому и первому индексам.
2. Это нормально, но при нескольких значениях выглядит шумно.

Через деструктуризацию:

```javascript
const colors = ['red', 'green', 'blue'];

const [primary, secondary] = colors;

console.log(primary, secondary);
```

Разберём по шагам:

1. Квадратные скобки слева `[primary, secondary]` означают распаковку массива.
2. Первый элемент массива идёт в `primary`.
3. Второй элемент идёт в `secondary`.
4. Здесь порядок критичен. Если поменять местами переменные, поменяется и результат.

Это как посадка людей в поезд. Первое кресло идёт первому пассажиру, второе — второму. Не по имени, а по месту.

### Прогрессия 3: пропуски, default values и переименование

Вот тут деструктуризация начинает быть по-настоящему полезной.

#### Пример с массивом: пропустить середину и дать дефолт

```javascript
const colors = ['red', 'green'];

const [first, , third = 'black'] = colors;

console.log(first); // 'red'
console.log(third); // 'black'
```

Разберём по шагам:

1. Первый элемент `'red'` попадает в `first`.
2. Вторую позицию мы пропускаем запятой.
3. Третьего элемента в массиве нет, значит значение было бы `undefined`.
4. Поэтому срабатывает default value `'black'`.

#### Пример с объектом: переименование и значение по умолчанию

Старый способ:

```javascript
const employee = { id: 10, name: 'Olga' };

const employeeName = employee.name;
const department = employee.department ?? 'IT';

console.log(employeeName, department);
```

Разберём по шагам:

1. Мы вручную переименовали `name` в `employeeName`.
2. Для `department` отдельно применили запасное значение.
3. Всё это рабоче, но разбросано по двум строкам и двум разным правилам.

Через деструктуризацию:

```javascript
const employee = { id: 10, name: 'Olga' };

const { name: employeeName, department = 'IT' } = employee;

console.log(employeeName, department);
```

Разберём по шагам:

1. `name: employeeName` означает: возьми свойство `name`, но положи его в локальную переменную `employeeName`.
2. `department = 'IT'` означает: если `department` равно `undefined`, подставь `'IT'`.
3. Всё собрано в одной точке.

Важное правило: default value срабатывает только на `undefined`. Не на `null`, не на `0`, не на `false`, не на пустую строку.

### Прогрессия 4: вложенная деструктуризация

Теперь чуть сложнее. Часто данные приходят от API в виде матрёшки. И тебе нужен один кусочек глубоко внутри.

Старый способ:

```javascript
const response = {
  meta: { total: 10 },
  data: {
    user: {
      profile: {
        email: 'dev@example.com'
      }
    }
  }
};

const total = response.meta.total;
const email = response.data.user.profile.email;

console.log(total, email);
```

Разберём по шагам:

1. Мы вручную дошли до `meta.total`.
2. Потом вручную дошли до глубоко вложенного email.
3. Код читается, но путь длинный и повторяющийся.

Через деструктуризацию:

```javascript
const response = {
  meta: { total: 10 },
  data: {
    user: {
      profile: {
        email: 'dev@example.com'
      }
    }
  }
};

const {
  meta: { total },
  data: {
    user: {
      profile: { email }
    }
  }
} = response;

console.log(total, email);
```

Разберём по шагам:

1. Мы говорим: из `meta` вытащи `total`.
2. Из `data.user.profile` вытащи `email`.
3. В итоге локально появляются переменные `total` и `email`.
4. Путь описан прямо в шаблоне распаковки.

Стоп. Но тут важно не увлечься. Если выражение уже похоже на лабиринт, иногда лучше остановиться и вынести промежуточную переменную. Читаемость важнее трюка.

### Прогрессия 5: деструктуризация в параметрах функции

Это очень популярный современный стиль. Функция сразу говорит, какие поля ей нужны.

Старый способ:

```javascript
function printUserCard(user) {
  const name = user.name;
  const role = user.role ?? 'USER';

  return `${name} (${role})`;
}
```

Разберём по шагам:

1. Функция получает весь объект `user`.
2. Внутри отдельно вытаскивает нужные поля.
3. Для `role` ставит дефолт.

Современный способ:

```javascript
function printUserCard({ name, role = 'USER' }) {
  return `${name} (${role})`;
}
```

Разберём по шагам:

1. Прямо в сигнатуре мы распаковали `name` и `role`.
2. Для `role` сразу задали значение по умолчанию.
3. Функция стала короче и честнее: ей реально нужны только эти поля.

Для Java-разработчика это похоже на метод, который принимает DTO, но язык позволяет сразу «расписать» интересующие поля при входе.

## Блок 3: Разворачивать и собирать

### 🎯 5.3. Spread и Rest операторы (Interview Focus)

#### Почему спрашивают

Потому что внешне символ один и тот же — `...`, а смысл у него противоположный. На интервью этого достаточно, чтобы быстро понять, человек реально понимает направление данных или просто запомнил «три точки что-то делают».

Давай сразу очень телесно. **`spread`** — это когда ты высыпаешь яблоки из пакета на стол. Один пакет раскрылся и превратился во много отдельных яблок. **`rest`** — это когда ты после пикника собираешь оставшиеся яблоки обратно в пакет. Много отдельных вещей собираются в один контейнер.

Ещё одна аналогия: `spread` — это веер карт на столе, а `rest` — рука, которая сгребает оставшиеся карты в стопку. В Java-духе `rest` особенно похож на `varargs`.

📚 **Источники:**
- [MDN: Spread syntax](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Operators/Spread_syntax)
- [MDN: Rest parameters](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Functions/rest_parameters)
- [javascript.info: Rest parameters и spread](https://learn.javascript.ru/rest-parameters-spread-operator)

### `spread`: развернуть содержимое

#### Зачем нужен `spread`

Ты хочешь объединить два массива ролей, скопировать массив перед сортировкой, поверхностно склеить конфиги, добавить элемент в начало или конец без мутации. Во всех этих случаях `spread` ведёт себя как «высыпать содержимое контейнера».

#### Пример: объединяем списки ролей

```javascript
const backendRoles = ['ADMIN', 'USER'];
const frontendRoles = ['VIEWER'];
const allRoles = [...backendRoles, ...frontendRoles];

console.log(allRoles); // ['ADMIN', 'USER', 'VIEWER']
```

Разберём по шагам:

1. `...backendRoles` разворачивает массив `backendRoles` в отдельные элементы.
2. `...frontendRoles` делает то же самое.
3. Новый массив `allRoles` собирается из всех этих элементов.
4. Исходные массивы не меняются.

#### Пример: поверхностное копирование объекта

```javascript
const baseConfig = { retries: 3, timeout: 1000 };
const requestConfig = { ...baseConfig, timeout: 3000 };

console.log(requestConfig); // { retries: 3, timeout: 3000 }
```

Разберём по шагам:

1. `{ ...baseConfig }` копирует верхний уровень свойств из `baseConfig`.
2. Затем `timeout: 3000` перезаписывает старое значение `timeout`.
3. Получается новый объект-конфиг.

📚 **Источники:**
- [MDN: Spread syntax](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Operators/Spread_syntax)
- [javascript.info: Rest parameters и spread](https://learn.javascript.ru/rest-parameters-spread-operator)

### Важнейшая ловушка: `spread` делает shallow copy

Вот это место надо прожевать. Поверхностная копия — значит копируются свойства верхнего уровня, но вложенные объекты остаются по тем же ссылкам.

```javascript
const original = { meta: { page: 1 } };
const copy = { ...original };

copy.meta.page = 2;

console.log(original.meta.page); // 2
```

Разберём по шагам:

1. `copy` — новый объект верхнего уровня.
2. Но свойство `meta` внутри него указывает на тот же вложенный объект, что и в `original`.
3. Когда мы меняем `copy.meta.page`, мы меняем общий внутренний объект.
4. Поэтому `original.meta.page` тоже становится `2`.

Стоп. Это один из самых частых багов в фронтенде и Node.js-коде. Человек думает: «Я же сделал копию через `...`». Да, но только неглубокую.

Для глубокого копирования в современном JS используй `structuredClone(obj)` (ES2022, Node 17+, все браузеры). Он рекурсивно копирует вложенные объекты, `Map`, `Set`, `Date`, `RegExp`. Ограничения: не копирует функции и DOM-элементы.

📚 **Источники:**
- [MDN: Spread syntax](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Operators/Spread_syntax)
- [MDN: structuredClone()](https://developer.mozilla.org/en-US/docs/Web/API/Window/structuredClone)
- [javascript.info: Rest parameters и spread](https://learn.javascript.ru/rest-parameters-spread-operator)

### `rest`: собрать остаток

#### Зачем нужен `rest`

Если `spread` высыпает содержимое, то `rest` собирает то, что осталось. Он живёт в двух типичных местах:

1. в параметрах функции — чтобы собрать много аргументов в массив;
2. в деструктуризации — чтобы собрать оставшиеся элементы или свойства.

#### Пример: остаток массива

```javascript
const [first, ...others] = [10, 20, 30, 40];

console.log(first);  // 10
console.log(others); // [20, 30, 40]
```

Разберём по шагам:

1. Первый элемент идёт в `first`.
2. Всё, что осталось, собирается в новый массив `others`.
3. Здесь `...` работает как пылесос для остатка.

#### Пример: убрать пароль из объекта

```javascript
const { password, ...safeUser } = {
  id: 1,
  name: 'Admin',
  password: 'secret'
};

console.log(password); // 'secret'
console.log(safeUser); // { id: 1, name: 'Admin' }
```

Разберём по шагам:

1. Свойство `password` вытаскивается отдельно.
2. Все остальные свойства собираются в объект `safeUser`.
3. Это очень популярный приём перед логированием или отдачей объекта наружу.

#### Rest-параметры и аналогия с Java varargs

```javascript
function sum(...numbers) {
  return numbers.reduce((acc, value) => acc + value, 0);
}

console.log(sum(10, 20, 30)); // 60
```

Разберём по шагам:

1. Все переданные аргументы собираются в массив `numbers`.
2. Внутри функции `numbers` — это настоящий массив, а не псевдомассив.
3. Мы используем знакомый `reduce` из §5.1 и складываем числа.

Для Java-разработчика это почти `int... numbers`, только в JS ты сразу получаешь полноценный массив с его методами.

📚 **Источники:**
- [MDN: Rest parameters](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Functions/rest_parameters)
- [javascript.info: Rest parameters и spread](https://learn.javascript.ru/rest-parameters-spread-operator)

### Как запомнить направление

Если `...` **разваливает одно значение на части**, это `spread`.
Если `...` **собирает много частей в одно значение**, это `rest`.

Повтори ещё раз через предметы:

- пакет яблок высыпали на стол → `spread`;
- оставшиеся яблоки собрали обратно в пакет → `rest`;
- колоду карт веером разложили → `spread`;
- колоду сгребли в стопку → `rest`.

### История типичной ошибки

Разработчик Дима был уверен, что `{ ...obj }` делает полную независимую копию. Он скопировал объект настроек пользователя, поменял вложенное поле темы, а у исходного пользователя тема тоже изменилась.

Причина? Помни: `spread` копирует только верхний уровень. Всё вложенное остаётся общим по ссылке.

#### Типичные вопросы

1. **В чём разница между `spread` и `rest`?** → Один разворачивает, другой собирает.
2. **Делает ли `{ ...obj }` глубокую копию?** → Нет.
3. **Чем `...args` лучше `arguments`?** → Это настоящий массив и более предсказуемый синтаксис.

#### 🚫 Red flags

- «`spread` и `rest` — это одно и то же».
- «Копия через `...` независима на всех уровнях вложенности».
- «`arguments` всегда равен rest-параметрам».

#### ✅ Green flags

- Кандидат объясняет разницу через направление данных.
- Упоминает shallow copy.
- Видит аналогию rest-параметров с varargs.

#### ✅ Проверь себя

**1. Что будет в `others`?**
```javascript
const [first, ...others] = [1, 2, 3];
```
- A) `1`
- B) `[2, 3]`
- C) `[[2, 3]]`
- D) `undefined`

<details><summary>Ответ</summary>B) `[2, 3]`. `rest` собирает оставшиеся элементы в новый массив.</details>

**2. Что выведет код?**
```javascript
const a = { nested: { count: 1 } };
const b = { ...a };
b.nested.count = 5;
console.log(a.nested.count);
```
- A) `1`
- B) `5`
- C) `undefined`
- D) Ошибка

<details><summary>Ответ</summary>B) `5`. Копирование поверхностное, ссылка на вложенный объект общая.</details>

**3. Где `...` работает как `spread`?**
- A) `function log(...args) {}`
- B) `const [first, ...rest] = arr`
- C) `const copy = [...arr]`
- D) `const { a, ...rest } = obj`

<details><summary>Ответ</summary>C) В `const copy = [...arr]` массив разворачивается в элементы.</details>

## Блок 4: Словари и множества

### 5.4. `Map` и `Set` (vs объекты и массивы)

#### Почему это важно

Очень частая ошибка начинающего JS-разработчика: использовать `Object` вообще для всего, потому что «ну это же тоже структура с ключами». Формально да. Но по смыслу нет.

**`Object`** — это в первую очередь модель сущности с полями: `user.name`, `config.timeout`, `task.status`. **`Map`** — это специализированная структура данных «ключ → значение». Это уже словарь, картотека, таблица соответствий, касса хранения. В Java аналог почти прямой: `HashMap`.

**`Set`** — это множество уникальных значений. Список гостей, в который нельзя вписать одного человека дважды. В Java — `HashSet`.

Стоп. Если ты из раза в раз делаешь что-то вроде `if (!arr.includes(x)) arr.push(x)`, задай себе вопрос: «А не нужен ли мне `Set`?» Если ты динамически создаёшь словарь, в котором ключом должен быть объект, функция или число, спроси себя: «А не нужен ли мне `Map`?»

### `Map`: когда объекта уже мало

#### Реальный сценарий: кеш по объекту пользователя

Представь, что у тебя есть объект пользователя, и ты хочешь хранить рядом с ним вычисленные права доступа. Если использовать обычный объект `{}`, ключ будет приведён к строке. Это уже красный флаг.

```javascript
const userCache = new Map();
const userObject = { id: 42, name: 'Nina' };

userCache.set(userObject, { permissions: ['READ'] });

console.log(userCache.get(userObject));
// { permissions: ['READ'] }
```

Разберём по шагам:

1. Создаём `Map`.
2. Берём объект пользователя как ключ.
3. Кладём по этому ключу данные о правах.
4. Получаем значение обратно по тому же объекту-ключу.

Это честная словарная семантика. У объекта `{}` так делать неудобно и часто бессмысленно.

#### Почему `Map` часто лучше `Object`

- ключи могут быть любого типа;
- есть удобные методы `set`, `get`, `has`, `delete`;
- есть свойство `size`;
- порядок вставки сохраняется при итерации;
- семантически это именно словарь, а не сущность с полями.

Справедливости ради: обычный `Object` в современных движках тоже сохраняет порядок для строковых ключей (в порядке вставки). Но для числовых ключей порядок сначала числовой! `Map` всегда гарантирует порядок вставки для любых ключей.

📚 **Источники:**
- [MDN: Map](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Map)
- [javascript.info: Map и Set](https://learn.javascript.ru/map-set)
- [ECMA-262: Map Objects](https://tc39.es/ecma262/#sec-map-objects)

### Когда объект всё ещё лучше

Если у тебя DTO, JSON-структура, конфиг или сущность с известными строковыми полями, объект проще и естественнее.

```javascript
const user = {
  id: 1,
  name: 'Anna',
  role: 'ADMIN'
};
```

Разберём по шагам:

1. Здесь `user` — не словарь «какой угодно ключ → какое угодно значение».
2. Это конкретная доменная сущность с известными полями.
3. Значит, обычный объект уместнее, чем `Map`.

Взрослое правило звучит так: **не менять всё на `Map`, а выбирать `Map`, когда нужен именно словарь**.

📚 **Источники:**
- [MDN: Map](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Map)
- [javascript.info: Map и Set](https://learn.javascript.ru/map-set)
- [MDN: Object.keys()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Object/keys)

### `Set`: когда важна уникальность

#### Реальный сценарий: уникальные теги статьи

У редактора есть статья. К ней добавляют теги: `javascript`, `backend`, `javascript`, `node`. Если хранить это в массиве, появятся дубли. А с `Set` семантика сразу честная: повтор не добавляется.

```javascript
const tags = new Set();
tags.add('javascript');
tags.add('backend');
tags.add('javascript');

tags.add('node');

console.log(tags.size); // 3
```

Разберём по шагам:

1. Создаём пустое множество.
2. Добавляем `javascript`.
3. Добавляем `backend`.
4. Пытаемся ещё раз добавить `javascript`.
5. Размер не увеличивается, потому что значение уже есть.
6. Добавляем `node`.

#### Пример: убрать дубликаты из массива

```javascript
const ids = [1, 2, 2, 3, 3, 3, 4];
const uniqueIds = [...new Set(ids)];

console.log(uniqueIds); // [1, 2, 3, 4]
```

Разберём по шагам:

1. `new Set(ids)` строит множество на основе массива.
2. Повторы автоматически отбрасываются.
3. `...` разворачивает элементы множества в новый массив.
4. В итоге получаем массив уникальных значений.

📚 **Источники:**
- [MDN: Set](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Set)
- [javascript.info: Map и Set](https://learn.javascript.ru/map-set)
- [ECMA-262: Set Objects](https://tc39.es/ecma262/#sec-set-objects)

### `Set` vs массив

Массив нужен, если важны порядок и дубликаты. `Set` нужен, если важна уникальность и быстрая проверка наличия.

Представь лист гостей на вечеринке. Если один человек записан три раза, это странный список. Здесь естественнее `Set`. А если у тебя очередь песен в плейлисте, где повторять трек можно, это уже массив.

📚 **Источники:**
- [MDN: Set](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Set)
- [javascript.info: Map и Set](https://learn.javascript.ru/map-set)
- [MDN: Array.prototype.includes()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Array/includes)

### Маленькая история ошибки

Разработчик месяцами держал уникальные email в массиве и каждый раз делал:

```javascript
if (!emails.includes(email)) {
  emails.push(email);
}
```

Разберём по шагам:

1. Код рабочий.
2. Но семантически это вручную реализованный `Set`.
3. Код шумнее, чем нужно.
4. Намерение спрятано.

С `Set` было бы честнее:

```javascript
const emails = new Set(['a@b.com', 'c@d.com']);
const email = 'e@f.com';

emails.add(email);
```

Разберём по шагам:

1. Мы прямо говорим структуре данных: добавь элемент, если его ещё нет.
2. Меньше шума, больше смысла.

## Блок 5: Слабые коллекции и GC

### 🎯 5.5. `WeakMap` и `WeakSet` (Interview Focus)

#### Почему спрашивают

Потому что это быстрый тест на понимание памяти. Человек может идеально знать синтаксис `map`, `filter`, `spread` и всё ещё не понимать, почему программа вдруг удерживает объекты в памяти дольше, чем надо.

Если говорить совсем по-человечески, тема `WeakMap` — это тема о том, **кто кого держит за рукав**. Для Java-разработчика это прямой мостик к `WeakHashMap`, а в JavaScript чаще всего всплывает в DOM, кешах и метаданных.

#### Напоминание про GC

**Garbage Collection (GC)** — это уборщик памяти. Как робот-пылесос в офисе, который ночью проходит по комнатам и убирает всё, что никому больше не нужно. Если до объекта больше нельзя добраться из живого кода, он становится кандидатом на удаление.

Важно: GC не обязан срабатывать мгновенно. Если ты занулил ссылку, объект **может** быть удалён позже, но не обязательно прямо сейчас.

📚 **Источники:**
- [MDN: WeakMap](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakMap)
- [MDN: WeakSet](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakSet)
- [javascript.info: WeakMap и WeakSet](https://learn.javascript.ru/weakmap-weakset)
- [MDN: WeakRef](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakRef)

### История 1: обычный `Map` как незаметный удерживатель памяти

Представь интерфейс с большим количеством DOM-элементов карточек товара. Ты хочешь хранить рядом с каждым элементом вычисленные подсказки: координаты, состояние hover, служебные данные для анимации.

Ты можешь сделать так:

```javascript
const elementMetadata = new Map();
let cardElement = { id: 'product-card-1' };

elementMetadata.set(cardElement, { hoverCount: 0, tooltip: 'Новинка' });
cardElement = null;
```

Разберём по шагам:

1. Мы создали `Map` и положили туда запись по ключу-объекту `cardElement`.
2. Потом переменная `cardElement` стала `null`.
3. Но объект всё ещё удерживается внутри `Map` как ключ.
4. Значит, для GC это всё ещё достижимый объект через коллекцию.

И вот здесь становится неприятно. Тебе кажется, что карточка уже «ушла со сцены», а память всё ещё занята. Как будто сотрудник уже уволился, а его шкафчик всё ещё закрыт за ним ключом, который лежит у тебя в сейфе.

📚 **Источники:**
- [MDN: WeakMap](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakMap)
- [javascript.info: WeakMap и WeakSet](https://learn.javascript.ru/weakmap-weakset)
- [MDN: Map](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Map)

### История 2: `WeakMap` как служебная записка, а не цепь на ноге

Теперь та же идея через `WeakMap`.

```javascript
const metadata = new WeakMap();
let button = { id: 'save' };

metadata.set(button, { clicked: 0 });
button = null;
```

Разберём по шагам:

1. Мы создали `WeakMap` и сохранили метаданные по ключу-объекту `button`.
2. Потом убрали последнюю сильную ссылку: `button = null`.
3. Если на этот объект больше нигде нет обычных ссылок, GC **может** удалить его.
4. А вместе с ним исчезнет и запись в `WeakMap`.

Вот почему говорят, что `WeakMap` не удерживает ключ в памяти силой. Это как стикер на коробке, который существует, пока сама коробка ещё в комнате. Если коробку унесли и больше нигде не хранят, стикер вместе с ней перестаёт иметь смысл.

📚 **Источники:**
- [MDN: WeakMap](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakMap)
- [javascript.info: WeakMap и WeakSet](https://learn.javascript.ru/weakmap-weakset)

### Реальный сценарий 1: кеш вычислений

Представь тяжёлую функцию, которая получает объект пользователя и вычисляет сложную аналитику только для UI. Ты хочешь кешировать результат, пока объект пользователя жив, но не хочешь навсегда накапливать такие записи.

```javascript
const analyticsCache = new WeakMap();

function getDashboardStats(user) {
  if (analyticsCache.has(user)) {
    return analyticsCache.get(user);
  }

  const stats = {
    score: user.orders.length * 10,
    level: user.orders.length > 5 ? 'gold' : 'silver'
  };

  analyticsCache.set(user, stats);
  return stats;
}
```

Разберём по шагам:

1. Функция получает объект `user`.
2. Если статистика для него уже вычислена, берём её из `WeakMap`.
3. Если нет — считаем и кладём в кеш.
4. Пока объект `user` жив где-то в приложении, кеш может работать.
5. Когда пользовательский объект больше никому не нужен, запись не обязана жить вечно.

Это очень «вкусный» сценарий для `WeakMap`: метаданные и кеш, которые должны жить **не дольше, чем живёт сам объект**.

📚 **Источники:**
- [MDN: WeakMap](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakMap)
- [javascript.info: WeakMap и WeakSet](https://learn.javascript.ru/weakmap-weakset)

### Реальный сценарий 2: DOM-элемент и подсказка

Представь страницу с сотней кнопок. Для каждой нужно хранить служебные данные. В обычном `Map` удалённые кнопки могут продолжать висеть в памяти через ключи. В `WeakMap` метаданные живут не дольше самих кнопок.

```javascript
const tooltipState = new WeakMap();

function attachTooltip(element, text) {
  tooltipState.set(element, {
    text,
    hoverCount: 0
  });
}

function registerHover(element) {
  const state = tooltipState.get(element);
  if (!state) {
    return;
  }

  state.hoverCount += 1;
}
```

Разберём по шагам:

1. Каждому элементу мы привязываем служебное состояние.
2. Состояние не хранится прямо на самом элементе как пользовательское свойство.
3. Оно живёт снаружи, в `WeakMap`.
4. Если элемент исчезнет из DOM и на него больше не останется сильных ссылок, состояние не будет удерживать его в памяти.

Вот это и есть «висцеральное» понимание. `WeakMap` нужен не ради экзотики, а ради того, чтобы служебные данные **не превращались в липкую память**.

📚 **Источники:**
- [MDN: WeakMap](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakMap)
- [javascript.info: WeakMap и WeakSet](https://learn.javascript.ru/weakmap-weakset)

### Ограничения `WeakMap` — и почему это не недоработка

У `WeakMap`:

- `WeakMap` принимает **объекты** и **нерегистрированные Symbol** в качестве ключей (т.е. обычные `Symbol()`, но НЕ `Symbol.for('...')`);
- нет `size`;
- нет нормальной итерации;
- нельзя перечислить всё содержимое как у обычного `Map`.

```javascript
const wm = new WeakMap();

const sym = Symbol('private');
wm.set(sym, 'data');        // ✅ работает (нерегистрированный Symbol)

const obj = {};
wm.set(obj, 'data');        // ✅ работает (объект)

wm.set(Symbol.for('x'), 1); // ❌ TypeError (registered Symbol = глобальный, не GC-able)
wm.set('string', 1);        // ❌ TypeError (примитив)
```

Почему? Потому что содержимое слабой коллекции не должно быть полностью наблюдаемым. Если GC может в любой момент убрать часть записей, то размер и полный перебор становятся концептуально нестабильными.

Это не «забыли дописать API». Это осознанный дизайн.

📚 **Источники:**
- [MDN: WeakMap](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakMap)
- [javascript.info: WeakMap и WeakSet](https://learn.javascript.ru/weakmap-weakset)
- [MDN: WeakRef](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakRef)

### `WeakSet`

`WeakSet` — это слабое множество, которое принимает **объекты** и **нерегистрированные Symbol** (т.е. обычные `Symbol()`, но НЕ `Symbol.for('...')`). Полезно, когда нужно пометить объекты как «уже обработанные», но не держать их в памяти насильно.

```javascript
const visited = new WeakSet();
let node = { type: 'div' };

visited.add(node);
console.log(visited.has(node)); // true
node = null;
```

Разберём по шагам:

1. В `WeakSet` можно добавлять объекты и нерегистрированные `Symbol()`.
2. Мы отметили `node` как посещённый.
3. Пока на `node` есть обычные ссылки, `visited.has(node)` может вернуть `true`.
4. Когда сильные ссылки исчезнут, объект может быть собран GC.

📚 **Источники:**
- [MDN: WeakSet](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakSet)
- [javascript.info: WeakMap и WeakSet](https://learn.javascript.ru/weakmap-weakset)

### Когда слабые коллекции не нужны

Если тебе нужно:

- итерировать по коллекции,
- видеть `size`,
- сериализовать данные,
- показывать их пользователю,
- хранить строки, числа и другие обычные примитивные ключи,

то это почти точно **не** `WeakMap` и не `WeakSet`.

Помнишь §5.4 про выбор структуры по задаче? Слабые коллекции — это очень специальный инструмент. Не «улучшенный `Map`», а другой класс задач.

📚 **Источники:**
- [MDN: WeakMap](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakMap)
- [MDN: WeakSet](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/WeakSet)
- [javascript.info: WeakMap и WeakSet](https://learn.javascript.ru/weakmap-weakset)

### Типичные вопросы

1. **Чем `WeakMap` отличается от `Map`?** → Ключи — объекты и нерегистрированные `Symbol()`, нет итерации и `size`.
2. **Почему ключом не может быть строка?** → Потому что идея слабости завязана на значениях, чей жизненный цикл можно корректно связывать с GC; строки и другие обычные примитивы под это не подходят.
3. **Как это связано с GC?** → `WeakMap` не мешает удалению ключа при отсутствии других ссылок.

#### 🚫 Red flags

- «`WeakMap` просто быстрее `Map`».
- «GC удаляет объект мгновенно в той же строке».
- «В `WeakMap` можно хранить любые примитивы».

#### ✅ Green flags

- Кандидат говорит про weak references.
- Упоминает Java-аналог `WeakHashMap`.
- Понимает, почему слабую коллекцию нельзя нормально перечислять.

<details><summary>Ответ</summary>B) Объект или нерегистрированный Symbol. `WeakMap` принимает объекты и обычные `Symbol()`, но не строки, не числа и не `Symbol.for(...)`, потому что зарегистрированный Symbol глобален и не подходит под идею weak-ключа.</details>

## Блок 6: Итераторы и протокол обхода

### 5.6. Итераторы и `Symbol.iterator`

#### Почему это нужно знать

Даже если ты не будешь каждый день писать свой итератор, понимание `Symbol.iterator` связывает в одну картину `for...of`, `spread`, `Set` и деструктуризацию массива.

**Итератор (`iterator`)** — это закладка в книге, которая помнит, где ты остановился. Или официант, который выносит блюда по одному, а не весь буфет сразу. Он не дублирует коллекцию, а просто умеет выдавать следующий элемент. В Java аналог понятен: `Iterator<T>`.

В JavaScript итератор — это объект с методом `next()`, который возвращает объект `{ value, done }`.

```javascript
const iterator = ['A', 'B'][Symbol.iterator]();

console.log(iterator.next());
console.log(iterator.next());
console.log(iterator.next());
```

Разберём по шагам:

1. `['A', 'B'][Symbol.iterator]()` создаёт итератор массива.
2. Первый `next()` возвращает `{ value: 'A', done: false }`.
3. Второй `next()` возвращает `{ value: 'B', done: false }`.
4. Третий `next()` возвращает `{ value: undefined, done: true }`.

📚 **Источники:**
- [MDN: Iteration protocols](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Iteration_protocols)
- [MDN: Symbol.iterator](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Symbol/iterator)
- [javascript.info: Iterable objects](https://learn.javascript.ru/iterable)

### `Symbol.iterator`

**`Symbol.iterator`** — это специальный служебный ключ, по которому язык понимает: объект умеет выдавать элементы по одному. Если объект реализует `[Symbol.iterator]`, его можно использовать в `for...of`, `spread`, `Array.from()` и деструктуризации.

То есть это как договор: «Если у тебя есть такой метод, я, язык JavaScript, знаю, как тебя обходить».

📚 **Источники:**
- [MDN: Symbol.iterator](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Symbol/iterator)
- [MDN: Iteration protocols](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Iteration_protocols)
- [javascript.info: Iterable objects](https://learn.javascript.ru/iterable)

### Собственный iterable-объект

```javascript
const range = {
  from: 1,
  to: 3,
  [Symbol.iterator]() {
    let current = this.from;
    const last = this.to;

    return {
      next() {
        if (current <= last) {
          return { value: current++, done: false };
        }
        return { value: undefined, done: true };
      }
    };
  }
};

console.log([...range]); // [1, 2, 3]
```

Разберём по шагам:

1. Объект `range` сам по себе не массив.
2. Но у него есть метод `[Symbol.iterator]()`.
3. Этот метод возвращает объект-итератор с `next()`.
4. Пока `current <= last`, возвращаются очередные числа.
5. Когда числа заканчиваются, приходит `{ done: true }`.
6. Благодаря этому `spread` умеет превратить `range` в массив `[1, 2, 3]`.

Это почти ручная реализация `Iterable<Integer>` из Java, только в стиле JavaScript.

📚 **Источники:**
- [MDN: Iteration protocols](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Iteration_protocols)
- [MDN: Symbol.iterator](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Symbol/iterator)
- [javascript.info: Iterable objects](https://learn.javascript.ru/iterable)

### `for...of` vs `for...in`

Это ещё один частый interview-вопрос.

- `for...of` идёт по значениям и работает с iterable-объектами.
- `for...in` идёт по именам свойств.

Для массивов чаще всего нужен именно `for...of`.

`for...in` проходит не только по собственным свойствам, но и по **унаследованным** enumerable-свойствам из прототипа! Поэтому для объектов используй `Object.keys()`/`Object.entries()`, а для массивов — `for...of`.

```javascript
const roles = ['ADMIN', 'USER'];

for (const role of roles) {
  console.log(role);
}
```

Разберём по шагам:

1. `for...of` получает значения массива: `ADMIN`, потом `USER`.
2. Это естественный способ пройти по элементам iterable-структуры.

📚 **Источники:**
- [MDN: for...of](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Statements/for...of)
- [MDN: for...in](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Statements/for...in)
- [MDN: Object.entries()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Object/entries)

### История ошибки

Новичок использовал `for...in` для массива и удивлялся, почему получает `0`, `1`, `2` вместо значений.

Мораль: `for...in` — про ключи/имена свойств, `for...of` — про значения.

### 5.7. Строки: методы, шаблонные литералы, tagged templates

#### Почему этот блок живёт рядом со структурами данных

Потому что строки постоянно пересекаются с массивами и объектами: `split` делает массив, `join` собирает строку обратно, шаблонные литералы строят текст для UI, JSON превращает данные в текст.

**Строка (`string`)** — это цепочка символов, как состав из маленьких вагонов. В Java это напоминает `String`, но в JavaScript строки теснее вплетены в синтаксис языка через шаблонные литералы.

Чаще всего тебе пригодятся `includes`, `startsWith`, `endsWith`, `slice`, `trim`, `split`, `replace`, `toUpperCase`, `toLowerCase`.

```javascript
const title = '  JavaScript for Java  ';

console.log(title.trim());
console.log(title.includes('Java'));
console.log(title.slice(2, 12));
```

Разберём по шагам:

1. `trim()` убирает пробелы по краям строки.
2. `includes('Java')` проверяет, есть ли подстрока `Java`.
3. `slice(2, 12)` берёт кусок строки по индексам.

📚 **Источники:**
- [MDN: String](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/String)

- [w3schools: Template Strings](https://www.w3schools.com/js/js_string_templates.asp)

### `split` и `join`: мост между строкой и массивом

```javascript
const csv = 'ADMIN,USER,VIEWER';
const roles = csv.split(',');
const backToString = roles.join(' | ');

console.log(roles);        // ['ADMIN', 'USER', 'VIEWER']
console.log(backToString); // 'ADMIN | USER | VIEWER'
```

Разберём по шагам:

1. `split(',')` режет строку по запятым.
2. Получается массив ролей.
3. `join(' | ')` склеивает массив обратно в строку с новым разделителем.

Помнишь §5.1? Вот ещё один пример, как данные спокойно перетекают между формами.

- [javascript.info: Методы массивов](https://learn.javascript.ru/array-methods)

### Шаблонные литералы

**Шаблонный литерал (`template literal`)** — это строка в обратных кавычках, как конструктор фразы с готовыми окошками для вставки значений. Это не просто «красивее, чем плюсики». Это заметно легче для чтения.

```javascript
const userName = 'Olga';
const tasksCount = 3;
const message = `${userName}, у тебя ${tasksCount} задачи`;

console.log(message);
```

Разберём по шагам:

1. Внутри `${...}` можно вставлять выражения.
2. Значения подставляются прямо в строку.
3. Не надо вручную склеивать куски через `+`.

Сравни с более шумным вариантом:

```javascript
const message = userName + ', у тебя ' + tasksCount + ' задачи';
```

Разберём по шагам:

1. Результат тот же.
2. Но читать такую строку тяжелее.
3. Шаблонный литерал ближе к естественной фразе.

📚 **Источники:**
- [MDN: Template literals](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Template_literals)
- [w3schools: Template Strings](https://www.w3schools.com/js/js_string_templates.asp)

### Tagged templates

Вот здесь становится интересно. **Tagged template** — это шаблонный литерал, который сначала попадает в функцию-обработчик. Как письмо к редактору перед публикацией: текст ещё можно проверить, подчистить, экранировать, обернуть логикой.

```javascript
function escapeHtml(strings, ...values) {
  return strings.reduce((result, part, index) => {
    const value = values[index] ?? '';
    const escaped = String(value)
      .replaceAll('&', '&amp;')
      .replaceAll('<', '&lt;')
      .replaceAll('>', '&gt;');

    return result + part + escaped;
  }, '');
}

const unsafe = '<script>alert(1)</script>';
const html = escapeHtml`<p>${unsafe}</p>`;

console.log(html); // <p>&lt;script&gt;alert(1)&lt;/script&gt;</p>
```

Разберём по шагам:

1. Функция `escapeHtml` вызывается не обычным способом, а как тег перед шаблонным литералом.
2. `strings` получает куски шаблона между вставками.
3. `values` получает сами подставляемые значения.
4. Внутри мы экранируем опасные символы.
5. Потом собираем строку обратно через `reduce` — привет из §5.1.
6. В результате опасный HTML-код превращается в безопасный текст.

Это полезно для экранирования HTML, построения SQL-шаблонов, маленьких DSL и любых случаев, когда значения нужно обрабатывать отдельно от каркаса строки.

## Блок 8: JSON как формат обмена

### 5.8. JSON: `parse`, `stringify`, `toJSON`, `reviver`

#### Почему важно проговорить это отдельно

Потому что новички часто думают: «JSON — это просто объект JavaScript в текстовом виде». Это опасное упрощение. JSON — **формат данных**, а не слепок всей объектной модели. Для Java-мира это очень похоже на Jackson: сериализация в строку и десериализация обратно в обычные структуры данных.

📚 **Источники:**
- [MDN: JSON](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/JSON)
- [javascript.info: JSON](https://learn.javascript.ru/json)
- [w3schools: JSON](https://www.w3schools.com/js/js_json.asp)

### `JSON.stringify`

**`JSON.stringify`** упаковывает объект в строку.

```javascript
const task = { id: 1, title: 'Deploy', done: false };
const json = JSON.stringify(task);

console.log(json); // {"id":1,"title":"Deploy","done":false}
```

Разберём по шагам:

1. Берём обычный JS-объект `task`.
2. `JSON.stringify` превращает его в текстовую строку JSON.
3. Эта строка уже подходит для передачи по сети, записи в файл или localStorage.

⚠️ `JSON.stringify` молча отбрасывает свойства со значением `undefined`, функции и Symbol-ключи. Это не ошибка — просто JSON как формат не поддерживает эти типы.

```javascript
const obj = { name: 'Alice', greet: () => 'hi', age: undefined };
JSON.stringify(obj); // '{"name":"Alice"}' — greet и age исчезли!
```

📚 **Источники:**
- [MDN: JSON.stringify()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/JSON/stringify)
- [javascript.info: JSON](https://learn.javascript.ru/json)

### `JSON.parse`

**`JSON.parse`** делает обратное — собирает объект из строки.

```javascript
const parsedTask = JSON.parse('{"id":1,"title":"Deploy","done":false}');

console.log(parsedTask.title); // Deploy
```

Разберём по шагам:

1. На входе строка JSON.
2. `JSON.parse` превращает её в обычный JS-объект.
3. После этого можно обращаться к полям как обычно.

📚 **Источники:**
- [MDN: JSON.parse()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/JSON/parse)
- [javascript.info: JSON](https://learn.javascript.ru/json)

### `toJSON`

**`toJSON`** — это личный пресс-секретарь объекта: он решает, в каком виде объект должен уйти в JSON.

```javascript
const user = {
  id: 5,
  password: 'secret',
  toJSON() {
    return { id: this.id };
  }
};

console.log(JSON.stringify(user)); // {"id":5}
```

Разберём по шагам:

1. У объекта есть метод `toJSON`.
2. При сериализации `JSON.stringify` использует результат этого метода.
3. Поэтому пароль не попадает в JSON.

Это очень полезно, когда тебе нужно контролировать внешний вид данных при сохранении или отправке.

📚 **Источники:**
- [MDN: JSON.stringify()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/JSON/stringify)
- [MDN: Date.prototype.toJSON()](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Date/toJSON)
- [javascript.info: JSON](https://learn.javascript.ru/json)

### `reviver`

Вот тонкое, но важное место. После `JSON.parse` даты не становятся объектами `Date` автоматически. Они приходят как строки. Если ты хочешь восстановить тип, нужен `reviver`.

```javascript
const rawEvent = '{"title":"Release","createdAt":"2026-06-21T12:00:00.000Z"}';

const event = JSON.parse(rawEvent, (key, value) => {
  if (key === 'createdAt') {
    return new Date(value);
  }
  return value;
});

console.log(event.createdAt instanceof Date); // true
```

Разберём по шагам:

1. `JSON.parse` идёт по полям разбираемого объекта.
2. Для каждого поля вызывает `reviver`.
3. Если ключ — `createdAt`, мы превращаем строку в `Date`.
4. Все остальные значения возвращаем как есть.
5. В итоге `createdAt` становится настоящим объектом `Date`.

Помнишь §5.1 про методы массивов как конвейер? Здесь похожая идея: данные тоже проходят стадию преобразования, только не на массиве, а на этапе парсинга.

## Блок 9: Время и локали

### 5.9. `Date` и `Intl` (основы)

#### Почему здесь легко наделать ошибок

Потому что время коварно даже для опытных разработчиков: часовые пояса, локали, ISO-строки и месяцы с нуля любят кусать тех, кто решил «да тут же просто дата».

**`Date`** — это обёртка над временной меткой, как будильник, внутри которого на самом деле лежит число миллисекунд от Unix epoch. Для Java-сознания это смесь идей `Instant` и старого `java.util.Date`.

```javascript
const now = new Date();
const fromIso = new Date('2026-06-21T12:00:00Z');
const fromParts = new Date(2026, 5, 21);

console.log(now);
console.log(fromIso);
console.log(fromParts);
```

Разберём по шагам:

1. `new Date()` создаёт дату «сейчас».
2. `new Date('2026-06-21T12:00:00Z')` парсит ISO-строку.
3. `new Date(2026, 5, 21)` создаёт дату из частей.
4. Ловушка: месяц `5` здесь означает июнь, потому что месяцы нумеруются с нуля.

📚 **Источники:**
- [MDN: Date](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Date)
- [MDN: Intl](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Intl)
- [w3schools: JavaScript Dates](https://www.w3schools.com/js/js_dates.asp)

### `Intl`

**`Intl`** — это переводчик локалей. Он знает, как в разных странах показывать даты, числа, валюты и другие форматируемые данные.

```javascript
const formatter = new Intl.DateTimeFormat('ru-RU', {
  dateStyle: 'long',
  timeStyle: 'short'
});

console.log(formatter.format(new Date()));
```

Разберём по шагам:

1. Создаём форматтер для локали `ru-RU`.
2. Говорим, что хотим длинный формат даты и короткий формат времени.
3. Передаём ему объект `Date`.
4. Получаем красивую строку для пользователя.

📚 **Источники:**
- [MDN: Intl](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Intl)
- [MDN: Intl.DateTimeFormat](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Intl/DateTimeFormat)
- [MDN: Intl.NumberFormat](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Intl/NumberFormat)

### Форматирование валюты и чисел

```javascript
const priceFormatter = new Intl.NumberFormat('de-DE', {
  style: 'currency',
  currency: 'EUR'
});

console.log(priceFormatter.format(1234.56));
```

Разберём по шагам:

1. Создаём форматтер чисел для немецкой локали.
2. Указываем, что хотим валютный формат.
3. Передаём число `1234.56`.
4. Получаем строку в нужном локальном представлении.

### Практическое правило

Очень здоровая дисциплина звучит так:

- храни и передавай даты в ISO UTC;
- показывай пользователю через `Intl`.

Это почти та же хорошая привычка, что и в Java/Spring-приложениях: отделять внутреннее представление времени от пользовательского формата отображения.

### История ошибки

Разработчик вручную собирал строку даты через `getMonth() + 1`, `getDate()`, `getHours()` и через пару месяцев команда обнаружила, что для другого региона формат выглядит неестественно.

Мораль: как только нужна локализация и красивое отображение, `Intl` почти всегда надёжнее ручной сборки.

### 🏋️ Мини-упражнения

#### 🏋️ Мини-упражнение 1: Фильтрация и сортировка

У тебя есть массив задач. Получи новый массив только из незавершённых задач, отсортированный по приоритету по убыванию, не мутируя исходный массив.

```javascript
const tasks = [
  { id: 1, title: 'Fix login', done: false, priority: 3 },
  { id: 2, title: 'Write docs', done: true, priority: 1 },
  { id: 3, title: 'Deploy', done: false, priority: 5 }
];
```

Разберём условие по шагам:

1. Нужен **новый** массив, значит мутации исходника лучше избегать.
2. Нужны только незавершённые задачи, значит сначала напрашивается `filter`.
3. Потом нужна сортировка по приоритету по убыванию.
4. Помни из §5.1: если не хочешь мутировать, лучше `toSorted`.

<details><summary>Возможное решение</summary>

```javascript
const result = tasks
  .filter(task => !task.done)
  .toSorted((a, b) => b.priority - a.priority);
```

Разберём по шагам:

1. `filter(task => !task.done)` оставляет только незавершённые задачи.
2. `toSorted((a, b) => b.priority - a.priority)` сортирует их по убыванию приоритета.
3. Исходный массив `tasks` при этом не меняется.
</details>

#### 🏋️ Мини-упражнение 2: Группировка через `reduce`

Сгруппируй пользователей по роли в объект вида `{ ADMIN: [...], USER: [...] }`.

```javascript
const users = [
  { id: 1, name: 'Anna', role: 'ADMIN' },
  { id: 2, name: 'Boris', role: 'USER' },
  { id: 3, name: 'Clara', role: 'USER' }
];
```

Разберём условие по шагам:

1. На выходе нужен один итоговый объект.
2. Внутри этого объекта по ключу роли должен лежать массив пользователей.
3. Это типичная группировка — как раз тот случай, о котором мы говорили в §5.1 про `reduce`.

<details><summary>Возможное решение</summary>

```javascript
const groupedByRole = users.reduce((acc, user) => {
  if (!acc[user.role]) {
    acc[user.role] = [];
  }
  acc[user.role].push(user);
  return acc;
}, {});
```

Разберём по шагам:

1. Стартуем с пустого объекта `{}`.
2. Если для роли ещё нет массива, создаём его.
3. Добавляем пользователя в нужную группу.
4. Возвращаем аккумулятор.
</details>

#### 🏋️ Мини-упражнение 3: Безопасная распаковка

Напиши функцию `printTaskSummary`, которая через деструктуризацию достаёт `title`, `status` и `assignee`, а для `assignee` ставит `'unassigned'` по умолчанию.

Разберём условие по шагам:

1. Функция должна сразу показать, какие поля ей важны.
2. Это удобный случай для деструктуризации прямо в параметрах.
3. Для `assignee` нужно дефолтное значение.

<details><summary>Возможное решение</summary>

```javascript
function printTaskSummary({ title, status, assignee = 'unassigned' }) {
  return `${title} [${status}] -> ${assignee}`;
}
```

Разберём по шагам:

1. Объект распаковывается прямо в параметрах функции.
2. Если `assignee` отсутствует и даёт `undefined`, сработает `'unassigned'`.
3. Возвращаем удобную строку через шаблонный литерал.
</details>

#### 🏋️ Мини-упражнение 4: Восстановление даты из JSON

Распарсь JSON-строку так, чтобы `createdAt` стал объектом `Date`.

```javascript
const raw = '{"title":"Release","createdAt":"2026-06-21T10:00:00.000Z"}';
```

Разберём условие по шагам:

1. После обычного `JSON.parse` дата осталась бы строкой.
2. Значит, нужен `reviver`.
3. Он должен сработать именно на ключе `createdAt`.

<details><summary>Возможное решение</summary>

```javascript
const parsed = JSON.parse(raw, (key, value) => {
  if (key === 'createdAt') {
    return new Date(value);
  }
  return value;
});
```

Разберём по шагам:

1. `JSON.parse` вызывает функцию для каждого ключа.
2. Когда встречаем `createdAt`, превращаем строку в `Date`.
3. Все остальные значения возвращаем без изменений.
</details>
