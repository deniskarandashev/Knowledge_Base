# Модуль 3: Объекты и прототипы

## Для кого этот модуль

Этот модуль для тебя, если после Java и Spring ты смотришь на JavaScript-объекты и чувствуешь странный микс из знакомого и непривычного. С одной стороны, слово «объект» знакомо. С другой — в JavaScript нет того же жёсткого ощущения «сначала класс, потом экземпляр». Здесь часто всё начинается сразу с объекта: вот данные, вот методы, вот связи с другими объектами.

Именно в этот момент многие Java-разработчики слегка тормозят. В Java ты привык, что структура данных заранее закреплена классом. В JavaScript объект похож не столько на экземпляр класса, сколько на живую карточку, к которой можно приколоть новое поле прямо на ходу. Сначала это кажется удобным. Потом — опасным. А потом, когда ты понимаешь прототипы, всё неожиданно собирается в цельную картину.

Если в Модуле 2 ты разобрался с функциями, областями видимости, замыканиями и `this`, то сейчас увидишь, где эти знания начинают сцепляться друг с другом. Объекты в JavaScript — это место, где функции превращаются в методы, ссылки начинают влиять на поведение программы, а прототипы показывают, как язык экономит память и делится поведением.

**Ожидаемый уровень на входе**: ты прошёл Модуль 1 и Модуль 2, знаешь примитивы, функции, `this`, стрелочные функции и базовый синтаксис.

## Каким должен быть результат

После этого модуля ты сможешь:
- уверенно создавать объекты и читать объектные литералы;

## Главная установка модуля

**В JavaScript объект — это самостоятельная сущность, которая хранит свойства и умеет делегировать поиск недостающих свойств своему прототипу.** В Java наследование похоже на чертёж классов. В JavaScript наследование чаще ощущается как живая цепочка объектов, которые одалживают поведение друг у друга.

## Блок 1: Объекты как базовая единица данных

### 3.1. Объекты: создание, свойства, вычисляемые ключи, shorthand

Прежде чем говорить о прототипах, надо очень спокойно поставить фундамент. Объект в JavaScript — это структура «ключ → значение». Но сухое определение не цепляет. Поэтому давай через образы.

Первая метафора: объект — это шкаф с подписанными ящиками. На каждом ящике есть наклейка: `title`, `done`, `priority`. Открываешь нужный ящик — достаёшь значение. Вторая метафора: объект — это анкета, где каждый пункт имеет имя и заполненное содержимое. Третья метафора: объект — это карточка в CRM, которую можно дополнить прямо во время разговора с клиентом. В Java ближайшая аналогия зависит от задачи: иногда это похоже на DTO с полями, иногда — на `Map<String, Object>`, а иногда — на сущность, у которой можно было бы динамически добавить поле во время выполнения, если бы Java так позволяла.

Теперь важный вопрос: **зачем язык вообще делает объекты такими гибкими?** Потому что реальный код редко состоит только из чисел и строк. Тебе нужно собрать данные пользователя, задачи, HTTP-ответа, конфигурации, состояния формы. Если бы всё это приходилось раскладывать по десяткам отдельных переменных, код быстро становился бы нечитаемым. Объект решает эту боль: он позволяет держать связанный набор данных в одной коробке.

Есть и вторая боль, которую объект снимает. Когда ты работаешь с сущностью вроде задачи, тебе хочется обращаться к данным по смыслу, а не по позиции. Сравни: «вторая ячейка массива — это статус» против «`task.status` — это статус». Первый вариант хрупкий. Второй говорит сам за себя. Поэтому объект — это не просто контейнер. Это контейнер с именами, а имена делают код объяснимым.

Наконец, объект — это отличный мост между данными и поведением. Уже в следующем параграфе ты увидишь методы. Но даже сейчас полезно помнить: объект — не только «что я храню», но и «что я умею делать». В Java к этой мысли легко привыкнуть через класс. В JavaScript она начинается ещё раньше — на уровне обычного литерала.

Чаще всего объект создают через **объектный литерал**. Литерал — это готовая запись значения прямо в коде.

```javascript
const task = {
  id: 1,
  title: 'Изучить прототипы',
  done: false
};
```

Разберём код очень медленно.

В первой строке мы объявляем константу `task`. Обрати внимание: `const` запрещает переприсвоить саму переменную, но не делает объект автоматически неизменяемым. Это важное зерно — оно прорастёт в §3.2 и §3.3.

Дальше после `=` начинается объектный литерал: фигурные скобки создают новый объект прямо на месте. Внутри объекта каждое свойство записывается как `ключ: значение`.

Строка `id: 1` создаёт свойство `id` со значением числа `1`. Строка `title: 'Изучить прототипы'` создаёт строковое свойство `title`. Строка `done: false` добавляет булево свойство `done`.

Последняя строка с `};` закрывает литерал и завершает инструкцию. На этом этапе у тебя в памяти появился объект с тремя собственными свойствами.

`id`, `title` и `done` — это **свойства**. Свойство — именованный карман объекта. В Java набор полей фиксирует класс. В JavaScript объект можно менять на лету: добавлять, удалять и заменять свойства.

```javascript
task.priority = 'high';
task['assignedTo'] = 'Denis';
```

Что тут происходит?

В первой строке через точку мы обращаемся к объекту `task` и создаём у него новое свойство `priority`. Если такого свойства раньше не было, оно появляется. Если было — значение заменится.

Во второй строке используется доступ через квадратные скобки. Это другой синтаксис доступа к свойству. Здесь имя свойства записано как строка `'assignedTo'`.

Обрати внимание на важную деталь: и точка, и скобки в итоге работают со свойствами объекта. Разница не в механике хранения, а в форме записи. Точка удобна, когда имя известно заранее и является корректным идентификатором. Скобки нужны, когда имя вычисляется динамически, содержит пробелы или приходит из переменной.

```javascript
const fieldName = 'status';
const taskCard = {
  ['task-' + 1]: 'Первая карточка',
  [fieldName]: 'in_progress'
};
```

Сначала мы создаём переменную `fieldName` и кладём в неё строку `'status'`. Пока это просто значение, не имя свойства.

Дальше начинается новый объект `taskCard`. В первой строке внутри литерала используется конструкция `['task-' + 1]`. JavaScript сначала вычисляет выражение внутри квадратных скобок. Получается строка `'task-1'`. Именно она и становится именем свойства.

Во второй строке `[fieldName]` означает: «возьми значение переменной `fieldName` и используй его как ключ». Поскольку в переменной лежит `'status'`, объект получает свойство `status`.

Такой ключ называется **вычисляемым ключом**. Это как печатать имя на бейджике в последний момент, когда ты уже знаешь, кто пришёл на встречу. В Java похожее чувство бывает при работе с `Map`, когда ключ собирается из переменных.

Есть и короткая запись — **property shorthand**. Если у тебя уже есть переменная `title`, ты можешь написать не `title: title`, а просто `title`.

```javascript
const title = 'Почитать про Object.create';
const priority = 'medium';

const taskInfo = {
  title,
  priority
};
```

Первые две строки создают обычные переменные `title` и `priority`.

Когда ниже ты пишешь объект `taskInfo`, запись `title,` внутри объекта означает то же самое, что `title: title`. JavaScript понимает: имя свойства должно быть `title`, а значение нужно взять из переменной `title`.

То же самое происходит со строкой `priority,`. Это не магия и не сокращение «для красоты». Это способ убрать дублирование и сделать код чище.

Для методов есть похожее сокращение. Вместо `print: function() {}` можно писать `print() {}`.

```javascript
const taskPrinter = {
  prefix: '[TASK]',
  print(task) {
    console.log(this.prefix, task.title);
  }
};
```

В первой строке создаётся объект `taskPrinter`.

Свойство `prefix` хранит строку `'[TASK]'`. Это обычные данные.

Дальше идёт метод `print(task) { ... }`. Обрати внимание: это сокращённая запись метода. Под капотом там всё ещё функция, просто синтаксис короче и приятнее читается.

Внутри метода строка `console.log(this.prefix, task.title);` делает две вещи. Во-первых, через `this.prefix` она берёт значение из самого объекта `taskPrinter`. Во-вторых, через `task.title` она читает свойство объекта, который пришёл аргументом.

Добавим промежуточный пример между «совсем просто» и «уже с методами».

```javascript
const key = 'deadline';
const deadline = '2026-06-30';
const assignee = 'Denis';

const extendedTask = {
  title: 'Подготовить демо',
  [key]: deadline,
  assignee,
  status: 'todo'
};
```

Сначала создаются три переменные: `key`, `deadline`, `assignee`.

В объекте `extendedTask` свойство `title` задано обычным способом. Свойство `[key]` вычисляется: JavaScript подставляет туда строку `'deadline'`. Свойство `assignee` использует shorthand. Свойство `status` снова задано явно.

Получается объект, в котором смешаны три техники: обычный ключ, вычисляемый ключ и shorthand. Это очень жизненный пример: именно так реальные объекты обычно и выглядят.

#### История о типичной ошибке

Представь разработчика Серёжу. Он получил с бэкенда объект пользователя и решил добавить поле `isOnline`. Через неделю другой разработчик увидел этот же объект и стал использовать его как будто `isOnline` всегда есть. Потом пришёл второй источник данных, где этого поля уже не было. Начались странные `undefined`. Ошибка не в том, что JavaScript плохой. Ошибка в том, что гибкость без договорённостей превращает объект в склад, где каждый оставил коробку без подписи.

Поэтому практическое правило такое: объект можно менять динамически, но делать это надо осознанно. Если структура важна, договорись о ней. Иначе в одном месте у тебя `user.name`, в другом `user.fullName`, а в третьем — `user.profile.name`.

Ещё одна связь с будущими темами. Помнишь из §3.2, что переменная с объектом хранит ссылку? Вот здесь это означает: если ты передал объект дальше и кто-то добавил ему новое свойство, это увидят все, у кого была ссылка на тот же объект. Стоп. Это не просто синтаксис. Это поведение программы.

Главная мысль раздела простая: в JavaScript объект — живая структура, а не заранее закрытая коробка. Это даёт гибкость, но требует дисциплины. Чем лучше ты понимаешь свойства сейчас, тем легче пойдут ссылки, копирование и прототипы дальше.

### 🎯 3.2. Ссылки vs значения: примитивы и объекты

> **Почему спрашивают**: если здесь есть путаница, дальше поплывут мутации, клонирование, `this`, React-state и половина вопросов про поведение объектов.

Сейчас будет один из самых важных участков модуля. Если бы я сидел рядом, я бы реально попросил тебя замедлиться. Ссылки — это не «ещё одна тема». Это механизм, который объясняет, почему объект внезапно изменился «сам», почему shallow copy опасен и почему вызов функции может мутировать внешний мир.

Начнём с боли, которую эта тема решает. Когда разработчик говорит: «Я же поменял копию, почему сломался оригинал?», почти всегда проблема именно в ссылках. Когда человек говорит: «Но я же передал объект в функцию, почему она изменила данные снаружи?», снова почти всегда проблема в ссылках. Без этой модели JavaScript кажется капризным. С ней — становится логичным.

Вторая причина понять ссылки — это умение отличать **сам объект** от **способа на него попасть**. Это звучит философски, но на практике всё просто. Дом стоит на улице. Адрес дома — это не сам дом. Можно иметь два листка с одним и тем же адресом. Тогда оба человека придут в один и тот же дом. Вот так и с переменными.

Третья причина особенно важна для Java-разработчика. В Java ты уже встречал ссылочные типы, но там язык сильнее подталкивает к классовой модели. В JavaScript объектов больше, они создаются легче, меняются быстрее, и поэтому путаница с ссылками встречается чаще. Чем раньше ты увидишь это глазами, тем спокойнее будет дальше.

**Примитив** — это простое значение, которое хранится само по себе. Представь наклейку с числом `5`. Скопировал наклейку — получил ещё одну независимую наклейку. **Объект** — составная сущность. Удобная метафора: в переменной лежит не сам дом, а его адрес.

Вот ещё три образа, чтобы это закрепилось.

- Метафора 1: примитив — это стакан воды, который ты реально держишь в руке; объект — это номер камеры хранения, по которому лежит коробка.
- Метафора 2: примитив — это листок с числом; объект — это ключ от шкафчика.
- Метафора 3: примитив — это монета у тебя в кармане; объект — это номер парковочного места, где стоит машина.
- Java-аналогия: `int a = 5;` хранит значение напрямую, а `User user = new User();` хранит ссылку на объект `User`.

```javascript
let a = 10;
let b = a;
b = 20;

console.log(a); // 10
console.log(b); // 20
```

В первой строке создаётся переменная `a` со значением `10`.

Во второй строке `let b = a;` берёт текущее значение `a` и копирует его в `b`. Поскольку это примитив, копируется само число.

Третья строка `b = 20;` не меняет `a`. Она просто кладёт в `b` новое значение.

В конце `console.log(a)` печатает `10`, а `console.log(b)` — `20`. У примитивов копии независимы.

Теперь объект.

```javascript
const task1 = { title: 'Изучить JS' };
const task2 = task1;

task2.title = 'Изучить прототипы';

console.log(task1.title); // 'Изучить прототипы'
```

В первой строке создаётся объект с одним свойством `title`.

Во второй строке `const task2 = task1;` не создаёт новый объект. Вот здесь многие мысленно спотыкаются. Переменная `task2` получает то же значение, что и `task1`. А значением здесь является ссылка на один и тот же объект.

Строка `task2.title = ...` не переприсваивает `task2`. Она идёт по ссылке к общему объекту и меняет его внутреннее свойство `title`.

Поэтому `console.log(task1.title)` тоже видит обновлённое значение. Не было двух домов. Был один дом и два листка с его адресом.

Давай нарисуем это в голове. Представь двух людей, которые держат одну и ту же верёвку, привязанную к коробке. Люди — это переменные `task1` и `task2`. Коробка — сам объект. Если один человек дёргает за верёвку и перекрашивает коробку, второй видит ту же перекрашенную коробку. Это и есть визуальная модель ссылки. Два человека. Одна верёвка. Одна коробка.

На интервью полезно формулировать точно: **JavaScript всегда передаёт аргументы по значению, но для объекта значением является ссылка на объект**. Эту модель часто называют *call by sharing*.

```javascript
function renameTask(task) {
  task.title = 'Новое имя';
}

const myTask = { title: 'Старое имя' };
renameTask(myTask);
console.log(myTask.title); // 'Новое имя'
```

Сначала объявляется функция `renameTask(task)`. Параметр `task` — это локальная переменная внутри функции.

Внутри функции строка `task.title = 'Новое имя';` не перенаправляет параметр на другой объект, а меняет свойство у объекта, на который параметр сейчас ссылается.

Ниже создаётся объект `myTask`.

Когда вызывается `renameTask(myTask)`, в параметр `task` попадает копия значения `myTask`. Но это значение — ссылка на тот же объект.

Поэтому после вызова `console.log(myTask.title)` печатает `'Новое имя'`. Функция получила доступ к тому же самому объекту.

Теперь вторая половина истины. Мутация объекта видна снаружи. Но переприсваивание параметра — нет.

```javascript
function replaceTask(task) {
  task = { title: 'Новый объект' };
}

const myTask = { title: 'Исходный объект' };
replaceTask(myTask);
console.log(myTask.title); // 'Исходный объект'
```

Внутри функции `replaceTask` строка `task = { title: 'Новый объект' };` создаёт новый объект и заставляет локальную переменную `task` ссылаться на него.

Но снаружи переменная `myTask` по-прежнему держит старую ссылку. Между ними нет телепатии.

Поэтому после вызова внешний объект остаётся прежним. Это очень похоже на Java: локальную копию ссылки можно перенаправить, но внешняя переменная от этого не меняет объект.

Добавим промежуточный пример, где одновременно видна и мутация, и переприсваивание.

```javascript
function reconfigure(settings) {
  settings.theme = 'dark';
  settings = { theme: 'light', compact: true };
  return settings;
}

const appSettings = { theme: 'blue', compact: false };
const result = reconfigure(appSettings);

console.log(appSettings); // { theme: 'dark', compact: false }
console.log(result);      // { theme: 'light', compact: true }
```

Внутри функции первая строка `settings.theme = 'dark';` мутирует исходный объект, на который ссылается аргумент.

Потом строка `settings = { ... }` перенаправляет только локальную переменную `settings` на новый объект.

`return settings;` возвращает уже новый объект.

Поэтому `appSettings` снаружи сохранил мутацию `theme: 'dark'`, но не превратился в полностью новый объект. А переменная `result` получила именно тот новый объект, который функция создала позже.

#### История о типичной ошибке

Представь разработчицу Аню. Она пишет функцию `prepareUser(user)`, внутри добавляет `user.isValid = true`, а потом удивляется, почему объект изменился в месте вызова. Она была уверена, что «функция получила копию». И это правда — функция получила копию значения. Но значением была ссылка на тот же объект. Ошибка не в функции, а в неточной модели в голове.

Помнишь §3.1, где мы добавляли свойства объекту на лету? Вот здесь становится ясно, почему это чувствительно. Если два участка кода держат ссылку на один объект, любой из них может добавить новое поле, и второй это увидит. Гибкость объекта и ссылочная модель вместе дают и силу, и риск.

### 🎯 3.3. Копирование объектов: shallow clone vs deep clone

> **Почему спрашивают**: ошибка в копировании часто рождает «призрачные» баги, когда ты меняешь копию, а мутируется оригинал или соседний state.

Если §3.2 был про «почему два имени могут смотреть на один объект», то этот раздел — про следующий естественный вопрос: **как сделать так, чтобы они смотрели не на один и тот же объект?** Именно здесь рождается тема клонирования.

Почему это вообще важно? Потому что в реальном коде ты постоянно хочешь сделать «почти такой же объект, но чуть-чуть другой». Например, обновить state в UI, подготовить DTO для отправки, не трогая исходник, создать новую конфигурацию на основе старой. Если ты копируешь неаккуратно, то получаешь один из самых неприятных типов багов: визуально кажется, что объект новый, а фактически внутри у него торчат старые общие ссылки.

Вторая боль — ложное чувство безопасности. Многие видят spread-оператор `{ ...obj }` и думают: «Ну всё, копия есть». Часто это правда только наполовину. Внешняя коробка новая, но вложенные коробки внутри могут оказаться теми же самыми. И тогда баги становятся особенно коварными: верхний уровень ведёт себя как независимый, а глубже всё внезапно связанное.

Третья причина разобраться в теме — производительность и осознанность. Deep copy звучит как «наверное, всегда лучше», но это не так. Глубокое копирование дороже по памяти и времени. Хороший разработчик не только знает разницу, но и понимает, когда какая глубина копии действительно нужна.

**Клонирование** — это создание копии объекта. Но копия может быть разной глубины. **Shallow copy** — поверхностная копия, как ксерокс только обложки папки. **Deep copy** — глубокая копия, как полное дублирование всей папки с вложенными документами.

Вот ещё три метафоры.

- Метафора 1: shallow copy — ты купил новый рюкзак, но все внутренние карманы в нём почему-то ведут в старый рюкзак. Deep copy — новый рюкзак и новые карманы тоже.
- Метафора 2: shallow copy — новый оглавлениец папки, но сами файлы внутри остались теми же. Deep copy — новая папка и новые файлы.
- Метафора 3: shallow copy — новый стол с теми же выдвижными ящиками; deep copy — новый стол и новые ящики.
- Java-аналогия: copy constructor, который скопировал поля верхнего уровня, но оставил тот же `List<Address>` по ссылке, делает shallow copy.

```javascript
const original = {
  title: 'Sprint planning',
  meta: { author: 'Anna' }
};

const shallow = { ...original };
shallow.meta.author = 'Denis';

console.log(original.meta.author); // 'Denis'
```

Первые три строки создают объект `original`. У него есть строковое поле `title` и вложенный объект `meta`.

Строка `const shallow = { ...original };` создаёт новый внешний объект. На верхнем уровне свойства копируются в новую оболочку.

Но вот ключевой момент: значение свойства `meta` — это объект. Spread не заходит внутрь него рекурсивно. Он просто переносит ссылку на тот же вложенный объект.

Поэтому строка `shallow.meta.author = 'Denis';` меняет общий внутренний объект `meta`.

В результате `console.log(original.meta.author)` печатает `'Denis'`. Внешний контейнер новый, вложенная коробка — старая.

Если представить историю, это выглядит так. У тебя есть проектная папка `original`. Ты сделал копию обложки и списка разделов, но сам файл `meta.docx` внутри оставил тем же физическим файлом. Когда в «копии» кто-то исправил автора, в оригинале тоже всё изменилось, потому что файл был один и тот же.

Для shallow copy обычно используют spread или `Object.assign`.

```javascript
const clone1 = { ...original };
const clone2 = Object.assign({}, original);
```

Первая строка создаёт поверхностную копию через spread.

Вторая строка делает почти то же самое через `Object.assign`. Пустой объект `{}` выступает как новая цель, а `original` — как источник свойств.

Оба варианта хороши для верхнего уровня. Оба не делают deep copy автоматически.

Теперь посмотрим на промежуточный вариант. Не всегда нужен full deep clone всего объекта. Иногда достаточно отдельно скопировать только проблемную ветку.

```javascript
const saferCopy = {
  ...original,
  meta: { ...original.meta }
};

saferCopy.meta.author = 'Olga';
console.log(original.meta.author); // 'Anna'
```

Здесь первая строка снова копирует верхний уровень `original`.

Но дальше свойство `meta` мы явно пересоздаём как новый объект через `{ ...original.meta }`.

Теперь у `saferCopy` свой собственный вложенный объект `meta`, независимый от `original.meta`.

Строка `saferCopy.meta.author = 'Olga';` меняет только копию. Оригинал остаётся нетронутым.

Это отличный практический паттерн: копируй ровно те уровни структуры, которые реально собираешься менять.

🔗 **Связь с Angular (OnPush Change Detection)**

При `OnPush` Angular не запускает проверку компонента автоматически при каждом CD-цикле. Компонент обновляется, когда: (1) изменилась ссылка `@Input`, (2) сработало событие внутри компонента, (3) сработал `AsyncPipe`, (4) вызван `markForCheck()` или `detectChanges()`, (5) изменился signal, используемый в шаблоне. Именно поэтому иммутабельные обновления объектов — не просто «хороший стиль», а практическая необходимость для предсказуемого поведения при OnPush.

```javascript
// ❌ НЕ сработает с OnPush — ссылка та же
this.user.name = 'New Name';

// ✅ Сработает — новый объект, новая ссылка
this.user = { ...this.user, name: 'New Name' };
```

Для **deep copy** в современном JS лучше знать `structuredClone`.

```javascript
const deep = structuredClone(original);
deep.meta.author = 'Olga';

console.log(original.meta.author); // 'Anna'
```

Строка `structuredClone(original)` просит движок создать глубокую структурную копию данных.

В переменной `deep` оказывается новый объект, у которого и верхний уровень, и вложенный `meta` независимы от оригинала.

Поэтому изменение `deep.meta.author` не влияет на `original.meta.author`.

⚠️ **Важное ограничение**: `structuredClone` теряет цепочку прототипов! Если клонировать экземпляр класса, результат будет plain object:

```javascript
class Task {
  constructor(title) { this.title = title; }
  getTitle() { return this.title; }
}

const original = new Task('Learn JS');
const clone = structuredClone(original);

console.log(clone.title);      // 'Learn JS' — данные скопированы
console.log(clone.getTitle);   // undefined — метод потерян!
console.log(clone instanceof Task); // false — это уже не Task
```

Для клонирования экземпляров классов с сохранением прототипа нужен кастомный метод `clone()` или паттерн factory.

И ещё одна аккуратность: `structuredClone` не клонирует функции и DOM-элементы. `RegExp` он клонирует, а вот с `Error`-объектами лучше не обещать одинаковое поведение во всех движках. Для объектов с поведением и особыми инвариантами всё равно нужен кастомный подход.

Старый приём `JSON.parse(JSON.stringify(obj))` полезно знать как исторический костыль, но не как универсальное решение.

```javascript
const user = {
  name: 'Anna',
  createdAt: new Date(),
  skip: undefined
};

const jsonClone = JSON.parse(JSON.stringify(user));
console.log(jsonClone);
```

Первая часть `JSON.stringify(user)` превращает объект в JSON-строку.

Вторая часть `JSON.parse(...)` создаёт из строки новый объект.

Проблема в том, что JSON умеет не всё. `Date` превращается в строку. `undefined` теряется. Функции, `Map`, `Set`, `BigInt` и некоторые другие значения либо исчезают, либо ломают процесс.

Поэтому этот подход можно знать как «исторический трюк», но нельзя держать в голове как надёжный deep clone на все случаи жизни.

### 3.4. Методы объекта и `this` в методах

После свойств, ссылок и копирования логично перейти к поведению. Объект в JavaScript — это не только данные. Он может хранить функции, а функции внутри объекта мы называем **методами**.

Почему это удобно? Потому что данные и операции над ними начинают жить рядом. Если у тебя есть задача, ей естественно уметь `complete()`, `reopen()`, `print()`. Если у тебя есть корзина, ей естественно уметь `addItem()` и `clear()`. Код становится ближе к предметной области, а не распадается на набор несвязанных функций.

Но есть и источник боли. В Java метод жёстко привязан к экземпляру класса, и `this` кажется чем-то стабильным и скучным. В JavaScript `this` зависит от способа вызова. Поэтому многие сначала думают: «Ну метод же внутри объекта, значит он всегда знает хозяина». Нет. И это надо прочувствовать именно на простых примерах, пока ещё не начались колбэки и события.

Ещё одна причина понять методы сейчас — связь с прототипами. Позже ты увидишь, что методы часто живут не в самом объекте, а в его прототипе. Но вызываются они всё равно как будто принадлежат самому объекту. Чтобы это не казалось магией, надо уже сейчас хорошо почувствовать, как работает вызов и откуда берётся `this`.

**Метод** — это функция, лежащая в свойстве объекта. Представь кнопку на панели устройства: кнопка находится в самом устройстве и запускает действие. В Java это привычный метод класса. В JavaScript метод технически остаётся функцией, но вызывается через объект.

```javascript
const task = {
  title: 'Прочитать про дескрипторы',
  done: false,
  complete() {
    this.done = true;
  }
};

task.complete();
console.log(task.done); // true
```

В первых строках создаётся объект `task`.

Свойства `title` и `done` — обычные данные.

Метод `complete()` — это функция, которая живёт в свойстве `complete`.

Внутри метода строка `this.done = true;` говорит: «возьми объект, от имени которого меня вызвали, и поставь у него `done` в `true`».

Дальше строка `task.complete();` вызывает функцию через объект `task`. Именно форма вызова `task.complete()` делает `this` внутри метода равным `task`.

Поэтому `console.log(task.done)` печатает `true`.

Полезная метафора: `this` — это бейдж текущего владельца вызова. Кто подошёл к терминалу и нажал кнопку, того имя и висит на бейдже.

Вот ещё три образа.

- Метафора 1: метод — это кнопка на кофемашине; `this` — конкретная кофемашина, на которой ты нажал кнопку.
- Метафора 2: метод — это сценарий разговора; `this` — человек, который сейчас произносит реплику.
- Метафора 3: метод — это инструмент в ящике; `this` — ящик, из которого его достали и используют.
- Java-аналогия: обычный метод экземпляра, только привязка к экземпляру определяется не местом объявления, а способом вызова.

Теперь посмотрим на больной момент.

```javascript
const user = {
  name: 'Anna',
  sayHello() {
    console.log(`Привет, я ${this.name}`);
  }
};

const fn = user.sayHello;
fn();
```

Сначала объект `user` получает метод `sayHello()`.

Внутри метода используется `this.name`. Пока всё выглядит безопасно.

Но затем строка `const fn = user.sayHello;` берёт саму функцию и кладёт её в отдельную переменную `fn`.

Когда ниже выполняется `fn();`, функция вызывается уже **не как метод объекта**, а как обычная функция. Бейдж владельца потерян.

В strict mode `this` внутри такой функции будет `undefined`. Поэтому попытка читать `this.name` приведёт к проблеме.

Стоп. Это один из тех моментов, где хочется сказать: «Но функция же всё ещё та же самая!» Да, функция та же. Но `this` определяется не тем, где функция лежит, а тем, **как её вызвали**.

Стрелочную функцию как метод стоит использовать осторожно.

```javascript
const broken = {
  name: 'Broken',
  say: () => console.log(this.name)
};
```

Здесь объект `broken` создаётся нормально.

Но свойство `say` получает стрелочную функцию. У стрелки нет собственного `this`.

Поэтому `this.name` внутри стрелки не означает «свойство объекта `broken`». Стрелка возьмёт `this` из внешнего лексического окружения, где она была создана.

Если тебе нужен именно объект-владелец, по умолчанию выбирай обычный метод.

#### История о типичной ошибке

Разработчик Олег сделал объект `modal` с методом `open()`. Потом передал `modal.open` как колбэк в кнопку. При клике метод вызвался, но внутри `this` оказался не `modal`. В консоли — `Cannot read properties of undefined`. Олег полчаса ругал браузер. Потом понял: он передал не вызов объекта, а голую функцию. Это типичный сценарий, и он очень часто случается именно у тех, кто приходит из Java.

Помнишь §3.1, где у нас был метод `print()` внутри объекта? Тогда мы использовали его прямо как `taskPrinter.print(task)`, и всё было хорошо. Здесь мы увидели, что важно не только «где хранится метод», но и «как он вызван». Эта мысль станет особенно полезной, когда в §3.7 мы будем вызывать методы, пришедшие из прототипа.

#### ✅ Проверь себя

**1. Что выведет код?**
```javascript
const account = {
```

### 3.5. Опциональная цепочка `?.` — глубже, чем просто защита от null

На первый взгляд optional chaining выглядит как маленький синтаксический сахар. На практике это один из самых человечных операторов языка. Он спасает от боли, которая раньше приводила к километрам проверок `if`.

Какая именно боль? Представь, что данные приходят не полностью. Пользователь может не иметь профиля. Профиль может не иметь адреса. Плагин может быть не подключён. Массив тегов может отсутствовать. Если ты пишешь прямой доступ вроде `user.profile.address.city`, JavaScript ожидает, что каждая ступенька существует. Если где-то пусто, код падает.

Почему это особенно важно сейчас, в модуле про объекты? Потому что реальный объектный мир почти никогда не бывает идеально заполненным. И когда ты читаешь свойства вложенных объектов, тебе нужно не только знание синтаксиса, но и осторожность пути: существует ли следующая дверь?

Есть ещё одна причина полюбить `?.`: он делает намерение видимым. Старый длинный `if (user && user.profile && user.profile.address)` говорил: «Я боюсь, что чего-то не будет». Оператор `?.` говорит то же самое короче и аккуратнее. Код становится проще читать глазами.

**Опциональная цепочка** — это безопасный проход по вложенной структуре. Представь коридор с дверями. Обычный доступ `user.profile.address.city` — это пробежка вслепую: если одной двери нет, ты врежешься в стену и получишь `TypeError`. `?.` — это датчик на двери: если прохода нет, ты просто останавливаешься и получаешь `undefined`.

```javascript
const city = user?.profile?.address?.city;
```

В этой строке JavaScript идёт слева направо.

Сначала он смотрит на `user`. Если `user` равен `null` или `undefined`, вычисление сразу прекращается и результатом становится `undefined`.

Если `user` существует, движок идёт к `profile`. Затем точно так же проверяет `address`. Затем — `city`.

Обрати внимание: оператор не «лечит» объект и не создаёт недостающие уровни. Он просто безопасно останавливается.

`?.` работает не только со свойствами через точку, но и с индексами и вызовами.

```javascript
const firstTag = task?.tags?.[0];
const result = plugin?.run?.();
```

В первой строке `task?.tags?.[0]` безопасно проходит к массиву `tags`, а потом берёт первый элемент через индекс.

Во второй строке `plugin?.run?.()` делает две проверки: существует ли объект `plugin` и существует ли у него свойство `run`.

Если `plugin` отсутствует, вызова не будет вообще. Если `run` отсутствует, тоже вернётся `undefined` без падения.

Но тут есть тонкий момент. Если `plugin` существует, `run` тоже существует, но это не функция, попытка вызвать `run?.()` всё равно завершится ошибкой на этапе вызова неподходящего значения. Optional chaining проверяет наличие `null`/`undefined`, а не «подходит ли тип под мои ожидания».

`?.` нельзя использовать слева от присваивания. Запись `user?.name = 'Anna'` синтаксически запрещена.

Почему? Потому что оператор создан для безопасного чтения и условного вызова, а не для двусмысленной записи. Если объекта нет, непонятно, куда именно нужно писать.

Для Java можно провести аналогию с длинной цепочкой `if (user != null && user.getProfile() != null ...)`, только JavaScript дал для этого короткий встроенный синтаксис.

#### История о типичной ошибке

Разработчица Лена получила ответ API, где `profile` иногда отсутствовал для новых пользователей. Код `user.profile.avatar.url` начал ронять страницу только на части аккаунтов. Она быстро поставила `user?.profile?.avatar?.url` — и баг исчез. Но потом решила вызвать `user?.profile?.loadAvatar?.()`, не убедившись, что `loadAvatar` действительно функция. В одном случае это оказалось строкой, и ошибка вернулась. Мораль: `?.` защищает от отсутствия звена, но не от неверного типа значения.

Помнишь §3.1 про доступ к свойствам через точку и скобки? Здесь те же операции, только с ремнём безопасности. А если вспомнить §3.2, optional chaining особенно полезен именно потому, что объекты могут быть получены из разных мест и иметь разную форму.

## Блок 2: Прототипы — сердце объектной модели JavaScript

### 🎯 3.6. Прототипы и `__proto__` — Interview Focus

> **Почему спрашивают**: если человек знает только `class`, но не понимает, что под ним живут прототипы, его знания выглядят косметическими. Прототипы — это фундамент JavaScript, а не экзотика.

Сейчас мы подходим к вершине модуля. Если бы это был белыйboard-урок, я бы встал и начал рисовать стрелки между коробками. Почему? Потому что прототипы редко понимают по одному определению. Их нужно увидеть как механизм движения: объект ищет свойство у себя, потом идёт выше, потом ещё выше.

Сначала ответим на вопрос **зачем**. Почему JavaScript вообще не копирует методы в каждый объект отдельно? Представь тысячу задач, у каждой есть метод `complete`. Если бы каждая задача хранила собственную отдельную копию этой функции, это было бы расточительно. Гораздо выгоднее иметь одно общее место, где лежит поведение, и позволить экземплярам пользоваться им совместно.

Вторая причина — гибкость наследования. В Java мышление обычно такое: есть класс-родитель, есть класс-потомок, есть иерархия типов. В JavaScript важнее не «какой чертёж у класса», а «у кого объект занимает недостающие свойства». Это модель делегирования. Объект не обязан быть копией класса. Он может просто говорить: «Если у меня чего-то нет, спроси у этого другого объекта».

Третья причина — понимание того, что `class` в JavaScript не создаёт новый отдельный мир. Он просто даёт более приятный синтаксис поверх прототипов. Поэтому интервьюеры и любят эту тему: она быстро показывает, понимаешь ли ты язык на уровне механики или только на уровне красивого фасада.

**Прототип** — это объект, к которому другой объект обращается за недостающими свойствами. Представь несколько метафор сразу.

- Метафора 1: запасная копия инструкции. У тебя на столе лежит короткая памятка. Если на ней нет нужного шага, ты идёшь к полной инструкции на полке.
- Метафора 2: старший коллега. Джун сначала смотрит в свои заметки. Не нашёл ответа — спрашивает сеньора. Не потому что сеньор «владеет» джуном, а потому что джун умеет делегировать вопрос дальше.
- Метафора 3: библиотечная полка. У тебя есть личный блокнот. Если нужной формулы там нет, ты идёшь к общей полке команды.
- Метафора 4: резервный источник питания. Пока у устройства всё есть локально, оно работает само. Чего не хватает — добирает из внешнего общего источника.
- Java-аналогия: `Task.prototype` похож на общее место, где лежат методы экземпляров, как будто это набор методов класса, но реализованный через объект и ссылки, а не через байткодовый шаблон класса.

У обычного объекта внутри есть скрытая ссылка `[[Prototype]]`. Это внутренний механизм. Исторически к нему часто обращались через `__proto__`, но важно понимать: `__proto__` — это не «магия языка», а legacy-API для доступа к `[[Prototype]]`.

Стоп. Здесь очень легко смешать три похожих слова. Давай разделим их заранее, прежде чем смотреть код.

- `[[Prototype]]` — внутренний скрытый слот объекта. Это не обычное свойство, а часть механики языка.
- `__proto__` — исторический способ посмотреть или поменять `[[Prototype]]` у конкретного объекта. Использовать для объяснения можно, но в коде лучше предпочитать явные API.
- `prototype` — обычное свойство **функции-конструктора**, которое будет использовано как прототип будущих экземпляров, созданных через `new`.

Если сейчас это кажется одинаковым набором слов, не пугайся. Через пару примеров они разведутся по местам.

Начнём с самого простого прототипного делегирования между двумя обычными объектами.

```javascript
const animal = {
  eats: true,
  walk() {
    console.log('walk');
  }
};

const dog = {
  bark() {
    console.log('woof');
  }
};

Object.setPrototypeOf(dog, animal);

console.log(dog.eats); // true
```

Разбираем, будто рисуем стрелки на доске.

Сначала создаётся объект `animal`. У него есть собственное свойство `eats` и собственный метод `walk()`.

Потом создаётся объект `dog` со своим методом `bark()`.

Строка `Object.setPrototypeOf(dog, animal);` делает главное действие: теперь внутренний `[[Prototype]]` объекта `dog` указывает на объект `animal`.

⚠️ `Object.setPrototypeOf` — это «горячая замена» прототипа на лету. Движки оптимизируют объекты под их начальную форму, и смена прототипа ломает эти оптимизации. На практике лучше задавать прототип **сразу при создании** через `Object.create(proto)`, а `setPrototypeOf` использовать только для понимания механики.

Когда ниже выполняется `console.log(dog.eats)`, JavaScript начинает поиск свойства `eats`.

Шаг 1: смотрит в самом `dog`. Свойства `eats` нет.

Шаг 2: идёт по ссылке `[[Prototype]]` к объекту `animal`.

Шаг 3: в `animal` свойство `eats` найдено. Значит, результат — `true`.

Вот этот маршрут нужно уметь проговаривать вслух: **JS looks here → not found → looks here → found**. Если ты можешь так объяснить поиск свойства, ты уже звучишь сильно на интервью.

Добавим ещё пример и снова проговорим lookup пошагово.

```javascript
console.log(dog.walk);
dog.walk();
```

На первой строке JavaScript ищет свойство `walk`.

Сначала он проверяет `dog`. Не находит.

Потом поднимается к `animal`. Находит функцию `walk`.

На второй строке эта функция вызывается как `dog.walk()`. И вот здесь тонкость: хотя метод физически хранится в `animal`, `this` внутри него будет связан с `dog`, потому что вызов происходит от имени `dog`. Помнишь §3.4? Тот же принцип, только метод приехал через прототип.

Теперь главный interview-узел: `prototype` и `__proto__` — не одно и то же. Посмотрим на функцию-конструктор.

```javascript
function Task(title) {
  this.title = title;
}
```

Здесь `Task` — обычная функция.

Но у функций-конструкторов есть особое свойство `prototype`. Это обычный объект, который будет назначен как прототип экземплярам, созданным через `new Task()`.

Пока мы ещё не создали экземпляр, полезно уже мысленно отделить: `Task.prototype` — это свойство самой функции `Task`, а не внутренний прототип будущего объекта `task`.

```javascript
Task.prototype.print = function() {
  console.log(this.title);
};

const task = new Task('Learn JS');
console.log(task.__proto__ === Task.prototype); // true
```

Строка `Task.prototype.print = function() { ... }` кладёт метод `print` в объект `Task.prototype`. Не в каждый будущий экземпляр отдельно, а в общую библиотеку методов.

Строка `const task = new Task('Learn JS');` делает несколько действий сразу. Сейчас прямо разложим их в белыйboard-режиме.

1. JavaScript создаёт новый пустой объект.
2. Назначает его внутренний `[[Prototype]]` равным `Task.prototype`.
3. Вызывает функцию `Task` с `this`, указывающим на новый объект.
4. Записывает в объект свойство `title`.
5. Возвращает этот новый объект.

Строка `console.log(task.__proto__ === Task.prototype)` проверяет, что прототип конкретного экземпляра действительно указывает на объект `Task.prototype`.

Вот здесь нужно запомнить тройку, почти как мантру:

- `Task.prototype` — свойство функции-конструктора;
- `task.__proto__` — прототип конкретного объекта;
- обычно `task.__proto__ === Task.prototype`.

Стоп! Перечитай. Очень многие путают `prototype` у функции и `__proto__` у объекта. Это разные вещи, просто они часто связаны между собой равенством.

Теперь нарисуем белыйboard-сценарий поиска метода `task.print()`.

1. JS видит вызов `task.print()`.
2. Сначала смотрит: есть ли у `task` собственное свойство `print`?
3. Не нашёл.
4. Идёт в `task.__proto__`.
5. `task.__proto__` равен `Task.prototype`.
6. В `Task.prototype` свойство `print` найдено.
7. Функция вызывается с `this = task`.

Вот это и есть делегирование в чистом виде. Экземпляр не хранит метод у себя, но умеет занять его у прототипа.

Современный код предпочитает `Object.getPrototypeOf(obj)` и `Object.setPrototypeOf(obj, proto)` вместо `__proto__`.

```javascript
console.log(Object.getPrototypeOf(task) === Task.prototype); // true
```

Здесь `Object.getPrototypeOf(task)` явно спрашивает движок: «Какой фактический прототип у объекта `task`?».

Сравнение с `Task.prototype` показывает `true`, потому что экземпляр действительно связан именно с этим объектом прототипа.

Этот API важен не только потому, что он современнее. Он ещё и честнее читается в коде: ты не как будто лезешь в скрытую кишку объекта, а явно спрашиваешь про его прототип.

Встроенные типы работают так же. Массив не хранит у себя копию `push` и `map`. Он находит их через `Array.prototype`.

```javascript
const items = [1, 2, 3];
console.log(Object.getPrototypeOf(items) === Array.prototype); // true
console.log(Object.getPrototypeOf(Array.prototype) === Object.prototype); // true
```

Первая строка создаёт массив `items`.

Во второй строке мы проверяем, что прототип конкретного массива — это `Array.prototype`.

В третьей строке идём ещё на уровень выше и видим, что сам `Array.prototype` наследуется от `Object.prototype`.

Это важный aha-момент: prototype chain есть не только у твоих объектов, но и у встроенных структур языка.

#### Типичная ошибка новичка

Разработчик видит `instance.prototype` и думает, что так можно добраться до прототипа объекта. Неа. У обычного объекта `instance` свойства `prototype` обычно нет. `prototype` — это в первую очередь свойство функций-конструкторов. У экземпляра есть фактический прототип, который читается через `Object.getPrototypeOf(instance)` или исторически через `instance.__proto__`.

Помнишь §3.4 про метод и `this`? Вот тут связь особенно красивая: метод может лежать в прототипе, но `this` при вызове всё равно будет указывать на конкретный объект, который вызов сделал. Поэтому `prototype` экономит память, но не мешает методу работать с данными экземпляра.

И ещё одна interview-мысль. Встроенные прототипы лучше не патчить без очень серьёзной причины. Добавить `Array.prototype.first` технически можно, но это как переписать общее поведение `ArrayList` для всего приложения. Может неожиданно повлиять на другие части кода и на сторонние библиотеки.

#### Типичные вопросы

- Что такое прототип?
- Чем `prototype` отличается от `__proto__`?
- Где массив берёт `map`?
- Почему `__proto__` считают legacy?

#### Red flags

- «Прототип — это просто класс».
- «У каждого объекта есть полезное свойство `prototype`».
- Полная путаница между `Task.prototype` и `task.__proto__`.

#### Green flags

- Человек различает `[[Prototype]]`, `__proto__` и `prototype`.
- Может пошагово описать lookup свойства.
- Помнит про встроенные прототипы вроде `Array.prototype`.

### 🎯 3.7. Прототипная цепочка наследования

> **Почему спрашивают**: interviewer проверяет, можешь ли ты мысленно выполнить поиск свойства и объяснить shadowing, делегирование и конец цепочки.

Если §3.6 отвечал на вопрос «что такое прототип?», то здесь мы отвечаем на вопрос «как именно работает поиск свойства по цепочке?». Это любимая зона интервьюеров, потому что тут мгновенно видно, понимает ли человек механику или просто знает пару слов.

Зачем тебе это в реальной жизни? Во-первых, чтобы не удивляться, откуда у объекта взялся метод, которого в нём самом нет. Во-вторых, чтобы понимать shadowing: почему локальное свойство перекрывает свойство прототипа. В-третьих, чтобы не путать место хранения метода с объектом, на котором этот метод работает.

И ещё одна важная причина: прототипная цепочка — это не отдельная экзотическая тема. Она соединяет всё, что мы уже проходили. Ссылки из §3.2 объясняют, как объекты связаны стрелками. Методы и `this` из §3.4 объясняют, как работает вызов найденной функции. Свойства из §3.1 объясняют, что именно ищет движок на каждом шаге.

**Прототипная цепочка** — это последовательность объектов, по которой JavaScript поднимается при поиске свойства.

```javascript
const baseTask = {
  type: 'task',
  printType() {
    console.log(this.type);
  }
};

const priorityTask = Object.create(baseTask);
priorityTask.priority = 'high';

console.log(priorityTask.priority); // 'high'
console.log(priorityTask.type);     // 'task'
priorityTask.printType();           // 'task'
```

Разберём этот пример по шагам, как будто рисуем на доске три коробки и стрелки между ними.

Сначала создаётся объект `baseTask`. У него есть собственное свойство `type` и метод `printType()`.

Потом строка `const priorityTask = Object.create(baseTask);` создаёт новый объект, у которого `baseTask` становится прототипом. Важно: свойства `type` и `printType` не копируются внутрь `priorityTask`. Он просто получает ссылку на `baseTask` как на запасной слой.

Строка `priorityTask.priority = 'high';` добавляет собственное свойство `priority` уже на сам `priorityTask`.

Теперь первая проверка: `console.log(priorityTask.priority)`. JavaScript смотрит в сам `priorityTask`, находит `priority` сразу и печатает `'high'`.

Вторая проверка: `console.log(priorityTask.type)`. Шаг 1 — поиск в `priorityTask`: нет свойства `type`. Шаг 2 — переход к прототипу `baseTask`: свойство найдено, результат `'task'`.

Третья проверка: `priorityTask.printType()`. Сначала ищется свойство `printType`: в `priorityTask` нет, в `baseTask` есть. Потом найденная функция вызывается **как метод `priorityTask`**, поэтому `this` внутри метода указывает на `priorityTask`.

Вот тут важный момент. Внутри `printType()` написано `console.log(this.type)`. Поиск `this.type` начинается уже от `priorityTask`, потому что `this = priorityTask`. У самого `priorityTask` свойства `type` нет, поэтому поиск поднимается к `baseTask` и находит `'task'`.

Давай прямо проговорим lookup словами интервьюера:

- `priorityTask.printType()` → JS ищет `printType` в `priorityTask` → не находит → идёт в `baseTask` → находит функцию.
- Вызывает её с `this = priorityTask`.
- Внутри функции ищет `this.type` → смотрит в `priorityTask` → не находит → идёт в `baseTask` → находит `'task'`.

Если у дочернего объекта есть свойство с тем же именем, что и у прототипа, происходит **shadowing** — затенение.

```javascript
const animal = { sound: 'generic' };
const cat = Object.create(animal);
cat.sound = 'meow';

console.log(cat.sound);    // 'meow'
console.log(animal.sound); // 'generic'
```

В первой строке создаётся прототип `animal` со свойством `sound`.

Во второй строке `cat` получает `animal` как прототип.

Третья строка `cat.sound = 'meow';` создаёт **собственное** свойство `sound` у `cat`.

Теперь при чтении `cat.sound` JavaScript находит собственное свойство сразу и даже не идёт к прототипу.

Но `animal.sound` при этом остаётся `'generic'`. Мы не изменили прототип. Мы просто положили поверх него своё локальное значение с тем же именем.

Это похоже на ситуацию, когда в общей библиотеке на полке лежит справочник `generic`, а ты кладёшь на свой стол личную заметку `meow`. Когда ищешь на своём столе, до полки уже не доходишь.

Добавим более детальный whiteboard-пример с тремя звеньями.

```javascript
const root = { a: 1 };
const middle = Object.create(root);
middle.b = 2;
const leaf = Object.create(middle);
leaf.c = 3;

console.log(leaf.a); // 1
```

Сначала `root` получает свойство `a`.

Потом `middle` создаётся с прототипом `root` и получает собственное свойство `b`.

Потом `leaf` создаётся с прототипом `middle` и получает собственное свойство `c`.

Когда выполняется `leaf.a`, поиск идёт так:

1. JS смотрит в `leaf` → свойства `a` нет.
2. Идёт в прототип `middle` → свойства `a` нет.
3. Идёт в прототип `root` → свойство `a` найдено.
4. Возвращает `1`.

Вот это и есть prototype chain: не один прыжок, а последовательный подъём по лестнице.

#### Своё свойство или унаследованное? Object.hasOwn()

Когда ты ищешь свойство через `obj.prop`, JavaScript идёт по всей цепочке прототипов. Но иногда нужно проверить: это свойство принадлежит ИМЕННО этому объекту, или оно пришло из прототипа?

```javascript
const animal = { eats: true };
const rabbit = Object.create(animal);
rabbit.jumps = true;

console.log('jumps' in rabbit);         // true — своё
console.log('eats' in rabbit);          // true — унаследованное
console.log(Object.hasOwn(rabbit, 'jumps')); // true
console.log(Object.hasOwn(rabbit, 'eats'));  // false — не своё!
```

`Object.hasOwn(obj, key)` (ES2022) — современная замена `obj.hasOwnProperty(key)`. Используй её при итерации, когда нужны только собственные свойства.

Для Java-разработчика важно держать в голове разницу. В Java наследование завязано на классы как blueprint. В JavaScript объект не обязан рождаться из класса в привычном смысле. Он может просто делегировать чтение свойств другому объекту.

Цепочка для обычных объектов заканчивается на `Object.prototype`, а дальше стоит `null` — знак «идти больше некуда».

```javascript
console.log(Object.getPrototypeOf(Object.prototype)); // null
```

Здесь `Object.getPrototypeOf(Object.prototype)` возвращает `null`. Это финал маршрута. Если свойство не найдено даже там, обычное чтение даст `undefined`.

Обрати внимание: отсутствие свойства и ошибка — не одно и то же. Ошибка обычно случается, когда ты пытаешься продолжить чтение у `undefined` или вызвать несуществующее как функцию. А просто `obj.missing` вернёт `undefined` после полного прохода по цепочке.

Ещё практический момент: часто менять прототип существующего объекта — плохая идея. Для движков это неудобно оптимизировать. Лучше создавать объект сразу с правильным прототипом через `Object.create` или конструктор.

#### История о типичной ошибке

Разработчик Паша увидел, что `child.role = 'admin'` не изменило `parent.role`, и решил, что в JavaScript «наследование копирует свойства». Нет. У `child` просто появилось собственное свойство `role`, которое затенило прототипное. Через неделю он пытался «обновить базовое значение для всех детей», меняя свойство только у одного потомка, и удивлялся, почему остальные не изменились. Shadowing — один из самых частых скрытых поводов для путаницы.

Помнишь §3.6, где мы различали `Task.prototype` и `task.__proto__`? Здесь применяется та же идея, только шире: цепочка — это последовательность переходов от конкретного объекта к его прототипу, потом ещё выше. А если вспомнить §3.4, метод из прототипа при вызове вовсе не становится методом прототипа по `this`; он работает с тем объектом, который его вызвал.

### 3.8. `Object.create` и `Object.getPrototypeOf`

После всех разговоров про прототипы очень полезно потрогать инструменты руками. `Object.create` и `Object.getPrototypeOf` — это два честных, прямых API, которые помогают работать с прототипной моделью без лишней дымовой завесы.

Почему это важно? Потому что до этого момента ты мог воспринимать прототип как что-то полумистическое, что «само как-то появляется» у экземпляра. `Object.create` убирает магию: вот я явно создаю объект, вот явно задаю ему прототип. А `Object.getPrototypeOf` позволяет не гадать, а спрашивать движок напрямую.

Вторая причина — обучение. Через `class` и `new` люди часто понимают поверхность, но не механизм. Через `Object.create` механизм становится почти физически осязаемым: есть объект методов, есть объект данных, между ними натянута прототипная стрелка.

Третья причина — практическая. Иногда тебе действительно нужен объект без `class`, без конструктора, просто с правильным делегированием. Или наоборот, нужен «чистый словарь» без унаследованных служебных свойств. Для этого и существуют эти инструменты.

`Object.create` создаёт новый объект с явно указанным прототипом. Представь, что ты сразу назначаешь новому сотруднику наставника. Собственных свойств у новичка пока мало, но он уже знает, к кому идти за недостающим поведением.

```javascript
const taskMethods = {
  complete() {
    this.done = true;
  },
  reopen() {
    this.done = false;
  }
};

const task = Object.create(taskMethods);
task.title = 'Сделать ревью';
task.done = false;

task.complete();
console.log(task.done); // true
```

Сначала создаётся объект `taskMethods`. В нём лежат два метода: `complete()` и `reopen()`.

Внутри `complete()` строка `this.done = true;` будет менять свойство объекта, который вызвал метод. То же зеркально делает `reopen()`.

Строка `const task = Object.create(taskMethods);` создаёт новый объект `task`, у которого `taskMethods` становится прототипом.

Дальше мы добавляем в сам `task` собственные данные: `title` и `done`.

Когда вызывается `task.complete();`, JavaScript сначала ищет `complete` в `task`, не находит, идёт в `taskMethods`, находит метод и вызывает его с `this = task`.

Поэтому `console.log(task.done)` печатает `true`.

Это очень наглядный способ почувствовать прототипную модель без `class` и даже без `new`. Поведение лежит отдельно, данные — отдельно, а объект связывает их через прототип.

`Object.getPrototypeOf(obj)` — это явный вопрос: «У кого этот объект одалживает недостающие свойства?»

```javascript
console.log(Object.getPrototypeOf(task) === taskMethods); // true
```

Функция `Object.getPrototypeOf(task)` достаёт фактический прототип объекта `task`.

Сравнение с `taskMethods` даёт `true`, потому что именно этот объект мы назначили в `Object.create`.

Если бы я сидел рядом, я бы сказал: «Заметь, как мало магии осталось. Мы явно создали объект, явно привязали прототип, явно прочитали его обратно».

Связь с §3.6 и §3.7 прямая. Там мы говорили о прототипах теоретически и рисовали цепочки. Здесь ты руками создаёшь эту цепочку и проверяешь её через API. Это важный шаг от «понял на словах» к «могу использовать без страха».

## Блок 3: Свойства под микроскопом

### 🎯 3.9. Свойства объекта: `enumerable`, `configurable`, `writable`

До этого момента мы вели себя так, будто свойство — это просто «имя и значение». На практике у свойства есть ещё и правила поведения. Можно ли его изменить? Будет ли оно видно при перечислении? Можно ли его удалить или перенастроить? Именно эти правила и описываются дескриптором свойства.

Почему это важно? Потому что иногда тебе нужно не просто хранить данные, а задавать режим доступа к ним. Например, сделать поле read-only, спрятать техническое свойство из `Object.keys`, запретить удаление или подготовить вычисляемое свойство через getter/setter.

Вторая причина — понимание странных случаев в чужом коде. Если свойство читается через `obj.x`, но не появляется в `Object.keys(obj)`, это не обязательно магия. Возможно, оно просто `enumerable: false`. Без знания дескрипторов такие вещи кажутся необъяснимыми.

Третья причина — точность мышления. JavaScript-объект — это не только набор значений, но и набор правил. А дескриптор — это способ эти правила сделать явными.

**Дескриптор свойства** — это паспорт свойства. Если свойство — дверь, то дескриптор говорит, можно ли её перекрасить, снять с петель и увидит ли её посетитель на плане здания.

Ещё три образа.

- Метафора 1: это пропуск на территорию. `writable` — можно ли менять запись, `enumerable` — видно ли тебя в списке гостей, `configurable` — можно ли перепечатать сам пропуск.
- Метафора 2: это карточка книги в библиотеке. Видна ли книга в каталоге, можно ли переписать описание, можно ли убрать её из фонда.
- Java-аналогия: смесь из `final`, модификаторов доступа, reflection-настроек и поведения коллекций, только в JavaScript всё это можно регулировать на уровне свойства во время выполнения.

У обычного свойства есть `value`, `writable`, `enumerable`, `configurable`.

- `value` — само значение.
- `writable` — можно ли заменить значение через обычное присваивание.
- `enumerable` — будет ли свойство видно в перечислении (`Object.keys`, `for...in`).
- `configurable` — можно ли удалить свойство и менять его дескриптор.

```javascript
const task = {};
Object.defineProperty(task, 'id', {
  value: 1,
  writable: false,
  enumerable: true,
  configurable: false
});

task.id = 2;
console.log(task.id); // 1
```

⚠️ **Ловушка для новичка**: если ты создаёшь свойство через `defineProperty` и не указал `writable` и `configurable`, они будут `false` по умолчанию! Это отличается от обычного присваивания `obj.prop = value`, где все дескрипторы по умолчанию `true`. Всегда указывай все флаги явно.

Первая строка создаёт пустой объект `task`.

Строка `Object.defineProperty(task, 'id', { ... })` добавляет свойство `id` не обычным присваиванием, а через явное описание дескриптора.

`value: 1` задаёт исходное значение.

`writable: false` запрещает менять его через `task.id = ...`.

`enumerable: true` говорит, что свойство будет видно в перечислении.

`configurable: false` делает свойство жёстким: удалить или перенастроить его будет нельзя или очень ограниченно.

Дальше строка `task.id = 2;` пытается изменить значение. В нестрогом режиме изменение просто не сработает, а в strict mode было бы исключение.

Поэтому `console.log(task.id)` печатает `1`.

Если `enumerable: false`, свойство можно читать по имени, но оно не появится в `Object.keys`.

```javascript
const user = {};
Object.defineProperty(user, 'secret', {
  value: 'token',
  enumerable: false
});

console.log(user.secret);       // 'token'
console.log(Object.keys(user)); // []
```

Сначала создаётся пустой объект `user`.

Потом через `Object.defineProperty` добавляется свойство `secret`.

`value: 'token'` задаёт содержимое.

`enumerable: false` скрывает свойство из обычного перечисления.

Поэтому `user.secret` читается нормально, но `Object.keys(user)` возвращает пустой массив.

Стоп. Это важная мысль: **non-enumerable не означает private**. Свойство не видно в списке ключей, но по имени его всё ещё можно читать.

`configurable: false` делает свойство жёстким: его нельзя удалить и трудно перенастроить.

```javascript
delete task.id;
console.log(task.id); // 1
```

Строка `delete task.id` пытается удалить свойство.

Но если для него задано `configurable: false`, удаление не происходит.

В strict mode попытка удалить `non-configurable` свойство выбросит `TypeError`, а не просто молча вернёт `false`.

Поэтому чтение `task.id` по-прежнему вернёт `1`.

Для чтения паспорта свойства есть `Object.getOwnPropertyDescriptor`.

```javascript
const descriptor = Object.getOwnPropertyDescriptor(user, 'secret');
console.log(descriptor);
```

Первая строка достаёт полный дескриптор собственного свойства `secret` у объекта `user`.

Во второй строке мы печатаем этот дескриптор и можем увидеть флаги вроде `enumerable`.

Это очень полезный диагностический инструмент: когда объект ведёт себя странно, можно не гадать, а посмотреть на фактические правила свойства.

### 3.10. Геттеры и сеттеры

Мы дошли до интересного места, где свойство перестаёт быть просто контейнером значения и начинает вести себя как маленький интерфейс. Геттеры и сеттеры позволяют сделать чтение и запись свойства умнее.

Зачем это вообще нужно? Первая причина — вычисляемые значения. Иногда тебе не хочется хранить `fullName` отдельно, потому что оно и так получается из `firstName` и `lastName`. Хранить дубликат опасно: значения могут разъехаться.

Вторая причина — валидация и нормализация входа. Хочешь запретить отрицательный возраст? Хочешь автоматически обрезать пробелы? Сеттер даёт тебе точку контроля. Причём снаружи API выглядит как обычное свойство, что удобно для потребителя объекта.

Третья причина — инкапсуляция намерения. Да, в JavaScript нет той же жёсткой классической приватности в старом объектном стиле, что в Java. Но можно хотя бы отделить «внутреннее поле» от «внешнего способа доступа».

**Геттер** — это свойство, которое при чтении выполняет функцию. **Сеттер** — это свойство, которое при записи выполняет функцию. Метафора простая: вместо обычной двери у тебя умная дверь с датчиком, которая реагирует на вход и выход.

Ещё несколько образов.

- Метафора 1: геттер — это информационное табло, которое показывает вычисленный результат; сеттер — сотрудник на стойке, который проверяет, что ты подал корректную форму.
- Метафора 2: геттер — это кофемашина, выдающая напиток по запросу; сеттер — фильтр качества перед загрузкой ингредиентов.
- Метафора 3: геттер — это окно, которое показывает пейзаж, не храня его внутри; сеттер — турникет, пропускающий только валидные данные.
- Java-аналогия: пара `getX()/setX()`, только синтаксис использования снаружи выглядит как обращение к полю.

```javascript
const user = {
  firstName: 'Anna',
  lastName: 'Ivanova',
  get fullName() {
    return `${this.firstName} ${this.lastName}`;
  },
  set fullName(value) {
    const [first, last] = value.split(' ');
    this.firstName = first;
    this.lastName = last;
  }
};
```

Сначала создаётся объект `user` с двумя обычными полями: `firstName` и `lastName`.

Дальше идёт `get fullName() { ... }`. Это геттер. Когда кто-то читает `user.fullName`, JavaScript вызовет эту функцию.

Внутри геттера строка ``return `${this.firstName} ${this.lastName}`;`` собирает строку из двух полей объекта.

Потом определяется `set fullName(value) { ... }`. Это сеттер. Когда кто-то пишет `user.fullName = 'Olga Petrova'`, JavaScript передаст строку в параметр `value`.

Строка `const [first, last] = value.split(' ');` разбивает входную строку на части.

Дальше `this.firstName = first;` и `this.lastName = last;` обновляют внутренние поля объекта.

Снаружи `user.fullName` выглядит как обычное свойство, но внутри исполняется код. В Java это напоминает пару `getX()/setX()`, только синтаксис доступа похож на поле.

Геттеры удобны для вычисляемых значений, чтобы не хранить дублирующиеся данные. Сеттеры удобны для валидации и нормализации входа.

⚠️ **Осторожно с геттерами в Angular templates!**

Геттер вызывается при каждой проверке change detection. Если он содержит тяжёлую логику, это замедлит UI:

```typescript
// ❌ Вызывается десятки раз в секунду при CD
get filteredTasks() {
  return this.tasks.filter(t => t.status === 'active'); // создаёт новый массив каждый раз
}

// ✅ Используйте computed signal или мемоизацию
filteredTasks = computed(() => this.tasks().filter(t => t.status === 'active'));
```

```javascript
const user = {
  _age: 0,
  get age() {
    return this._age;
  },
  set age(value) {
    if (value < 0) {
      throw new Error('Возраст не может быть отрицательным');
    }
    this._age = value;
  }
};
```

Свойство `_age` используется как внутреннее хранилище.

Геттер `age()` просто возвращает текущее внутреннее значение.

Сеттер `age(value)` сначала проверяет, не пришло ли отрицательное число.

Если условие `value < 0` истинно, выбрасывается ошибка.

Только если значение прошло проверку, строка `this._age = value;` обновляет внутреннее поле.

Подчёркивание в `_age` — это соглашение: «не трогай напрямую». Это не настоящая приватность, а знак для разработчика.

И ещё важно помнить: accessor property с `get/set` — это отдельный режим свойства. Оно не живёт одновременно как обычное `value`-свойство. У свойства либо `value`/`writable`, либо `get`/`set`.

#### История о типичной ошибке

Разработчик сделал сеттер `fullName`, который внутри присваивал `this.fullName = value`. Получилась бесконечная рекурсия: запись в `fullName` вызывала тот же сеттер снова и снова. Правильный путь — хранить данные в другом внутреннем поле, вроде `_fullName` или в разложенном виде по нескольким полям.

Помнишь §3.9 про дескрипторы? Геттеры и сеттеры — это не отдельная вселенная, а разновидность дескрипторов. Просто вместо `value` и `writable` у свойства появляются функции `get` и `set`. А если вспомнить §3.4, внутри геттера и сеттера всё ещё работает `this`, потому что это тоже функции, вызываемые в контексте объекта.

## 🚀 Задание по проекту: продолжаем `task-manager.js`

В модуле 2 у тебя был менеджер задач с замыканием, которое прятало массив задач. Теперь добавь к нему прототипное наследование. Представь, что обычная задача — это базовый тип, а приоритетная задача — его расширение. В Java ты бы написал `class PriorityTask extends Task`. Здесь сделай то же самое через функции-конструкторы и прототипы, чтобы руками почувствовать, как работает механизм под капотом.

**Что нужно сделать**: создать конструктор `Task`, конструктор `PriorityTask`, связать `PriorityTask.prototype` с `Task.prototype` и научить менеджер создавать оба типа задач.

```javascript
// task-manager.js

function Task(id, title) {
  this.id = id;
  this.title = title;
  this.done = false;
  this.createdAt = new Date().toISOString();
}

Task.prototype.complete = function() {
  this.done = true;
};

Task.prototype.getSummary = function() {
  return `#${this.id}: ${this.title} [${this.done ? 'done' : 'pending'}]`;
};

function PriorityTask(id, title, priority) {
  Task.call(this, id, title);
  this.priority = priority;
}

PriorityTask.prototype = Object.create(Task.prototype);
PriorityTask.prototype.constructor = PriorityTask;

PriorityTask.prototype.getSummary = function() {
  return `${Task.prototype.getSummary.call(this)} priority=${this.priority}`;
};

function createTaskManager() {
  let tasks = [];
  let nextId = 1;

  function findTask(id) {
    return tasks.find(task => task.id === id);
  }

  return {
    add(title) {
      const task = new Task(nextId++, title);
      tasks.push(task);
      return task;
    },

    addPriority(title, priority = 'medium') {
      const task = new PriorityTask(nextId++, title, priority);
      tasks.push(task);
      return task;
    },

    complete(id) {
      const task = findTask(id);
      if (!task) return null;
      task.complete();
      return task;
    },

    list() {
      return tasks.slice();
    },

    summaries() {
      return tasks.map(task => task.getSummary());
    }
  };
}

const tm = createTaskManager();
tm.add('Прочитать про descriptors');
tm.addPriority('Починить production баг', 'high');
tm.complete(2);

console.log(tm.summaries());
console.log(tm.list()[1] instanceof PriorityTask); // true
console.log(Object.getPrototypeOf(tm.list()[1]) === PriorityTask.prototype); // true
```

Разберём этот проектный кусок как наставник рядом с ноутбуком.

Строка `function Task(id, title) { ... }` создаёт функцию-конструктор базовой задачи. Внутри неё свойства `id`, `title`, `done`, `createdAt` записываются прямо в конкретный экземпляр через `this`.

`Task.prototype.complete = function() { ... }` добавляет общий метод `complete` в прототип всех задач. Благодаря этому каждая задача не хранит отдельную копию функции.

`Task.prototype.getSummary = function() { ... }` точно так же кладёт общий метод формирования строки в `Task.prototype`.

Конструктор `PriorityTask(id, title, priority)` внутри вызывает `Task.call(this, id, title);`. Это ручной аналог идеи `super(...)`: мы запускаем логику базового конструктора так, чтобы она записала поля в текущий объект.

После этого строка `this.priority = priority;` добавляет уже специфичное поле приоритетной задачи.

Критически важная строка `PriorityTask.prototype = Object.create(Task.prototype);` строит prototype chain. Новый объект прототипа для `PriorityTask` делегирует к `Task.prototype`.

Строка `PriorityTask.prototype.constructor = PriorityTask;` возвращает корректную ссылку на конструктор после переназначения прототипа. Иначе поле `constructor` осталось бы унаследованным от `Task`.

Метод `PriorityTask.prototype.getSummary = function() { ... }` переопределяет базовый метод. Внутри через `Task.prototype.getSummary.call(this)` мы явно вызываем родительскую реализацию для текущего объекта, а потом дописываем `priority`.

Дальше функция `createTaskManager()` создаёт замыкание с приватным массивом `tasks` и счётчиком `nextId`. Это приятная связь с модулем 2: данные менеджера по-прежнему защищены замыканием.

Метод `add(title)` создаёт обычную задачу через `new Task(...)`.

Метод `addPriority(title, priority = 'medium')` создаёт приоритетную задачу через `new PriorityTask(...)`.

Метод `complete(id)` ищет задачу, вызывает общий метод `complete()` и возвращает обновлённый объект.

`list()` возвращает копию массива через `slice()`, чтобы внешний код не мутировал внутреннее хранилище напрямую. Помни: `slice()` создаёт новый массив, но элементы в нём — те же объекты по ссылке. Если внешний код сделает `list[0].done = true`, это изменит и внутреннее состояние.

`summaries()` проходит по задачам и вызывает `getSummary()` у каждой. Обрати внимание: полиморфизм уже работает. Для обычной задачи вызовится версия из `Task.prototype`, для приоритетной — версия из `PriorityTask.prototype`.

`Task.call(this, ...)` здесь играет роль ручного `super(...)`. Строка `PriorityTask.prototype = Object.create(Task.prototype)` строит prototype chain. Строка `PriorityTask.prototype.constructor = PriorityTask` возвращает корректный `constructor` после переназначения прототипа.

**Критерий готовности**: менеджер умеет создавать обычные и приоритетные задачи, обе имеют `complete()`, обе имеют `getSummary()`, а у `PriorityTask` summary дополняется приоритетом. Проверка через `Object.getPrototypeOf(...)` показывает правильную цепочку.

**Идея для усложнения**: добавь `DeadlineTask`, которая тоже наследуется от `Task`, но хранит `deadline` и показывает его в `getSummary()`.

## Контрольные вопросы (открытые)

1. **Объясни разницу между объектом и ссылкой на объект. Почему это важно при передаче аргументов?**

<details><summary>Ожидаемый ответ</summary>Объект — это сама структура в памяти, а ссылка — значение, которое указывает на эту структуру. В переменной хранится ссылка. Поэтому функция может мутировать тот же объект, но переприсваивание параметра не меняет внешнюю переменную.</details>

2. **Почему spread не делает deep clone? Какой реальный баг отсюда вырастает?**

<details><summary>Ожидаемый ответ</summary>Spread копирует только верхний уровень собственных enumerable-свойств. Вложенные объекты и массивы остаются теми же ссылками. Из-за этого изменение вложенного поля в «копии» неожиданно меняет оригинал.</details>

3. **Объясни разницу между `prototype`, `__proto__` и `[[Prototype]]`.**

<details><summary>Ожидаемый ответ</summary>`[[Prototype]]` — внутренний механизм объекта, ссылка на его прототип. `__proto__` — legacy-доступ к этому механизму. `prototype` — свойство функции-конструктора, которое станет прототипом экземпляров, созданных через `new`.</details>

4. **Чем прототипное наследование в JavaScript отличается от наследования классов в Java?**

<details><summary>Ожидаемый ответ</summary>В Java наследование строится вокруг классов-шаблонов. В JavaScript объект может делегировать поиск свойств другому объекту напрямую. Поэтому ключевая идея — не blueprint, а delegation.</details>

5. **Когда стоит использовать `Object.defineProperty`?**

<details><summary>Ожидаемый ответ</summary>Когда нужно сделать свойство read-only, скрыть его из перечисления, жёстко зафиксировать правила изменения или описать accessor через `get/set`.</details>

6. **Когда геттеры и сеттеры помогают, а когда вредят?**

<details><summary>Ожидаемый ответ</summary>Помогают для вычисляемых значений, валидации и нормализации. Вредят, когда под видом обычного свойства скрывается тяжёлая логика или неожиданные побочные эффекты.</details>
