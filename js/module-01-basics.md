## Блок 1: Переменные и объявления

### 1.1. Зачем три способа объявлять переменные?

Давай начнём с вопроса, который раздражает почти всех, кто приходит из Java: почему в JavaScript вообще три способа объявлять переменные? Неужели нельзя было сделать один нормальный? В каком-то смысле твоя интуиция права: если бы язык проектировали сегодня с нуля, скорее всего, `var` бы просто не существовал. Но JavaScript рос исторически, как старый район города: сначала построили кривые улочки, потом достраивали развязки, а потом начали объяснять, почему некоторые повороты лучше объезжать.

Представь офис. `var` — это как общий маркер на кухне: кто угодно взял, кто угодно подписал контейнер, потом кто-то другой стёр, потом третий вообще утащил маркер в переговорку. `let` — это уже подписанный ящик в твоём столе: доступен в пределах твоего рабочего места, логично организован. `const` — это пломба на документе: документ лежит на столе, к нему можно обращаться, но подменять его нельзя.

Если тебе ближе Java-аналогия, то думай так: `let` — это обычная локальная переменная внутри блока, а `const` — примерно как `final` для ссылки. А вот `var` — это переменная из странной альтернативной Java, где локальная переменная, объявленная внутри `if`, внезапно видна во всей функции. Уже неприятно, да?

Технически разница сводится к трём вещам: **область видимости** (scope), **возможность переприсваивания** и **hoisting** (подъём объявлений).

#### `var` — функциональная область видимости

Прежде чем смотреть на код, давай ответим на вопрос «зачем это вообще когда-то существовало?». В раннем JavaScript язык был намного проще: никакого `let`, никакого `const`, никаких современных модулей. Код часто писали маленькими скриптами для страниц. Тогда `var` казался достаточным. Проблема началась позже, когда на языке стали писать большие приложения. Оказалось, что переменная, «утекающая» из блока наружу, в маленьком скрипте просто неприятна, а в крупной системе — источник тихих и мерзких багов.

Без понимания `var` разработчик часто думает так: «Я объявил переменную внутри `if`, значит вне `if` её нет». Это здравая, почти универсальная интуиция, воспитанная Java, C#, Kotlin и вообще половиной индустрии. Но JavaScript со старым `var` отвечает: «Сюрприз, она есть». И человек тратит время не на бизнес-логику, а на расследование того, почему код увидел значение из неожиданного места.

```javascript
function greetUser(isVip) {
  if (isVip) {
    var greeting = 'Добро пожаловать, VIP!';
  }
  console.log(greeting);
}

greetUser(true);  // "Добро пожаловать, VIP!"
greetUser(false); // undefined
```

Разбор по строкам:
- В первой строке мы объявляем функцию `greetUser`, у которой есть параметр `isVip`.
- Во второй строке начинается блок `if`. Мы проверяем, передали ли `true`.
- В третьей строке объявляется `var greeting`. Обрати внимание: хотя строка находится **внутри** блока `if`, сама переменная будет принадлежать **всей функции**, а не только блоку.
- В пятой строке мы делаем `console.log(greeting)`. Здесь видно особенность `var`: переменная доступна и за пределами `if`.
- В седьмой строке вызываем функцию с `true`, поэтому присваивание произошло, лог выводит строку.
- В восьмой строке вызываем с `false`, блок `if` не выполнился, присваивание не случилось, но переменная всё равно существует и равна `undefined`.

Если ты пришёл из Java, это ощущается так, будто локальная переменная цикла внезапно просочилась наружу. Именно поэтому современный код избегает `var`.

История типичной ошибки. Разработчик пишет функцию расчёта скидок, создаёт внутри `if` переменную `discount`, а потом ниже использует её как будто она гарантированно есть. На тестовых данных всё проходит, потому что условие почти всегда true. Потом приходит реальный кейс, условие ложно, и расчёт уезжает в `NaN` или в странную строку, потому что `undefined` незаметно попал в формулу. Ошибка не в математике. Ошибка в ментальной модели `var`.

#### `let` — блочная область видимости

Теперь — зачем появился `let`. Когда JavaScript вырос из набора скриптов в язык для приложений, разработчикам понадобилось более предсказуемое поведение. Люди хотели, чтобы переменная жила там, где её объявили. Чтобы блок `{}` действительно был границей. Чтобы код читался сверху вниз, без ощущения, что переменные телепортируются.

В жизни это как переход от «общего шкафа в офисе» к ящикам с ключом. Если вещь лежит в конкретном ящике, ты не ожидаешь найти её в коридоре. Так и `let`: объявил внутри блока — живёт внутри блока. Не просачивается наружу. Не маскируется под удобство. Не устраивает сюрпризы.

```javascript
function greetUser(isVip) {
  if (isVip) {
    let greeting = 'Добро пожаловать, VIP!';
    console.log(greeting);
  }
  console.log(greeting);
}
```

Разбор по строкам:
- В первой строке снова объявляем функцию `greetUser`.
- Во второй строке открываем блок `if`.
- В третьей строке объявляем `greeting` через `let`. Теперь переменная принадлежит **только** этому блоку.
- В четвёртой строке `console.log(greeting)` работает, потому что мы всё ещё внутри блока.
- В шестой строке происходит попытка обратиться к `greeting` снаружи блока.
- Результат — `ReferenceError`, потому что переменной там не существует.

Это полезное поведение: ошибка возникает сразу, и проблему проще заметить, чем при тихом получении `undefined`, как с `var`.

Вот это уже поведение, которое легко держать в голове. И именно поэтому в современном коде `let` используется везде, где нужно переприсваивание.

#### `const` — блочная область + запрет переприсваивания

Теперь самый любимый инструмент современного JavaScript — `const`. Но давай сначала про «почему». Без `const` любой кусок кода потенциально может переписать значение переменной дальше по файлу. А когда код большой, эта свобода превращается в тревогу. Ты читаешь выражение и не уверен: «А это значение случайно не перезапишут через 50 строк?» Чем больше таких сомнений, тем тяжелее сопровождать код.

Java-аналогия очень полезна: `const` для объекта похож на `final` для ссылки. И вот тут многие попадаются. Они слышат слово «константа» и мысленно переводят его как «абсолютно неизменяемое». Но это неверно. `const` делает неизменяемой **ссылку**, а не содержимое объекта.

```javascript
const config = { debug: true };
config.debug = false;
console.log(config.debug);

config = {};
```

Разбор по строкам:
- В первой строке мы создаём объект и сохраняем ссылку на него в переменную `config`, объявленную через `const`.
- Во второй строке меняем свойство `debug` у самого объекта. Это разрешено.
- В третьей строке убеждаемся, что значение свойства действительно стало `false`.
- В пятой строке пытаемся присвоить переменной `config` новую ссылку на другой объект.
- Именно здесь получаем ошибку, потому что `const` запрещает переприсваивание переменной.

То есть `const` не говорит: «Объект нельзя трогать». Он говорит: «Эта переменная всегда будет ссылаться на тот же самый объект». Это тонкость, на которой очень любят ловить на собеседованиях.

Java-аналог:

```java
final Map<String, Boolean> config = new HashMap<>();
config.put("debug", false);
config = new HashMap<>();
```

Разбор по строкам:
- В первой строке объявляем `config` как `final`, то есть ссылку переприсваивать нельзя.
- Во второй строке мутируем содержимое объекта `Map`. Это разрешено.
- В третьей строке пытаемся дать переменной новую ссылку — и получили бы ошибку компиляции.

История типичной ошибки. Разработчик видит `const settings = {...}` и расслабляется: «Отлично, настройки неизменяемые». Потом в другом месте кто-то делает `settings.theme = 'dark'`. Ошибка не в `const`, а в том, что его перепутали с глубокой иммутабельностью.

#### Что выбирать?

Теперь соберём правило в простую практику. В современном коде почти всегда думай так: «Могу ли я не менять эту переменную?» Если да — используй `const`. Это уменьшает количество подвижных частей в программе. Меньше подвижных частей — меньше случайных состояний.

Если ты знаешь, что значение будет переприсваиваться, бери `let`. Например, счётчик, аккумулятор результата, временная ссылка, которую ты действительно меняешь по ходу алгоритма.

`var` оставь для чтения legacy-кода и собеседований. Это как старый лифт в доме: полезно понимать, как он устроен, но жить хочется всё-таки с новым.

Правило простое:
1. По умолчанию — `const`. Если значение не нужно менять, не давай возможности это сделать.
2. Если нужно переприсваивание — `let`.
3. `var` — никогда в новом коде. Только при работе с legacy.

**1. Что выведет этот код?**
```javascript
console.log(x);
var x = 10;
```
- A) 10
- B) ReferenceError
- C) undefined
- D) null

<details><summary>Ответ</summary>C) undefined — объявление `var x` «всплывает» (hoists) до начала, но присваивание `= 10` остаётся на месте. В момент `console.log` переменная существует, но ещё не получила значение. Это классическое проявление hoisting.</details>

**2. Что произойдёт при выполнении?**
```javascript
const arr = [1, 2, 3];
arr.push(4);
console.log(arr.length);
```
- A) TypeError — нельзя менять const
- B) 4
- C) 3
- D) ReferenceError

<details><summary>Ответ</summary>B) 4 — `const` запрещает переприсваивание ссылки (`arr = ...`), но не мутацию объекта. `push` мутирует массив, не переприсваивая переменную. Это частая ловушка для тех, кто думает, что `const` означает «неизменяемый».</details>

```javascript

```

```javascript

```

```javascript

```
- A) Только вариант B
- B) Варианты A и B
- C) Все три
- D) Ни один

<details><summary>Ответ</summary>A) Только вариант B — `const` запрещает переприсваивание примитивов. `let` и `var` позволяют менять значение свободно.</details>

### 🎯 1.2. Hoisting — подъём объявлений (Interview Focus)

> **Почему спрашивают на собеседованиях**: Hoisting проверяет, понимает ли кандидат разницу между фазой компиляции и фазой исполнения. Это фундамент для понимания scope, замыканий и temporal dead zone.

Hoisting — тема, о которую спотыкаются даже люди с опытом. Почему? Потому что код читается сверху вниз, а движок готовит его к выполнению не совсем так, как читаешь ты глазами. Ты видишь сначала `console.log`, потом объявление. Движок сначала регистрирует объявления, а потом уже исполняет инструкции. Из-за этого мозг новичка и реальность рантайма расходятся.

Представь театр. До начала спектакля рабочие сцены расставляют декорации и подписывают, где кто должен стоять. Это подготовительный этап. А потом уже выходят актёры и играют сцену. Hoisting — это вот эта подготовка. Декорация уже существует, но актёр ещё не произнёс реплику. У `var` декорация есть, но пустая. У `function declaration` декорация и актёр уже на месте. У `let` и `const` на полу уже лежит табличка «место занято, но подходить рано» — это TDZ.

Java-аналогия неточная, но полезная: компилятор Java не даст обратиться к локальной переменной до объявления вообще. JavaScript тоже «знает» о переменной заранее, но ведёт себя по-разному для разных видов объявлений. Вот эту разницу и нужно держать в голове.

Hoisting (подъём) — это механизм JavaScript, при котором **объявления** переменных и функций «перемещаются» в начало своей области видимости ещё до выполнения кода. Но это не буквальное физическое перемещение строк. Для понимания удобна модель двух фаз: фаза создания окружения (когда движок «замечает» объявления) и фаза исполнения (когда код выполняется построчно). Это упрощённая ментальная модель, а не буквальное описание работы V8/SpiderMonkey.

#### Как hoisting работает для разных объявлений

| Объявление | Что «всплывает» | Значение до присваивания |
|------------|------------------|--------------------------|
| `var x = 5` | Только `var x` | `undefined` |
| `let x = 5` | Знание о существовании | TDZ (ошибка при обращении) |
| `const x = 5` | Знание о существовании | TDZ (ошибка при обращении) |
| `function f() {}` | Всё целиком (имя + тело) | Полностью работоспособна |
| `const f = () => {}` | Как `const` | TDZ |

Эту таблицу мало просто прочитать. Её нужно «почувствовать». `var` — как пустой контейнер, который уже привезли на склад. `let`/`const` — как контейнер, место под который уже зарезервировали, но пломбу не сняли. `function declaration` — как готовый инструмент: бери и работай.

**TDZ (Temporal Dead Zone)** — временная мёртвая зона — это промежуток от начала блока до строки с объявлением `let`/`const`. Если обратишься к переменной в этой зоне — получишь `ReferenceError`.

```javascript
{
  console.log(greeting);
  let greeting = 'Привет';
  console.log(greeting);
}
```

Разбор по строкам:
- В первой строке начинается новый блок `{}`. Это и есть область видимости для `let greeting`.
- Во второй строке мы пытаемся прочитать `greeting` до её инициализации.
- Движок уже знает, что такая переменная будет, но она находится в TDZ, поэтому сразу бросает `ReferenceError`.
- В третьей строке происходит объявление и инициализация `greeting`.
- В четвёртой строке доступ уже разрешён, и мы получаем `'Привет'`.

Очень важно: TDZ — не значит «переменной не существует». Наоборот, движок **знает**, что она существует. Именно поэтому обращение не даёт `undefined`, а выбрасывает ошибку. Это защитный механизм против слишком раннего использования.

Вот здесь и рождается важное различие: `var` даёт «тихую пустоту», `let`/`const` дают «громкую ошибку». В современном коде чаще полезнее именно громкая ошибка.

#### Hoisting функций

Теперь разберём самую приятную часть hoisting: объявления функций через `function`. Зачем это поведение вообще полезно? Потому что оно позволяет организовывать код более свободно. Ты можешь сначала описывать сценарий, а подробности реализации вынести ниже. Это как оглавление книги, которое идёт раньше полного текста главы.

```javascript
const result = calculateDiscount(100, 0.15);
console.log(result);

function calculateDiscount(price, discountRate) {
  return price * (1 - discountRate);
}
```

Разбор по строкам:
- В первой строке вызываем `calculateDiscount` ещё до того, как визуально дошли до её объявления.
- Это работает, потому что `function calculateDiscount(...)` всплывает целиком вместе с телом.
- Во второй строке выводим результат вычисления.
- В четвёртой строке объявляется функция.
- В пятой строке происходит расчёт скидки и возврат значения.

Теперь сравни с function expression:

```javascript
const result = calculateDiscount(100, 0.15);

const calculateDiscount = (price, discountRate) => {
  return price * (1 - discountRate);
};
```

Разбор по строкам:
- В первой строке мы пытаемся вызвать `calculateDiscount`.
- Но сама переменная `calculateDiscount` объявлена через `const` только ниже.
- До строки объявления она находится в TDZ.
- Поэтому уже на первой строке получаем ошибку, и до тела функции дело вообще не дойдёт.
- В третьей строке только происходит объявление переменной и присваивание стрелочной функции.

Стоп. Тут очень частая ловушка на интервью: люди говорят «стрелочные функции не hoistятся». Формально это неточно. **Hoistится объявление переменной**, потому что это `const`, но доступ к ней до инициализации запрещён из-за TDZ. Это тонкий, но важный ответ.

Видишь разницу? Для `const`/`let` — `ReferenceError`, потому что TDZ. Для `var` — переменная есть, но содержит `undefined`, поэтому при вызове уже `TypeError`.

#### 💼 Типичные вопросы на собеседовании

1. **Что такое hoisting?** → Механизм, при котором объявления переменных и функций «поднимаются» в начало scope до исполнения кода. `var` поднимается как `undefined`, `let`/`const` попадают в TDZ, function declarations поднимаются целиком.

2. **Что такое TDZ?** → Temporal Dead Zone — зона от начала блока до объявления `let`/`const`, в которой обращение к переменной выбрасывает ReferenceError.

3. **Почему `var` в цикле создаёт проблемы?** → Потому что `var` имеет function scope, а не block scope. Все итерации цикла делят одну переменную. Классический пример с `setTimeout` мы подробно увидим в модуле про функции, когда дойдём до замыканий.

#### 🚫 Red flags в ответе
- «Hoisting — это когда код физически перемещается наверх» (нет, это просто модель)
- «let не hoists» (hoists, но в TDZ)
- «Если код работает с function declaration, он обязан работать и со стрелочной функцией выше по файлу»

#### ✅ Green flags в ответе
- Упоминает две фазы: compilation + execution
- Объясняет TDZ для `let`/`const`
- Приводит пример разницы function declaration vs function expression
- Различает `ReferenceError` и `TypeError` в зависимости от вида объявления


<details><summary>Ответ</summary>C) "Hello!" — function declarations всплывают целиком (и имя, и тело). Функция уже полностью доступна до строки объявления.</details>

### 1.3. Типы данных в JavaScript

Прежде чем спорить о сравнениях и приведении, нужно разобраться, **что вообще хранится в переменных**. Потому что очень много «магии» JavaScript на самом деле начинается с довольно простого факта: в языке мало базовых типов, но правила их преобразования очень гибкие.

Если сравнивать с Java, JavaScript похож на чемодан с меньшим количеством отделений, но с очень хитрыми карманами внутри. В Java у тебя много отдельных примитивов: `int`, `long`, `float`, `double` и так далее. В JavaScript всё заметно компактнее: один `number` для большинства чисел, отдельный `bigint` для очень больших целых, строки, булевы значения, `undefined`, `null`, `symbol` — и объектный мир.

Помнишь, в §1.1 мы говорили, что в JavaScript многое решается во время выполнения? Вот типы — прямое продолжение этой идеи. Переменная не «пришита» к одному типу навсегда. Сегодня в ней число, завтра строка, потом объект. Это даёт гибкость, но требует дисциплины от разработчика.

#### Примитивные типы

| Тип | Пример | Аналог в Java |
|-----|--------|---------------|
| `number` | `42`, `3.14`, `NaN`, `Infinity` | `double` (но один тип на всё) |
| `bigint` | `9007199254740993n` | `BigInteger` |
| `string` | `'hello'`, `"world"`, `` `шаблон` `` | `String` |
| `boolean` | `true`, `false` | `boolean` |
| `undefined` | `undefined` | нет аналога (ближе всего — `null`) |
| `null` | `null` | `null` |
| `symbol` | `Symbol('id')` | нет прямого аналога |

Примитивы можно представить как «значения сами по себе». Они не хранят внутри настраиваемый набор полей, как объект. Это маленькие самостоятельные сущности. Как монета, карточка, талон, ярлык — у каждой свой формат и свои правила использования.

**Ключевое отличие от Java**: в JavaScript нет разделения на `int` и `double`. Тип `number` — это всегда 64-битное число с плавающей точкой (IEEE 754). Это значит, что `5` и `5.0` — это одно и то же значение. И это же причина знаменитой «проблемы»:

```javascript
console.log(0.1 + 0.2);
console.log(0.1 + 0.2 === 0.3);
```

Разбор по строкам:
- В первой строке складываем два десятичных числа, которые не представимы в двоичной форме идеально точно.
- Поэтому результатом становится `0.30000000000000004`.
- Во второй строке мы сравниваем это значение с `0.3`.
- Сравнение вернёт `false`, потому что внутри памяти числа немного различаются.

Это не баг JavaScript. Это особенность стандарта IEEE 754, которую ты можешь встретить и в Java с `double`. Просто в Java у тебя чаще есть соблазн использовать `int` для целых и `BigDecimal` для денег, а в JS почти всё по умолчанию живёт в мире `number`.

#### `undefined` vs `null`

Вот здесь начинается особенно важная часть для практики. В Java у тебя в основном одно «пустое» значение — `null`. В JavaScript пустот две: `undefined` и `null`. И новичок часто думает: «Ну окей, две разные записи одного и того же». Нет. Они похожи, но семантически разные.

Представь гостиницу. `undefined` — это номер, в который ещё никого не заселили и даже не решили, кто там будет. `null` — это номер, который администратор сознательно отметил как пустой: «здесь никого нет, это решение принято». Или другая аналогия: `undefined` — пустая графа в анкете, `null` — явно поставленный прочерк.

Java-аналогия такая: `undefined` — это что-то, чего в Java почти нет в явном виде для локальных ссылок; `null` — это привычное «ссылка никуда не указывает». Поэтому для Java-разработчика `undefined` часто выглядит как дополнительный источник путаницы. И да, так и есть.

```javascript
let user;
let deletedUser = null;

console.log(user);
console.log(deletedUser);
```

Разбор по строкам:
- В первой строке объявляем `user`, но ничего туда не кладём. После выхода из TDZ переменная получит `undefined`.
- Во второй строке явно присваиваем `null` переменной `deletedUser`.
- В четвёртой строке увидим `undefined`.
- В пятой строке увидим `null`.

Практическая договорённость в командах обычно такая: `undefined` часто означает «значение не пришло», а `null` — «значение намеренно очищено».

Важный нюанс для Java-разработчика: если ты обратишься к несуществующему свойству объекта, результат тоже будет `undefined` — не ошибка, а тихое «тут ничего нет». Это ключевое отличие от Java, где `obj.missingField` не скомпилируется.

```javascript
const user = { name: 'Alice' };
console.log(user.age); // undefined — свойства нет, но ошибки тоже нет
```

#### `typeof` — проверка типа

Когда человек впервые узнаёт о динамической типизации, первым рефлексом становится: «А как мне узнать тип прямо во время выполнения?» Для этого есть оператор `typeof`. Он возвращает строку с именем типа. Казалось бы, просто. Но JavaScript, как всегда, не отпускает без пары исторических сюрпризов.

```javascript
typeof 42;
typeof 'hello';
typeof true;
typeof undefined;
typeof Symbol();
typeof 10n;
typeof {};
typeof [];
typeof null;
typeof function(){};
```

Разбор по строкам:
- Первая строка вернёт `"number"`.
- Вторая строка вернёт `"string"`.
- Третья строка вернёт `"boolean"`.
- Четвёртая строка вернёт `"undefined"`.
- Пятая строка вернёт `"symbol"`.
- Шестая строка вернёт `"bigint"`.
- Седьмая строка вернёт `"object"`.
- Восьмая строка тоже вернёт `"object"`, потому что массив — это объект.
- Девятая строка вернёт `"object"` для `null` — и это исторический баг.
- Десятая строка вернёт `"function"`, хотя функция в JS тоже объект, просто с особым внутренним поведением.

Два момента, которые почти гарантированно всплывут на собеседовании:

1. **`typeof null === 'object'`** — это баг из первой версии JavaScript (1995). В оригинальной реализации значения хранились с type tag, и у `null` был тот же tag, что у объектов. Баг не исправляют, потому что это сломает миллионы сайтов.

2. **`typeof []` возвращает `"object"`** — массив в JavaScript — это объект с числовыми ключами и свойством `length`. Для проверки массива используй `Array.isArray(arr)`.

#### `NaN` — Not a Number

А теперь один из самых любимых персонажей JavaScript-хоррора: `NaN`. Название звучит как «не число», и мозг автоматически делает вывод: «Наверное, это особый тип». Нет. `NaN` — это значение типа `number`. Да, число, которое означает, что числовая операция провалилась.

Java-аналогия отличная: `Double.NaN`. И там тоже `NaN` не равен самому себе. Это не причуда JS, а следствие IEEE 754.

```javascript
typeof NaN;
NaN === NaN;
isNaN('hello');
Number.isNaN(NaN);
```

Разбор по строкам:
- В первой строке `typeof NaN` вернёт `"number"`.
- Во второй строке `NaN === NaN` даст `false`.
- В третьей строке `isNaN('hello')` вернёт `true`, потому что старая функция сначала пытается привести аргумент к числу.
- В четвёртой строке `Number.isNaN(NaN)` надёжно вернёт `true`, потому что не делает лишнего приведения.

Запомни практическое правило: если проверяешь именно значение `NaN`, используй `Number.isNaN`. Старый глобальный `isNaN` хорош для исторических артефактов и неожиданных сюрпризов, но не для точной проверки.

#### ✅ Проверь себя

**1. Что вернёт `typeof null`?**
- A) "null"
- B) "undefined"
- C) "object"
- D) TypeError

<details><summary>Ответ</summary>C) "object" — это исторический баг JavaScript с 1995 года. `null` должен был иметь свой тип, но из-за особенностей внутреннего представления данных в первом движке JS он получил type tag объекта. Баг сохранён для обратной совместимости.</details>

**2. Чему равно выражение `NaN === NaN`?**
- A) true
- B) false
- C) TypeError
- D) undefined

<details><summary>Ответ</summary>B) false — NaN — единственное значение в JavaScript, которое не равно самому себе. Это следствие стандарта IEEE 754. Для проверки используй `Number.isNaN(value)`.</details>

**3. Каков тип переменной `x` после `let x;`?**
- A) "null"
- B) "undefined"
- C) "uninitialized"
- D) ReferenceError при проверке

<details><summary>Ответ</summary>B) "undefined" — переменная объявлена через `let`, но не инициализирована. JavaScript автоматически присваивает `undefined` таким переменным (после выхода из TDZ).</details>

### 🎯 1.4. Приведение типов (Type Coercion) — Interview Focus

> **Почему спрашивают на собеседованиях**: Приведение типов — одна из самых контринтуитивных частей JavaScript. Интервьюер проверяет, предсказуем ли для кандидата поведение языка в edge cases и знает ли он правила, а не просто зазубрил примеры.

Вот мы и дошли до той части, после которой JavaScript либо начинают уважать, либо начинают бояться. Обычно сначала бояться. Потому что приведение типов похоже на очень услужливого официанта, который не стал уточнять заказ и сам решил, что ты имел в виду. Иногда он угадывает. Иногда приносит кетчуп к десерту.

В Java ты привык к жёсткости: если типы не сходятся, компилятор останавливает тебя заранее. В JavaScript язык говорит: «Не переживай, я сам что-нибудь придумаю». Эта «помощь» экономит пару символов в простых сценариях, но создаёт немало сюрпризов в сложных. И именно поэтому тему нужно не зубрить, а понять как алгоритм.

Помнишь, в §1.3 мы обсуждали `undefined`, `null`, `number`, `string` и `boolean`? Вот здесь все они начинают активно сталкиваться друг с другом.

Приведение типов (type coercion) — это автоматическое преобразование значения из одного типа в другой, которое JavaScript делает «за кулисами» при операциях с разными типами.

#### Три типа приведения

JavaScript приводит значения к одному из трёх типов:

1. **К строке (String)** — при конкатенации с `+` или при `String(value)`
2. **К числу (Number)** — при математических операциях (`-`, `*`, `/`) или при `Number(value)`
3. **К булеву (Boolean)** — в условиях (`if`, `while`, `? :`) или при `Boolean(value)`

#### Приведение к строке

Почему вообще существует приведение к строке? Потому что в реальной жизни мы постоянно собираем текст из разных кусочков данных: «Возраст: 25», «Пользователь: Ivan», «Сумма: 1500». Если бы язык требовал вручную приводить всё к строке каждый раз, это было бы утомительно. Поэтому JS делает это автоматически при работе `+` со строкой.

Но в этой удобной идее сидит ловушка. Оператор `+` в JavaScript перегружен: он может и складывать числа, и склеивать строки. Поэтому каждый раз, когда рядом появляется строка, ты должен на секунду остановиться и спросить себя: «Это сейчас математика или текст?» Не делай вид, что это мелочь. Огромное количество багов в обработке форм и query-параметров начинается именно здесь.

```javascript
'Возраст: ' + 25;
'5' + 3;
'5' + null;
'5' + undefined;
'5' + true;
```

Разбор по строкам:
- В первой строке строка `'Возраст: '` склеивается с числом `25`, которое предварительно превращается в строку `'25'`.
- Во второй строке строка `'5'` заставляет число `3` стать строкой `'3'`, результат — `'53'`.
- В третьей строке `null` превращается в строку `'null'`, результат — `'5null'`.
- В четвёртой строке `undefined` превращается в строку `'undefined'`, результат — `'5undefined'`.
- В пятой строке `true` превращается в строку `'true'`, результат — `'5true'`.

Главное правило: **оператор `+` при наличии строки всегда конкатенирует**. Это отличает его от `-`, `*`, `/`.

Что ты думаешь, почему `1 + 2 + '3'` и `'1' + 2 + 3` дают разные результаты? Потому что приведение зависит от того, что происходит **на каждом конкретном шаге слева направо**.

#### Приведение к числу

Теперь про вторую большую зону. Почему `-`, `*`, `/` ведут себя иначе? Потому что у них нет «строкового режима». Они созданы для математики, а значит пытаются загнать операнды в числа.

Без понимания этого правила люди удивляются: почему `'5' + 3` даёт `'53'`, а `'5' - 3` даёт `2`? Ответ простой: у `+` есть два режима, а у `-` — только числовой.

```javascript
'6' - 2;
'6' * '2';
'abc' - 1;
true + true;
null + 5;
undefined + 5;
```

Разбор по строкам:
- В первой строке строка `'6'` приводится к числу `6`, результат `4`.
- Во второй строке обе строки приводятся к числам, результат `12`.
- В третьей строке `'abc'` не удаётся превратить в число, получается `NaN`.
- В четвёртой строке `true` превращается в `1`, второй `true` тоже в `1`, результат `2`.
- В пятой строке `null` приводится к `0`, результат `5`.
- В шестой строке `undefined` превращается в `NaN`, поэтому итог тоже `NaN`.

Таблица приведения к числу:

| Значение | Number(...) | Комментарий |
|----------|-------------|-------------|
| `"123"` | 123 | Строка с числом → число |
| `""` | 0 | Пустая строка → 0 |
| `"abc"` | NaN | Не-числовая строка → NaN |
| `true` | 1 | |
| `false` | 0 | |
| `null` | 0 | Внимание! |
| `undefined` | NaN | Отличается от null! |
| `[]` | 0 | Сначала [] → "" → 0 |
| `[5]` | 5 | Сначала [5] → "5" → 5 |
| `{}` | NaN | Сначала {} → "[object Object]" → NaN |

Самая опасная пара здесь — `null` и `undefined`: `null` превращается в `0`, а `undefined` — в `NaN`.

#### Приведение к булеву (Falsy/Truthy)

Вот ещё один участок, где Java-разработчик часто получает культурный шок. В Java условие `if` принимает только `boolean`. В JavaScript в условие можно подставить почти что угодно, а язык сам решит, считать это истинным или ложным.

Зачем это вообще нужно? Потому что удобно. Можно писать `if (user)` вместо `if (user !== null && user !== undefined)`. Можно использовать короткие идиомы с `&&` и `||`. Но удобство работает только пока ты отлично помнишь список falsy-значений. (Также существует исторический артефакт `document.all`, который ведёт себя как falsy в браузерах, но это экзотика, которую не нужно запоминать.) Иначе начинаются ошибки серии «я думал, пустой массив — это false».

```javascript
Boolean(0);
Boolean(-0);
Boolean('');
Boolean(null);
Boolean(undefined);
Boolean(false);
Boolean(NaN);
Boolean(0n);
```

Разбор по строкам:
- Первая строка даёт `false`, потому что ноль — falsy.
- Вторая строка тоже `false`, потому что `-0` тоже falsy.
- Третья строка возвращает `false`, так как пустая строка falsy.
- Четвёртая строка возвращает `false` для `null`.
- Пятая строка возвращает `false` для `undefined`.
- Шестая строка возвращает `false` для самого булева значения `false`.
- Седьмая строка возвращает `false` для `NaN`.
- Восьмая строка возвращает `false` для `0n`.

**Falsy-значения** (приводятся к `false`) — их нужно запомнить, их восемь:
- `0`
- `-0`
- `""`
- `null`
- `undefined`
- `false`
- `NaN`
- `0n`

Всё остальное — truthy, включая значения, которые психологически кажутся «пустыми»:

```javascript
Boolean('0');
Boolean(' ');
Boolean([]);
Boolean({});
Boolean('false');
```

Разбор по строкам:
- В первой строке `'0'` — непустая строка, значит `true`.
- Во второй строке строка с пробелом тоже непустая, значит `true`.
- В третьей строке пустой массив — объект, а объекты truthy.
- В четвёртой строке пустой объект тоже truthy.
- В пятой строке `'false'` — всё равно просто непустая строка, значит `true`.

🔗 **Связь с Angular**

В Angular-шаблонах выражение `*ngIf="value"` использует те же правила truthy/falsy, что и обычный JavaScript. Поэтому пустой массив `[]` всё равно truthy — это частая ловушка: `*ngIf="items"` будет истинным даже при `items = []`. Если хочешь проверить, что массив не пустой, пиши `*ngIf="items.length"`, а в новом control flow — `@if (items.length) { ... }`.

#### Знаменитые «головоломки» приведения

```javascript
[] + [];
[] + {};
{} + [];
true + true;
'ba' + + 'a';
```

Разбор по строкам:
- Первая строка: каждый пустой массив превращается в пустую строку `""`, затем строки склеиваются, получается `""`.
- Вторая строка: `[]` превращается в `""`, `{}` — в `"[object Object]"`, итог `"[object Object]"`.
- Третья строка коварная: если выражение стоит в начале строки, `{}` может быть воспринят как пустой блок, а `+[]` как унарный плюс, который даёт `0`.
- Четвёртая строка: `true` превращается в `1`, второй `true` тоже в `1`, результат `2`.
- Пятая строка: `+'a'` пытается привести `'a'` к числу, получает `NaN`, потом `'ba' + NaN` даёт строку `'baNaN'`.

Не нужно зубрить все эти случаи — нужно понимать **алгоритм**:
1. Если оба операнда — примитивы одного типа, операция выполняется напрямую.
2. Если один из них — объект, вызывается `valueOf()` или `toString()` для приведения к примитиву.
3. Для `+`: если хоть один операнд — строка, второй приводится к строке.
4. Для `-`, `*`, `/`: оба приводятся к числу.

История типичной ошибки: значения из формы приходят строками, разработчик пишет `price + quantity` и получает `'105'` вместо `15`. Поэтому для внешних данных явное приведение обычно безопаснее.

#### 💼 Типичные вопросы на собеседовании

1. **Что такое type coercion?** → Неявное преобразование типа, которое JS выполняет автоматически при операциях с разнотипными значениями.
2. **Перечислите все falsy-значения** → `0`, `-0`, `""`, `null`, `undefined`, `false`, `NaN`, `0n`.
3. **Что выведет `[] == false`?** → `true`. Цепочка: `[]` → `""` → `0`, `false` → `0`, `0 == 0` → true.
4. **Чем отличается приведение для `+` и для `-`?** → `+` конкатенирует при наличии строки, `-` всегда приводит к числу.

#### 🚫 Red flags в ответе
- «Я просто не использую `==` и всё» (уход от понимания)
- Не может объяснить, почему `"" == false` даёт true
- Путает falsy и null/undefined
- Не знает, что `null` и `undefined` по-разному ведут себя при приведении к числу

#### ✅ Green flags в ответе
- Описывает алгоритм Abstract Equality Comparison
- Упоминает `ToPrimitive`, `valueOf()`, `toString()`
- Знает разницу между `null` и `undefined` при приведении к числу (0 vs NaN)
- Не просто выдаёт ответ, а раскладывает выражение на шаги

#### ✅ Проверь себя

**1. Что выведет `'5' + 3`?**
- A) 8
- B) "53"
- C) NaN
- D) TypeError

<details><summary>Ответ</summary>B) "53" — оператор `+` при наличии строки выполняет конкатенацию. Число 3 приводится к строке "3", результат — "53". Это одна из самых частых ловушек JS.</details>

**2. Что выведет `'5' - 3`?**
- A) "53"
- B) 2
- C) NaN
- D) "2"

<details><summary>Ответ</summary>B) 2 — оператор `-` всегда приводит операнды к числу. "5" → 5, затем 5 - 3 = 2. В отличие от `+`, тут нет конкатенации.</details>

**3. Какое из этих значений truthy?**
- A) `0`
- B) `""`
- C) `[]`
- D) `null`

<details><summary>Ответ</summary>C) `[]` — пустой массив truthy! Это частая ловушка. Массив — это объект, а все объекты truthy (даже пустые). Чтобы проверить пустоту массива, используй `arr.length === 0`.</details>

### 🎯 1.5. Операторы сравнения: `==` vs `===` — Interview Focus

> **Почему спрашивают на собеседованиях**: Это базовый вопрос, который проверяет, понимает ли кандидат приведение типов и умеет ли писать предсказуемый код.

Если бы меня попросили назвать один маленький оператор, из-за которого написано непропорционально много статей, мемов и багов, я бы без колебаний выбрал `==`. Потому что внешне он выглядит безобидно: ну подумаешь, сравнение. Но внутри него живёт целый алгоритм приведения типов, который срабатывает не так, как подсказывает бытовая логика.

В Java всё проще: для примитивов `==` сравнивает значения, для объектов — ссылки, а для содержимого есть `.equals()`. В JavaScript же у нас два разных оператора равенства: один строгий, другой «с творчеством». И если ты не различаешь, когда язык начинает «помогать», то неизбежно сталкиваешься с сюрпризами.

**`===` (строгое равенство)** — сравнивает без приведения типов. Если типы разные — сразу `false`.

**`==` (нестрогое равенство)** — перед сравнением пытается привести операнды к одному типу.

```javascript
5 === '5';
5 == '5';

null === undefined;
null == undefined;

0 === false;
0 == false;
```

Разбор по строкам:
- Первая строка вернёт `false`, потому что `number` и `string` — разные типы, а `===` не приводит значения.
- Вторая строка вернёт `true`, потому что `'5'` будет приведена к числу `5`.
- Четвёртая строка вернёт `false`, потому что `null` и `undefined` — разные типы.
- Пятая строка вернёт `true` из-за специального правила спецификации: `null` и `undefined` считаются равными при `==`.
- Седьмая строка вернёт `false`, потому что число и булево значение имеют разные типы.
- Восьмая строка вернёт `true`, потому что `false` приводится к `0`.

#### Объекты сравниваются по ссылке, а не по содержимому

Здесь Java-разработчику полезно включить знакомую ментальную модель: в Java `==` для объектов сравнивает ссылки, а `.equals()` — содержимое. В JavaScript похожая история: и `==`, и `===` для двух объектов сравнивают **ссылку на объект в памяти**, а не «похожесть полей».

```javascript
const a = { x: 1 };
const b = { x: 1 };
console.log(a === b); // false — разные объекты в памяти

const c = a;
console.log(a === c); // true — одна и та же ссылка
```

Если у двух объектов одинаковые поля, это ещё не делает их равными. Они будут равны только если это буквально один и тот же объект. Поэтому в JS нет встроенного аналога Java `.equals()` для обычных объектов — сравнение по значению приходится писать отдельно.

#### Алгоритм `==` (Abstract Equality Comparison)

Очень соблазнительно просто сказать: «Да ну его, всегда используй `===`». Практически это хороший совет. Но на собеседовании и в отладке этого мало. Нужно уметь объяснить, **почему** выражение дало такой результат.

Когда ты пишешь `a == b`, JavaScript следует этому алгоритму:

1. Если типы одинаковые → сравни как `===`
2. `null == undefined` → `true` (и наоборот)
3. `null` или `undefined` с чем-то другим → `false`
4. Число == строка → приведи строку к числу
5. Булев == что-то → приведи булев к числу, повтори
6. Объект == примитив → приведи объект к примитиву через `valueOf()`/`toString()`

Благодаря правилу 2, конструкция `x == null` проверяет одновременно и `null`, и `undefined` — это **единственный** случай, где `==` бывает полезнее `===`:

```javascript
if (value === null || value === undefined) {
  // ...
}

if (value == null) {
  // ...
}
```

Разбор по строкам:
- В первой конструкции мы явно проверяем два случая: `null` и `undefined`.
- Во второй конструкции используем специальное поведение `==`.
- Строка `value == null` сработает только для `null` и `undefined`, но не для `0`, `false` или `''`.
- Именно поэтому это редкое, но допустимое исключение из общего правила «предпочитай `===`».

#### Таблица неочевидных сравнений

| Выражение | Результат | Почему |
|-----------|-----------|--------|
| `"" == false` | true | `""` → 0, false → 0 |
| `" " == false` | true | `" "` → 0, false → 0 |
| `"0" == false` | true | `"0"` → 0, false → 0 |
| `[] == false` | true | `[]` → `""` → 0, false → 0 |
| `[1] == true` | true | [1] → `"1"` → 1, true → 1 |
| `[2] == true` | false | [2] → `"2"` → 2, true → 1, 2≠1 |
| `NaN == NaN` | false | NaN не равен ничему, включая себя |

Давай разберём один особенно любимый пример интервьюеров:

```javascript
[] == ![];
```

Разбор по шагам:
- Сначала вычисляется `![]`.
- Пустой массив `[]` — truthy, поэтому `![]` становится `false`.
- Получаем выражение `[] == false`.
- Массив приводится к примитиву: `[]` → `""`.
- Пустая строка приводится к числу: `""` → `0`.
- `false` приводится к числу: `false` → `0`.
- Итог: `0 == 0`, то есть `true`.

#### Рекомендация

**Всегда используй `===`**, кроме одного случая: `value == null` для проверки `null`/`undefined`. Линтер ESLint с правилом `eqeqeq` помогает это контролировать.

Чем меньше скрытых преобразований во время сравнения, тем предсказуемее код.

TypeScript с `strict: true` иногда ловит заведомо несовместимые сравнения на этапе компиляции, но это не замена правилу «предпочитай `===`» — многие случаи TypeScript не отловит.

#### 💼 Типичные вопросы на собеседовании

1. **В чём разница `==` и `===`?** → `===` сравнивает без приведения (типы должны совпадать), `==` приводит типы по алгоритму Abstract Equality Comparison.
2. **Когда допустимо использовать `==`?** → При проверке `value == null` (одновременно `null` и `undefined`).
3. **Почему `[] == ![]` возвращает true?** → `![]` → false (массив truthy, отрицание → false). Затем `[] == false` → `"" == 0` → `0 == 0` → true.

#### ✅ Проверь себя

**1. Что вернёт `null === undefined`?**
- A) true
- B) false
- C) TypeError
- D) NaN

<details><summary>Ответ</summary>B) false — оператор `===` не делает приведения. `null` и `undefined` — это разные типы, поэтому строгое сравнение возвращает false. А вот `null == undefined` вернёт true — специальное правило спецификации.</details>

**2. Что вернёт `'' == false`?**
- A) true
- B) false
- C) TypeError
- D) undefined

<details><summary>Ответ</summary>A) true — `false` приводится к числу (0), пустая строка `""` приводится к числу (0), затем `0 == 0` → true. Именно из-за таких неочевидных случаев рекомендуется всегда использовать `===`.</details>

## Блок 2: Операторы и выражения

### 1.6. Логические операторы и их особенности

На этом месте у Java-разработчика обычно происходит второй сильный культурный шок после `==`. В Java логические операторы почти всегда про `boolean`. В JavaScript они часто используются как **операторы выбора значения**. То есть они возвращают не `true`/`false`, а один из исходных операндов.

Так сделано ради кратких выражений: дефолтных значений, условного доступа и ленивого вычисления. Но это удобно только если хорошо понимать правила truthy/falsy из §1.4.

#### Оператор `||` (ИЛИ) — возвращает первое truthy

Почему `||` так любят? Потому что им удобно задавать значения по умолчанию. У тебя есть пользовательское имя? Возьми его. Нет? Покажи «Аноним». Всё просто, пока ты не сталкиваешься с валидными falsy-значениями вроде `0` и `""`.

```javascript
const name = userName || 'Аноним';
```

Разбор по строкам:
- В первой строке оператор `||` смотрит на `userName`.
- Если `userName` truthy, выражение сразу вернёт его.
- Если `userName` falsy, оператор перейдёт к правой части и вернёт `'Аноним'`.

Алгоритм `||`: идёт слева направо, возвращает первый truthy-операнд. Если все falsy — возвращает последний.

```javascript
null || 0 || '' || 'Привет' || 42;
null || 0 || '';
```

Разбор по строкам:
- В первой строке `null`, `0` и `''` — falsy, поэтому поиск продолжается, а `'Привет'` — первый truthy, он и вернётся.
- Во второй строке все значения falsy, поэтому оператор отдаст последнее из них, то есть пустую строку `''`.

Это удобно, но только если пустая строка **не считается** валидным значением. Если считается — лучше уже `??`.

#### Оператор `&&` (И) — возвращает первое falsy

`&&` часто используют для условного продолжения вычисления. Идея такая: «делай следующий шаг, только если предыдущий шаг успешен». Это похоже на проход по коридору с несколькими дверями: как только одна дверь закрыта, дальше не идёшь.

```javascript
const user = getUser() && getUser().name;
```

Разбор по строкам:
- В первой строке сначала вызывается `getUser()`.
- Если он вернёт `null`, `undefined` или другое falsy-значение, на этом всё остановится и выражение вернёт это значение.
- Если он вернёт truthy-объект, тогда вычислится правая часть `getUser().name`.

Но тут есть практический нюанс: вызывать `getUser()` дважды может быть плохо, если функция дорогая или имеет побочные эффекты. Лучше сначала сохранить результат в переменную, а потом уже применять `&&`.

Алгоритм `&&`: идёт слева направо, возвращает первый falsy-операнд. Если все truthy — возвращает последний.

```javascript
1 && 2 && 3;
1 && 0 && 3;
```

Разбор по строкам:
- В первой строке все значения truthy, поэтому возвращается последний операнд `3`.
- Во второй строке первым falsy оказывается `0`, на нём вычисление останавливается.

#### Оператор `??` (Nullish Coalescing) — ES2020

Вот теперь добрались до оператора, который очень часто лечит злоупотребление `||`. Проблема `||` в том, что он считает отсутствием **любое** falsy-значение: `0`, `''`, `false`, `NaN`, `null`, `undefined`. Но в реальной жизни многие из них вполне валидны.

`??` заменяет значение по умолчанию только если слева **действительно пусто** в смысле `null` или `undefined`, а не просто falsy.

```javascript
const port = config.port || 3000;
```

Разбор по строкам:
- В первой строке `||` проверяет `config.port` на truthy/falsy.
- Если `config.port === 0`, это falsy-значение.
- Поэтому результатом станет `3000`, хотя `0` мог быть осознанным значением.

Правильнее так:

```javascript
const port = config.port ?? 3000;
```

Разбор по строкам:
- В первой строке `??` проверяет только два случая: `null` и `undefined`.
- Если `config.port === 0`, это валидное значение, и оно будет сохранено.
- Если `config.port === null` или `undefined`, только тогда сработает `3000`.

В Java 8+ аналог — `Optional.ofNullable(port).orElse(3000)`. Не идеально один в один, но по смыслу близко.

#### Optional Chaining `?.` — ES2020

До появления `?.` разработчики писали длинные цепочки проверок на `null` и `undefined`. Это было многословно и утомительно. А утомительный код рождает две беды: его хуже читают и в нём чаще забывают одну из проверок.

```java
String city = null;
if (user != null && user.getAddress() != null) {
    city = user.getAddress().getCity();
}
```

Разбор по строкам:
- В первой строке объявляем переменную `city` и инициализируем `null`.
- Во второй строке вручную проверяем `user` и `user.getAddress()` на `null`.
- В третьей строке только после проверок достаём `city`.

JavaScript-версия короче:

```javascript
const city = user?.address?.city;
```

Разбор по строкам:
- В первой строке `user?.address` означает: если `user` равен `null` или `undefined`, верни `undefined` и не иди дальше.
- Если `user` существует, тогда читаем `address`.
- Затем `?.city` делает такую же проверку для `address`.
- В итоге, если что-то в цепочке отсутствует, исключения не будет, результат просто `undefined`.

Работает с:
- Свойствами: `obj?.prop`
- Методами: `obj?.method()`
- Индексами: `arr?.[0]`

```javascript
const users = null;
users?.length;
users?.[0]?.name;
users?.map(u => u.name);
```

Разбор по строкам:
- В первой строке переменная `users` равна `null`.
- Во второй строке `users?.length` вернёт `undefined`, а не выбросит ошибку.
- В третьей строке попытка взять первый элемент и его имя тоже безопасно даст `undefined`.
- В четвёртой строке метод `map` не будет вызван, потому что слева `null`.

История типичной ошибки до появления `?.`: одно отсутствующее поле в ответе API — и приложение падает с `Cannot read properties of undefined`. Optional chaining не лечит плохую схему данных, но делает код устойчивее.

#### ✅ Проверь себя

**1. Что вернёт `0 || 'default'`?**
- A) 0
- B) "default"
- C) true
- D) false

<details><summary>Ответ</summary>B) "default" — оператор `||` возвращает первый truthy-операнд. `0` — falsy, поэтому переходит к следующему операнду. "default" — truthy, возвращает его.</details>

**2. Что вернёт `0 ?? 'default'`?**
- A) 0
- B) "default"
- C) null
- D) undefined

<details><summary>Ответ</summary>A) 0 — оператор `??` срабатывает только на `null` и `undefined`. `0` — это ни null, ни undefined, поэтому возвращается левый операнд. В этом ключевое отличие `??` от `||`.</details>

**3. Что вернёт выражение `null?.toString()`?**
- A) "null"
- B) TypeError
- C) undefined
- D) ""

<details><summary>Ответ</summary>C) undefined — optional chaining `?.` видит, что слева `null`, и немедленно возвращает `undefined` без вызова метода. Без `?.` был бы TypeError.</details>

**3. Какое значение НЕ является falsy?**
- A) `""`
- B) `0`
- C) `" "` (строка с пробелом)
- D) `null`

<details><summary>Ответ</summary>C) `" "` — строка с пробелом непустая, значит truthy. Единственная falsy-строка — пустая (`""`). Даже `"0"` и `"false"` — truthy!</details>

**4. Что выведет `console.log(1 + '2' + 3)`?**
- A) 6
- B) "123"
- C) "33"
- D) 15

<details><summary>Ответ</summary>B) "123" — операции выполняются слева направо. `1 + '2'` → "12" (конкатенация, т.к. один операнд — строка). Затем `"12" + 3` → "123" (снова конкатенация).</details>

**5. Что вернёт `[] == ![]`?**
- A) true
- B) false
- C) TypeError
- D) undefined

<details><summary>Ответ</summary>A) true — `![]` выполняется первым: массив truthy, отрицание → `false`. Затем `[] == false`: `[]` → `""` → `0`, `false` → `0`. Итого `0 == 0` → true. Это классический пример неинтуитивного приведения.</details>

**6. Какой оператор использовать для проверки «значение — null или undefined»?**
- A) `value === null || value === undefined`
- B) `value == null`
- C) `!value`
- D) A и B — оба правильные, B — короче

<details><summary>Ответ</summary>D) — Оба варианта эквивалентны. `value == null` по спецификации проверяет именно `null` и `undefined` (специальное правило). Вариант C (`!value`) неправильный, потому что он также сработает для `0`, `""`, `NaN`, `false`.</details>

**7. В чём разница между `undefined` и «необъявленная переменная»?**
- A) Никакой разницы
- B) `undefined` — объявлена, но без значения; необъявленная — не существует (ReferenceError)
- C) `undefined` — ошибка; необъявленная — null
- D) Обе возвращают null

<details><summary>Ответ</summary>B) — `let x;` создаёт переменную со значением `undefined`. Обращение к необъявленной переменной `y` (без `let`/`var`/`const`) выбросит ReferenceError. Исключение: `typeof y` для необъявленной вернёт `"undefined"` без ошибки.</details>

#### 🏋️ Мини-упражнение 2: Безопасное получение значения

**Задание**: Напиши функцию `getConfigValue(config, key, defaultValue)`, которая:
- Возвращает `config[key]`, если он не `null` и не `undefined`
- Иначе возвращает `defaultValue`
- Должна корректно работать, когда `config[key]` равен `0`, `""` или `false`

```javascript
// Тесты:
getConfigValue({ port: 0 }, 'port', 3000);      // → 0 (не 3000!)
getConfigValue({ name: '' }, 'name', 'default'); // → '' (не 'default'!)
getConfigValue({ debug: false }, 'debug', true); // → false (не true!)
getConfigValue({}, 'port', 3000);                // → 3000
getConfigValue({ x: null }, 'x', 42);            // → 42
```

<details><summary>Подсказка</summary>Используй оператор `??` (nullish coalescing). Он проверяет только `null`/`undefined`, в отличие от `||`, который считает falsy все «пустые» значения.</details>

<details><summary>Решение</summary>

**Предусловие**: предполагаем, что `config` — всегда объект. Если нужна защита от `null/undefined` config, используй `config?.[key] ?? defaultValue`.

```javascript
function getConfigValue(config, key, defaultValue) {
  return config[key] ?? defaultValue;
}
```

Разбор по строкам:
- В первой строке объявляем функцию с тремя параметрами: объект настроек, ключ и запасное значение.
- Во второй строке читаем `config[key]`.
- Оператор `??` проверяет: если слева `null` или `undefined`, верни `defaultValue`.
- Если слева `0`, `''` или `false`, они сохраняются как есть.
- В третьей строке функция завершается.

Разбор идеи: оператор `??` возвращает правый операнд только если левый — `null` или `undefined`. Значения `0`, `""`, `false` — не `null` и не `undefined`, поэтому возвращаются как есть.

Если бы мы написали `config[key] || defaultValue`, тесты с `0`, `""` и `false` провалились бы — все они falsy для `||`.
</details>

#### 🏋️ Мини-упражнение 3: Определи тип

**Задание**: Напиши функцию `getType(value)`, которая возвращает *реальный* тип значения, исправляя недостатки `typeof`:
- Для `null` → `"null"` (а не `"object"`)
- Для массива → `"array"` (а не `"object"`)
- Для всего остального → результат `typeof`

```javascript
getType(null);       // → "null"
getType([1, 2]);     // → "array"
getType({});         // → "object"
getType(42);         // → "number"
getType('hello');    // → "string"
getType(undefined);  // → "undefined"
```

<details><summary>Подсказка</summary>Используй `=== null` для проверки `null` и `Array.isArray()` для массивов. Порядок проверок важен!</details>

<details><summary>Решение</summary>

```javascript
function getType(value) {
  if (value === null) return 'null';
  if (Array.isArray(value)) return 'array';
  return typeof value;
}
```

Разбор по строкам:
- В первой строке объявляем функцию `getType`.
- Во второй строке первым делом проверяем `value === null`, потому что `typeof null` нам не поможет.
- В третьей строке используем `Array.isArray(value)`, это надёжная проверка на массив.
- В четвёртой строке для всех остальных значений возвращаем обычный `typeof`.

Разбор идеи: порядок важен — проверка на `null` идёт первой, потому что `null` — это и `typeof "object"`, и не массив. `Array.isArray()` — единственный надёжный способ отличить массив от объекта (не используй `instanceof Array` — он ломается между iframe'ами).
</details>

## Контрольные вопросы (открытые)

1. **Почему в современном коде рекомендуется не использовать `var`? Назови минимум три причины.**

<details><summary>Ожидаемый ответ</summary>1) Function scope вместо block scope — переменная «утекает» из блоков `if`/`for`. 2) Hoisting с `undefined` — можно случайно использовать до присваивания без ошибки. 3) Позволяет повторное объявление в том же scope (маскирует ошибки). 4) Не создаёт TDZ — нет защиты от использования до инициализации.</details>

2. **Объясни, почему `typeof null === 'object'` и почему это не исправляют.**

<details><summary>Ожидаемый ответ</summary>В первой реализации JavaScript (1995, Brendan Eich) значения хранились как пара: type tag + data. Для объектов type tag был 0. `null` представлялся как нулевой указатель (все биты — 0), включая type tag. Отсюда `typeof null === 'object'`. Не исправляют, потому что это breaking change: миллионы сайтов проверяют `typeof x === 'object'` и полагаются на то, что `null` проходит эту проверку.</details>

3. **В чём практическая разница между `||` и `??`? Приведи пример, где `||` даёт баг, а `??` — нет.**

<details><summary>Ожидаемый ответ</summary>`||` возвращает правый операнд для любого falsy значения (`0`, `""`, `false`, `null`, `undefined`, `NaN`). `??` — только для `null` и `undefined`. Пример бага: `const count = userInput || 10;` — если пользователь ввёл `0` (валидное значение), `||` заменит его на `10`. С `??`: `const count = userInput ?? 10;` — `0` останется `0`, default сработает только для `null`/`undefined`.</details>

4. **Как JavaScript приводит объект к примитиву? Опиши алгоритм ToPrimitive.**

<details><summary>Ожидаемый ответ</summary>Алгоритм `ToPrimitive`: 1) Если есть метод `[Symbol.toPrimitive]`, вызвать его с hint (`"number"`, `"string"`, `"default"`). 2) Для hint `"number"`: сначала `valueOf()`, если не примитив — `toString()`. 3) Для hint `"string"`: сначала `toString()`, если не примитив — `valueOf()`. 4) Для hint `default` (оператор `+`, `==`): для большинства объектов ведёт себя как `number`, но есть важное исключение — `Date`, который при `default` ведёт себя как `string`. Если ни один метод не вернул примитив — `TypeError`.</details>
