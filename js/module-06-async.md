# Модуль 6: Асинхронность — Event Loop, Promises, async/await

## Для кого этот модуль

Этот модуль для тебя, если ты пришёл из Java/Spring и привык мыслить потоками, `Thread`, `ExecutorService`, `Future` и `CompletableFuture`. В JavaScript логика другая: главный поток один, но задач «в воздухе» может быть очень много. Именно поэтому тема асинхронности — самая важная на собеседованиях: она показывает, понимаешь ли ты язык на уровне механики, а не только на уровне красивого синтаксиса.

Здесь мы будем идти не по принципу «вот API, запомни». Мы будем идти по принципу наставника у доски: сначала **зачем это вообще появилось**, потом **какую боль решает**, потом **как это работает внутри**, и только после этого — код. Твоя цель после модуля не просто написать `await`, а видеть внутренний маршрут: что сейчас в стеке, что ушло в очередь, что именно произойдёт на следующем тике Event Loop.

Если коротко: этот модуль надо не просто прочитать. Его надо прожевать, проговорить, прорисовать и несколько раз перепроверить примерами. **СТОП! Это самая важная тема на собеседованиях.** Очень часто кандидат нормально пишет CRUD, но сыпется на вопросе «а почему здесь такой порядок вывода?».

## Каким должен быть результат

После модуля ты сможешь:

- объяснить, почему однопоточность JavaScript не делает язык слабым;
- нарисовать Event Loop по памяти и проговорить его без шпаргалки;
- предсказывать порядок вывода в задачах с `setTimeout`, `Promise.resolve().then(...)`, `queueMicrotask(...)` и `async/await`;
- объяснять, почему `await` не блокирует весь поток;
- уверенно сравнивать `Promise.all`, `Promise.allSettled`, `Promise.race` и `Promise.any`;
- объяснять асинхронность Java-разработчику через аналогии с `CompletableFuture` и очередями задач.

## Главная установка модуля

**JavaScript на main thread не выполняет твой код параллельно. Он координирует короткие куски работы через Event Loop. Это не многопоточность JVM, а cooperative scheduling: если ты долго держишь поток, всё остальное ждёт.**

Запомни этот тезис как главный рефрен всего модуля. Мы ещё много раз будем к нему возвращаться в блоках про таймеры, промисы, `await` и микрозадачи. Если этот тезис не встал в голову, все остальные детали начинают казаться магией.

## Блок 1: Однопоточность JavaScript — почему это не проблема

### 6.1. Один повар, один кассир, один главный поток

Когда человек слышит «однопоточный язык», у него часто возникает тревога: «как же он вообще может обслуживать современное приложение?». Это понятная реакция. Особенно если ты пришёл из Java, где почти любой разговор про производительность быстро приводит к потокам, пулам, блокировкам, `synchronized`, `CompletableFuture` и борьбе за shared state.

Но у JavaScript другая философия. Он как маленькая, но очень хорошо организованная кухня. Представь ресторан, где в зоне выдачи стоит один шеф. Он физически не может **одновременно** двумя руками оформлять три блюда. Зато он может поставить мясо в духовку, передать мытьё посуды помощнику, попросить бариста приготовить кофе, а сам в это время оформить салат. То есть сам шеф не размножается на три потока, но система в целом обслуживает много процессов сразу.

Есть ещё бытовая аналогия. Один человек собирает документы для визы. Он не может в одну и ту же миллисекунду и стоять в очереди, и разговаривать с банком, и печатать анкету. Но он может подать запрос, дождаться звонка, пока ждёт — сделать следующий шаг, потом вернуться. Асинхронность — это искусство **не стоять без дела в ожидании**. Вот почему один поток — не катастрофа.

Для Java-разработчика полезна третья аналогия: представь один worker-thread, который исполняет только твой JS-код, а все ожидания I/O и системные штуки обслуживаются вокруг него инфраструктурой среды. Это не точная модель JVM, но она помогает не пытаться искать второй «секретный JS-поток» там, где его нет.

### Что именно значит «один поток»

Однопоточность JavaScript означает: на главном потоке твой пользовательский код исполняется по одной инструкции за раз. Сейчас выполняется одна функция. Потом следующая. Потом следующая. Две строки JavaScript не исполняются одновременно на одном и том же main thread.

Важно не перепутать две разные вещи:

1. **Параллелизм** — два куска твоего JS-кода реально идут одновременно.
2. **Конкурентность** — у приложения много начатых работ, но выполнение координируется так, чтобы не простаивать.

JavaScript на main thread силён именно во втором. Он хорошо справляется с ожиданием сети, таймеров, кликов, завершения файловых операций, готовности данных. Пока одна операция «едет», твой стек свободен для других коротких шагов.

- [MDN: Event loop](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Event_loop)

### Где новички путаются

Новички часто думают так: «Если `fetch` асинхронный, значит у запроса есть отдельный JavaScript-поток». Нет. У запроса есть работа внешней среды: браузера, Node.js, системных библиотек, сетевого стека. Но когда результат готов, продолжение всё равно вернётся на главный поток в виде новой задачи.

Это важный сдвиг мышления. В Java ты часто говоришь: «я создам ещё один поток, чтобы не блокировать текущий». В JavaScript ты чаще говоришь: «я запущу асинхронную операцию и быстро освобожу стек, чтобы Event Loop потом вернул мне продолжение».

### Простой пример

```javascript
console.log('1. Старт');

setTimeout(() => {
  console.log('3. Таймер сработал');
}, 1000);

console.log('2. Конец');
```

Разбор по шагам:

1. Движок начинает выполнять файл сверху вниз.
2. `console.log('1. Старт')` выполняется сразу и печатает первую строку.
3. `setTimeout(...)` **не** запускает колбэк прямо сейчас. Он только регистрирует таймер во внешней среде.
4. Движок идёт дальше и выполняет `console.log('2. Конец')`.
5. Только спустя минимум 1000 мс среда получает право поставить колбэк таймера в очередь задач.
6. Когда стек освободится и Event Loop дойдёт до этой задачи, тогда напечатается `3. Таймер сработал`.

Обрати внимание на главную мысль: код не «спал» одну секунду внутри стека. Он быстро дошёл до конца синхронной части и освободил поток. В Java по смыслу это ближе к `ScheduledExecutorService`, чем к `Thread.sleep()`.

### Почему проблема не в одном потоке, а в долгом удержании потока

Плохая новость не в том, что поток один. Плохая новость в том, что ты можешь занять его надолго. Если написал тяжёлый цикл или сложные вычисления прямо на main thread, интерфейс зависнет, клики перестанут обрабатываться, таймеры начнут ждать, а анимации заикаться.

Представь того же шефа на кухне, который вместо делегирования внезапно решил сам полчаса точить ножи. В этот момент неважно, сколько вокруг заказов, помощников и приборов. Узкое место — он сам. Точно так же в JavaScript узкое место — твой длинный синхронный кусок.

### Частая interview-мысль

На собеседовании сильный ответ звучит так: **«JavaScript однопоточен на main thread, но не блокируется на I/O, потому что ожидание обслуживает среда, а продолжения возвращаются через Event Loop. Опасность не в одном потоке как таковом, а в длинном синхронном CPU-bound коде»**.

Если тебе всё же нужен настоящий параллелизм (CPU-intensive задачи), существуют **Web Workers** (в браузере) и **worker_threads** (в Node.js). Они запускают JS в отдельном потоке с собственным Event Loop, но обмениваются данными только через сообщения — shared memory ограничена `SharedArrayBuffer`.

### История разработчика: как ломают интерфейс

Мидл-разработчик Денис сделал поиск по большому массиву в обработчике `input`. Каждый символ запускал тяжёлый синхронный фильтр на сотни тысяч записей. «Но это же всего лишь цикл», — думал он. На демо поле ввода начало лагать, потому что каждый символ держал поток. Проблема была не в `input`, не в браузере и не в React. Проблема была в длинной синхронной работе на main thread.

## Блок 2: 🎯🎯🎯 Event Loop — главная interview тема

### 6.2. Диспетчер, который решает «кто следующий»

**СТОП! Это самая важная тема на собеседованиях.** Если ты уверенно объясняешь Event Loop, ты производишь впечатление человека, который реально понимает JavaScript, а не просто копирует `await` из Stack Overflow.

Event Loop — это механизм координации, который решает, **когда** исполнить следующую порцию JavaScript-кода. Самый частый провал на собеседовании выглядит так: кандидат знает, что есть «очередь» и «что-то про промисы», но не может чётко проговорить порядок. А интервьюеру нужен именно порядок.

Представь диспетчера на вокзале. Поезда не едут все разом в одну и ту же секунду. У каждого есть состояние: один уже на платформе, другой только подходит, третий экстренный, четвёртый стоит в резерве. Диспетчер не создаёт поезда и не чинит рельсы. Он координирует, кого пустить следующим. Event Loop — такой же диспетчер для задач.

Теперь любимая метафора этого модуля — **ресторанная кухня**. Это очень сильная визуализация для Event Loop. В зале заказы принимаются постоянно. На кухне один главный повар, который может делать только одну активную операцию руками в текущий момент. Рядом есть:

- стол активной работы — это **call stack**;
- обычный список заказов — это **macrotask queue**;
- короткие срочные записки от менеджера — это **microtask queue**;
- момент выдачи блюд в зал — это похожий образ для **rendering**.

И ключевое правило кухни такое: пока повар режет текущий заказ на рабочем столе, он не берёт новый. Когда закончил текущую порцию работы, он сначала разбирает **все короткие срочные записки**, и только потом берёт следующий обычный заказ. Вот тебе Event Loop в одном абзаце.

### Четыре сущности, без которых дальше нельзя

#### 1. Call stack

**Call stack** — это стек вызовов. Активная функция кладётся на вершину. Когда функция вызывает другую функцию, сверху кладётся новый фрейм. Когда функция завершилась, фрейм снимается.

Метафора из жизни: стопка тарелок в буфете. Последнюю положенную тарелку ты снимаешь первой. Метафора из офиса: папка на столе сверху — это текущая задача, под ней лежат те, к которым вернёшься позже.

Java-аналогия: это похоже на обычный стек вызовов внутри одного потока JVM. Здесь почти нет сюрпризов — сюрпризы начинаются, когда стек пустеет, и Event Loop получает право выбрать новую задачу.

#### 2. Macrotask queue

Это обычная очередь задач. Сюда попадают таймеры, события, некоторые сообщения среды. В бытовой аналогии это основная очередь клиентов в банке. В ресторанной аналогии — длинный список обычных заказов.

Сюда, например, часто думают о `setTimeout`, `setInterval`, кликах и прочих host-задачах. Важно: таймер не прыгает в стек сам по себе. Он сначала получает право стать готовым, потом попадает в очередь, и только потом Event Loop когда-нибудь возьмёт его оттуда.

#### 3. Microtask queue

Это очередь микрозадач повышенного приоритета. Сюда попадают обработчики промисов (`then`, `catch`, `finally`) и `queueMicrotask(...)`.

Метафора: менеджер ресторана периодически подсовывает короткие VIP-записки: «перед тем как брать следующий обычный заказ, закрой вот эти маленькие срочные пункты». В офисе это выглядит как короткие стикеры от директора: «подпиши акт, отправь письмо, подтверди оплату — и только потом бери нового клиента».

Самая важная фраза этого блока: **после завершения текущего кода или macrotask движок вычищает всю очередь microtasks**. Не одну задачу. Не «одну-две». Всю.

#### 4. Rendering

В браузере есть ещё перерисовка интерфейса. Условно: после выполнения порции работы браузер получает шанс обновить экран. Это не «обычный JavaScript-код», а отдельная часть жизненного цикла среды.

Метафора: блюда уже готовы, но их ещё надо вынести в зал. Или табло на вокзале уже обновили внутренне, но его нужно показать пассажирам. Если ты бесконечно держишь главный поток или без конца подбрасываешь microtasks, экран может долго не обновляться.

### Детальная схема Event Loop

```text
┌───────────────────────────────────────────────────────────────────┐
│                      Среда исполнения (Browser)                   │
│                                                                   │
│  Host APIs / Web APIs                                             │
│  ┌──────────────┐  ┌──────────────┐  ┌─────────────────────────┐  │
│  │ setTimeout   │  │ fetch / XHR  │  │ DOM events / click / IO│  │
│  └──────┬───────┘  └──────┬───────┘  └────────────┬────────────┘  │
│         │                 │                       │               │
│         ▼                 ▼                       ▼               │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │                 Task Queue / Macrotasks                    │  │
│  │   [timer] [click handler] [message] [interval tick] ...   │  │
│  └─────────────────────────────────────────────────────────────┘  │
│                                                                   │
│  ┌─────────────────────────────────────────────────────────────┐  │
│  │                    Microtask Queue                         │  │
│  │   [promise then] [catch] [finally] [queueMicrotask] ...   │  │
│  └─────────────────────────────────────────────────────────────┘  │
└───────────────────────────────────────────────────────────────────┘
                              │
                              ▼
                ┌──────────────────────────────────┐
                │           Event Loop             │
                │ 1. Ждёт, пока stack пуст         │
                │ 2. Берёт 1 macrotask             │
                │ 3. Выполняет её до конца         │
                │ 4. Полностью чистит microtasks   │
                │ 5. Даёт шанс rendering           │
                │ 6. Повторяет цикл                │
                └────────────────┬─────────────────┘
                                 │
                                 ▼
                      ┌──────────────────────┐
                      │      Call Stack      │
                      │ main()               │
                      │ foo()                │
                      │ console.log()        │
                      └──────────────────────┘
```

Разбор по шагам:

1. Верхняя часть схемы — это среда исполнения, то есть то, что умеет ждать таймеры, сеть, события и прочие внешние штуки.
2. Слева и справа находятся источники задач: таймеры, сеть, клики, сообщения.
3. Готовые обычные задачи попадают в **macrotask queue**.
4. Короткие продолжения промисов и `queueMicrotask` попадают в **microtask queue**.
5. Event Loop не запускает всё подряд. Он ждёт, пока **call stack станет пустым**.
6. Когда стек пуст, берётся **одна** macrotask.
7. После её завершения Event Loop обязан вычистить **все** microtasks.
8. Затем браузер может сделать rendering.
9. Потом цикл повторяется.

Если тебе сложно, не спеши дальше. Перерисуй эту схему от руки. Серьёзно. Один рисунок, нарисованный самостоятельно, даёт больше понимания, чем пять пассивных прочтений.

### Белая доска: как рассказывать Event Loop на интервью

На whiteboard хороший ответ можно строить так:

1. «Вот call stack — здесь исполняется текущий синхронный код».  
2. «Вот host APIs / среда — она умеет ждать таймеры, сеть и события».  
3. «Вот macrotask queue — сюда приходят обычные задачи».  
4. «Вот microtask queue — сюда приходят продолжения промисов и `queueMicrotask`».  
5. «Пока стек не пуст, никакие очереди не обслуживаются».  
6. «После завершения текущего кода или macrotask движок вычищает всю очередь microtasks».  
7. «Только потом возможен rendering и следующая macrotask».  

Это тот случай, где нужна почти заученная формулировка. Не бойся, это не зубрёжка ради зубрёжки. Это базовый протокол работы языка.

### Первый конкретный пример

```javascript
console.log('A');

setTimeout(() => console.log('B'), 0);
Promise.resolve().then(() => console.log('C'));
console.log('D');
```

Разбор по шагам:

1. Выполняется `console.log('A')`, поэтому сразу видим `A`.
2. `setTimeout(..., 0)` регистрирует таймер. Колбэк **не** идёт в стек прямо сейчас.
3. `Promise.resolve().then(...)` создаёт microtask для `console.log('C')`.
4. Выполняется `console.log('D')`, поэтому видим `D`.
5. Синхронный код файла закончился, стек пуст.
6. Event Loop проверяет microtask queue и выполняет `console.log('C')`.
7. Только после полного опустошения microtask queue можно взять следующую macrotask.
8. Берётся колбэк таймера и печатает `B`.

Итоговый порядок: `A`, `D`, `C`, `B`.

💡 **Event Loop: браузер vs Node.js**

Базовая модель одинакова, но есть отличия:
- **Браузер**: microtasks → rendering → macrotasks. Есть `requestAnimationFrame` между макрозадачами и отрисовкой
- **Node.js**: phases (`timers → pending → idle → poll → check → close`). `setImmediate` ≈ следующий tick `poll` phase
- **Для собеседования**: обычно спрашивают браузерную модель. Главное — microtask queue (`Promise`) опустошается **полностью** перед следующей macrotask

### Почему `setTimeout(..., 0)` не означает «немедленно»

Это один из самых популярных interview-вопросов. Нулевая задержка не означает мгновенное исполнение. Она означает: **поставь задачу в обычную очередь как можно раньше, когда стек освободится и когда дойдёт очередь до этой macrotask**.

Метафора из поликлиники: ты взял талончик «без ожидания», но перед кабинетом всё равно сидят люди, и врач сейчас занят. Метафора из ресторана: официант сказал «передам на кухню срочно», но повар всё равно не разорвётся на две части.

Java-аналогия: это не `runNow()`, а скорее «зарегистрируй задачу, чтобы worker-thread взял её, когда освободится». Только здесь worker-thread — это тот же главный поток JS.

### Run-to-completion — правило, без которого всё ломается

В JavaScript задача, попавшая в стек, выполняется по модели **run-to-completion**. Это значит: её не прервут посередине, чтобы вставить другой JS-колбэк. Если задача началась, она дойдёт до конца, пока сама не вернёт управление.

Представь шефа, который уже начал нарезать рыбу. Менеджер не врывается каждую миллисекунду и не говорит: «Стоп, теперь нарежь половину салата, потом вернёшься». Нет. Текущий маленький кусок работы доводится до конца. Поэтому длинный синхронный цикл — это реальная беда.

### Tick за tick: разберём реальный сценарий как на доске

```javascript
console.log('script start');

setTimeout(() => {
  console.log('timer 1');
}, 0);

Promise.resolve()
  .then(() => {
    console.log('promise 1');
  })
  .then(() => {
    console.log('promise 2');
  });

setTimeout(() => {
  console.log('timer 2');
}, 0);

console.log('script end');
```

Разбор по шагам:

1. Стартует основной скрипт. Это текущая macrotask — условно «выполнение файла».
2. Печатается `script start`.
3. Первый `setTimeout` регистрирует колбэк `timer 1` для будущей macrotask.
4. `Promise.resolve().then(...)` ставит в microtask queue обработчик `promise 1`.
5. Второй `.then(...)` пока **не ставится** сразу. Он появится только когда завершится предыдущий шаг цепочки.
6. Второй `setTimeout` регистрирует `timer 2` как ещё одну будущую macrotask.
7. Печатается `script end`.
8. Синхронный код закончен. Стек пуст.
9. Event Loop смотрит microtask queue. Там лежит `promise 1`.
10. Выполняется `promise 1`, печатается `promise 1`.
11. Завершение первого `then` делает следующий `then` готовым, и он ставится в microtask queue.
12. Event Loop продолжает чистить microtasks, потому что правило говорит «очистить всю очередь».
13. Выполняется `promise 2`, печатается `promise 2`.
14. Очередь microtasks пуста.
15. Теперь можно перейти к macrotask queue. Там первым стоит `timer 1`.
16. Выполняется `timer 1`, печатается `timer 1`.
17. После macrotask движок снова проверяет microtask queue. Она пуста.
18. Берётся следующая macrotask `timer 2`.
19. Выполняется `timer 2`, печатается `timer 2`.

Итоговый порядок: `script start`, `script end`, `promise 1`, `promise 2`, `timer 1`, `timer 2`.

### Ресторанная кухня: глубокая аналогия для Event Loop

Давай развернём метафору кухни глубже, потому что она реально помогает.

- **Call stack** — стол, на котором повар прямо сейчас режет, жарит, оформляет.
- **Macrotask queue** — общий список заказов: борщ, паста, десерт, напиток.
- **Microtask queue** — короткие записки: «проверить соль в супе», «подписать чек поставщика», «добавить соус к уже готовому блюду».
- **Rendering** — момент, когда блюдо выносят в зал и клиент реально видит результат.

Как работает кухня:

1. Пока на столе идёт работа над текущим блюдом, новое блюдо не начинают.
2. Когда этот кусок закончен, менеджер говорит: «Сначала быстро закрой все короткие срочные записки».
3. Повар быстро выполняет все эти маленькие действия.
4. Только потом берёт следующий полноценный заказ.
5. Между этими этапами зал получает шанс увидеть новые блюда на выдаче.

Теперь смотри, почему microtasks опасны. Если менеджер постоянно подсовывает всё новые «очень короткие, но срочные» бумажки, повар может долго не перейти к новым обычным заказам. Это и есть starvation — голодание менее приоритетных задач.

### Rendering и голодание

В браузере после порции JS-работы должен происходить rendering. Но если ты очень долго не отдаёшь управление или бесконечно накидываешь microtasks, пользователь может не увидеть обновление экрана вовремя. Ему кажется, что приложение зависло, хотя «где-то внутри» оно занято.

Представь табло на вокзале. Диспетчер постоянно получает срочные бумажки и не успевает нажать кнопку обновления табло. Внутри станции уже всё поменялось, а пассажир всё ещё видит старую картинку. Тот же эффект бывает в браузере.

### Java-аналогия

По образу Event Loop похож на один поток из `ExecutorService`, который берёт задачи из очереди. Но это только образ, а не точное совпадение. В JVM поток может быть вытеснен ОС, там много деталей планировщика, а в JavaScript колбэк обычно выполняется по модели run-to-completion.

По Promises можно думать примерно как о сильно упрощённом родственнике `CompletableFuture`. По `await` — как о синтаксической обёртке над цепочкой продолжений. Но если ты начнёшь буквально переносить модель Java-потоков на JS, быстро запутаешься.

### Common mistakes — истории разработчиков

**История 1.** Разработчик говорит: «Промисы работают в отдельном потоке». Ему кажется, что `.then()` запускается «где-то параллельно». На практике это ломает понимание порядка вывода.

**История 2.** Разработчица уверена, что `setTimeout(..., 0)` — это «сразу после текущей строки». Поэтому она ждёт, что таймер опередит `.then()`. На собеседовании такой ответ почти всегда подсвечивает слабое понимание очередей.

**История 3.** Мидл знает про Event Loop, но забывает, что microtask queue очищается целиком. Из-за этого он неправильно решает задачи, где microtask порождает новую microtask.

## Блок 3: setTimeout и setInterval — macrotasks

### 6.3. Таймер — это не секундомер для CPU

Когда ты только начинаешь писать асинхронный код, `setTimeout` кажется почти магическим: «скажи число миллисекунд, и функция запустится через столько времени». Но это опасное упрощение. Таймер не даёт железной гарантии старта в конкретную миллисекунду. Он даёт гарантию только одного типа: **раньше указанного времени — не запущу**.

Представь талончик в поликлинике. На нём написано 14:00. Это не значит, что в 14:00 ты уже сидишь в кресле врача. Это значит, что **не раньше** 14:00 тебе вообще можно рассчитывать на приём, а дальше всё зависит от очереди и занятости кабинета. С `setTimeout` ровно так же.

В ресторанной метафоре таймер — это как духовка с сигналом. Сигнал говорит: «блюдо можно доставать». Но если шеф в этот момент обеими руками украшает другой заказ, он не телепортируется к духовке. Блюдо подождёт, пока освободится стол активной работы.

Java-аналогия: таймер не похож на жёсткий real-time scheduler. Он скорее похож на постановку задачи в очередь с минимальной задержкой, после которой задача **может стать готовой**, но всё ещё зависит от доступности основного потока.

```javascript
setTimeout(() => console.log('timer'), 0);

for (let i = 0; i < 1_000_000_000; i++) {
  // долгий цикл
}
```

Разбор по шагам:

1. `setTimeout(..., 0)` регистрирует таймер с минимальной задержкой.
2. Затем начинается длинный синхронный цикл.
3. Пока цикл крутится, стек занят.
4. Даже если таймер уже готов, его колбэк не может прервать этот цикл.
5. Только когда цикл закончится и стек освободится, таймер получит шанс выполниться.

Вывод: `0` миллисекунд — не магическая кнопка обхода очереди. Это всего лишь «поставь задачу как можно раньше после освобождения потока».

### Почему `setInterval` коварнее, чем кажется

`setInterval` звучит удобно: «запускай функцию каждые N миллисекунд». Но в реальности он часто создаёт ложное чувство контроля. Если операция внутри интервала выполняется дольше, чем сам интервал, новые тики не начинают идти параллельно на том же JS-потоке. Они просто начинают опаздывать, а логика может наползать сама на себя по смыслу.

Представь охранника, который должен каждые 5 минут обходить здание, но один обход занимает 8 минут. Формально расписание красивое, но в жизни оно уже сломано. Так же и с `setInterval`.

```javascript
setInterval(() => {
  console.log('tick start');
  const start = Date.now();

  while (Date.now() - start < 1500) {
    // блокирующая работа 1.5 секунды
  }

  console.log('tick end');
}, 1000);
```

Разбор по шагам:

1. Интервал просит запускать колбэк каждые 1000 мс.
2. Внутри колбэка происходит блокирующая работа на 1500 мс.
3. Пока колбэк выполняется, следующий JS-колбэк не может стартовать на этом потоке.
4. В результате тики будут не «идеально раз в секунду», а с отставанием.
5. При сложной логике это приводит к накоплению задержек и странному поведению.

- [javascript.info: setTimeout и setInterval](https://learn.javascript.ru/settimeout-setinterval)

### Почему для polling часто лучше рекурсивный `setTimeout`

Когда ты делаешь polling — то есть периодически спрашиваешь сервер «ну что там?» — хочется написать `setInterval`. Но часто безопаснее рекурсивный `setTimeout`: новый цикл планируется только **после завершения предыдущего**.

Это как доставка курьером. `setInterval` — ты отправляешь нового курьера каждые 10 минут, даже если прошлый ещё не вернулся. Рекурсивный `setTimeout` — ты отправляешь следующего курьера только когда предыдущий закончил маршрут и отчитался.

```javascript
function poll() {
  setTimeout(async () => {
    console.log('Проверяем статус');
    await fetchStatus();
    poll();
  }, 2000);
}

poll();
```

Разбор по шагам:

1. `poll()` запускает таймер на 2 секунды.
2. Когда таймер срабатывает, начинается асинхронная проверка статуса.
3. Следующий вызов `poll()` происходит **после** завершения `await fetchStatus()`.
4. Это защищает тебя от логического наложения нескольких проверок одновременно.
5. Такой стиль особенно полезен, когда длительность запроса непредсказуема.

📚 **Источники:**
- [MDN: setTimeout()](https://developer.mozilla.org/ru/docs/Web/API/Window/setTimeout)
- [javascript.info: setTimeout и setInterval](https://learn.javascript.ru/settimeout-setinterval)

### requestAnimationFrame — таймер для визуальных обновлений

`setTimeout` не гарантирует выполнение перед следующим кадром отрисовки. Для анимаций и визуальных обновлений существует `requestAnimationFrame`:

```javascript
let position = 0;

function animate() {
  element.style.left = position + 'px';
  position += 2;

  if (position < 300) {
    requestAnimationFrame(animate); // вызовется перед следующей отрисовкой (~60fps)
  }
}

requestAnimationFrame(animate);
```

**Почему не `setTimeout` для анимаций?**
- `requestAnimationFrame` синхронизирован с refresh rate монитора
- Автоматически приостанавливается, когда вкладка неактивна (экономит CPU)
- Гарантирует, что изменения DOM применятся перед отрисовкой кадра

**Аналогия**: `setTimeout` — это будильник на произвольное время. `requestAnimationFrame` — это «позови меня ровно перед тем, как художник начнёт рисовать следующий кадр».

📚 **Источники:**

## Блок 4: Callback Hell и зачем нужны промисы

### 6.4. Лестница вправо как запах кода

Чтобы по-настоящему понять Promises, надо сначала почувствовать боль, которую они лечат. Иначе промис кажется просто «ещё одним способом написать асинхронность». На самом деле он появился не от хорошей жизни.

До промисов асинхронный код часто строили на callback'ах: «когда закончишь — вызови эту функцию». Идея сама по себе нормальная. Ты оставляешь номер мастеру по ремонту, и он звонит, когда всё готово. Или отдаёшь костюм в химчистку, а потом получаешь сообщение, что можно забирать. В этом нет ничего плохого.

Проблема начинается, когда шагов много. Нужно загрузить пользователя, потом его заказы, потом рекомендации по этим заказам, потом обновить интерфейс, а на каждом этапе ещё обработать ошибки. Код начинает сползать вправо, а мозг устаёт держать уровни вложенности.

```javascript
loadUser(userId, (userError, user) => {
  if (userError) return handleError(userError);

  loadOrders(user.id, (ordersError, orders) => {
    if (ordersError) return handleError(ordersError);

    loadRecommendations(orders, (recError, recs) => {
      if (recError) return handleError(recError);
      renderPage(user, orders, recs);
    });
  });
});
```

Разбор по шагам:

1. Сначала вызывается `loadUser`.
2. Когда пользователь загружен, внутри его callback вызывается `loadOrders`.
3. Когда заказы загружены, внутри следующего callback вызывается `loadRecommendations`.
4. Только на самом глубоком уровне вызывается `renderPage`.
5. На каждом уровне отдельно проверяется ошибка.
6. Визуально код уезжает вправо, а логика становится сложной для чтения и сопровождения.

📚 **Источники:**
- [javascript.info: Callbacks](https://learn.javascript.ru/callbacks)
- [w3schools: Callback Functions](https://www.w3schools.com/js/js_callback.asp)

### Почему callback hell раздражает не только глазами

Проблема не только в отступах. Глубокая вложенность мешает композиции. Ты уже не можешь легко вернуть «будущий результат» из функции, объединить несколько процессов, централизованно провести ошибки через одну трубу. Каждый новый шаг как будто встраивается в лабиринт.

Ещё одна боль — контроль ошибок. В callback-стиле ошибка часто передаётся первым аргументом, и тебе приходится на каждом уровне руками повторять один и тот же шаблон `if (err) return ...`. Код превращается в офис, где на каждом этаже стоит отдельный охранник с одним и тем же списком вопросов.

Java-разработчику это напоминает каскад вложенных слушателей, анонимных классов или колбэков без хорошей композиции. Не то чтобы невозможно, но тяжело и хрупко.

📚 **Источники:**
- [javascript.info: Callbacks](https://learn.javascript.ru/callbacks)
- [w3schools: Callback Functions](https://www.w3schools.com/js/js_callback.asp)

### Promise как идея, а не только как API

**Promise** — это объект, представляющий будущий результат. Это очень важная мысль. Промис — не «функция, которая ждёт». Промис — это **контейнер судьбы результата**: либо когда-то придёт значение, либо когда-то придёт ошибка.

Метафора 1: номерок из химчистки. У тебя ещё нет вещи в руках, но у тебя есть официальный документ, с которым ты потом либо получишь костюм, либо услышишь, что возникла проблема.

Метафора 2: номер заказа в доставке. Самой еды ещё нет, но по номеру заказа ты можешь следить, объединять статусы, показывать пользователю прогресс.

Java-аналогия: по смыслу Promise похож на маленького родственника `CompletableFuture`. Не идентичен, но концептуально близок: это не финальный результат, а обещание будущего результата.

📚 **Источники:**
- [MDN: Promise](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise)
- [javascript.info: Promise basics](https://learn.javascript.ru/promise-basics)
- [ECMA-262: Promise Objects](https://tc39.es/ecma262/#sec-promise-objects)

### Зачем это меняет архитектуру кода

Как только результат становится объектом, который можно вернуть наружу, передать дальше, объединить с другими и централизованно обработать, код становится площе. Ты перестаёшь думать «кого куда вложить» и начинаешь думать «какую цепочку шагов построить».

Вот почему Promises — это не просто «красивее callbacks». Это другой способ организации асинхронного потока данных и ошибок.

📚 **Источники:**
- [MDN: Promise](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise)
- [javascript.info: Promise basics](https://learn.javascript.ru/promise-basics)

### Common mistakes — истории разработчиков

**История 1.** Разработчик говорит: «Я и с callback'ами справляюсь, зачем мне Promises?». Потом в коде появляется пять вложенных уровней, параллельный запрос внутри третьего колбэка и дублирование обработки ошибок в четырёх местах.

**История 2.** Джун думает, что Promise нужен только чтобы писать `await`. А потом не понимает, что без понимания самого Promise он не поймёт ни `await`, ни порядок исполнения, ни маршрутизацию ошибок.

**1. В чём главный минус Callback Hell?**
- A) Колбэки медленные
- B) Глубокая вложенность и плохая композиция
- C) Колбэки запрещены в браузере
- D) Колбэки всегда текут по памяти

<details><summary>Ответ</summary>B) Проблема не в самой callback-функции, а в читабельности и управляемости сложной цепочки.</details>

- D) Он автоматически отменяет ошибки

## Блок 5: 🎯🎯 Promises — создание, then/catch/finally, цепочки

### 6.5. Обещание будущего результата — с самого нуля

Теперь начинаем Promises **с нуля**, как будто человек вообще их ещё не понимает. Это правильно, потому что многие запоминают внешний синтаксис и пропускают саму концепцию.

Promise — это объект с будущим результатом асинхронной операции. Пока результат не готов, промис находится в ожидании. Когда успех наступил — промис выполнен успешно. Когда произошла ошибка — промис отклонён. Самое важное: промис описывает **судьбу результата**, а не «идущий сейчас поток».

Метафора 1: заказ в интернет-магазине. Статус может быть «ожидает обработки», «доставлен», «отменён».

Метафора 2: экзаменационная работа. Пока проверяют — `pending`. Если проверили и зачли — `fulfilled`. Если обнаружили нарушение — `rejected`.

Java-аналогия: это напоминает `CompletableFuture`, где значение ещё не пришло, но у тебя уже есть объект, с которым можно работать дальше.

📚 **Источники:**
- [MDN: Promise](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise)
- [javascript.info: Promise basics](https://learn.javascript.ru/promise-basics)
- [ECMA-262: Promise Objects](https://tc39.es/ecma262/#sec-promise-objects)
- [w3schools: Promise](https://www.w3schools.com/js/js_promise.asp)

### Три состояния Promise

У промиса есть три состояния:

- **pending** — результат ещё неизвестен;
- **fulfilled** — операция завершилась успешно;
- **rejected** — операция завершилась ошибкой.

И здесь очень важна мысль про **необратимость**. После того как промис перешёл из `pending` в `fulfilled` или `rejected`, он больше не меняет судьбу. Это как отправленная посылка: ты не можешь сделать так, чтобы уже доставленная посылка снова стала «в ожидании».

📚 **Источники:**
- [MDN: Promise](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise)
- [javascript.info: Promise basics](https://learn.javascript.ru/promise-basics)
- [ECMA-262: Promise Objects](https://tc39.es/ecma262/#sec-promise-states)

### Создание промиса

```javascript
const delay = new Promise((resolve) => {
  setTimeout(() => resolve('Готово'), 1000);
});
```

Разбор по шагам:

1. Создаётся объект Promise.
2. В конструктор передаётся функция-исполнитель — её называют **executor**.
3. `executor` запускается **сразу**, синхронно в момент создания промиса.
4. Внутри `executor` регистрируется таймер.
5. Когда таймер сработает, будет вызван `resolve('Готово')`.
6. После `resolve` промис перейдёт из `pending` в `fulfilled` со значением `'Готово'`.

📚 **Источники:**
- [MDN: Promise](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise)
- [javascript.info: Promise basics](https://learn.javascript.ru/promise-basics)

### Ловушка: executor синхронный

Это очень любят спрашивать на интервью. Люди видят `new Promise(...)` и думают, что всё внутри него «асинхронное по определению». Нет. Асинхронность не в факте создания промиса, а в том, что внутри него происходит.

```javascript
console.log('A');
new Promise((resolve) => {
  console.log('B');
  resolve();
});
console.log('C');
```

Разбор по шагам:

1. Печатается `A`.
2. Создаётся Promise.
3. `executor` запускается сразу, поэтому печатается `B`.
4. Внутри `executor` вызывается `resolve()`. Это меняет состояние промиса, но не запускает будущие `then` синхронно прямо здесь.
5. После завершения конструктора выполняется `console.log('C')`.

Итоговый порядок: `A`, `B`, `C`.

📚 **Источники:**
- [MDN: Promise()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/Promise)
- [javascript.info: Promise basics](https://learn.javascript.ru/promise-basics)

### Почему `then` не выполняется синхронно даже на resolved promise

Вот это место ломает голову многим. Кажется логичным: «если промис уже resolved, почему бы не выполнить `then` сразу?». Ответ: потому что тогда модель стала бы хаотичной. Движок гарантирует, что обработчики промисов идут асинхронно через microtask queue. Это делает поведение предсказуемым.

```javascript
console.log('1');

Promise.resolve('ok').then((value) => {
  console.log('2', value);
});

console.log('3');
```

Разбор по шагам:

1. Печатается `1`.
2. `Promise.resolve('ok')` создаёт уже fulfilled promise.
3. Но обработчик `.then(...)` не выполняется немедленно; он ставится в microtask queue.
4. Печатается `3`.
5. Синхронный код завершён.
6. Выполняется microtask и печатает `2 ok`.

Итоговый порядок: `1`, `3`, `2 ok`.

📚 **Источники:**
- [MDN: Promise.prototype.then()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/then)
- [javascript.info: Promise chaining](https://learn.javascript.ru/promise-chaining)
- [javascript.info: Microtask queue](https://learn.javascript.ru/microtask-queue)

### `then`: цепочка шагов

`then` — это способ сказать: «когда значение придёт, сделай следующий шаг». Очень важно мыслить не как «вызови потом вот эту функцию», а как «построй следующий этап конвейера». Именно здесь Promises начинают выигрывать у callback hell.

```javascript
fetchUser()
  .then((user) => user.name)
  .then((name) => console.log(name));
```

Разбор по шагам:

1. `fetchUser()` возвращает промис пользователя.
2. Первый `then` ждёт пользователя и превращает объект `user` в `user.name`.
3. Результат первого `then` становится входом для второго `then`.
4. Второй `then` получает уже имя и печатает его.
5. Код идёт не «матрёшкой», а плоской цепочкой слева направо.

📚 **Источники:**
- [MDN: Promise.prototype.then()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/then)
- [javascript.info: Promise chaining](https://learn.javascript.ru/promise-chaining)

### Самое важное свойство: `then()` возвращает новый Promise

Вот здесь лежит основа всей композиции. Каждый `then` возвращает **новый** промис. Не тот же самый. Не «ничего». Новый.

Если ты вернул обычное значение, следующий `then` получит его как результат. Если ты вернул промис, цепочка подождёт его завершения. Это и делает код выразительным.

```javascript
Promise.resolve(2)
  .then((x) => x * 10)
  .then((x) => console.log(x));
```

Разбор по шагам:

1. Создаётся resolved promise со значением `2`.
2. Первый `then` получает `2` и возвращает `20`.
3. Поскольку вернулось обычное значение, движок автоматически оборачивает его в fulfilled promise.
4. Следующий `then` получает `20` и печатает его.

```javascript
Promise.resolve(2)
  .then((x) => Promise.resolve(x * 10))
  .then((x) => console.log(x));
```

Разбор по шагам:

1. Исходный promise содержит `2`.
2. Первый `then` возвращает **другой promise**, который позже даст `20`.
3. Цепочка не передаёт во второй `then` «promise как объект». Она ждёт, пока он завершится.
4. Когда внутренний promise fulfilled со значением `20`, второй `then` получает `20`.
5. Печатается `20`.

На интервью полезно сказать фразу: **«`then` умеет расплющивать возвращённый promise и ждать его»**.

📚 **Источники:**
- [MDN: Promise.prototype.then()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/then)
- [javascript.info: Promise chaining](https://learn.javascript.ru/promise-chaining)

### Что происходит при `throw` внутри `then`

Если внутри `then` выбросить исключение, соответствующий новый промис становится `rejected`. Это ещё одна центральная идея маршрутизации ошибок.

```javascript
Promise.resolve('{')
  .then((json) => JSON.parse(json))
  .catch((error) => console.log('Ошибка:', error.message));
```

Разбор по шагам:

1. Исходный promise успешно даёт строку `'{'`.
2. Внутри `then` вызывается `JSON.parse('{')`.
3. `JSON.parse` выбрасывает исключение, потому что JSON некорректен.
4. Это исключение не теряется. Оно превращает следующий promise в `rejected`.
5. `.catch(...)` ловит этот reject и печатает текст ошибки.

📚 **Источники:**
- [MDN: Promise.prototype.then()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/then)
- [MDN: Promise.prototype.catch()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/catch)
- [javascript.info: Promise error handling](https://learn.javascript.ru/promise-error-handling)

### `catch` и `finally`

`catch` — это удобная форма для обработки ошибки в цепочке. `finally` выполняется и при успехе, и при ошибке. Это место для уборки: выключить спиннер, снять блокировку формы, закрыть progress bar.

```javascript
loadTasks()
  .then(renderTasks)
  .catch((error) => showError(error.message))
  .finally(hideLoader);
```

Разбор по шагам:

1. `loadTasks()` возвращает промис.
2. Если всё хорошо, `renderTasks` получает данные и отображает задачи.
3. Если где-то в цепочке произошла ошибка, управление идёт в `catch`.
4. Независимо от успеха или ошибки вызывается `hideLoader`.
5. `finally` не заменяет `catch`, а отвечает именно за «уборку после сценария».

Два правила, которые часто забывают: (1) callback в `finally` **не получает аргументов** — ты не знаешь, был ли resolve или reject. (2) Если callback в `finally` выбрасывает ошибку или возвращает rejected promise, он **перезаписывает** исходный результат цепочки.

📚 **Источники:**
- [MDN: Promise.prototype.catch()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/catch)
- [MDN: Promise.prototype.finally()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/finally)
- [javascript.info: Promise error handling](https://learn.javascript.ru/promise-error-handling)

### Порядок выполнения цепочки

```javascript
console.log('1');

Promise.resolve()
  .then(() => console.log('2'))
  .then(() => console.log('3'));

console.log('4');
```

Разбор по шагам:

1. Печатается `1`.
2. Для первого `then` создаётся microtask.
3. Второй `then` пока не готов — он зависит от завершения первого.
4. Печатается `4`.
5. Синхронный код заканчивается.
6. Выполняется первая microtask → `2`.
7. Завершение первого `then` ставит второй `then` в microtask queue.
8. Event Loop продолжает чистить очередь microtasks.
9. Выполняется вторая microtask → `3`.

Итог: `1`, `4`, `2`, `3`.

📚 **Источники:**
- [MDN: Promise.prototype.then()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/then)
- [javascript.info: Promise chaining](https://learn.javascript.ru/promise-chaining)
- [javascript.info: Microtask queue](https://learn.javascript.ru/microtask-queue)

### Promise — это не старт операции, а контракт результата

Есть важная мысль, которую часто пропускают. Promise сам по себе не обязательно «запускает» операцию. Часто операция стартует там, где ты вызываешь API, а промис — это просто контракт, через который ты узнаешь судьбу результата.

То есть промис — не мотор и не двигатель. Он скорее накладная на груз: по ней ты узнаёшь, приехал груз или нет, и можешь выстроить дальнейшие действия.

📚 **Источники:**
- [MDN: Promise](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise)
- [javascript.info: Promise basics](https://learn.javascript.ru/promise-basics)

### 🔗 От Promise к Observable (мост в Angular)

Когда ты придёшь в Angular, рядом с `Promise` почти сразу появится `Observable` из RxJS. У них похожий дух «получу результат позже», но характер разный.

| | Promise | Observable (RxJS) |
|---|---|---|
| Значения | Одно | Поток (0..∞) |
| Ленивость | Операция обычно уже запущена к моменту получения Promise (нет lazy-подписки) | Lazy — операция запускается только при `subscribe()` |
| Отмена | `AbortController` (внешняя) | `unsubscribe` (встроенная) |
| Операторы | `.then/.catch/.finally` | `pipe(map, filter, switchMap...)` |

```javascript
// Promise — одно значение
const user = await fetch('/api/user').then(r => r.json());

// Observable — поток + мощные операторы
this.http.get<User>('/api/user').pipe(
  retry(3),
  catchError(err => of(fallbackUser))
).subscribe(user => this.user = user);
```

**Когда что использовать?**
- Разовый запрос без отмены → `Promise` / `async-await` достаточно
- Поток событий, отмена, retry, debounce → `Observable`
- В Angular: `HttpClient` возвращает `Observable`, но для простых случаев можно `toSignal()` или `lastValueFrom()`

### Common mistakes — истории разработчиков

**История 1.** Джун считает, что `.then()` возвращает тот же самый promise. Потом удивляется, почему можно строить разные независимые цепочки и почему ошибки текут по-разному.

**История 2.** Разработчик пишет внутри `new Promise` тяжёлую синхронную работу и думает, что «раз это Promise, значит поток не блокируется». Блокируется. Если внутри `executor` тяжёлый код, он идёт синхронно.

**История 3.** Кто-то думает, что resolved promise вызывает `then` мгновенно и синхронно. Из-за этого ломаются все predict-the-output задачи.

### 🎯 Interview Focus: Promises

**Почему спрашивают.** Потому что `async/await` стоит поверх промисов. Если ты не понимаешь состояния, цепочки и propagation ошибок, красивый синтаксис не спасёт.

**Типичные вопросы.** Когда запускается executor? Что возвращает `then()`? Почему `then` у resolved promise не выполняется синхронно? Что происходит, если из `then` вернуть другой промис?

**Red flags.** «`then` у resolved promise выполнится сразу в той же строке», «`then()` возвращает тот же promise», «ошибки из `then` не попадают в `catch`».

**Green flags.** «Executor синхронный, а `then` всегда асинхронен через microtask; каждый `then` строит новый промис; throw внутри `then` превращается в reject».

- D) В отдельном потоке

## Блок 6: Promise.all, Promise.race, Promise.allSettled, Promise.any

### 6.6. Как координировать несколько обещаний сразу

Когда ты освоил один Promise, почти сразу возникает новый вопрос: а что делать, если промисов несколько? Как ждать их вместе? Как завершиться на первом? Как собрать отчёт по каждому? Тут появляются комбинаторы.

Комбинатор — это функция, которая берёт несколько промисов и собирает их в один общий сценарий. Представь, что у тебя несколько курьеров: один везёт еду, другой документы, третий пропуск. Иногда тебе нужны **все трое**, иногда достаточно **первого**, иногда нужен **первый успешный**, а иногда нужен **отчёт по каждому**. Разные бизнес-правила — разные комбинаторы.

📚 **Источники:**
- [MDN: Promise.all()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/all)
- [javascript.info: Promise API](https://learn.javascript.ru/promise-api)

### `Promise.all`

`Promise.all` говорит: «жду всех, если хотя бы один упал — общий сценарий падает». Это правильный выбор, когда без полного набора данных продолжать бессмысленно.

```javascript
const [user, tasks, settings] = await Promise.all([
  fetchUser(),
  fetchTasks(),
  fetchSettings()
]);
```

Разбор по шагам:

1. Все три функции запускаются практически одновременно по отношению друг к другу.
2. `Promise.all` создаёт общий promise.
3. Он ждёт, пока завершатся все три участника.
4. Если все успешны, результатом становится массив значений в исходном порядке.
5. Если любой из них rejected, общий promise сразу становится rejected.

Важный нюанс: если один промис в `Promise.all` reject-ится, остальные **продолжают выполняться** — они не отменяются автоматически. `Promise.all` просто перестаёт ждать их результаты. Для реальной отмены нужен `AbortController`.

📚 **Источники:**
- [MDN: Promise.all()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/all)
- [javascript.info: Promise API](https://learn.javascript.ru/promise-api)
- [w3schools: Promise](https://www.w3schools.com/js/js_promise.asp)

### Почему `Promise.all` часто лучше трёх последовательных `await`

Если операции независимы, последовательные `await` заставляют тебя ждать по очереди: сначала первая, потом вторая, потом третья. Это как трижды ходить в разные магазины по одному, хотя можно было отправить трёх курьеров одновременно.

Сильный practical insight: **`await` делает код красивым, но не автоматически быстрым**. Скорость зависит от того, запускаешь ли ты операции параллельно там, где это безопасно.

📚 **Источники:**
- [MDN: Promise.all()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/all)
- [javascript.info: Promise API](https://learn.javascript.ru/promise-api)
- [javascript.info: Async/await](https://learn.javascript.ru/async-await)

### `Promise.race`

`Promise.race` говорит: «беру первый завершившийся результат — неважно, успех или ошибка». Это полезно для таймаутов, выбора самого быстрого источника или конкурентных гонок.

```javascript
function withTimeout(promise, ms) {
  return Promise.race([
    promise,
    new Promise((_, reject) => setTimeout(() => reject(new Error('Timeout')), ms))
  ]);
}
```

Разбор по шагам:

1. В `race` участвуют два промиса: исходная операция и таймер-ошибка.
2. Если исходная операция завершается раньше — `race` возвращает её результат.
3. Если таймер успевает раньше — `race` завершается ошибкой `Timeout`.
4. Важно: проигравшая операция **не отменяется автоматически**. Она может продолжать жить где-то в фоне.

#### Реальная отмена: AbortController

`Promise.race` с таймером НЕ отменяет проигравший промис — он продолжает работать. Для реальной отмены fetch/операции используй `AbortController`:

```javascript
async function fetchWithTimeout(url, ms = 5000) {
  const controller = new AbortController();
  const timeoutId = setTimeout(() => controller.abort(), ms);

  try {
    const response = await fetch(url, { signal: controller.signal });
    return await response.json();
  } catch (err) {
    if (err.name === 'AbortError') {
      console.log('Запрос отменён по таймауту');
    }
    throw err;
  } finally {
    clearTimeout(timeoutId); // всегда убираем таймер
  }
}
```

- [MDN: Promise.race()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/race)
- [javascript.info: Promise API](https://learn.javascript.ru/promise-api)
- [MDN: AbortController](https://developer.mozilla.org/ru/docs/Web/API/AbortController)
- [MDN: AbortSignal.timeout()](https://developer.mozilla.org/en-US/docs/Web/API/AbortSignal/timeout_static)

### `Promise.allSettled`

`Promise.allSettled` говорит: «не падай на первом reject, дай мне судьбу каждого участника». Это полезно для админок, мониторинга, пакетных операций, где важно видеть полный отчёт.

```javascript
const results = await Promise.allSettled([
  fetchUser(),
  fetchTasks(),
  fetchSettings()
]);
```

```javascript
// Каждый элемент результата имеет одну из двух форм:
// { status: 'fulfilled', value: результат }
// { status: 'rejected', reason: ошибка }
```

Разбор по шагам:

1. Все операции запускаются.
2. Общий promise ждёт завершения всех участников.
3. Даже если один или несколько упали, общий promise не падает на первом reject.
4. В результате ты получаешь массив объектов вида `{ status, value }` или `{ status, reason }`.
5. Это удобно, когда тебе нужен отчёт, а не сценарий «всё или ничего».

📚 **Источники:**
- [MDN: Promise.allSettled()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/allSettled)
- [javascript.info: Promise API](https://learn.javascript.ru/promise-api)

### `Promise.any`

`Promise.any` говорит: «дай мне первый успешный результат; ошибки игнорируй, пока есть шанс, что кто-то ещё завершится успешно». Это полезно, например, для запроса к нескольким зеркалам или CDN.

```javascript
const data = await Promise.any([
  fetch('/mirror-1/data'),
  fetch('/mirror-2/data'),
  fetch('/mirror-3/data')
]);
```

Разбор по шагам:

1. Все запросы стартуют.
2. Если один из них первым успешно завершился, `Promise.any` сразу резолвится этим результатом.
3. Ошибки отдельных участников не ломают сценарий, пока остаётся шанс на успех.
4. Если упали вообще все, вернётся `AggregateError`.

⚠️ Помни: `fetch()` НЕ reject-ится на HTTP 4xx/5xx ошибки — только на сетевые проблемы. Результат — объект `Response`, и нужно проверять `response.ok` перед чтением тела.

📚 **Источники:**
- [MDN: Promise.any()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/any)
- [javascript.info: Promise API](https://learn.javascript.ru/promise-api)

### Как отвечать на интервью

Если спросили разницу между комбинаторами, удобно отвечать через вопрос: **какое бизнес-правило завершения мне нужно?**

- Нужны все, иначе бессмысленно → `Promise.all`
- Нужен первый любой итог → `Promise.race`
- Нужен отчёт по каждому → `Promise.allSettled`
- Нужен первый успех → `Promise.any`

### Common mistakes — истории разработчиков

**История 1.** Разработчик использует `Promise.all` для пяти независимых блоков статистики на дашборде. Один блок падает — и ломается весь экран, хотя можно было показать остальные через `allSettled`.

**История 2.** Кто-то думает, что `Promise.race` отменяет проигравшие операции. Нет. Без отдельного механизма отмены проигравшая операция может продолжать выполняться.

### ✅ Проверь себя

**1. Что выбрать, если важен статус каждого промиса независимо от ошибок?**
- A) `Promise.all`
- B) `Promise.race`
- C) `Promise.allSettled`
- D) `Promise.any`

<details><summary>Ответ</summary>C) Только `allSettled` гарантированно даст судьбу каждого участника.</details>

**2. Чем `Promise.race` отличается от `Promise.any`?**
- A) `race` ждёт всех
- B) `race` берёт первый любой итог, `any` — первый успешный
- C) `any` работает только с fetch
- D) Разницы нет

<details><summary>Ответ</summary>B) `race` может завершиться и ошибкой, если ошибка пришла первой. `any` ждёт первый успех.</details>

## Блок 7: 🎯🎯 async/await — синтаксический сахар над промисами

### 6.7. Линейный вид поверх асинхронной природы

Теперь мы подходим к самому популярному синтаксису в ежедневной работе. Но здесь легко ошибиться: из-за красивого линейного вида кажется, будто `async/await` создал новую модель выполнения. Не создал. Он просто сделал Promises удобнее для чтения.

Метафора 1: синтаксический сахар — это молния вместо пуговиц. Куртка та же, физика одежды не изменилась.

Метафора 2: это как перевести техническую схему в человеческий маршрут. Дороги те же, но читать карту проще.

Java-аналогия: `async/await` по смыслу напоминает ситуацию, когда сложная цепочка `CompletableFuture` завернута в более линейный стиль чтения. Но под капотом всё равно живут продолжения, а не магический синхронный блокировщик мира.

📚 **Источники:**
- [MDN: async function](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Statements/async_function)
- [javascript.info: Async/await](https://learn.javascript.ru/async-await)
- [w3schools: Async](https://www.w3schools.com/js/js_async.asp)

### Что возвращает `async function`

`async function` всегда возвращает Promise. Даже если внутри ты написал `return 42`, наружу уйдёт `Promise.resolve(42)`. Если внутри выбросил ошибку, наружу уйдёт rejected promise.

```javascript
async function getAnswer() {
  return 42;
}
```

Разбор по шагам:

1. Объявляется `async function`.
2. При вызове `getAnswer()` функция вернёт не просто число, а promise.
3. Значение `42` автоматически оборачивается в fulfilled promise.
4. Поэтому снаружи ты можешь писать либо `.then(...)`, либо `await getAnswer()`.

📚 **Источники:**
- [MDN: async function](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Statements/async_function)
- [javascript.info: Async/await](https://learn.javascript.ru/async-await)

### Что делает `await`

`await` — это оператор ожидания промиса внутри `async`-функции. Но здесь очень важно проговорить точную формулировку: **`await` приостанавливает текущую async-функцию, а не весь JavaScript**.

Представь, что ты читаешь книгу и ставишь закладку. Ты не уничтожил книгу и не остановил жизнь во вселенной. Ты просто отметил место, до которого дочитал, и вернёшься, когда появится следующая глава. `await` делает именно это с телом функции.

```javascript
async function showTasks() {
  const user = await fetchUser();
  const tasks = await fetchTasks(user.id);
  return tasks;
}
```

Разбор по шагам:

1. `showTasks()` вызывается и сразу возвращает promise наружу.
2. Внутри начинается выполнение до первого `await fetchUser()`.
3. `await fetchUser()` **всегда** приостанавливает `showTasks` и ставит продолжение как microtask — даже если промис уже resolved.
4. Продолжение функции после `await` сохраняется как будущая microtask.
5. Когда выполнится microtask продолжения, в `user` попадёт результат `fetchUser()`.
6. Затем код дойдёт до второго `await fetchTasks(user.id)` и снова передаст продолжение через microtask-механизм.
7. После второго возобновления функция вернёт `tasks`, который будет обёрнут в fulfilled promise.

📚 **Источники:**
- [MDN: await](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Operators/await)
- [javascript.info: Async/await](https://learn.javascript.ru/async-await)

### Под капотом: `await` — это sugar над `.then()`

Очень полезно уметь переписывать один и тот же код в двух формах. Тогда ты реально понимаешь, что это одна и та же модель.

```javascript
fetchUser()
  .then((user) => fetchTasks(user.id))
  .then((tasks) => {
    renderTasks(tasks);
  })
  .catch((error) => {
    showError(error);
  });
```

Разбор по шагам:

1. `fetchUser()` возвращает promise пользователя.
2. Когда пользователь пришёл, первый `then` запускает `fetchTasks(user.id)`.
3. Поскольку `then` вернул promise задач, цепочка ждёт его завершения.
4. Когда задачи пришли, следующий `then` вызывает `renderTasks(tasks)`.
5. Ошибки из любой части цепочки собираются в `catch`.

```javascript
async function renderUserTasks() {
  try {
    const user = await fetchUser();
    const tasks = await fetchTasks(user.id);
    renderTasks(tasks);
  } catch (error) {
    showError(error);
  }
}
```

Разбор по шагам:

1. `renderUserTasks()` сразу возвращает promise.
2. Внутри выполняется `await fetchUser()` — это эквивалент ожидания первого `then`.
3. После получения `user` выполняется `await fetchTasks(user.id)` — это эквивалент следующего звена цепочки.
4. Если оба шага успешны, вызывается `renderTasks(tasks)`.
5. Если любой `await` получает rejected promise, управление переходит в `catch`.

На собеседовании сильный ответ: **«`async/await` — это не другая модель исполнения, а более удобная форма записи promise-цепочек»**.

📚 **Источники:**
- [MDN: await](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Operators/await)
- [MDN: Promise.prototype.then()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/then)
- [javascript.info: Async/await](https://learn.javascript.ru/async-await)

### Ловушка последовательного ожидания

Это одна из самых практичных тем. Красивый код не всегда означает эффективный код.

```javascript
const user = await fetchUser();
const tasks = await fetchTasks();
```

Разбор по шагам:

1. Сначала стартует `fetchUser()`.
2. Пока пользователь не придёт, `fetchTasks()` даже не начнётся.
3. Если операции независимы, ты потерял время на лишнюю последовательность.

Часто лучше так:

```javascript
const userPromise = fetchUser();
const tasksPromise = fetchTasks();

const [user, tasks] = await Promise.all([userPromise, tasksPromise]);
```

Разбор по шагам:

1. `fetchUser()` и `fetchTasks()` стартуют почти одновременно.
2. Ты не ждёшь первую операцию, чтобы только потом запустить вторую.
3. `Promise.all` ждёт обе и возвращает результаты вместе.
4. Это быстрее, если между задачами нет зависимости.

📚 **Источники:**
- [MDN: Promise.all()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/all)
- [javascript.info: Async/await](https://learn.javascript.ru/async-await)
- [javascript.info: Promise API](https://learn.javascript.ru/promise-api)

### Common mistakes — истории разработчиков

**История 1.** Разработчик говорит: «`await` блокирует поток». Из-за этого он начинает бояться `await` как чего-то тяжёлого. На самом деле блокируется только продолжение текущей async-функции, а Event Loop живёт дальше.

**История 2.** Джун пишет пять `await` подряд для независимых запросов и не понимает, почему экран грузится на 5 секунд дольше.

**История 3.** Кто-то думает, что `async function` возвращает «обычное значение, если внутри return». Нет. Снаружи это всё равно Promise.

### 🎯 Interview Focus: async/await

**Почему спрашивают.** Потому что многие пишут `await` ежедневно, но не понимают, что происходит под капотом.

**Типичные вопросы.** Что возвращает `async function`? Блокирует ли `await` поток? Чем `await` отличается от `.then()`? Почему несколько `await` подряд могут быть медленнее `Promise.all`?

**Red flags.** «`await` останавливает весь JavaScript», «`async/await` работает без промисов», «если код выглядит последовательным, значит он и выполняется как синхронный Java-код».

**Green flags.** «`await` приостанавливает только текущую async-функцию; продолжение идёт через microtask; выбор между `.then()` и `await` — это вопрос читабельности и композиции, а не другой модели исполнения».

- B) Всегда `Promise`
- C) Только `undefined`

<details><summary>Ответ</summary>B) Это фундаментальное свойство `async function`.</details>

## Блок 8: Error handling в async — try/catch, .catch(), unhandled rejection

### 6.8. Ошибка тоже путешествует по правилам

Когда разработчик начинает писать асинхронный код, он часто сосредотачивается только на happy path: данные пришли, экран нарисовался, всё красиво. Но зрелость начинается там, где ты думаешь не только о «как получить результат», но и о «как аккуратно провести ошибку до места обработки».

Метафора 1: течь в трубе. Важно не только заметить воду, но и понимать, по каким каналам она пойдёт.

Метафора 2: сбойная посылка в логистике. Нужен маршрут, по которому информация о проблеме доедет до менеджера, а не потеряется на складе.

В промисах ошибка живёт как `rejected`-состояние. В `async/await` она выглядит почти как обычное исключение, поэтому `try/catch` становится очень удобным.

```javascript
async function loadDashboard() {
  try {
    const user = await fetchUser();
    const tasks = await fetchTasks(user.id);
    renderDashboard(user, tasks);
  } catch (error) {
    showError(error.message);
  }
}
```

Разбор по шагам:

1. `loadDashboard()` запускается и возвращает promise.
2. Внутри `try` выполняется `await fetchUser()`.
3. Если `fetchUser()` завершился reject, управление сразу переходит в `catch`.
4. Если пользователь получен, дальше выполняется `await fetchTasks(user.id)`.
5. Если второй шаг падает, управление тоже идёт в `catch`.
6. Если оба шага успешны, рендерится дашборд.

📚 **Источники:**
- [MDN: try...catch](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Statements/try...catch)
- [javascript.info: Promise error handling](https://learn.javascript.ru/promise-error-handling)
- [MDN: Promise.prototype.catch()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/catch)

### То же самое в `.then()`-стиле

```javascript
loadTasks()
  .then(renderTasks)
  .catch((error) => showError(error.message));
```

Разбор по шагам:

1. `loadTasks()` возвращает promise.
2. Если он fulfilled, `renderTasks` получает данные.
3. Если он rejected или если `renderTasks` бросит исключение, управление идёт в `catch`.
4. `catch` в promise-цепочке — это такой же маршрут ошибок, просто в другой форме записи.

📚 **Источники:**
- [MDN: Promise.prototype.catch()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/catch)
- [javascript.info: Promise error handling](https://learn.javascript.ru/promise-error-handling)

### `unhandled rejection`

**Unhandled rejection** — это отклонённый promise, у которого никто не обработал ошибку. Метафора: пожарная сигнализация сработала, а дежурный не взял трубку. Это не нормальный сценарий, а баг.

```javascript
async function broken() {
  throw new Error('Boom');
}

broken();
```

Разбор по шагам:

1. Вызывается `broken()`.
2. Поскольку функция `async`, выброшенная ошибка превращает возвращаемый promise в `rejected`.
3. Но результат вызова никто не `await`-ит и не обрабатывает через `.catch()`.
4. В итоге возникает `unhandled rejection`.

📚 **Источники:**
- [MDN: Promise rejection events](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Using_promises#promise_rejection_events)
- [javascript.info: Promise error handling](https://learn.javascript.ru/promise-error-handling)

### Что не ловит обычный `try/catch`

Вот очень важная ловушка: `try/catch` не ловит всё подряд просто по факту своего существования. Он ловит только то, что происходит **внутри его текущего синхронного контекста** или через `await` в той же async-функции.

```javascript
try {
  setTimeout(() => {
    throw new Error('Не поймается этим try/catch');
  }, 0);
} catch (error) {
  console.log('Сюда не попадём');
}
```

Разбор по шагам:

1. Внутри `try` только регистрируется таймер.
2. Сам колбэк таймера выполнится позже, в другой macrotask.
3. К тому моменту исходный `try/catch` уже давно завершился.
4. Поэтому ошибка внутри колбэка не может быть поймана этим внешним `try/catch`.

Чтобы поймать такую ошибку, нужен обработчик **внутри самого колбэка** или другая модель построения асинхронного кода.

📚 **Источники:**
- [MDN: try...catch](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Statements/try...catch)
- [javascript.info: Promise error handling](https://learn.javascript.ru/promise-error-handling)

### `finally` в error handling

Не забывай про `finally`. Это место для кода, который должен выполниться и при успехе, и при ошибке: спрятать loader, вернуть кнопке состояние, закрыть подключение.

```javascript
async function saveForm() {
  showLoader();

  try {
    await submitForm();
    showSuccess();
  } catch (error) {
    showError(error.message);
  } finally {
    hideLoader();
  }
}
```

Разбор по шагам:

1. Лоадер включается до начала операции.
2. Если `submitForm()` успешен, показывается успех.
3. Если операция падает, показывается ошибка.
4. В обоих случаях `finally` выключает лоадер.
5. Это защищает интерфейс от «вечного спиннера».

📚 **Источники:**
- [MDN: Promise.prototype.finally()](https://developer.mozilla.org/ru/docs/Web/JavaScript/Reference/Global_Objects/Promise/finally)
- [javascript.info: Promise error handling](https://learn.javascript.ru/promise-error-handling)

### Common mistakes — истории разработчиков

**История 1.** Разработчик зовёт `async`-функцию без `await` и без `.catch()`, а потом удивляется `unhandled rejection` в консоли.

**История 2.** Мидл оборачивает `setTimeout` в `try/catch` снаружи и думает, что этого достаточно. Потом ошибка внутри таймера улетает мимо.

**История 3.** Кто-то выключает loader только в успешной ветке и забывает про `finally`, из-за чего при ошибке интерфейс остаётся «вечно загружающимся».

### 🎯 Interview Focus: Error handling

**Почему спрашивают.** Потому что асинхронный код без корректного маршрута ошибок быстро становится минным полем.

**Типичные вопросы.** Ловит ли `try/catch` ошибку из `await`? А из `setTimeout`? Что такое unhandled rejection? Когда лучше `.catch()`, а когда `try/catch`?

**Red flags.** «Любой `try/catch` ловит вообще все асинхронные ошибки».

**Green flags.** «Ошибка должна быть обработана либо через `await` внутри `try/catch`, либо через `.catch()` на промисе; unhandled rejection — баг».

### ✅ Проверь себя

**1. Как поймать ошибку rejected promise внутри async-функции?**
- A) `try/catch` вокруг `await`
- B) `for`
- C) `setTimeout`
- D) `map`

<details><summary>Ответ</summary>A) `await` делает rejected promise похожим на throw.</details>

**2. Что такое unhandled rejection?**
- A) Ошибка сборки
- B) Отклонённый промис без обработчика ошибки
- C) Таймер с нулевой задержкой
- D) Пустая очередь микрозадач

<details><summary>Ответ</summary>B) Это ошибка, которую никто не забрал.</details>

## Блок 9: 🎯 Microtasks vs Macrotasks — порядок выполнения

### 6.9. Две очереди, два уровня срочности

Если Block 2 был про общую картину Event Loop, то здесь мы зумимся в один из самых любимых interview-подвохов: **microtasks vs macrotasks**. Именно тут кандидат либо начинает уверенно считать шаги, либо говорит «ну вроде как промисы раньше таймеров» и дальше теряется.

Microtask — это маленькая задача повышенного приоритета, которая выполняется сразу после завершения текущего стека и до следующей macrotask. Macrotask — это обычная задача основной очереди. Самое важное различие не в размере кода, а в **моменте приоритета**.

Ресторанная кухня снова помогает. Macrotasks — это полноценные заказы. Microtasks — короткие срочные бумажки: «доложи соус», «подпиши чек», «подтверди выдачу». После каждого большого заказа шеф обязан разобрать все короткие срочные бумажки, и только потом берёт следующий полноценный заказ.

Java-аналогия: если грубо, macrotask — как обычная задача в очереди worker'a, а microtask — как приоритетное короткое продолжение, которое планировщик обязан выпустить перед следующим обычным заданием. Повторю: это аналогия, не буквальный механизм JVM.

📚 **Источники:**
- [MDN: Microtask guide](https://developer.mozilla.org/en-US/docs/Web/API/HTML_DOM_API/Microtask_guide)
- [javascript.info: Microtask queue](https://learn.javascript.ru/microtask-queue)
- [ECMA-262: Jobs](https://tc39.es/ecma262/#sec-jobs)

### Главное правило — выучи дословно

**После завершения текущего синхронного кода или одной macrotask движок полностью вычищает очередь microtasks.**

Это правило надо уметь проговаривать как таблицу умножения. Потому что почти все predict-the-output задачи крутятся вокруг него.

### `queueMicrotask`

`queueMicrotask` — это явный способ сказать: «выполни этот код асинхронно, но очень скоро — сразу после текущего кода и раньше следующей macrotask».

```javascript
console.log('A');
queueMicrotask(() => console.log('B'));
console.log('C');
```

Разбор по шагам:

1. Печатается `A`.
2. `queueMicrotask` ставит `B` в очередь microtasks.
3. Печатается `C`.
4. Синхронный код завершён.
5. Event Loop очищает microtask queue и печатает `B`.

Ответ: `A`, `C`, `B`.

- [javascript.info: Microtask queue](https://learn.javascript.ru/microtask-queue)

### `Promise.resolve().then(...)` как microtask

`then` у уже resolved promise тоже идёт в очередь микрозадач. Поэтому в задачах на порядок вывода `queueMicrotask` и `Promise.resolve().then(...)` часто ведут себя очень похоже.

```javascript
console.log('A');
Promise.resolve().then(() => console.log('B'));
console.log('C');
```

Разбор по шагам:

1. Печатается `A`.
2. Обработчик `.then(...)` ставится в microtask queue.
3. Печатается `C`.
4. После завершения синхронного кода выполняется microtask `B`.

Ответ: `A`, `C`, `B`.

### Нумерованный сценарий: «что происходит по шагам»

Возьмём базовый пример и пройдём его в формате, который любят интервьюеры.

```javascript
console.log('start');
setTimeout(() => console.log('timeout'), 0);
Promise.resolve().then(() => console.log('promise'));
queueMicrotask(() => console.log('microtask'));
console.log('end');
```

Разбор по шагам:

1. JS выполняет строку `console.log('start')` → печатается `start`.
2. JS выполняет `setTimeout(...)` → колбэк уходит в механизм таймера и позже попадёт в macrotask queue.
3. JS выполняет `Promise.resolve().then(...)` → обработчик ставится в microtask queue.
4. JS выполняет `queueMicrotask(...)` → ещё один обработчик ставится в microtask queue после промиса.
5. JS выполняет `console.log('end')` → печатается `end`.
6. Синхронный код завершён, call stack пуст.
7. Event Loop проверяет microtask queue.
8. Выполняется первая microtask → печатается `promise`.
9. Выполняется вторая microtask → печатается `microtask`.
10. Очередь microtasks пуста.
11. Event Loop берёт следующую macrotask.
12. Выполняется таймер → печатается `timeout`.

Ответ: `start`, `end`, `promise`, `microtask`, `timeout`.

### Predict the output

#### Задача 1

```javascript
console.log(1);
setTimeout(() => console.log(2), 0);
Promise.resolve().then(() => console.log(3));

console.log(5);
```

Разбор по шагам:

1. Печатается `1`.
2. Таймер с `2` идёт в будущие macrotasks.
3. Promise с `3` идёт в microtask queue.
4. `queueMicrotask` с `4` идёт в microtask queue после `3`.
5. Печатается `5`.
6. Синхронная часть закончилась.

8. Выполняется microtask `4`.
9. Потом выполняется macrotask `2`.

Ответ: `1`, `5`, `3`, `4`, `2`.

#### Задача 2

```javascript
console.log('start');
setTimeout(() => {
  console.log('timeout-1');
  Promise.resolve().then(() => console.log('promise-in-timeout'));
}, 0);

Promise.resolve()
  .then(() => {
    console.log('promise-1');
    setTimeout(() => console.log('timeout-2'), 0);
  })
  .then(() => console.log('promise-2'));

console.log('end');
```

Разбор по шагам:

1. Печатается `start`.
2. Регистрируется `timeout-1` как macrotask.
3. Первый `then` ставится в microtask queue.
4. Второй `then` пока ждёт завершения первого.
5. Печатается `end`.
6. Начинается очистка microtasks.
7. Выполняется `promise-1`, печатается `promise-1`.
8. Внутри `promise-1` регистрируется новый таймер `timeout-2`.
9. Завершение первого `then` делает готовым следующий `then`, он идёт в microtask queue.
10. Выполняется `promise-2`, печатается `promise-2`.
11. Теперь очередь microtasks пуста.
12. Выполняется первый таймер `timeout-1`, печатается `timeout-1`.
13. Внутри него ставится microtask `promise-in-timeout`.
14. Macrotask таймера заканчивается.
15. Event Loop очищает microtask queue → печатается `promise-in-timeout`.
16. Потом выполняется следующий таймер `timeout-2`.

Ответ: `start`, `end`, `promise-1`, `promise-2`, `timeout-1`, `promise-in-timeout`, `timeout-2`.

3. Печатается `A`.

5. Завершение первого `then` ставит следующий `then` как microtask `C`.


Разбор по шагам:

1. Регистрируется `timer-1`.
2. Первый `then` ставится в очередь microtasks.

### Starvation

**Starvation** — это голодание менее приоритетных задач. Если ты бесконечно добавляешь microtasks, macrotasks и rendering могут долго не получить шанс.

```javascript
function loop() {
  queueMicrotask(loop);
}
```

Разбор по шагам:

1. `loop` ставит саму себя в очередь microtasks.
2. Когда microtask выполняется, она снова ставит себя в очередь.
3. Очередь microtasks никогда не становится пустой.
4. Значит, Event Loop не может нормально перейти к следующей macrotask и rendering.
5. Это пример того, как «маленькие срочные задачи» могут задушить всё остальное.

### Common mistakes — истории разработчиков

**История 1.** Разработчик думает, что после каждой macrotask выполняется только одна microtask. Из-за этого он неверно предсказывает задачи с вложенными `.then()`.

**История 2.** Джун считает `queueMicrotask` почти тем же, что `setTimeout(..., 0)`. На самом деле разница принципиальная: один идёт в приоритетную очередь, другой — в обычную.

**История 3.** Кто-то бесконечно планирует microtask внутри microtask и удивляется, почему интерфейс перестал реагировать.

### 🎯 Interview Focus: Microtasks vs Macrotasks

**Почему спрашивают.** Потому что это лакмусовая бумажка реального понимания Event Loop.

**Типичные вопросы.** Порядок вывода. Разница между `queueMicrotask`, `Promise.resolve().then(...)` и `setTimeout(..., 0)`. Момент рендеринга.

**Red flags.** «Таймер с нулём всегда раньше промиса», «из microtask queue берётся только одна задача».

**Green flags.** «Сначала синхронный стек, потом все microtasks FIFO, потом следующая macrotask».

## Блок 10: Generators и async iterators — основы

### 6.10. Функция, которая умеет ставить себя на паузу

Этот блок не такой критичный для собеседований, как Event Loop или Promises, но он помогает расширить картину. Иногда полезно понимать, что в JavaScript есть не только «выполнилась до конца и умерла», но и функции, которые умеют отдавать управление по частям.

**Generator** — это функция, которая может останавливаться и потом продолжаться, отдавая значения по одному. Представь сериал: после каждой серии ты решаешь, запускать ли следующую. Обычная функция — это фильм, который идёт от начала до конца. Генератор — сериал с кнопкой «продолжить».

Java-аналогия: это немного похоже на ленивый `Iterator`, только с более выразительным синтаксисом и возможностью писать логику между выдачами значений.

```javascript
function* ids() {
  yield 1;
  yield 2;
  yield 3;
}
```

Разбор по шагам:

1. Объявляется generator-функция `ids`.
2. При вызове `ids()` она не исполняется до конца сразу.
3. Вместо этого возвращается generator-объект.
4. Каждый вызов `next()` продвигает выполнение до следующего `yield`.
5. `yield` отдаёт значение наружу и ставит функцию на паузу.

📚 **Источники:**
- [MDN: AsyncGenerator](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/AsyncGenerator)
- [javascript.info: Async iterators and generators](https://learn.javascript.ru/async-iterators-generators)

### Async iterator

**Async iterator** — это итератор, который поставляет значения со временем. То есть следующего элемента может ещё не быть прямо сейчас, за ним надо подождать.

Метафора: коробки приезжают по одной на склад, и кладовщик не может взять следующую коробку, пока она физически не приехала.

```javascript
async function* streamTasks() {
  yield await Promise.resolve('task-1');
  yield await Promise.resolve('task-2');
}

(async () => {
  for await (const task of streamTasks()) {
    console.log(task);
  }
})();
```

Разбор по шагам:

1. Объявляется async generator `streamTasks`.
2. Он умеет отдавать значения не мгновенно, а через `await`.
3. `for await...of` последовательно ждёт каждый следующий элемент.
4. Сначала приходит `task-1`, потом `task-2`.
5. Такой стиль удобен для стримов, пагинации, последовательного чтения асинхронных данных.

📚 **Источники:**
- [MDN: AsyncGenerator](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/AsyncGenerator)
- [MDN: for await...of](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Statements/for-await...of)
- [javascript.info: Async iterators and generators](https://learn.javascript.ru/async-iterators-generators)

### Зачем это знать

Тебя редко попросят руками реализовать async iterator на обычном собеседовании по фронтенду начального уровня. Но если ты понимаешь идею, тебе легче читать современные API стримов, асинхронных источников данных и библиотечный код.

📚 **Источники:**
- [MDN: AsyncGenerator](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/AsyncGenerator)
- [javascript.info: Async iterators and generators](https://learn.javascript.ru/async-iterators-generators)

## Блок 11: Top-level await

### 6.11. await прямо на верхнем уровне модуля

Top-level await — это возможность использовать `await` прямо на верхнем уровне ES-модуля, не заворачивая код в отдельную `async`-функцию. Механизм полезный, но его надо применять осознанно.

Метафора: раньше звонить можно было только из переговорки, а теперь из общего холла — но только если ты находишься в правильной зоне офиса. Правильная зона здесь — **ES module context**.

```javascript
const response = await fetch('/config.json');
const config = await response.json();
```

Разбор по шагам:

1. Код выполняется на верхнем уровне модуля.
2. Первый `await` приостанавливает инициализацию этого модуля до получения ответа.
3. Второй `await` ждёт парсинг JSON.
4. Пока модуль не готов, зависящие от него модули могут стартовать позже.

📚 **Источники:**
- [MDN: await (top-level await)](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Operators/await#top_level_await)
- [MDN: JavaScript modules](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Modules#top_level_await)

### Почему это и полезно, и опасно

Полезно — когда модуль реально не может быть корректно инициализирован без конфигурации, ключей SDK или стартовых данных. Опасно — когда ты без нужды ставишь асинхронный барьер на загрузку целой цепочки модулей.

Java-аналогия: это похоже на асинхронный барьер при инициализации конфигурации приложения. Если барьер нужен — хорошо. Если нет — ты просто тормозишь старт.

📚 **Источники:**
- [MDN: await (top-level await)](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Operators/await#top_level_await)
- [MDN: JavaScript modules](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Modules#top_level_await)

## Отдельный interview-блок: `async/await` vs `.then()`

На собеседовании не стоит отвечать в стиле «`.then()` старый, `await` новый и поэтому лучше». Это слабый ответ. Оба подхода живут поверх Promises. `async/await` удобнее для линейной истории и локального `try/catch`. `.then()` хорош, когда тебе нравится композиция шагов, короткие трансформации или функциональный стиль.

Сильный ответ звучит так: **«модель исполнения одна и та же; различается форма записи и удобство в конкретном сценарии»**.

Если хочешь ещё более зрелый ответ, добавь: **«`await` часто улучшает читаемость, но может случайно сделать независимые операции последовательными; `.then()` иногда компактнее для коротких трансформаций»**.

## Большой блок «Предскажи вывод»

Ниже — специальная тренировка. Не просто читай ответы. Сначала реально прогнозируй порядок сам, потом сверяйся. Это один из самых окупаемых форматов подготовки перед интервью.

### Задача A

```javascript
async function demo() {
  console.log('1');
  await Promise.resolve();
  console.log('2');
}

console.log('0');
demo();
console.log('3');
```

Разбор по шагам:

1. Печатается `0`.
2. Вызывается `demo()`.
3. Внутри `demo` сразу печатается `1`.
4. На `await Promise.resolve()` функция приостанавливается; продолжение после `await` уходит в microtask.
5. Внешний код продолжается и печатает `3`.
6. Синхронная часть завершена.
7. Выполняется microtask продолжения `demo` и печатается `2`.

Ответ: `0`, `1`, `3`, `2`.

### Задача B

```javascript
async function demo() {
  console.log('A');
  setTimeout(() => console.log('B'), 0);
  await Promise.resolve();
  console.log('C');
}

demo();
console.log('D');
```

Разбор по шагам:

1. Вызывается `demo()`.
2. Внутри сразу печатается `A`.
3. Регистрируется таймер `B`.
4. На `await Promise.resolve()` функция приостанавливается и ставит продолжение в microtask.
5. Внешний код печатает `D`.
6. Синхронный код завершён.
7. Выполняется microtask продолжения `demo` → печатается `C`.
8. Затем выполняется macrotask таймера → печатается `B`.

Ответ: `A`, `D`, `C`, `B`.

### Задача C

```javascript
console.log('start');
setTimeout(() => console.log('timeout-1'), 0);

(async function () {
  console.log('async-start');
  await null;
  console.log('async-end');
})();

Promise.resolve().then(() => console.log('promise'));
console.log('finish');
```

Разбор по шагам:

1. Печатается `start`.
2. Регистрируется таймер `timeout-1`.
3. IIFE `async function` запускается сразу и печатает `async-start`.
4. `await null` эквивалентен ожиданию `Promise.resolve(null)`, поэтому продолжение функции уходит в microtask.
5. Затем ставится microtask `promise`.
6. Печатается `finish`.
7. Синхронный код завершён.
8. Выполняется microtask продолжения async-функции → `async-end`.
9. Выполняется следующая microtask промиса → `promise`.
10. Только потом выполняется таймер `timeout-1`.

Ответ: `start`, `async-start`, `finish`, `async-end`, `promise`, `timeout-1`.

### Задача D

```javascript
Promise.resolve()
  .then(() => {
    console.log('1');
    return Promise.resolve();
  })
  .then(() => console.log('2'));

Promise.resolve().then(() => console.log('3'));
```

Разбор по шагам:

1. Первая цепочка ставит в очередь microtask для первого `then`.
2. Вторая независимая строка ставит в очередь microtask для `3`.
3. Синхронный код завершён.
4. Выполняется первый `then` первой цепочки → печатается `1`.
5. `return Promise.resolve()` не выполняет следующий `then` синхронно: движок сначала усыновляет возвращённый promise, а продолжение цепочки ставит новой microtask **после уже стоящих**.
6. Следующей в очереди идёт microtask с `3` → печатается `3`.
7. Потом приходит очередь второй части первой цепочки → печатается `2`.

Ответ: `1`, `3`, `2`.

### Задача E

```javascript
setTimeout(() => {
  console.log('timer');
  queueMicrotask(() => console.log('micro-in-timer'));
}, 0);

queueMicrotask(() => {
  console.log('micro-1');
  queueMicrotask(() => console.log('micro-2'));
});
```

Разбор по шагам:

1. Регистрируется таймер `timer`.
2. Ставится microtask с `micro-1`.
3. Синхронный код заканчивается.
4. Выполняется `micro-1`.
5. Внутри неё ставится новая microtask `micro-2`.
6. Очередь microtasks очищается полностью, поэтому выполняется `micro-2`.
7. Затем выполняется macrotask таймера → `timer`.
8. Внутри таймера ставится microtask `micro-in-timer`.
9. После завершения macrotask таймера Event Loop очищает microtasks → `micro-in-timer`.

Ответ: `micro-1`, `micro-2`, `timer`, `micro-in-timer`.

### Задача F

```javascript
console.log('S');

(async function () {
  console.log('A1');
  await Promise.resolve();
  console.log('A2');
  await Promise.resolve();
  console.log('A3');
})();

Promise.resolve().then(() => console.log('P1'));
setTimeout(() => console.log('T1'), 0);
console.log('E');
```

Разбор по шагам:

1. Печатается `S`.
2. Запускается async IIFE, печатает `A1`.
3. На первом `await` её продолжение `A2 -> await -> A3` уходит в microtask queue.
4. Ставится независимая microtask `P1`.
5. Регистрируется таймер `T1`.
6. Печатается `E`.
7. Синхронный код закончился.
8. Выполняется microtask продолжения async-функции → печатается `A2`.
9. Внутри неё встречается второй `await`, поэтому остаток (`A3`) ставится как новая microtask в конец очереди.
10. Следующая microtask, которая уже стояла раньше, — это `P1` → печатается `P1`.
11. Затем выполняется microtask `A3`.
12. Только потом выполняется таймер `T1`.

Ответ: `S`, `A1`, `E`, `A2`, `P1`, `A3`, `T1`.

### Задача G

```javascript
setTimeout(() => {
  console.log('outer timer');
  setTimeout(() => console.log('inner timer'), 0);
  Promise.resolve().then(() => console.log('promise in outer timer'));
}, 0);

Promise.resolve().then(() => {
  console.log('top promise');
});
```

Разбор по шагам:

1. Регистрируется `outer timer`.
2. Ставится microtask `top promise`.
3. Синхронный код заканчивается.
4. Выполняется `top promise`.
5. Затем идёт macrotask `outer timer` → печатается `outer timer`.
6. Внутри него регистрируется `inner timer` как новая macrotask.
7. Внутри него же ставится microtask `promise in outer timer`.
8. После завершения `outer timer` Event Loop очищает microtasks → печатается `promise in outer timer`.
9. Потом приходит очередь `inner timer`.

Ответ: `top promise`, `outer timer`, `promise in outer timer`, `inner timer`.

### Задача H

```javascript
Promise.resolve()
  .then(() => {
    console.log('p1');
    return 'value';
  })
  .then((value) => {
    console.log(value);
  });

queueMicrotask(() => console.log('qm'));
console.log('sync');
```

Разбор по шагам:

1. Первый `then` ставится в microtask queue.
2. `queueMicrotask` ставит `qm` следом.
3. Печатается `sync`.
4. Выполняется первая microtask цепочки → `p1`.
5. Возвращается `'value'`, поэтому продолжение второй части цепочки ставится как новая microtask в конец очереди.
6. Следующей выполняется `qm`.
7. Потом выполняется второй `then`, который печатает `value`.

Ответ: `sync`, `p1`, `qm`, `value`.

**11. Что выведет код?**

## Финальное резюме модуля

Если запомнить три вещи, пусть это будут они.

Во-первых, JavaScript на главном потоке не делает параллельное выполнение твоего кода, но отлично справляется с конкурентностью через Event Loop. Не ищи в центре модели второй секретный поток — ищи стек, очереди и короткие шаги.

Во-вторых, после каждой macrotask движок вычищает всю очередь microtasks. Именно это решает большинство interview-задач. Если ты понимаешь это правило и умеешь разложить пример по шагам, половина страха перед асинхронностью исчезает.

В-третьих, `async/await` не заменяет Promises, а лишь даёт более удобную форму записи. Под красивым синтаксисом всё равно живут те же обещания будущего результата, те же цепочки, те же microtasks и те же правила Event Loop.

Для Java/Spring-разработчика главный ментальный поворот такой: не ищи здесь в центре модели `Thread` и `synchronized`. Центральные абстракции JavaScript — это стек, очереди, обещания будущего результата и короткие неблокирующие шаги. Когда ты начнёшь думать именно так, тема асинхронности перестанет быть магией.

И ещё раз: **СТОП! Это самая важная тема на собеседованиях.** Если после этого модуля ты сможешь без паники, медленно и чётко проговорить порядок работы Event Loop на доске, ты уже очень сильно отличаешься от большинства кандидатов.
