# Модуль 2: Функции — scope, замыкания, `this` и стрелочные функции

## Для кого этот модуль

Этот модуль для тебя, если ты уже умеешь объявлять функцию и вызывать её, но пока чувствуешь, что JavaScript ведёт себя «слишком живо». В Java метод обычно сидит внутри класса, смотрит на поля объекта и живёт предсказуемо. В JavaScript функция намного самостоятельнее: её можно передавать, возвращать, хранить в переменной, создавать на лету, а ещё она умеет помнить окружение, в котором родилась.

Если сейчас тебе кажется, что слова **scope**, **closure** и **this** — это три разные туманные темы, хорошая новость: это одна история с трёх сторон. Scope отвечает на вопрос: **где переменная ищется**. Closure отвечает: **почему она всё ещё доступна позже**. `this` отвечает: **с каким объектом связана функция при вызове**.

**Ожидаемый уровень на входе**: ты уже прошёл Модуль 1, понимаешь разницу между `var`, `let`, `const`, знаешь про hoisting и TDZ. Если это плавает — лучше сначала освежить Модуль 1. Особенно раздел про hoisting. Он здесь будет нужен постоянно.

## Главная установка модуля

**Функция в JavaScript — это не просто кусок кода. Это код, который запускается в определённом окружении, может помнить это окружение и может быть вызван с разным контекстом `this`.**

Если хочешь совсем коротко:
- **scope** = откуда функция берёт переменные
- **closure** = почему она помнит эти переменные потом
- **this** = какой объект считается «текущим» в момент вызова

Давай остановимся и закрепим: если держать в голове три вопроса выше, почти любой пример начинает раскрываться.

## Блок 1: Способы объявления функций

### 2.1. Три способа создать функцию

В Java есть сильное ощущение, что функция принадлежит классу. Даже если это `static`-метод, он всё равно живёт внутри класса. В JavaScript функция — отдельная сущность. Её можно положить в переменную, передать в другую функцию, вернуть из функции и даже создать прямо посреди выражения.

Почему это важно? Потому что от формы объявления зависит не только синтаксис, но и поведение: hoisting, возможность использовать `new`, собственный `this`, читаемость и даже удобство отладки. На собеседовании легко поймать человека, который запомнил только «стрелочные короче».

Первая бытовая аналогия: функция в JavaScript — как инструмент в мастерской. Все они помогают работать, но используются по-разному. Java-аналогия: обычный метод, анонимный класс/лямбда и короткая лямбда.

#### Function Declaration (объявление функции)

```javascript
function calculateTax(amount, rate) {
  return amount * rate;
}
```

**Разбор по строкам:**
- `function calculateTax(amount, rate) {` — объявляем функцию с именем `calculateTax` и двумя параметрами.
- `return amount * rate;` — вычисляем результат и возвращаем его вызывающему коду.
- `}` — закрываем тело функции.

Это самый «классический» вид функции. Он ближе всего к тому, что ты ожидаешь после Java. Но ключевая особенность здесь не в привычности, а в **hoisting**: движок поднимает объявление функции целиком, поэтому её можно вызвать даже до строки, где она написана.

Почему это удобно? Можно сначала увидеть основной сценарий, а детали вынести ниже. Это похоже на оглавление книги: сначала ход мысли, потом реализация.

Промежуточный пример:

```javascript
greet();

function greet() {
  console.log('Привет!');
}
```

**Разбор по строкам:**
- `greet();` — вызываем функцию ещё до визуального места её объявления.
- `function greet() {` — движок уже знает о существовании функции благодаря hoisting.
- `console.log('Привет!');` — печатаем сообщение.
- `}` — завершаем функцию.

Здесь нет магии «выполнилось назад во времени». Движок сначала подготавливает окружение, регистрирует имя функции и связывает его с телом, а потом уже выполняет код. Если в Модуле 1 у тебя закрепилась мысль «hoisting — это про подготовительный этап, а не про перенос строк», сейчас она очень пригодилась.

#### Function Expression (функциональное выражение)

```javascript
const calculateTax = function(amount, rate) {
  return amount * rate;
};
```

**Разбор по строкам:**
- `const calculateTax =` — создаём константу, в которую положим значение.
- `function(amount, rate) {` — это сама функция без имени; она является выражением.
- `return amount * rate;` — возвращаем вычисленный результат.
- `};` — закрываем функцию и завершаем присваивание.

Здесь функция создаётся как значение и сохраняется в переменную. Это уже сильнее похоже на Java-лямбду: сначала есть значение-функция, потом есть переменная, которая на него ссылается. Отличие от `declaration` критично: имя переменной `calculateTax` поднимается как `const`, то есть попадает в TDZ.

Почему это важно? Потому что разработчик часто видит знакомое слово `function` и думает, что поведение будет таким же, как у declaration. Но это не так. Здесь поднимается **переменная**, а не готовая функция для безопасного вызова до объявления.

```javascript
sayHi();

const sayHi = function() {
  console.log('Hi');
};
```

**Разбор по строкам:**
- `sayHi();` — пытаемся вызвать переменную до инициализации.
- `const sayHi =` — только на этой строке переменная получает значение функции.
- `function() {` — создаётся объект функции.
- `console.log('Hi');` — тело функции.
- `};` — завершение.

Результат — `ReferenceError`, потому что `sayHi` находится в TDZ. Стоп. Перечитай это ещё раз. Ошибка здесь не потому, что «функция объявлена позже». Ошибка потому, что ты обращаешься к `const` до инициализации.

#### Arrow Function (стрелочная функция, ES6)

```javascript
const calculateTax = (amount, rate) => amount * rate;
```

**Разбор по строкам:**
- `const calculateTax =` — создаём переменную.
- `(amount, rate) =>` — объявляем стрелочную функцию с двумя параметрами.
- `amount * rate` — это неявно возвращаемое выражение.
- `;` — завершение выражения.

Очень опасная ловушка: стрелочная функция — **не просто короткая запись**. Главное отличие не в длине, а в семантике: нет собственного `this`, `arguments`, `prototype`, и её нельзя вызвать через `new`. Это другой инструмент, ближе к лямбде, чем к полноценной обычной функции.

Промежуточный пример:

```javascript
const square = n => n * n;
const sum = (a, b) => a + b;
const log = () => console.log('ready');
```

**Разбор по строкам:**
- `const square = n => n * n;` — один параметр можно писать без скобок.
- `const sum = (a, b) => a + b;` — для двух параметров нужны скобки.
- `const log = () => console.log('ready');` — если параметров нет, скобки обязательны.

#### Когда что использовать?

| Ситуация | Выбор | Почему |
|----------|-------|--------|
| Обычная функция верхнего уровня | `function declaration` | Удобно читать, работает hoisting |
| Колбэк без собственного контекста | `arrow function` | Кратко и безопасно для внешнего `this` |
| Метод объекта | Обычная функция / shorthand | Нужен собственный `this` |
| Конструктор | Обычная функция / `class` | Стрелочную нельзя вызывать через `new` |

Запомни не «что моднее», а «что семантически подходит». Это как выбирать не красивую отвёртку, а ту, у которой нужный наконечник.

#### История типичной ошибки

Молодой разработчик после пары статей решил: «Стрелочные везде лучше, они же современнее». Он переписал все методы объекта в стрелочные, запустил проект и получил загадочный `undefined` вместо свойств. Он не сделал ничего “синтаксически неправильного”. Он просто выбрал инструмент, который не предназначен для роли метода.

Если ты сейчас поймал себя на мысли «я бы тоже так сделал», отлично. Значит, вовремя.

### 2.2. Области видимости и лексическое окружение

Если сейчас что-то непонятно — перечитай §2.1 и вспомни Модуль 1. Замыкание без scope — это дом без фундамента. Нельзя понять, **что именно функция запоминает**, пока не ясно, **где вообще живут переменные**.

#### Зачем вообще нужен scope

Представь квартиру-студию без шкафов: scope делает с переменными то же, что система хранения с вещами — ограничивает, где что видно и где искать. Вторая аналогия: офис с комнатами отделов и общим ресепшеном. Java-аналогия: локальная переменная метода и поля класса, только в Java всё жёстче.

Это важно на практике, потому что многие ошибки со scope возникают из-за неверного понимания области видимости: разработчик ожидает одно значение, а по правилам языка берётся другое.

#### Что такое Lexical Environment

Каждый раз, когда движок исполняет код, он создаёт внутреннюю структуру, которую удобно представлять как **лексическое окружение** (`Lexical Environment`). У него две части:
1. **Environment Record** — таблица «имя → значение»
2. **ссылка на внешнее окружение** — указатель на родителя

Представь матрёшку или вложенные папки: сначала ищешь имя у себя, потом поднимаешься выше. Java-аналогия: немного похоже на стек вызовов плюс внешний контекст, но в JavaScript эта связь работает гибче.

```javascript
const globalVar = 'глобальная';

function outer() {
  const outerVar = 'внешняя';

  function inner() {
    const innerVar = 'внутренняя';
    console.log(innerVar);
    console.log(outerVar);
    console.log(globalVar);
  }

  inner();
}

outer();
```

**Разбор по строкам:**
- `const globalVar = 'глобальная';` — переменная в глобальном окружении.
- `function outer() {` — объявляем внешнюю функцию.
- `const outerVar = 'внешняя';` — переменная, доступная внутри `outer` и вложенных функций.
- `function inner() {` — объявляем внутреннюю функцию.
- `const innerVar = 'внутренняя';` — локальная переменная только для `inner`.
- `console.log(innerVar);` — поиск сразу успешен в текущем окружении.
- `console.log(outerVar);` — в `inner` нет, поднимаемся в окружение `outer`.
- `console.log(globalVar);` — ищем выше, пока не дойдём до глобального окружения.
- `}` — закрываем `inner`.
- `inner();` — запускаем внутреннюю функцию.
- `}` — закрываем `outer`.
- `outer();` — стартуем всю цепочку.

Визуально это можно представить так:

```
Global Environment
  globalVar -> "глобальная"
  outer -> function
        ↓
outer() Environment
  outerVar -> "внешняя"
  inner -> function
        ↓
inner() Environment
  innerVar -> "внутренняя"
```

Важно: поиск всегда идёт **изнутри наружу**, но никогда не наоборот. Внешняя функция не видит локальные переменные внутренней. Это как родитель может помнить, что у ребёнка есть комната, но не может автоматически взять вещь из закрытого ящика внутри неё.

#### Scope chain: как происходит поиск

Когда интерпретатор встречает имя переменной, он не «угадывает». Он проверяет:
1. есть ли имя в текущем окружении
2. если нет — есть ли во внешнем
3. если нет — ещё выше
4. если нигде нет — `ReferenceError`

```javascript
const x = 10;

function foo() {
  console.log(x);
}

function bar() {
  const x = 20;
  foo();
}

bar();
```

**Разбор по строкам:**
- `const x = 10;` — глобальная переменная.
- `function foo() {` — функция `foo` создаётся в глобальном окружении.
- `console.log(x);` — внутри `foo` поиск `x` пойдёт в её лексическое окружение создания.
- `}` — конец `foo`.
- `function bar() {` — объявляем `bar`.
- `const x = 20;` — локальная переменная внутри `bar`.
- `foo();` — вызываем `foo` изнутри `bar`.
- `}` — конец `bar`.
- `bar();` — запуск.

Что выведется? `10`. И это тот момент, где ломается интуиция человека после Java. Ему кажется: «ну я же вызываю `foo` изнутри `bar`, где `x = 20`». Но JavaScript использует **лексический scope**, а не динамический. Функция помнит не место вызова, а место создания.

Стоп! Перечитай. Это центральная мысль. **Функция не спрашивает: “кто меня сейчас вызвал?” при поиске обычных переменных. Она смотрит: “где я была создана?”**

#### Лексический vs динамический scope

- **Лексический**: переменные определяются расположением кода в файле
- **Динамический**: переменные определялись бы тем, откуда функцию вызвали

JavaScript использует лексический scope. Это хорошо, потому что код становится предсказуемым: можно читать файл и понимать поведение, не проигрывая всю историю вызовов.

Бытовая аналогия: твой домашний адрес определяется тем, где ты живёшь, а не тем, откуда тебе позвонили. Java-аналогия: лямбда захватывает значения из окружающего кода именно в точке объявления, а не в точке вызова. Но есть принципиальная разница: Java-лямбда захватывает **effectively final копию значения**. JS-замыкание захватывает **живую ссылку на переменную** — если она изменится снаружи, замыкание увидит новое значение. Это и есть причина классического бага с `var` в цикле.

#### Типичная история ошибки

Разработчик пишет helper-функцию в одном месте, а потом вызывает её внутри другого метода и ожидает, что helper увидит локальные переменные этого метода. Не видит. Человек злится. А на деле helper никогда и не обещал этого делать. Он был создан раньше и в другом окружении.

#### Провал в TDZ внутри scope

```javascript
let x = 1;

function f() {
  console.log(x);
  let x = 2;
}

f();
```

**Разбор по строкам:**
- `let x = 1;` — внешняя переменная.
- `function f() {` — объявляем функцию.
- `console.log(x);` — кажется, что сейчас напечатается внешняя `x`.
- `let x = 2;` — но внутри функции объявлена своя `x`.
- `}` — конец функции.
- `f();` — вызов.

Почему здесь `ReferenceError`, а не `1`? Потому что локальная `x` уже существует в этом scope с самого начала исполнения функции, но до строки инициализации находится в TDZ. Она **затеняет** внешнюю переменную. То есть поиск не идёт наружу, потому что локальное имя уже найдено — просто оно пока недоступно.

Это очень важная связь с Модулем 1. Если тема TDZ была “ну где-то там про `let`”, сейчас она становится реально прикладной.

### 🎯 2.3. Замыкания (Closures) — Interview Focus

> **Почему спрашивают на собеседованиях**: замыкания — лакмусовая бумажка. Если человек понимает замыкания, он понимает scope, lifetime переменных, сборку мусора, колбэки и половину асинхронного JavaScript.

#### Почему эта тема кажется мистической

Термин «замыкание» часто кажется абстрактным, поэтому разберём его на конкретном примере.

Представь рюкзак. Ты вышел из дома, но часть вещей из дома у тебя теперь с собой. Не весь дом, а доступ к нужным вещам. Это первая метафора. Вторая: банковская карта. Физически деньги не лежат у тебя в кармане, но карта даёт доступ к счёту. Третья: пульт от шлагбаума. Ты уже далеко от будки охраны, но кнопка у тебя осталась, и шлагбаум всё ещё реагирует. Java-аналогия: лямбда, которая захватила внешнюю переменную. Только в JS она может не просто читать, а ещё и менять состояние.

Ещё одна аналогия. Представь секретаря, который уволился из компании, но у него на руках остался действующий пропуск в архив. Компания формально «закрыла рабочий день», но доступ к архиву через пропуск сохранился. Замыкание — это такой пропуск. Внешняя функция закончилась, а внутренняя всё ещё может зайти к её переменным.

#### Что проблему решают замыкания

Почему вообще язык даёт такую возможность? Потому что без замыканий функции были бы гораздо тупее. Ты не смог бы удобно создавать счётчики, фабрики обработчиков, мемоизацию, приватное состояние, конфигурируемые функции и половину нормального функционального стиля.

Без замыканий пришлось бы чаще тащить глобальные переменные, лишние объекты или вручную передавать состояние через каждый вызов. Замыкание позволяет функции быть маленьким устройством с памятью.

#### Первая ключевая мысль

Замыкание — это **не** «функция внутри функции». Это слишком слабое определение. Функция внутри функции — обычное дело. Замыкание появляется тогда, когда внутренняя функция использует переменные внешней функции и продолжает иметь к ним доступ позже.

Вот ловушка, на которой валятся многие. Они говорят: «Ну раз функция объявлена внутри другой — значит closure». Не обязательно. Если внутренняя функция не использует внешние переменные, особой ценности в этом факте нет. Суть замыкания — в **запомненном окружении**.

#### Вторая ключевая мысль

Замыкание — это не копия значений, положенная в мешок. Это чаще всего **ссылка на живое окружение**. Поэтому если замкнутая переменная меняется, внутреняя функция увидит обновлённое значение.

Это важно. Представь табличку с номером кабинета. Ты не уносишь сам кабинет домой. Ты уносишь адрес. И каждый раз, когда приходишь, видишь актуальное содержимое кабинета. Java-аналогия: не snapshot, а ссылка на захваченное состояние.

#### Третья ключевая мысль

Замыкание возможно только потому, что JavaScript использует лексический scope. Если §2.2 был неясен, вернись туда. Сначала функция должна знать, **откуда брать переменные**, и только потом можно обсуждать, **почему доступ к ним сохранился после завершения внешней функции**.

#### Четвёртая ключевая мысль

Внешняя функция может завершиться, а её переменные — не исчезнуть. В JavaScript время жизни определяется не только стеком вызовов, но и тем, остались ли ссылки на окружение.

#### Пятая ключевая мысль

Замыкания в JavaScript не экзотика. Это обработчики событий, колбэки в `setTimeout`, функции-фабрики, middleware и конфигурируемые обработчики. То есть это не тема “для олимпиад”, а базовая рабочая реальность.

#### Шестая ключевая мысль

Очень часто люди интуитивно понимают closure, но не называют его. Например: «мы сохранили `prefix`, чтобы потом логировать с ним сообщения». Это и есть замыкание. Не нужно мистифицировать. Нужно увидеть паттерн.

#### Седьмая ключевая мысль

Замыкание — это сочетание двух вещей:
1. функции
2. её лексического окружения создания

Не функция отдельно. Не переменные отдельно. А именно пара. На собеседовании это формулировка почти золотая.

#### Восьмая ключевая мысль

Когда ты говоришь «функция помнит переменную», не представляй человеческую память. Представляй внутреннюю ссылку `[[Environment]]`. Это скрытая техническая связь между функцией и окружением, где она была создана.

#### Девятая ключевая мысль

Замыкание делает код мощнее, но может и навредить. Если ты случайно удержал тяжёлые данные через closure, память будет занята дольше, чем хотелось бы. Поэтому зрелое понимание темы — это не только «как круто сделать счётчик», но и «как не притащить в память целый грузовик». 

#### Десятая ключевая мысль

Если ты на собеседовании начинаешь ответ словами «Замыкание — это когда функция внутри функции», остановись и переформулируй. Лучше сказать: **«Замыкание — это функция вместе с запомненным лексическим окружением, благодаря которому она может обращаться к внешним переменным даже после завершения внешней функции»**.

Теперь можно к коду. Медленно.

#### Самый простой пример замыкания

```javascript
function outer() {
  const message = 'Привет';

  return function inner() {
    console.log(message);
  };
}

const fn = outer();
fn();
```

**Разбор по строкам:**
- `function outer() {` — объявляем внешнюю функцию.
- `const message = 'Привет';` — локальная переменная внешней функции.
- `return function inner() {` — возвращаем внутреннюю функцию.
- `console.log(message);` — внутренняя функция использует внешнюю переменную.
- `};` — возвращаем функцию как значение.
- `}` — конец `outer`.
- `const fn = outer();` — вызываем `outer` и получаем обратно функцию `inner`.
- `fn();` — запускаем её позже.

Что здесь важно заметить? `outer()` уже закончилась к моменту вызова `fn()`. Но `message` всё ещё доступна. Вот и есть тот самый «ага»-момент. Не переменная пришла в новую функцию как аргумент. Не глобальная область помогла. Именно замыкание сохранило доступ.

Если хочется совсем механически представить шаги:
1. вызывается `outer`
2. создаётся её окружение с `message`
3. создаётся `inner`, и у неё появляется ссылка `[[Environment]]` на окружение `outer`
4. `outer` возвращает `inner`
5. сама `outer` завершается
6. окружение `outer` не удаляется, потому что на него всё ещё ссылается `inner`
7. при вызове `fn()` поиск `message` идёт через эту ссылку

#### Ещё более постепенный пример

```javascript
function makePrinter(text) {
  return function() {
    console.log(text);
  };
}

const printHello = makePrinter('Hello');
const printBye = makePrinter('Bye');

printHello();
printBye();
```

**Разбор по строкам:**
- `function makePrinter(text) {` — фабрика, принимающая значение.
- `return function() {` — возвращаем функцию.
- `console.log(text);` — функция использует параметр внешней функции.
- `};` — конец внутренней функции.
- `}` — конец фабрики.
- `const printHello = makePrinter('Hello');` — создаём одно замыкание с `text = 'Hello'`.
- `const printBye = makePrinter('Bye');` — создаём другое замыкание с `text = 'Bye'`.
- `printHello();` — печатаем `Hello`.
- `printBye();` — печатаем `Bye`.

Это уже очень важный шаг. Кажется, что функция одна и та же по структуре. Но замыкания разные. У каждого вызова `makePrinter` — своё окружение. То есть функция-шаблон может производить много независимых «функций с памятью».

#### Счётчик: классика, которую нужно понимать, а не узнавать

```javascript
function createCounter() {
  let count = 0;

  return function() {
    count++;
    return count;
  };
}

const counter = createCounter();
console.log(counter());
console.log(counter());
console.log(counter());
```

**Разбор по строкам:**
- `function createCounter() {` — объявляем фабрику счётчика.
- `let count = 0;` — внутреннее состояние счётчика.
- `return function() {` — возвращаем функцию, которая будет менять это состояние.
- `count++;` — увеличиваем замкнутую переменную.
- `return count;` — возвращаем новое значение.
- `};` — конец внутренней функции.
- `}` — конец фабрики.
- `const counter = createCounter();` — получаем функцию с собственным состоянием `count`.
- `console.log(counter());` — первый вызов даёт `1`.
- `console.log(counter());` — второй вызов даёт `2`.
- `console.log(counter());` — третий вызов даёт `3`.

Вот здесь у многих случается настоящий «ага»-момент. Переменная `count` не глобальная и не лежит в объекте, но живёт между вызовами. Функция — это действительно маленький объект с памятью. Бытовая аналогия: автомат с напитками помнит, сколько монет уже внесли.

#### Независимые замыкания

```javascript
const counterA = createCounter();
const counterB = createCounter();

console.log(counterA());
console.log(counterA());
console.log(counterB());
console.log(counterA());
console.log(counterB());
```

**Разбор по строкам:**
- `const counterA = createCounter();` — первое независимое замыкание.
- `const counterB = createCounter();` — второе независимое замыкание.
- `console.log(counterA());` — `counterA` становится `1`.
- `console.log(counterA());` — `counterA` становится `2`.
- `console.log(counterB());` — `counterB` только начинает, поэтому `1`.
- `console.log(counterA());` — `counterA` дальше `3`.
- `console.log(counterB());` — `counterB` дальше `2`.

Это отличный промежуточный пример между «совсем просто» и «реально полезно». Он показывает, что closure — это не одна общая магическая память, а отдельная память на каждый вызов внешней функции.

#### Практическое применение 1: инкапсуляция

```javascript
function createWallet(initialBalance) {
  let balance = initialBalance;

  return {
    deposit(amount) {
      if (amount > 0) {
        balance += amount;
      }
    },
    withdraw(amount) {
      if (amount > 0 && amount <= balance) {
        balance -= amount;
        return amount;
      }
      return 0;
    },
    getBalance() {
      return balance;
    }
  };
}

const wallet = createWallet(100);
wallet.deposit(50);
console.log(wallet.getBalance());
console.log(wallet.balance);
```

**Разбор по строкам:**
- `function createWallet(initialBalance) {` — фабрика кошелька.
- `let balance = initialBalance;` — приватный баланс.
- `return {` — возвращаем публичный API.
- `deposit(amount) {` — метод пополнения.
- `if (amount > 0) {` — простая валидация.
- `balance += amount;` — меняем приватную переменную.
- `}` — конец `if`.
- `},` — конец метода `deposit`.
- `withdraw(amount) {` — метод снятия.
- `if (amount > 0 && amount <= balance) {` — проверяем условия.
- `balance -= amount;` — уменьшаем баланс.
- `return amount;` — возвращаем снятую сумму.
- `}` — конец успешной ветки.
- `return 0;` — если нельзя снять, возвращаем 0.
- `},` — конец `withdraw`.
- `getBalance() {` — метод чтения.
- `return balance;` — возвращаем текущий баланс.
- `}` — конец метода.
- `};` — возвращаем объект.
- `}` — конец фабрики.
- `const wallet = createWallet(100);` — создаём кошелёк.
- `wallet.deposit(50);` — пополняем.
- `console.log(wallet.getBalance());` — читаем баланс через публичный метод.
- `console.log(wallet.balance);` — прямого доступа к приватной переменной нет.

Почему это важно? Потому что ещё до `class #private` замыкания были одним из основных способов делать приватное состояние. И до сих пор это очень полезно, когда нужен не класс, а маленькая фабрика с закрытыми данными.

#### Практическое применение 2: каррирование и частичное применение

```javascript
function multiply(a) {
  return function(b) {
    return a * b;
  };
}

const double = multiply(2);
const triple = multiply(3);

console.log(double(5));
console.log(triple(5));
```

**Разбор по строкам:**
- `function multiply(a) {` — принимаем первый аргумент.
- `return function(b) {` — возвращаем вторую функцию.
- `return a * b;` — используем оба значения.
- `};` — конец внутренней функции.
- `}` — конец внешней.
- `const double = multiply(2);` — фиксируем `a = 2`.
- `const triple = multiply(3);` — фиксируем `a = 3`.
- `console.log(double(5));` — `2 * 5 = 10`.
- `console.log(triple(5));` — `3 * 5 = 15`.

Это уже путь к функциональному стилю. Ты создаёшь специализированные функции из более общей. В Java это часто ощущается тяжелее, а в JavaScript — очень естественно.

#### Практическое применение 3: фабрики логгеров

```javascript
function createLogger(prefix) {
  return function(message) {
    console.log(`[${prefix}] ${message}`);
  };
}

const errorLog = createLogger('ERROR');
const infoLog = createLogger('INFO');

errorLog('Database connection failed');
infoLog('User loaded');
```

**Разбор по строкам:**
- `function createLogger(prefix) {` — принимаем конфигурацию.
- `return function(message) {` — возвращаем реальный логгер.
- `console.log(`[${prefix}] ${message}`);` — используем замкнутый префикс.
- `};` — конец внутренней функции.
- `}` — конец фабрики.
- `const errorLog = createLogger('ERROR');` — логгер ошибок.
- `const infoLog = createLogger('INFO');` — логгер информации.
- `errorLog('Database connection failed');` — печатаем сообщение с префиксом.
- `infoLog('User loaded');` — печатаем другое сообщение.

В реальных проектах это встречается в middleware, конфигураторах API, генераторах обработчиков и валидации.

#### Практическое применение 4: мемоизация

```javascript
function memoize(fn) {
  const cache = new Map();

  return function(...args) {
    const key = JSON.stringify(args);

    if (cache.has(key)) {
      console.log('Из кэша!');
      return cache.get(key);
    }

    const result = fn.apply(this, args);
    cache.set(key, result);
    return result;
  };
}

const square = memoize(function(n) {
  console.log('Вычисляю...');
  return n * n;
});

square(5);
square(5);
```

**Разбор по строкам:**
- `function memoize(fn) {` — принимаем функцию, которую хотим кешировать.
- `const cache = new Map();` — создаём приватный кеш без риска случайно зацепить свойства из prototype chain.
- `return function(...args) {` — возвращаем обёртку, которая умеет принимать любое число аргументов.
- `const key = JSON.stringify(args);` — превращаем список аргументов в ключ для кеша.
- `if (cache.has(key)) {` — проверяем, есть ли готовый ответ именно в нашем кеше.
- `console.log('Из кэша!');` — показываем, что повторно не считаем.
- `return cache.get(key);` — отдаём сохранённый результат.
- `}` — конец ветки кеша.
- `const result = fn.apply(this, args);` — если нет кеша, вызываем исходную функцию и сохраняем её текущий `this`.
- `cache.set(key, result);` — кладём результат в `Map`.
- `return result;` — возвращаем его.
- `};` — конец внутренней функции.
- `}` — конец `memoize`.
- `const square = memoize(function(n) {` — создаём мемоизированную функцию.
- `console.log('Вычисляю...');` — сигнал о реальном вычислении.
- `return n * n;` — само вычисление.
- `});` — завершаем создание.
- `square(5);` — первый вызов считает.
- `square(5);` — второй берёт из кеша.

🔗 **Связь с Angular**

В Angular замыкания и функции высшего порядка встречаются постоянно. Сервисы и фабрики нередко держат приватное состояние через closure, RxJS-операторы `map`, `filter`, `switchMap` — это функции высшего порядка, а колбэки в событиях и подписках — база для `Observable.subscribe(...)`. Даже цепочка `pipe(map(...), filter(...))` — это чистая композиция HOF-ов.

#### Классическая interview-задача: цикл с `var`

Вот она. Та самая. Если ты идёшь на собеседование и не можешь объяснить этот пример по шагам, это почти гарантированный удар по уверенности.

```javascript
for (var i = 0; i < 3; i++) {
  setTimeout(function() {
    console.log(i);
  }, 100);
}
```

**Разбор по строкам:**
- `for (var i = 0; i < 3; i++) {` — создаём **одну** переменную `i` на весь функциональный scope.
- `setTimeout(function() {` — на каждой итерации создаём колбэк.
- `console.log(i);` — колбэк использует переменную `i` из внешнего scope.
- `}, 100);` — просим выполнить колбэк позже.
- `}` — конец цикла.

Что думаешь, что выведет? Очень хочется сказать `0 1 2`. Но нет. Будет `3 3 3`.

Теперь разложим по итерациям. На первой, второй и третьей итерации создаются три колбэка, но все они получают доступ к одной и той же переменной `i`, а не к отдельным копиям значений. После завершения цикла `i` становится `3`, и только потом запускаются таймеры. Поэтому каждый колбэк видит текущее значение одной общей переменной — `3`. Стоп! Это место, где обычно и приходит понимание: колбэк замкнулся не на **значение**, а на **переменную**.

#### Решение 1: `let`

```javascript
for (let i = 0; i < 3; i++) {
  setTimeout(function() {
    console.log(i);
  }, 100);
}
```

**Разбор по строкам:**
- `for (let i = 0; i < 3; i++) {` — `let` создаёт новую переменную `i` для каждой итерации.
- `setTimeout(function() {` — создаём колбэк.
- `console.log(i);` — колбэк замыкается уже на свою итерационную `i`.
- `}, 100);` — откладываем выполнение.
- `}` — конец цикла.

Результат: `0 1 2`. Почему? Потому что спецификация делает отдельное лексическое окружение на каждую итерацию `for` с `let`.

#### Решение 2: IIFE

```javascript
for (var i = 0; i < 3; i++) {
  (function(j) {
    setTimeout(function() {
      console.log(j);
    }, 100);
  })(i);
}
```

**Разбор по строкам:**
- `for (var i = 0; i < 3; i++) {` — по-прежнему `var`.
- `(function(j) {` — создаём и сразу вызываем функцию.
- `setTimeout(function() {` — создаём колбэк внутри неё.
- `console.log(j);` — колбэк использует параметр `j`, а не общую `i`.
- `}, 100);` — ставим таймер.
- `})(i);` — передаём текущее значение `i` как аргумент.
- `}` — конец цикла.

Это старый до-ES6 способ. Каждая итерация создаёт отдельный scope через немедленно вызываемую функцию.

#### Решение 3: стрелочная фабрика

```javascript
for (var i = 0; i < 3; i++) {
  setTimeout(((j) => () => console.log(j))(i), 100);
}
```

**Разбор по строкам:**
- `for (var i = 0; i < 3; i++) {` — тот же цикл.
- `((j) => () => console.log(j))(i)` — создаём функцию, которой передаём текущее значение `i`.
- `() => console.log(j)` — возвращаем колбэк, замкнутый на `j`.
- `setTimeout(..., 100);` — передаём уже готовый колбэк в таймер.
- `}` — конец цикла.

На собеседовании лучше сначала показать решение с `let`, потом с IIFE. Третье оставь как бонус, если хочешь блеснуть.

#### Замыкания и память

Теперь зрелая часть темы. Замыкание не бесплатное. Оно удерживает окружение живым, пока на функцию есть ссылки.

```javascript
function processData() {
  const hugeArray = new Array(1000000).fill('data');

  return function getLength() {
    return hugeArray.length;
  };
}

const fn = processData();
console.log(fn());
```

**Разбор по строкам:**
- `function processData() {` — внешняя функция.
- `const hugeArray = new Array(1000000).fill('data');` — создаём большой массив.
- `return function getLength() {` — возвращаем внутреннюю функцию.
- `return hugeArray.length;` — используем замкнутый массив.
- `};` — конец внутренней функции.
- `}` — конец внешней.
- `const fn = processData();` — получаем функцию и тем самым сохраняем доступ к окружению.
- `console.log(fn());` — вызываем её.

Проблема не в том, что это «плохо всегда». Проблема в том, что если массив больше не нужен, а замыкание его удерживает, память будет занята. Иногда движок умеет оптимизировать, но писать код в надежде на чудо оптимизатора — плохая стратегия.

Более аккуратный вариант:

```javascript
function processData() {
  const hugeArray = new Array(1000000).fill('data');
  const length = hugeArray.length;

  return function getLength() {
    return length;
  };
}
```

**Разбор по строкам:**
- `const hugeArray = ...` — создаём массив.
- `const length = hugeArray.length;` — сохраняем только то, что реально нужно.
- `return function getLength() {` — возвращаем функцию.
- `return length;` — теперь ей не нужен сам массив.
- `};` — конец.

#### История типичной ошибки

Типичный сценарий утечки: обработчик кнопки замыкает большой массив результатов поиска. Пока обработчик живёт, массив тоже остаётся в памяти, что может приводить к утечкам в долгоживущих компонентах.

#### 💼 Типичные вопросы на собеседовании

1. **Что такое замыкание?** → Функция вместе с её лексическим окружением, позволяющим обращаться к внешним переменным даже после завершения внешней функции.
2. **Как объяснить задачу с `var` в цикле?** → Все колбэки замыкаются на одну и ту же переменную `i`, а выполняются после завершения цикла.
3. **Где используется на практике?** → Инкапсуляция, фабрики функций, каррирование, мемоизация, обработчики событий, конфигурация middleware.
4. **Как closure связано с GC?** → Если функция всё ещё доступна, её окружение тоже может остаться в памяти.

#### 🚫 Red flags в ответе
- «Замыкание — это функция внутри функции»
- «Там просто копируется значение»
- «`var` в цикле выводит `3 3 3`, потому что `setTimeout` тормозит»
- полное отсутствие связи со scope

#### ✅ Green flags в ответе
- Упоминание лексического окружения
- Чёткое объяснение задачи с циклом
- Понимание, что замыкание удерживает состояние
- Практические примеры, а не только абстракция
- Упоминание связи с памятью и GC

### 🎯 2.4. Контекст `this` — Interview Focus

> **Почему спрашивают на собеседованиях**: `this` — зона, где у Java/C#-разработчика ломается автоматизм. Кто понимает `this`, тот умеет не угадывать, а вычислять поведение JavaScript.

**Важное допущение**: все примеры в этом разделе выполняются в strict mode (ES-модули или `'use strict'`). ES-модули автоматически работают в strict mode, даже без директивы `'use strict'`. В legacy browser scripts без strict mode «потерянный» this указывает на `window`, а не на undefined.

🔎 **Тонкий, но важный момент про ES-модули и `this`**

На верхнем уровне ES-модуля `this === undefined`. Но это не значит, что любой вызов метода внутри модуля «ломается». Если ты вызываешь `obj.method()`, то `this` всё равно будет `obj`, потому что правило метода сильнее места, где файл лежит. Проблема начинается, когда метод отрывают от объекта: `const fn = obj.method; fn();`. Тогда это уже обычный вызов функции, `this === undefined`, и попытка прочитать `this.property` заканчивается `TypeError`.

#### Почему тема `this` болезненна

В Java всё приятно: `this` — это текущий объект экземпляра. Он как бейджик, пришитый навсегда. В JavaScript бейджик съёмный. Один и тот же кусок кода-функция может быть вызван с разным `this`.

Первая бытовая аналогия: микрофон на конференции. Он один и тот же, но в разные моменты находится у разных людей. Java-аналогия: будто метод можно временно прикрепить к другому объекту в момент вызова.

Самая опасная ошибка — пытаться определять `this` по месту объявления. Нельзя. Для обычной функции `this` определяется **способом вызова**. Исключение — стрелочные функции, у которых собственного `this` нет вообще.

Давай построим железное правило: когда видишь `this`, задавай не вопрос «где функция написана?», а вопрос **«как она вызвана?»**.

#### Пять правил определения `this`

Мы разберём пять правил, и очень важно именно пройти их по порядку. Не просто увидеть названия, а реально почувствовать.

##### Правило 1: вызов как метода объекта → `this = объект перед точкой`

Это самое дружелюбное правило. Здесь всё почти как в Java.

**Пример 1:**

```javascript
const user = {
  name: 'Alice',
  greet() {
    console.log(`Привет, я ${this.name}`);
  }
};

user.greet();
```

**Разбор по строкам:**
- `const user = {` — создаём объект.
- `name: 'Alice',` — свойство объекта.
- `greet() {` — метод объекта.
- `console.log(`Привет, я ${this.name}`);` — обращаемся к `this.name`.
- `}` — конец метода.
- `};` — конец объекта.
- `user.greet();` — функция вызывается через точку, значит `this = user`.

Здесь всё просто: слева от точки стоит `user`, значит он и становится `this`.

**Пример 2:**

```javascript
const calculator = {
  value: 10,
  add(n) {
    this.value += n;
    return this.value;
  }
};

console.log(calculator.add(5));
```

**Разбор по строкам:**
- `value: 10` — начальное состояние.
- `add(n) {` — метод.
- `this.value += n;` — изменяем свойство текущего объекта.
- `return this.value;` — возвращаем новое значение.
- `console.log(calculator.add(5));` — метод вызван через `calculator`, значит `this = calculator`.

**Пример 3:**

```javascript
const team = {
  title: 'Frontend',
  printTitle() {
    return this.title;
  }
};

const anotherTeam = {
  title: 'Backend',
  printTitle: team.printTitle
};

console.log(anotherTeam.printTitle());
```

**Разбор по строкам:**
- `team.printTitle` — это ссылка на функцию.
- `anotherTeam.printTitle = team.printTitle` — ту же функцию кладём в другой объект.
- `anotherTeam.printTitle()` — теперь вызов идёт через `anotherTeam`.
- `this.title` — значит, берётся `'Backend'`.

Это прекрасный пример того, что функция сама по себе не «принадлежит навечно» объекту. Важен текущий способ вызова.

##### Правило 2: обычный вызов без объекта → `this = undefined` в strict mode

Вот где начинается боль.

**Пример 1:**

```javascript
function showThis() {
  console.log(this);
}

showThis();
```

**Разбор по строкам:**
- `function showThis() {` — обычная функция.
- `console.log(this);` — хотим узнать её контекст.
- `showThis();` — вызов без объекта слева от точки.

В современных модулях и строгом режиме здесь будет `undefined`. В старом нестрогом браузерном коде мог бы быть `window`. Для учебной модели лучше привыкай к `undefined` как к основной реальности.

**Пример 2: потеря контекста**

```javascript
const user = {
  name: 'Alice',
  greet() {
    console.log(this.name);
  }
};

const fn = user.greet;
fn();
```

**Разбор по строкам:**
- `user.greet` — берём ссылку на функцию.
- `const fn = user.greet;` — сохраняем её отдельно.
- `fn();` — теперь это обычный вызов, без объекта перед точкой.
- Само по себе правило такое: в strict mode `this === undefined`.
- А вот следующая деталь уже про конкретный код: попытка прочитать `this.name` у `undefined` заканчивается `TypeError`.

Важно различать две вещи: `user.greet()` по-прежнему работает нормально, потому что это вызов как метода и `this = user`. Ломается именно «оторванный» вызов `fn()`.

**Пример 3: setTimeout**

```javascript
const user = {
  name: 'Alice',
  greet() {
    console.log(`Привет, я ${this.name}`);
  }
};

setTimeout(user.greet, 100);
```

**Разбор по строкам:**
- `setTimeout(user.greet, 100);` — мы не вызываем метод объекта прямо сейчас.
- Мы передаём ссылку на функцию в `setTimeout`.
- Позже таймер вызовет её как обычную функцию.
- Значит, `this` потеряется.

На собеседовании здесь важно сказать не «`setTimeout` странный», а: **метод был передан как колбэк и затем вызван без объекта**.

##### Правило 3: вызов через `new` → `this = новый объект`

Теперь конструкция, которая похожа на Java-конструктор.

**Пример 1:**

```javascript
function User(name) {
  this.name = name;
}

const user = new User('Bob');
console.log(user.name);
```

**Разбор по строкам:**
- `function User(name) {` — функция-конструктор.
- `this.name = name;` — записываем свойство в новый объект.
- `new User('Bob')` — создаёт объект, привязывает его к `this`, выполняет функцию и возвращает объект.
- `console.log(user.name);` — получаем `'Bob'`.

**Пример 2:**

```javascript
function Point(x, y) {
  this.x = x;
  this.y = y;
}

const point = new Point(3, 4);
console.log(point.x, point.y);
```

**Разбор по строкам:**
- `this.x = x;` — кладём `x` в новый объект.
- `this.y = y;` — кладём `y` в новый объект.
- `new Point(3, 4)` — создаёт экземпляр.
- `console.log(point.x, point.y);` — видим `3 4`.

**Пример 3: что будет без `new`?**

```javascript
function User(name) {
  this.name = name;
}

User('Bob');
```

**Разбор по строкам:**
- функция та же самая
- но вызов обычный, без `new`
- значит, срабатывает правило 2
- в strict mode `this = undefined`
- попытка сделать `this.name = name` даст ошибку

Вот это классическая ловушка. Человек видит «конструкторную» функцию, но забывает `new`.

##### Правило 4: `call`, `apply`, `bind` — явное управление `this`

Иногда ты сам хочешь сказать функции, кем ей считать `this`. Для этого есть специальные методы.

**Пример 1: `call`**

```javascript
function greet(greeting) {
  console.log(`${greeting}, я ${this.name}`);
}

const alice = { name: 'Alice' };
greet.call(alice, 'Привет');
```

**Разбор по строкам:**
- `function greet(greeting) {` — обычная функция.
- `console.log(`${greeting}, я ${this.name}`);` — использует `this.name`.
- `const alice = { name: 'Alice' };` — объект-кандидат на роль `this`.
- `greet.call(alice, 'Привет');` — вызываем функцию немедленно с `this = alice`.

**Пример 2: `apply`**

```javascript
function greet(greeting, sign) {
  console.log(`${greeting}, я ${this.name}${sign}`);
}

const bob = { name: 'Bob' };
greet.apply(bob, ['Здорово', '!']);
```

**Разбор по строкам:**
- `greet.apply(bob, ['Здорово', '!']);` — `this = bob`, аргументы передаются массивом.
- По сути идея та же, что у `call`, отличается форма передачи аргументов.

**Пример 3: `bind`**

```javascript
function greet(greeting) {
  console.log(`${greeting}, я ${this.name}`);
}

const alice = { name: 'Alice' };
const greetAlice = greet.bind(alice);

greetAlice('Йо');
```

**Разбор по строкам:**
- `greet.bind(alice)` — не вызывает функцию сразу.
- Возвращает новую функцию, у которой `this` навсегда привязан к `alice`.
- `greetAlice('Йо');` — вызываем уже эту новую функцию.

Коротко: `call` — вызвать сейчас, `apply` — вызвать сейчас с массивом аргументов, `bind` — вернуть новую функцию с фиксированным `this`.

##### Правило 5: стрелочные функции не имеют собственного `this`

Вот правило, которое надо не просто знать, а чувствовать. Стрелочная функция **вообще не участвует в определении `this`**. Она берёт `this` из внешнего лексического окружения.

**Пример 1: колбэк внутри метода**

```javascript
const user = {
  name: 'Alice',
  friends: ['Bob', 'Charlie'],
  listFriends() {
    this.friends.forEach((friend) => {
      console.log(`${this.name} дружит с ${friend}`);
    });
  }
};

user.listFriends();
```

**Разбор по строкам:**
- `listFriends()` — обычный метод, значит `this = user`.
- `forEach((friend) => {` — стрелочная функция не создаёт новый `this`.
- `this.name` — берётся из внешнего метода `listFriends`, то есть это `user.name`.
- `user.listFriends();` — всё работает как ожидается.

**Пример 2: обычная функция вместо стрелочной**

```javascript
const user = {
  name: 'Alice',
  friends: ['Bob', 'Charlie'],
  listFriends() {
    this.friends.forEach(function(friend) {
      console.log(`${this.name} дружит с ${friend}`);
    });
  }
};
```

**Разбор по строкам:**
- `listFriends()` — здесь `this = user`.
- Но `function(friend) {` — это уже обычная функция.
- `forEach` вызовет её как простой колбэк.
- Значит, внутри неё `this` будет `undefined`.

**Пример 3: стрелочная как метод объекта — антипаттерн**

```javascript
const user = {
  name: 'Alice',
  greet: () => {
    console.log(this.name);
  }
};

user.greet();
```

**Разбор по строкам:**
- `greet: () => {` — метод объявлен стрелочной функцией.
- Она не получает `this = user` при вызове `user.greet()`.
- `this` берётся из внешнего окружения места создания.
- Поэтому `this.name` здесь почти наверняка `undefined`.

Вот здесь самая распространённая ошибка: стрелочные функции чинят `this` **во вложенных колбэках**, но не подходят как обычные методы объекта.

#### Приоритет правил

Если несколько идей будто бы конкурируют, держи иерархию:
1. `new`
2. `call` / `apply` / `bind`
3. вызов как метода объекта
4. обычный вызов
5. стрелочная функция не участвует, а просто берёт внешний `this`

Точнее: стрелочная функция стоит особняком. Для неё вопрос определения собственного `this` вообще не возникает.

#### Реальная история ошибки

Разработчик писал компонент и передал `this.loadData` в кнопку как колбэк. Внутри метода всё начало падать: `Cannot read properties of undefined`. Он два часа искал проблему в API. А проблема была не в данных, а в том, что `this` потерялся ещё до входа в метод.

#### Практический пример в стиле Angular/Promise

```typescript
class UserComponent {
  userName = 'Alice';

  loadData() {
    fetch('/api/user')
      .then(response => response.json())
      .then((data) => {
        this.userName = data.name;
      });
  }
}
```

**Разбор по строкам:**
- `class UserComponent {` — объявляем класс.
- `userName = 'Alice';` — поле экземпляра.
- `loadData() {` — метод класса.
- `fetch('/api/user')` — делаем запрос.
- `.then(response => response.json())` — короткий стрелочный колбэк.
- `.then((data) => {` — ещё один стрелочный колбэк.
- `this.userName = data.name;` — `this` берётся из `loadData`, то есть из экземпляра компонента.
- `});` — конец.

Если заменить второй `then` на `function(data) { ... }`, контекст сломается.

#### 💼 Типичные вопросы на собеседовании

1. **Что такое `this`?** → Контекст выполнения функции, определяемый способом вызова.
2. **Почему `setTimeout(obj.method, 0)` ломает `this`?** → Потому что передаётся ссылка на функцию, которая потом вызывается без объекта.
3. **Чем `call`, `apply`, `bind` отличаются?** → `call/apply` вызывают сразу, `bind` возвращает новую функцию.
4. **Почему стрелочная функция плоха как метод объекта?** → У неё нет собственного `this`.
5. **Когда стрелочная функция полезна?** → Во вложенных колбэках, где нужно сохранить внешний `this`.

- «`bind` вызывает функцию»
- неспособность объяснить, почему потерялся контекст

### 🎯 2.5. Стрелочные функции vs обычные — полный разбор

Теперь, когда мы уже знаем scope, closures и `this`, можно честно сравнить стрелочные и обычные функции. Стрелочная функция — это не «модная версия обычной», а отдельная конструкция с другой семантикой.

#### Полная таблица отличий

| Свойство | Обычная функция | Стрелочная функция |
|----------|----------------|--------------------|
| `this` | Определяется вызовом | Лексический, из внешнего scope |
| `arguments` | Есть | Нет |
| `new` | Можно | Нельзя |
| `prototype` | Есть | Нет |
| Подходит как метод объекта | Да | Обычно нет |
| Подходит как колбэк | Иногда, но осторожно | Часто да |

Ещё два важных interview-момента: (1) `call`/`apply`/`bind` **не меняют** this у стрелочной функции — лексический this не перебивается ничем. (2) У стрелочной функции нет собственного `super` и `new.target` — они тоже берутся из внешнего scope.

#### Отличие 1: `this`

```javascript
const user = {
  name: 'Alice',
  normal() {
    return this.name;
  },
  arrow: () => this.name
};

console.log(user.normal());
console.log(user.arrow());
```

**Разбор по строкам:**
- `normal()` — обычный метод, `this = user`.
- `arrow: () => this.name` — стрелочная функция не получает `this = user`.
- `user.normal()` вернёт `'Alice'`.
- `user.arrow()` почти наверняка вернёт `undefined`.

#### Отличие 2: `arguments`

```javascript
function normalSum() {
  return arguments[0] + arguments[1];
}

const arrowSum = () => {
  return arguments[0] + arguments[1];
};
```

**Разбор по строкам:**
- `function normalSum() {` — у обычной функции есть собственный `arguments`.
- `return arguments[0] + arguments[1];` — можно обратиться к переданным аргументам.
- `const arrowSum = () => {` — у стрелочной собственного `arguments` нет.
- `return arguments[0] + arguments[1];` — здесь будет взят внешний `arguments` или ошибка, что почти всегда не то, что нужно.

Правильный современный способ для стрелочных функций — rest parameters:

```javascript
const sum = (...args) => args.reduce((acc, n) => acc + n, 0);
```

**Разбор по строкам:**
- `(...args)` — собираем все аргументы в настоящий массив.
- `args.reduce(...)` — суммируем элементы.
- `0` — начальное значение аккумулятора.

#### Отличие 3: `new`

```javascript
function Person(name) {
  this.name = name;
}

const Animal = (name) => {
  this.name = name;
};

const person = new Person('Alice');
// const animal = new Animal('Cat');
```

**Разбор по строкам:**
- `function Person(name)` — обычная функция может быть конструктором.
- `const Animal = (name) => {` — стрелочная не имеет `[[Construct]]`.
- `new Person('Alice')` — корректно.
- `new Animal('Cat')` — вызвал бы `TypeError`.

#### Отличие 4: `prototype`

```javascript
function Foo() {}
const Bar = () => {};

console.log(typeof Foo.prototype);
console.log(typeof Bar.prototype);
```

**Разбор по строкам:**
- `function Foo() {}` — обычная функция получает `prototype`.
- `const Bar = () => {};` — стрелочная нет.
- `typeof Foo.prototype` — обычно `'object'`.
- `typeof Bar.prototype` — `'undefined'`.

#### Антипаттерн: стрелочная функция как метод

```javascript
const calculator = {
  value: 0,
  add: (n) => {
    this.value += n;
    return this.value;
  }
};
```

**Разбор по строкам:**
- `add: (n) => {` — метод записан стрелкой.
- `this.value += n;` — `this` не ссылается на `calculator`.
- Возникает ошибка или `NaN`, в зависимости от контекста.

Правильный вариант:

```javascript
const calculator = {
  value: 0,
  add(n) {
    this.value += n;
    return this.value;
  }
};
```

**Разбор по строкам:**
- `add(n) {` — обычный метод объекта.
- `this.value += n;` — теперь `this = calculator` при вызове через точку.
- `return this.value;` — возвращаем обновлённое значение.

#### Когда стрелочные функции особенно хороши

- короткие преобразования в `map`, `filter`, `reduce`
- вложенные колбэки внутри методов
- функции, которым не нужен собственный `this`
- фабрики, где важна компактность

#### Когда лучше обычная функция

- методы объектов
- конструкторы
- случаи, где нужен `arguments`
- ситуации, где нужен динамический `this`

### 2.6. IIFE и Module Pattern

До появления ES-модулей разработчикам тоже нужно было изолировать код и не засорять глобальную область. Поэтому появился паттерн IIFE — **Immediately Invoked Function Expression**: создал локальный scope, выполнил код и наружу выпустил только то, что решил вернуть.

Бытовая аналогия: одноразовый контейнер для доставки еды. Внутри есть всё нужное, но после использования он не превращается в постоянно открытую кухню. Java-аналогия: статический блок и приватные поля класса вместе, хотя сравнение неполное.

```javascript
(function() {
  const privateVar = 'секрет';
  console.log('Я выполнилась!');
})();
```

**Разбор по строкам:**
- `(function() {` — создаём function expression.
- `const privateVar = 'секрет';` — локальная переменная внутри неё.
- `console.log('Я выполнилась!');` — выполняем действие.
- `})();` — сразу же вызываем функцию.

Почему тут скобки? Потому что без них `function() {}` считалась бы declaration, а не expression. А IIFE нужно именно выражение, которое можно тут же вызвать.

#### Module Pattern

```javascript
const UserModule = (function() {
  let users = [];
  let nextId = 1;

  function validate(user) {
    return user.name && user.name.length > 0;
  }

  return {
    add(name) {
      const user = { id: nextId++, name };
      if (validate(user)) {
        users.push(user);
        return user;
      }
      throw new Error('Invalid user');
    },
    getAll() {
      return [...users];
    },
    getCount() {
      return users.length;
    }
  };
})();
```

**Разбор по строкам:**
- `const UserModule = (function() {` — создаём модуль через IIFE.
- `let users = [];` — приватный массив пользователей.
- `let nextId = 1;` — приватный счётчик.
- `function validate(user) {` — приватная функция.
- `return user.name && user.name.length > 0;` — простая проверка.
- `return {` — отдаём наружу только публичный API.
- `add(name) {` — публичный метод добавления.
- `const user = { id: nextId++, name };` — создаём пользователя и увеличиваем счётчик.
- `if (validate(user)) {` — валидируем.
- `users.push(user);` — сохраняем в приватный массив.
- `return user;` — возвращаем созданного пользователя.
- `throw new Error('Invalid user');` — если данные плохие.
- `getAll() { return [...users]; }` — отдаём копию, а не оригинал.
- `getCount() { return users.length; }` — читаем приватное состояние.
- `})();` — сразу исполняем и сохраняем результат.

Сегодня IIFE встречается реже, потому что есть `import`/`export`, но в legacy-коде он всё ещё жив.

#### 🏋️ Мини-упражнение 1: Создай генератор ID

**Задание**: Напиши функцию `createIdGenerator(prefix)`, которая возвращает функцию-генератор. Каждый вызов генератора возвращает уникальный ID в формате `prefix-N` (`N` начинается с 1).

```javascript
const userIdGen = createIdGenerator('user');
console.log(userIdGen());
console.log(userIdGen());

const orderIdGen = createIdGenerator('order');
console.log(orderIdGen());
console.log(userIdGen());
```

**Разбор по строкам:**
- `createIdGenerator('user')` должно создать отдельное замыкание для пользовательских ID.
- Повторные вызовы `userIdGen()` должны двигать только его счётчик.
- `createIdGenerator('order')` должно создать независимый счётчик.

<details><summary>Подсказка</summary>Используй замыкание: внешняя функция хранит `counter`, внутренняя — увеличивает и возвращает его.</details>

<details><summary>Решение</summary>

```javascript
function createIdGenerator(prefix) {
  let counter = 0;
  return function() {
    counter++;
    return `${prefix}-${counter}`;
  };
}
```

**Разбор по строкам:**
- `function createIdGenerator(prefix) {` — принимаем префикс.
- `let counter = 0;` — приватный счётчик.
- `return function() {` — возвращаем генератор.
- `counter++;` — увеличиваем значение.
- `return `${prefix}-${counter}`;` — собираем строку результата.
- `};` — конец внутренней функции.
- `}` — конец внешней.

Разбор идеи: каждый вызов `createIdGenerator` создаёт своё отдельное замыкание. Поэтому `userIdGen` и `orderIdGen` не мешают друг другу.
</details>

#### 🏋️ Мини-упражнение 2: Исправь потерю контекста

**Задание**: Этот код не работает — исправь его тремя разными способами.

```javascript
const timer = {
  seconds: 0,
  start() {
    setInterval(function() {
      this.seconds++;
      console.log(this.seconds);
    }, 1000);
  }
};
timer.start();
```

**Разбор по строкам:**
- `start()` вызывается как метод.
- Но внутренняя `function()` не наследует `this` автоматически.
- Поэтому `this.seconds` внутри колбэка не указывает на `timer.seconds`.

<details><summary>Решение</summary>

```javascript
// Способ 1: стрелочная функция
start() {
  setInterval(() => {
    this.seconds++;
    console.log(this.seconds);
  }, 1000);
}

// Способ 2: bind
start() {
  setInterval(function() {
    this.seconds++;
    console.log(this.seconds);
  }.bind(this), 1000);
}

// Способ 3: сохранение ссылки
start() {
  const self = this;
  setInterval(function() {
    self.seconds++;
    console.log(self.seconds);
  }, 1000);
}
```

**Разбор по строкам:**
- В способе 1 стрелочная функция берёт `this` из `start()`.
- В способе 2 `bind(this)` создаёт новую функцию с зафиксированным контекстом.
- В способе 3 мы вообще не используем `this` внутри колбэка, а работаем через замкнутую переменную `self`.

Все три подхода нужно понимать. На практике чаще всего используют стрелочную функцию.
</details>

#### 🏋️ Мини-упражнение 3: Once-функция

**Задание**: Напиши функцию `once(fn)`, которая принимает функцию и возвращает новую, которую можно вызвать только один раз. Повторные вызовы возвращают результат первого вызова.

```javascript
const initialize = once(() => {
  console.log('Инициализация!');
  return { ready: true };
});

console.log(initialize());
console.log(initialize());
console.log(initialize());
```

**Разбор по строкам:**
- `once(fn)` должна создать замыкание с флагом «уже вызывали?»
- При первом вызове нужно сохранить результат.
- При следующих вызовах — вернуть сохранённый результат без повторного запуска `fn`.

<details><summary>Подсказка</summary>Тебе понадобятся две переменные в замыкании: `called` и `result`.</details>

<details><summary>Решение</summary>

```javascript
function once(fn) {
  let called = false;
  let result;

  return function(...args) {
    if (!called) {
      called = true;
      result = fn.apply(this, args);
    }
    return result;
  };
}
```

**Разбор по строкам:**
- `function once(fn) {` — принимаем исходную функцию.
- `let called = false;` — флаг первого вызова.
- `let result;` — переменная для сохранённого результата.
- `return function(...args) {` — возвращаем обёртку.
- `if (!called) {` — проверяем, был ли уже вызов.
- `called = true;` — отмечаем, что первый вызов состоялся.
- `result = fn.apply(this, args);` — вызываем оригинальную функцию с текущим `this` и аргументами.
- `}` — конец `if`.
- `return result;` — всегда возвращаем один и тот же сохранённый результат.
- `};` — конец внутренней функции.
- `}` — конец `once`.

Обрати внимание на `apply(this, args)`: это делает решение аккуратнее, потому что мы не ломаем контекст и аргументы исходной функции.
</details>

### 🚀 Задание по проекту: Модуль управления задачами (консольный)

**Задание**: Создай файл `task-manager.js` — простой менеджер задач, использующий замыкания для инкапсуляции.

```javascript
// task-manager.js

function createTaskManager() {
  let tasks = [];
  let nextId = 1;

  function findTask(id) {
    return tasks.find(t => t.id === id);
  }

  return {
    add(title, priority = 'medium') {
      const task = {
        id: nextId++,
        title,
        priority,
        done: false,
        createdAt: new Date().toISOString()
      };
      tasks.push(task);
      return task;
    },

    complete(id) {
      const task = findTask(id);
      if (task) {
        task.done = true;
        return task;
      }
      return null;
    },

    remove(id) {
      const index = tasks.findIndex(t => t.id === id);
      if (index !== -1) {
        return tasks.splice(index, 1)[0];
      }
      return null;
    },

    list(filter = 'all') {
      switch (filter) {
        case 'done': return tasks.filter(t => t.done);
        case 'pending': return tasks.filter(t => !t.done);
        default: return [...tasks];
      }
    },

    stats() {
      return {
        total: tasks.length,
        done: tasks.filter(t => t.done).length,
        pending: tasks.filter(t => !t.done).length
      };
    }
  };
}

// === Тестирование ===
const tm = createTaskManager();

tm.add('Изучить замыкания', 'high');
tm.add('Написать тесты', 'medium');
tm.add('Сделать рефакторинг', 'low');

console.log('Все задачи:', tm.list());
console.log('Статистика:', tm.stats());

tm.complete(1);
console.log('Выполненные:', tm.list('done'));
console.log('В работе:', tm.list('pending'));

tm.remove(2);
console.log('После удаления:', tm.list());
console.log('Финальная статистика:', tm.stats());

console.log('Прямой доступ к tasks:', tm.tasks);
```

**Разбор по строкам:**
- `let tasks = [];` — приватное состояние модуля.
- `let nextId = 1;` — приватный счётчик ID.
- `function findTask(id)` — приватный helper.
- `return { ... }` — наружу отдаём только публичные методы.
- `add(...)` — создаёт новую задачу и сохраняет её.
- `complete(id)` — меняет состояние задачи.
- `remove(id)` — удаляет задачу из приватного массива.
- `list(filter = 'all')` — читает состояние через фильтры.
- `stats()` — считает агрегированную статистику.
- `const tm = createTaskManager();` — создаём один экземпляр менеджера.
- `tm.tasks` — должен быть `undefined`, потому что `tasks` закрыт в замыкании.

⚠️ **Важный нюанс**: `list()` через `[...tasks]` защищает сам массив от внешних push/splice, но сами объекты задач отдаются по ссылке. Внешний код может мутировать `task.title`. Для полной защиты нужно возвращать копии: `tasks.map(t => ({...t}))`.

**Запуск**: `node task-manager.js`

**Критерий готовности**: все операции работают корректно, `tm.tasks` возвращает `undefined`.

Почему это упражнение полезно? Потому что это уже не игрушечный `counter`, а маленький модуль с приватным состоянием.

## Контрольные вопросы (открытые)

1. **Объясни разницу между лексическим и динамическим scope. Какой использует JavaScript?**

<details><summary>Ожидаемый ответ</summary>Лексический scope означает, что функция видит переменные из окружения, где она была создана. Динамический scope означал бы зависимость от места вызова. JavaScript использует лексический scope.</details>

2. **Почему в Angular используются стрелочные функции в `subscribe` / `then`?**

<details><summary>Ожидаемый ответ</summary>Потому что стрелочная функция берёт `this` из внешнего метода компонента и не теряет контекст экземпляра.</details>

3. **Какую проблему решает `bind()` и когда он не нужен?**

<details><summary>Ожидаемый ответ</summary>`bind()` решает проблему потери контекста, создавая новую функцию с зафиксированным `this`. Он не нужен, если функция и так вызывается как метод объекта или если используется стрелочный колбэк, которому нужен внешний `this`.</details>

4. **Может ли замыкание вызвать утечку памяти? Приведи пример.**

<details><summary>Ожидаемый ответ</summary>Да. Если замыкание удерживает ссылку на окружение с большим объектом, этот объект может жить дольше, чем нужно. Например, функция возвращает колбэк, который использует одну переменную, но в том же scope лежит большой массив.</details>
